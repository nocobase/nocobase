# 사전 준비

클러스터 애플리케이션을 배포하기 전에 다음 준비 작업을 완료해야 합니다.

## 상용 플러그인 라이선스

NocoBase 애플리케이션을 클러스터 모드로 실행하려면 다음 플러그인의 지원이 필요합니다.

| 기능             | 플러그인                                                                            |
| ---------------- | ----------------------------------------------------------------------------------- |
| 캐시 어댑터      | 내장                                                                                |
| 동기화 신호 어댑터 | `@nocobase/plugin-pubsub-adapter-redis`                                             |
| 메시지 큐 어댑터 | `@nocobase/plugin-queue-adapter-redis` 또는 `@nocobase/plugin-queue-adapter-rabbitmq` |
| 분산 잠금 어댑터 | `@nocobase/plugin-lock-adapter-redis`                                               |
| 워커 ID 할당자   | `@nocobase/plugin-workerid-allocator-redis`                                         |

먼저 위 플러그인에 대한 라이선스를 확보했는지 확인해 주세요. (상용 플러그인 서비스 플랫폼에서 해당 플러그인 라이선스를 구매할 수 있습니다.)

## 시스템 구성 요소

애플리케이션 인스턴스 자체 외에도 클러스터 배포에는 데이터베이스, 미들웨어, 공유 스토리지, 로드 밸런싱과 같은 시스템 구성 요소가 필요합니다. 각 팀은 자체 운영 체계에 맞게 이러한 구성 요소의 구체적인 구현 방식을 선택할 수 있습니다.

### 데이터베이스

현재 클러스터 모드는 애플리케이션 인스턴스에만 적용되므로, 데이터베이스는 임시로 단일 노드만 지원합니다. 만약 마스터-슬레이브와 같은 데이터베이스 아키텍처를 사용한다면, 미들웨어를 통해 직접 구현해야 하며 NocoBase 애플리케이션에 투명하게 작동하도록 보장해야 합니다.

가용 영역이나 리전을 넘는 웜 스탠바이 또는 재해 복구가 필요하다면, 데이터베이스 동기화 및 전환 전략은 운영 팀이 직접 설계하고 구현해야 합니다.

### 미들웨어

NocoBase의 클러스터 모드는 클러스터 간 통신 및 조정을 위해 다음과 같은 미들웨어에 의존합니다.

- **캐시**: Redis 기반 분산 캐시 미들웨어를 사용하여 데이터 접근 속도를 향상시킵니다.
- **동기화 신호**: Redis의 스트림(stream) 기능을 사용하여 클러스터 간 동기화 신호 전달을 구현합니다.
- **메시지 큐**: Redis 또는 RabbitMQ 기반 메시지 큐 미들웨어를 사용하여 비동기 메시지 처리를 구현합니다.
- **분산 잠금**: Redis 기반 분산 잠금을 사용하여 클러스터 내 공유 리소스 접근의 안전성을 보장합니다.

모든 미들웨어 구성 요소가 Redis를 사용하는 경우, 클러스터 내부 네트워크(또는 Kubernetes)에서 단일 Redis 서비스를 시작할 수 있습니다. 또는 각 기능(캐시, 동기화 신호, 메시지 큐, 분산 잠금)별로 별도의 Redis 서비스를 활성화할 수도 있습니다.

**버전 권장 사항**

- Redis: 8.0 이상 또는 Bloom Filter 기능이 포함된 redis-stack 버전
- RabbitMQ: 4.0 이상

### 공유 스토리지

NocoBase는 시스템 관련 파일을 저장하기 위해 `storage` 디렉터리를 사용하며, 공유 스토리지는 클러스터 배포의 필수 구성 요소이기도 합니다. 다중 노드 모드에서는 인프라 환경에 따라 클라우드 디스크, NFS, EFS 등 다양한 구현 방식을 선택하여 여러 노드 간 공유 접근을 지원할 수 있습니다. 그렇지 않으면 시스템 파일이 자동으로 동기화되지 않아 애플리케이션이 정상적으로 동작하지 않습니다.

Kubernetes로 배포할 때는 [Kubernetes 배포: 공유 스토리지](./kubernetes#공유-스토리지) 섹션을 참고해 주세요.

#### `storage` 디렉터리에는 일반적으로 무엇이 저장되는가

`storage` 디렉터리의 내용은 활성화된 플러그인과 배포 방식에 따라 달라집니다. 현재 구현 기준으로 일반적인 내용은 다음과 같습니다.

| 경로 | 용도 | 사용 권장 사항 |
| --- | --- | --- |
| `storage/uploads` | 로컬 스토리지 모드에서 업로드된 파일 | 운영 환경의 클러스터에서는 S3 / OSS / COS 같은 오브젝트 스토리지를 우선 사용하는 것이 좋습니다 |
| `storage/plugins` | 런타임에 설치, 업로드 또는 탐지되는 로컬 플러그인 패키지 | 로컬 플러그인에 의존한다면 이 디렉터리는 반드시 공유되어야 하며, 플러그인이 이미지에 포함되어 있다면 이 의존성을 줄일 수 있습니다 |
| `storage/apps/<app>/jwt_secret.dat` | `APP_KEY` 가 명시적으로 설정되지 않았을 때 자동 생성되는 기본 토큰 비밀키 | 운영 환경에서는 이 파일에 의존하지 말고 `APP_KEY` 를 명시적으로 설정하세요 |
| `storage/apps/<app>/aes_key.dat` | `APP_AES_SECRET_KEY` 가 명시적으로 설정되지 않았을 때 자동 생성되는 기본 AES 키 | 운영 환경에서는 이 파일에 의존하지 말고 `APP_AES_SECRET_KEY` 를 명시적으로 설정하세요 |
| `storage/environment-variables/<app>/aes_key.dat` | 환경 변수 플러그인 시나리오에서 사용하는 AES 키 파일 | 읽기 전용으로 마운트한 키 파일 사용을 권장합니다 |
| `storage/logs` | 기본 로그 디렉터리 및 일부 마이그레이션 로그 | 향후 외부 로그 플랫폼과의 연계를 권장합니다 |
| `storage/tmp` | 가져오기, 내보내기, 마이그레이션 등의 임시 파일 | 임시 디렉터리일 수 있지만, 노드 간 재사용이 필요하다면 공유해야 하며, 그렇지 않으면 단일 관리 노드에서만 실행해야 합니다 |
| `storage/backups`, `storage/duplicator`, `storage/migration-manager` | 백업, 복원, 마이그레이션 관련 산출물 | 운영용 디렉터리로 간주하고 영구 저장하며 여러 노드에서 동시에 수정하지 않도록 해야 합니다 |

위 표가 모든 항목을 완전히 나열한 것은 아니지만, 중요한 점을 보여 줍니다. `storage` 에는 업무 파일, 키 파일, 플러그인 디렉터리, 로그, 운영 관련 임시 산출물이 함께 섞여 있습니다. 따라서 클러스터 배포에서는 일반적으로 `/app/nocobase/storage` 전체를 공유하고 영구 저장하는 것을 기본 기준으로 삼습니다.

#### 스토리지 관련 권장 사항

NocoBase의 클러스터 일관성은 주로 데이터베이스, Redis, 메시지 큐, 분산 잠금에 의해 보장되며, 공유 파일 시스템을 고동시성 조정 매체로 사용하는 것을 전제로 하지 않습니다.

따라서 다음을 권장합니다.

- 첨부 파일처럼 접근 빈도가 높은 업무 파일은 오브젝트 스토리지를 우선 사용하세요. 운영 환경의 클러스터에서 로컬 스토리지에 장기간 의존하는 것은 권장되지 않습니다.
- 공유 스토리지는 주로 `storage` 디렉터리를 담는 용도로 사용하고, 고처리량 파일 스토리지 서비스처럼 사용하지 마세요.
- 플러그인 설치, 플러그인 업그레이드, 백업, 복원, 마이그레이션과 같은 작업은 클러스터를 단일 노드로 축소한 후에 수행하고, 완료 후 다시 확장하세요.

### 로드 밸런싱

클러스터 모드는 요청 분배, 애플리케이션 인스턴스의 상태 확인 및 장애 조치를 위해 로드 밸런서를 필요로 합니다. 이 부분은 팀의 운영 요구사항에 따라 직접 선택하고 구성해야 합니다.

자체 구축 Nginx를 예로 들어, 설정 파일에 다음 내용을 추가합니다.

```
upstream myapp {
    server 172.31.0.1:13000; # 내부 노드 1
    server 172.31.0.2:13000; # 내부 노드 2
    server 172.31.0.3:13000; # 내부 노드 3
}

server {
    listen 80;

    location / {
        # 정의된 upstream을 사용하여 로드 밸런싱
        proxy_pass http://myapp;
        # ... 기타 설정
    }
}
```

이는 요청을 역방향 프록시하여 여러 서버 노드로 분산 처리한다는 의미입니다.

다른 클라우드 서비스 제공업체에서 제공하는 로드 밸런싱 미들웨어는 해당 서비스 제공업체의 설정 문서를 참고해 주세요.

고가용성 배포를 위해서는 다음을 권장합니다.

- 동일한 클러스터 내에서 최소 2개의 애플리케이션 인스턴스를 실행하고, 인스턴스 장애 조치는 로드 밸런서가 처리하도록 하세요.
- 로드 밸런서의 헬스 체크는 단순히 포트가 열려 있는지만이 아니라, 애플리케이션의 실제 가용성을 반영해야 합니다.
- 가용 영역이나 리전을 넘는 웜 스탠바이가 필요하다면 일반적으로 여러 개의 독립적인 클러스터를 배치해야 하며, 데이터베이스, 공유 스토리지 및 기타 인프라의 동기화와 전환은 운영 팀이 담당해야 합니다.

## 환경 변수 설정

클러스터 내의 모든 노드는 동일한 환경 변수 설정을 사용해야 합니다. NocoBase의 기본 [환경 변수](/api/cli/env) 외에도 다음 미들웨어 관련 환경 변수를 설정해야 합니다.

### 핵심 비밀키

미들웨어 관련 환경 변수 외에도, 클러스터의 모든 노드는 다음 핵심 비밀키를 동일한 값으로 명시적으로 설정해야 합니다.

```ini
APP_KEY=
APP_AES_SECRET_KEY=
# 또는 읽기 전용으로 마운트한 키 파일 사용
# APP_AES_SECRET_KEY_PATH=
```

- `APP_KEY` 는 token / JWT 서명에 사용됩니다. 이를 명시적으로 설정하지 않으면 애플리케이션은 `storage` 아래의 기본 키 파일로 폴백합니다.
- `APP_AES_SECRET_KEY` 는 데이터베이스의 민감한 필드를 복호화하는 데 사용됩니다. 이를 명시적으로 설정하지 않으면 역시 `storage` 아래의 기본 키 파일로 폴백합니다.
- 일시적인 컨테이너 환경이나 다중 노드 배포에서 자동 생성된 로컬 키 파일에 의존하면, 재시작 후 토큰이 무효화되거나 과거의 암호화된 데이터를 복호화하지 못할 수 있습니다.

:::info{title=팁}
`APP_AES_SECRET_KEY` 는 32바이트 AES-256 키여야 하며, 64개의 16진수 문자로 표현해야 합니다.

클라우드 환경에서는 Secrets Manager, SSM Parameter Store, Kubernetes Secret 또는 읽기 전용으로 마운트한 키 파일 등을 통해 이러한 값을 중앙에서 관리하는 것을 권장합니다.
:::

### 멀티코어 모드

애플리케이션이 멀티코어 노드에서 실행될 때, 노드의 멀티코어 모드를 활성화할 수 있습니다.

```ini
# PM2 멀티코어 모드 활성화
# CLUSTER_MODE=max # 기본적으로 비활성화되어 있으며, 수동으로 설정해야 합니다.
```

Kubernetes에서 애플리케이션 파드(pod)를 배포하는 경우, 이 설정을 무시하고 파드의 복제본(replica) 수를 통해 애플리케이션 인스턴스 수를 제어할 수 있습니다.

### 캐시

```ini
# 캐시 어댑터. 클러스터 모드에서는 redis로 설정해야 합니다. (설정하지 않으면 기본값은 인메모리입니다.)
CACHE_DEFAULT_STORE=redis

# Redis 캐시 어댑터 연결 URL. 직접 입력해야 합니다.
CACHE_REDIS_URL=
```

### 동기화 신호

```ini
# Redis 동기화 어댑터 연결 URL. 설정하지 않으면 기본값은 redis://localhost:6379/0입니다.
PUBSUB_ADAPTER_REDIS_URL=
```

### 분산 잠금

```ini
# 잠금 어댑터. 클러스터 모드에서는 redis로 설정해야 합니다. (설정하지 않으면 기본값은 인메모리 로컬 잠금입니다.)
LOCK_ADAPTER_DEFAULT=redis

# Redis 잠금 어댑터 연결 URL. 설정하지 않으면 기본값은 redis://localhost:6379/0입니다.
LOCK_ADAPTER_REDIS_URL=
```

### 메시지 큐

```ini
# Redis를 메시지 큐 어댑터로 활성화합니다. 설정하지 않으면 기본값은 인메모리 어댑터입니다.
QUEUE_ADAPTER=redis
# Redis 메시지 큐 어댑터 연결 URL. 설정하지 않으면 기본값은 redis://localhost:6379/0입니다.
QUEUE_ADAPTER_REDIS_URL=
```

### 워커 ID 할당자

NocoBase의 일부 시스템 컬렉션은 전역 고유 ID를 기본 키로 사용합니다. 클러스터 내에서 기본 키 충돌을 방지하려면 워커 ID 할당자를 통해 각 애플리케이션 인스턴스에 고유한 워커 ID를 할당해야 합니다. 현재 워커 ID 범위는 0-31이며, 이는 동일한 애플리케이션이 최대 32개의 노드에서 동시에 실행될 수 있음을 의미합니다. 전역 고유 ID 설계에 대한 자세한 내용은 [@nocobase/snowflake-id](https://github.com/nocobase/nocobase/tree/main/packages/core/snowflake-id)를 참고해 주세요.

```ini
# 워커 ID 할당자 Redis 연결 URL.
# 생략하면 임의의 워커 ID가 할당됩니다.
REDIS_URL=
```

:::info{title=팁}
일반적으로 관련 어댑터는 모두 동일한 Redis 인스턴스를 사용할 수 있지만, 잠재적인 키 충돌 문제를 피하기 위해 다른 데이터베이스를 구분하여 사용하는 것이 좋습니다. 예를 들면 다음과 같습니다.

```ini
CACHE_REDIS_URL=redis://localhost:6379/0
PUBSUB_ADAPTER_REDIS_URL=redis://localhost:6379/1
LOCK_ADAPTER_REDIS_URL=redis://localhost:6379/2
QUEUE_ADAPTER_REDIS_URL=redis://localhost:6379/3
REDIS_URL=redis://localhost:6379/4
```

현재 각 플러그인은 자체 Redis 관련 환경 변수를 사용하고 있습니다. 향후 `REDIS_URL`을 기본(fallback) 설정으로 통합하여 사용하는 것을 고려할 예정입니다.

:::

Kubernetes를 사용하여 클러스터를 관리하는 경우, 위 환경 변수들을 ConfigMap 또는 Secret에 설정할 수 있습니다. 더 자세한 내용은 [Kubernetes 배포](./kubernetes)를 참고해 주세요.
