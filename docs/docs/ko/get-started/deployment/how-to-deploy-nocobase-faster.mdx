# 더 빠른 배포 방법

많은 사용자가 배포 후 NocoBase가 예상보다 느리게 실행된다고 느낄 수 있습니다. 이는 일반적으로 네트워크 환경, 서버 구성 또는 배포 아키텍처의 영향 때문입니다. 최적화 기법을 소개하기 전에, 먼저 정상적인 NocoBase 로딩 속도의 참고 값을 살펴보고 불필요한 우려를 피하도록 하겠습니다.

### NocoBase 정상 로딩 속도 참고 값

다음은 NocoBase 데모 환경에서 테스트한 로딩 속도입니다:

* URL을 입력하고 애플리케이션에 처음 진입하는 데 걸리는 시간: 약 2초
* 애플리케이션 내에서 페이지를 전환하는 데 걸리는 시간: 약 50-300밀리초

## I. 네트워크 및 인프라 최적화

### 1. HTTP 프로토콜 버전: HTTP/2를 손쉽게 도입

【전제 조건】

- **HTTPS 지원 필수**: 매우 중요합니다! 거의 모든 최신 브라우저는 HTTPS 연결에서만 HTTP/2를 지원하므로 SSL 인증서를 먼저 구성해야 합니다.
- **서버 요구사항**: Nginx 1.9.5+ 또는 Apache 2.4.17+ 등 HTTP/2를 지원하는 서버 소프트웨어를 사용해야 합니다.
- **TLS 버전**: TLS 1.2 이상을 권장합니다(TLS 1.3이 가장 좋음). 오래된 SSL 버전은 HTTP/2를 지원하지 않습니다.

【설명】

기존 HTTP/1.1 프로토콜은 여러 요청을 처리할 때 제한이 있습니다. 일반적으로 동시에 6-8개의 연결만 가능하며, 이는 매표 줄을 서서 기다리는 것과 비슷하여 쉽게 지연이 발생합니다.
![250416_http1_en](https://static-docs.nocobase.com/250416_http1_en.png)

HTTP/2는 "멀티플렉싱" 기술을 사용해 여러 요청을 동시에 처리하여 리소스 로딩을 크게 가속화합니다. 최신 HTTP/3는 불안정한 네트워크에서 더 뛰어난 성능을 발휘합니다.

![250416_http2_en](https://static-docs.nocobase.com/250416_http2_en.png)

【최적화 제안】

- 웹 서버에 HTTP/2 지원이 활성화되어 있는지 확인하세요. 대부분의 서버(Nginx, Caddy 등)에서 간단히 설정할 수 있습니다.
- Nginx에서는 listen 지시문 뒤에 `http2` 매개변수를 추가하기만 하면 됩니다:

```nginx
listen 443 ssl http2;
```

【검증】

브라우저 개발자 도구의 "Network" 탭을 열고, 마우스 우클릭 후 "Protocol"을 체크하면 현재 연결 프로토콜 버전을 확인할 수 있습니다:
![20250407145442](https://static-docs.nocobase.com/20250407145442.png)

테스트 결과, 전체적인 속도가 약 10% 향상되며, 시스템에 블록과 리소스가 많을수록 성능 향상이 더 두드러집니다.

### 2. 네트워크 대역폭: 클수록 좋고, 유연한 과금 방식 추천

【설명】

고속도로가 일반 도로보다 원활하듯, 대역폭은 데이터 전송 효율을 결정합니다. NocoBase가 처음 로딩될 때 많은 프런트엔드 리소스를 다운로드해야 하므로, 대역폭이 부족하면 쉽게 병목이 됩니다.

【최적화 제안】

- 충분한 대역폭을 선택하세요(다수 사용자의 경우 50Mbps+ 권장). 이 중요한 리소스에 인색하지 마세요.
- "사용량 기반 과금"을 권장합니다: 많은 클라우드 공급자가 이 유연한 모델을 제공하므로, 피크 시간대에는 더 높은 대역폭을 누리면서 평상시에는 비용을 통제할 수 있습니다.

### 3. 네트워크 지연과 서버 지리적 위치: 가까울수록 빠르다

【설명】

지연(Latency)은 데이터 전송의 대기 시간입니다. 대역폭이 충분하더라도 서버가 사용자로부터 너무 멀면(예: 사용자는 중국에 있고 서버는 미국에 있는 경우), 각 요청-응답이 긴 거리 때문에 느려질 수 있습니다.

【최적화 제안】

- 가능한 한 주요 사용자층과 가까운 지역에 NocoBase를 배포하세요.
- 사용자가 전 세계에 분포되어 있다면, 글로벌 가속 서비스(예: Alibaba Cloud Global Accelerator 또는 AWS Global Accelerator)를 사용해 네트워크 경로를 최적화하고 지연을 줄이는 것을 고려하세요.

【검증】

ping 명령으로 여러 지역에서의 지연을 테스트합니다.
이 접근 방식은 가장 뚜렷한 개선 효과를 보여주며, 지역에 따라 접속 속도가 1-3배 또는 그 이상 빨라집니다.

## II. 배포 아키텍처 최적화

### 4. 서버 배포 및 프록시 방식: 가장 적합한 아키텍처 선택

【전제 조건】

- **서버 권한**: Nginx 같은 서비스를 구성하려면 root 또는 sudo 권한이 필요합니다.
- **기본 지식**: 서버 구성에 대한 기본 지식이 필요하지만, 걱정하지 마세요. 구체적인 설정 예시를 제공합니다.
- **포트 접근**: 방화벽에서 80(HTTP) 및 443(HTTPS) 포트 접근이 허용되어야 합니다.

【설명】

사용자가 NocoBase에 접속할 때 요청은 직접 서버로 전달됩니다. 적절한 배포 방식은 서버가 요청을 효율적으로 처리하고 더 빠른 응답을 제공할 수 있게 합니다.

【다양한 솔루션과 권장 사항】

**역방향 프록시를 사용하지 않고 NocoBase 시작(비권장):**

- 단점: 이 방식은 간단하지만 높은 동시성이나 정적 파일을 처리할 때 성능이 떨어집니다. 개발 및 테스트 용도로만 적합합니다.
- 권장: 가급적 이 방식은 피하세요.

> 참고: "[설치 문서](https://v2.docs.nocobase.com/get-started/quickstart)"

역방향 프록시 없이는 홈페이지 로딩에 약 6.1초가 소요됩니다.
![20250409131357](https://static-docs.nocobase.com/20250409131357.png)

**Nginx / Caddy를 역방향 프록시로 사용(강력 권장):**

- 장점: 역방향 프록시 서버는 동시 연결을 효율적으로 처리하고, 정적 파일을 제공하며, 로드 밸런싱을 구현할 수 있고 HTTP/2 구성도 간단합니다.
- 권장: 프로덕션 환경에서는 애플리케이션 배포(소스 코드 배포 / create-nocobase-app / Docker 이미지) 후 Nginx 또는 Caddy를 역방향 프록시로 사용하세요.

> 참고: "[배포 문서](https://v2.docs.nocobase.com/get-started/deployment/production)"

Nginx 프록시를 사용하면 홈페이지 로딩에 약 3-4초가 소요됩니다.
![20250409131541](https://static-docs.nocobase.com/20250409131541.png)

![20250416081410](https://static-docs.nocobase.com/20250416081410.png)

**부하 분산을 적용한 클러스터 배포(고동시성·고가용성 시나리오에 적합):**

- 장점: 여러 인스턴스를 배포해 요청을 처리하면 전체 시스템 안정성과 동시 처리 능력을 크게 향상시킬 수 있습니다.
- 구체적인 배포 방법은 **[클러스터 모드](https://docs.nocobase.com/welcome/getting-started/deployment/cluster-mode)**를 참고하세요.

### 5. CDN을 활용한 정적 리소스 가속

【전제 조건】

- **도메인 요구사항**: 등록된 도메인 이름과 DNS 설정을 관리할 수 있는 권한이 필요합니다.
- **SSL 인증서**: 대부분의 CDN 서비스는 SSL 인증서 구성이 필요합니다(무료 Let's Encrypt 인증서 사용 가능).
- **서비스 선택**: 사용자 지역에 따라 적절한 CDN 제공자를 선택하세요(중국 본토 사용자에게는 ICP 등록이 된 CDN이 필요합니다).

【설명】
CDN(Content Delivery Network)은 전 세계 노드에 정적 리소스를 캐싱하여 사용자가 가장 가까운 노드에서 리소스를 받을 수 있게 합니다. 가까운 수원지에서 물을 얻는 것과 같이 로딩 지연을 크게 줄입니다.

또한, CDN의 가장 큰 장점은 **애플리케이션 서버의 부하와 대역폭 압박을 크게 줄여주는 것**입니다. CDN 없이 많은 사용자가 NocoBase에 동시 접속하면, 모든 정적 리소스 요청(JavaScript, CSS, 이미지 등)이 서버로 직접 전달되어 대역폭 포화, 성능 저하, 심지어 서버 다운까지 일으킬 수 있습니다. 이러한 요청을 CDN으로 오프로딩하면 서버는 핵심 비즈니스 로직 처리에 집중할 수 있어 더 안정적인 사용자 경험을 제공합니다.

![20250416_0826](https://static-docs.nocobase.com/20250416_0826.png)

【최적화 제안】• CDN을 통해 정적 리소스 요청을 분산하도록 서버를 구성하세요.• 사용자 위치에 따라 적합한 CDN 제공자를 선택하세요:

- 글로벌 사용자: Cloudflare, Akamai, AWS CloudFront;
- 중국 본토 사용자: Alibaba Cloud CDN, Tencent Cloud CDN, Baidu Cloud Acceleration. 설정 예시:

```nginx
# 정적 리소스를 CDN 도메인으로 리다이렉트
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
    rewrite ^(.*)$ https://your-cdn-domain.com$1 permanent;
}
```

- 소규모 프로젝트라면 Cloudflare의 무료 플랜이 좋은 CDN 가속을 제공합니다:

1. Cloudflare 계정을 등록하고 도메인을 추가하세요;
2. DNS를 수정하여 도메인을 Cloudflare 서버로 지정하세요;
3. 제어판에서 적절한 캐시 레벨을 설정하세요.

**특별 안내**: 모든 사용자가 같은 지역에 있더라도, 서버 부담을 효과적으로 줄이고 특히 트래픽 피크 시 시스템 전체 안정성을 향상시키므로 CDN 사용을 강력히 권장합니다.

## III. 정적 리소스 최적화

### 6. 서버 압축 및 캐시 설정

【전제 조건】

- **CPU 리소스**: 압축은 서버 CPU 부하를 증가시키므로 서버에 충분한 처리 능력이 있어야 합니다.
- **Nginx 모듈 지원**: Gzip 압축은 일반적으로 내장되어 있지만, Brotli 압축은 추가 모듈 설치가 필요할 수 있습니다.
- **구성 권한**: 서버 설정을 수정할 수 있는 권한이 필요합니다.

【설명】

압축을 활성화하고 합리적인 캐시 전략을 적용하면 데이터 전송량과 반복 요청을 크게 줄일 수 있습니다. 본질적으로 리소스를 "다이어트"시켜 로딩 속도를 끌어올리는 것입니다.
![20250416081719](https://static-docs.nocobase.com/20250416081719.png)

【최적화 제안】

- 가장 간단한 방법: Cloudflare의 무료 CDN 서비스를 사용하면 Gzip 압축이 자동 활성화됩니다.
- Gzip 또는 Brotli 압축을 활성화하세요. Nginx에서 다음과 같이 설정할 수 있습니다:

```nginx
# Gzip 압축 활성화
gzip on;
gzip_comp_level 6;
gzip_min_length 1000;
gzip_proxied any;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript;

# Brotli 압축이 지원되면 더 효율적인 압축을 위해 활성화
brotli on;
brotli_comp_level 6;
brotli_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript;
```

- 정적 리소스에 적절한 캐시 헤더를 설정하여 반복 로딩을 줄이세요:

```nginx
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg)$ {
    expires 30d;
    add_header Cache-Control "public, max-age=2592000";
    access_log off;
}
```

### 7. SSL/TLS 사용 및 성능 최적화

【전제 조건】

- **SSL 인증서**: 유효한 SSL 인증서가 필요합니다(무료 Let's Encrypt 인증서 사용 가능).
- **서버 구성 권한**: SSL 설정을 수정할 수 있어야 합니다.
- **DNS 구성**: OCSP Stapling을 위해 안정적인 DNS 리졸버를 구성하세요.

【설명】

보안은 언제나 최우선이지만, 부적절한 HTTPS 구성은 약간의 지연을 추가할 수 있습니다. 아래는 보안을 유지하면서 높은 성능을 유지할 수 있는 최적화 팁입니다.

【최적화 제안】

- 현재 가장 빠른 TLS 버전인 TLS 1.3을 사용하세요. Nginx 설정:

```nginx
ssl_protocols TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers on;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
```

- OCSP Stapling을 활성화하여 인증서 검증 시간을 단축하세요:

```nginx
ssl_stapling on;
ssl_stapling_verify on;
resolver 8.8.8.8 8.8.4.4 valid=300s;
resolver_timeout 5s;
```

- 세션 재사용을 통해 반복 핸드셰이크 시간을 줄이세요:

```nginx
ssl_session_cache shared:SSL:10m;
ssl_session_timeout 10m;
```

## IV. 모니터링 및 문제 해결

### 8. 성능 모니터링 및 기본 분석

【전제 조건】

- **접근성**: 대부분의 온라인 테스트 도구를 사용하려면 웹사이트가 공개적으로 접근 가능해야 합니다.
- **기본 지식**: 성능 지표의 기본 의미를 이해해야 하지만, 각 주요 지표를 설명해드리겠습니다.

【설명】

병목이 어디인지 모르고 정확하게 최적화하는 것은 어렵습니다. 무료 도구로 웹사이트 성능을 모니터링하여 문제를 식별할 것을 권장합니다.

【최적화 제안】

**다음 무료 도구로 웹사이트 성능을 확인하세요:**

- [Google PageSpeed Insights](https://pagespeed.web.dev/)
- [WebPageTest](https://www.webpagetest.org/)
- [Pingdom](https://tools.pingdom.com/)

**다음 주요 지표에 집중하세요:**

- 페이지 로딩 시간
- 서버 응답 시간
- DNS 해석 시간
- SSL 핸드셰이크 시간

**자주 발생하는 문제 해결책:**

- DNS 해석이 느리다? DNS 서비스 변경 또는 DNS 사전 해석 활성화를 고려하세요.
- SSL 핸드셰이크가 느리다? SSL 구성을 최적화하고 세션 재사용을 활성화하세요.
- 서버 응답이 느리다? 서버 리소스를 점검하고 필요시 업그레이드하세요.
- 정적 리소스 로딩이 느리다? CDN을 도입하고 캐시 전략을 조정해보세요.

## 배포 최적화 빠른 체크리스트

다음 체크리스트는 NocoBase 배포를 빠르게 점검하고 최적화하는 데 도움이 됩니다:

1. **HTTP 버전 확인**

   - [ ]  HTTPS 활성화(HTTP/2의 전제 조건)
   - [ ]  HTTP/2 활성화
   - [ ]  조건이 허락하면 HTTP/3 지원 고려
2. **대역폭 평가**

   - [ ]  충분한 서버 대역폭(최소 10Mbps 권장, 50Mbps+ 선호)
   - [ ]  고정 대역폭이 아닌 사용량 기반 과금 모델 고려
3. **서버 위치 선택**

   - [ ]  서버 위치는 사용자 지역과 가까워야 함
   - [ ]  전 세계 사용자에게는 글로벌 가속 서비스 사용 고려
4. **배포 아키텍처**

   - [ ]  Nginx/Caddy를 역방향 프록시로 사용해 정적 리소스와 API 분리
   - [ ]  필요한 경우 멀티 인스턴스 배포 및 로드 밸런싱 기술 도입
5. **CDN 도입**

   - [ ]  CDN을 통한 정적 리소스 분산 가속
   - [ ]  적절한 캐시 전략 구성
   - [ ]  CDN이 HTTP/2 또는 HTTP/3 지원 확인
6. **압축 및 캐시**

   - [ ]  Gzip 또는 Brotli 압축 활성화
   - [ ]  적절한 브라우저 캐시 헤더 설정
7. **SSL/TLS 최적화**

   - [ ]  TLS 1.3을 사용해 핸드셰이크 속도 향상
   - [ ]  OCSP Stapling 활성화
   - [ ]  SSL 세션 재사용 구성
8. **성능 모니터링**

   - [ ]  정기적으로 성능 테스트 도구로 웹사이트 평가
   - [ ]  주요 지표 모니터링(로딩, 응답, 해석, 핸드셰이크 시간)
   - [ ]  식별된 문제에 기반한 최적화 진행

## 자주 묻는 질문

【Q】서버가 사용자와 다른 지역에 배포되어 있어 접속이 느립니다. 어떻게 해야 하나요?

【A】가장 좋은 해결책은 주요 사용자와 같은 지역의 클라우드 서버를 선택하는 것입니다. 그것이 불가능하다면, 다음도 고려할 수 있습니다:

1. CDN 서비스를 사용해 정적 리소스 가속;
2. 글로벌 가속 서비스를 활용해 네트워크 경로 최적화;
3. 가능한 한 모든 압축 및 캐시 최적화 조치 활성화.

【Q】왜 NocoBase는 처음 로딩이 느리고 그 후에는 빠른가요?

【A】처음 로딩이 느린 것은 정상입니다. 처음에 많은 리소스를 다운로드해야 하기 때문입니다.

저희 공식 Demo를 예로 들면, 처음 로딩은 일반적으로 약 3초가 걸립니다. 이후 일상적인 접속은 URL 입력 시 약 1-2초가 걸리며, 애플리케이션 내 페이지 전환은 약 50-300밀리초로 매우 빠르고 지연이 거의 없습니다.

![20250416130719](https://static-docs.nocobase.com/20250416130719.png)

과도하게 긴 로딩 시간이 있다면 최적화의 여지가 남아 있습니다:

1. HTTP/2가 활성화되었는지 확인;
2. CDN 가속 도입;
3. Gzip/Brotli 압축 활성화;
4. 서버 대역폭이 충분한지 점검.

【Q】공유 호스팅을 사용 중이라 Nginx 설정을 수정할 수 없습니다. 어떻게 해야 하나요?

【A】이 경우 최적화 옵션은 더 적지만 다음을 권장합니다:

1. CDN 서비스 사용 시도(예: Cloudflare);
2. 애플리케이션 내에서 조정 가능한 매개변수 최적화;
3. 조건이 허락하면 더 많은 사용자 정의 설정을 지원하는 VPS로 업그레이드 고려.

---

이러한 간단하면서도 실용적인 배포 최적화 전략을 통해 NocoBase 접속 속도를 크게 향상시키고 사용자에게 더 부드러운 경험을 제공할 수 있습니다. 많은 최적화 조치는 코드 변경 없이 몇 시간 내에 완료할 수 있으며 쉽게 효과를 확인할 수 있습니다.
