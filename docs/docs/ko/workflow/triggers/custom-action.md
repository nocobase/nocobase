# 사용자 정의 작업 이벤트

## 소개

NocoBase에는 일반적인 데이터 작업(추가, 삭제, 수정, 조회 등)이 내장되어 있습니다. 이러한 작업이 복잡한 비즈니스 요구 사항을 충족하지 못할 때, 워크플로우에서 사용자 정의 작업 이벤트를 사용하고 해당 이벤트를 페이지 블록의 "워크플로우 트리거" 버튼에 바인딩할 수 있습니다. 사용자가 클릭하면 사용자 정의 작업 워크플로우가 트리거됩니다.

## 워크플로우 생성

워크플로우를 생성할 때 "사용자 정의 작업 이벤트"를 선택합니다:

![“사용자 정의 작업 이벤트” 워크플로우 생성](https://static-docs.nocobase.com/20240509091820.png)

## 트리거 설정

### 컨텍스트 유형

> v.1.6.0+

컨텍스트 유형에 따라 해당 워크플로우를 바인딩할 수 있는 블록 버튼이 결정됩니다:

* 컨텍스트 없음: 전역 이벤트이며, 작업 패널이나 데이터 블록의 작업 버튼에 바인딩할 수 있습니다.
* 단일 데이터: 테이블 데이터 행, 폼, 상세 정보 등 데이터 블록의 작업 버튼에 바인딩할 수 있습니다.
* 다중 데이터: 테이블의 일괄 작업 버튼에 바인딩할 수 있습니다.

![트리거 설정_컨텍스트 유형](https://static-docs.nocobase.com/20250215135808.png)

### 컬렉션

컨텍스트 유형이 단일 데이터 또는 다중 데이터인 경우, 데이터 모델을 바인딩할 컬렉션을 선택해야 합니다:

![트리거 설정_컬렉션 선택](https://static-docs.nocobase.com/20250215135919.png)

### 사용할 관계 데이터

워크플로우에서 트리거된 데이터 행의 관계 데이터를 사용해야 하는 경우, 여기서 심층 관계 필드를 선택할 수 있습니다:

![트리거 설정_사용할 관계 데이터 선택](https://static-docs.nocobase.com/20250215135955.png)

이 필드들은 이벤트가 트리거된 후 워크플로우 컨텍스트에 자동으로 미리 로드되어 워크플로우에서 사용할 수 있게 됩니다.

## 작업 설정

워크플로우에 설정된 컨텍스트 유형에 따라 각 블록의 작업 버튼 설정이 달라집니다.

> v1.6.0+

버튼을 추가한 후 이전에 생성한 컨텍스트 없는 워크플로우를 바인딩합니다. 작업 패널의 버튼을 예로 들면 다음과 같습니다:

### 다중 데이터

> v1.6.0+

테이블 블록의 작업 표시줄에 "워크플로우 트리거" 버튼을 추가할 때, 컨텍스트 유형을 "컨텍스트 없음" 또는 "다중 데이터" 중에서 선택하는 추가 옵션이 있습니다:

![블록에 작업 버튼 추가_테이블](https://static-docs.nocobase.com/20250215222507.png)

"컨텍스트 없음"을 선택하면 전역 이벤트가 되며, 컨텍스트 없는 유형의 워크플로우만 바인딩할 수 있습니다.

"다중 데이터"를 선택하면 다중 데이터 유형의 워크플로우를 바인딩할 수 있으며, 데이터 다중 선택 후의 일괄 작업에 사용할 수 있습니다(현재 테이블만 지원). 이때 선택 가능한 워크플로우 범위는 현재 데이터 블록의 컬렉션과 일치하도록 설정된 워크플로우로 제한됩니다:

![20250215224436](https://static-docs.nocobase.com/20250215224436.png)

버튼을 클릭하여 트리거할 때, 반드시 테이블에서 일부 데이터 행을 선택해야 하며, 그렇지 않으면 워크플로우가 트리거되지 않습니다:

![20250215224736](https://static-docs.nocobase.com/20250215224736.png)

## 예시

예를 들어 "샘플" 컬렉션이 있고, "수집됨"(상태)인 샘플에 대해 "검사 제출" 작업을 제공해야 한다고 가정해 봅시다. 검사 제출은 먼저 샘플의 기본 정보를 확인한 후 "검사 기록" 데이터를 생성하고, 샘플 상태를 "검사 제출됨"으로 수정합니다. 이러한 일련의 과정은 단순한 "추가, 삭제, 수정, 조회" 버튼 클릭만으로는 완료할 수 없으며, 이때 사용자 정의 작업 이벤트를 사용하여 구현할 수 있습니다.

먼저 "샘플" 컬렉션과 "검사 기록" 컬렉션을 생성하고, 샘플 컬렉션에 기본적인 테스트 데이터를 입력합니다:

![예시_샘플 컬렉션](https://static-docs.nocobase.com/20240509172234.png)

그 다음 "사용자 정의 작업 이벤트" 워크플로우를 생성합니다. 작업 프로세스에 즉각적인 피드백이 필요한 경우 동기 모드를 선택할 수 있습니다(동기 모드에서는 수동 처리 등 비동기 유형의 노드를 사용할 수 없습니다):

![예시_워크플로우 생성](https://static-docs.nocobase.com/20240509173106.png)

트리거 설정에서 컬렉션으로 "샘플"을 선택합니다:

![예시_트리거 설정](https://static-docs.nocobase.com/20240509173148.png)

비즈니스 요구 사항에 따라 프로세스 로직을 구성합니다. 예를 들어 지표 매개변수가 `90`보다 클 때만 검사 제출을 허용하고, 그렇지 않으면 관련 문제를 알립니다:

![예시_비즈니스 로직 구성](https://static-docs.nocobase.com/20240509174159.png)

:::info{title=팁}
"[응답 메시지](../nodes/response-message.md)" 노드는 동기식 사용자 정의 작업 이벤트에서 클라이언트에 알림 메시지를 반환하는 데 사용할 수 있습니다. 비동기 모드에서는 사용할 수 없습니다.
:::

프로세스를 설정하고 활성화한 후, 테이블 인터페이스로 돌아가 테이블의 작업 열에 "워크플로우 트리거" 버튼을 추가합니다:

![예시_작업 버튼 추가](https://static-docs.nocobase.com/20240509174525.png)

그 다음 버튼의 설정 메뉴에서 워크플로우 바인딩을 선택하여 설정 팝업을 엽니다:

![예시_워크플로우 바인딩 팝업 열기](https://static-docs.nocobase.com/20240509174633.png)

이전에 활성화한 워크플로우를 추가합니다:

![예시_워크플로우 선택](https://static-docs.nocobase.com/20240509174723.png)

제출 후 버튼 텍스트를 "검사 제출"과 같은 작업 명칭으로 수정하면 설정 프로세스가 완료됩니다.

사용 시 테이블에서 샘플 데이터를 하나 선택하고 "검사 제출" 버튼을 클릭하면 사용자 정의 작업 이벤트가 트리거됩니다. 이전에 구성한 로직에 따라 샘플 지표 매개변수가 90 미만인 경우 클릭 시 다음과 같이 표시됩니다:

![예시_지표가 검사 제출 조건을 충족하지 않음](https://static-docs.nocobase.com/20240509175026.png)

지표 매개변수가 90보다 크면 프로세스가 정상적으로 실행되어 "검사 기록" 데이터를 생성하고 샘플 상태를 "검사 제출됨"으로 수정합니다:

![예시_검사 제출 성공](https://static-docs.nocobase.com/20240509175247.png)

이로써 간단한 사용자 정의 작업 이벤트가 완성되었습니다. 마찬가지로 주문 처리, 보고서 제출 등 복잡한 작업이 있는 비즈니스에 대해 사용자 정의 작업 이벤트를 통해 구현할 수 있습니다.

## 외부 호출

사용자 정의 작업 이벤트의 트리거는 사용자 인터페이스 작업에 국한되지 않으며, HTTP API 호출을 통해서도 트리거할 수 있습니다. 특히 사용자 정의 작업 이벤트는 모든 컬렉션 작업에 대해 워크플로우를 트리거하는 새로운 작업 유형인 `trigger`를 제공하며, NocoBase 표준 작업 API를 사용하여 호출할 수 있습니다.

:::info{title="팁"}
외부 호출도 사용자 신원을 기반으로 해야 하므로, HTTP API를 통해 호출할 때 일반 인터페이스에서 보내는 요청과 마찬가지로 `Authorization` 요청 헤더 또는 `token` 매개변수(로그인 시 획득한 토큰), 그리고 `X-Role` 요청 헤더(사용자의 현재 역할 이름) 등 인증 정보를 제공해야 합니다.
:::

### 컨텍스트 없음

컨텍스트 없는 워크플로우는 workflows 리소스에 대해 트리거 작업을 수행해야 합니다:

```bash
curl -X POST -H 'Authorization: Bearer <your token>' -H 'X-Role: <roleName>' \
  "http://localhost:3000/api/workflows:trigger?triggerWorkflows=workflowKey"
```

### 단일 데이터

예시와 같이 버튼으로 트리거되는 워크플로우는 다음과 같이 호출할 수 있습니다:

```bash
curl -X POST -H 'Authorization: Bearer <your token>' -H 'X-Role: <roleName>' \
  "http://localhost:3000/api/samples:trigger/<:id>?triggerWorkflows=workflowKey"
```

해당 작업은 단일 데이터를 대상으로 하므로, 기존 데이터에 대해 호출할 때 데이터 행의 ID를 지정하여 URL의 `<:id>` 부분을 대체해야 합니다.

폼에 대해 호출하는 경우(예: 추가 또는 업데이트), 데이터 추가 폼의 경우 ID를 전달하지 않아도 되지만, 실행 컨텍스트 데이터로서 제출된 데이터를 전달해야 합니다:

```bash
curl -X POST -H 'Authorization: Bearer <your token>' -H 'X-Role: <roleName>' -d \
  '{
    "title": "Sample 1",
    "id": 91
  }'
  "http://localhost:3000/api/samples:trigger?triggerWorkflows=workflowKey"
```

업데이트 폼의 경우 데이터 행의 ID와 업데이트된 데이터를 동시에 전달해야 합니다:

```bash
curl -X POST -H 'Authorization: Bearer <your token>' -H 'X-Role: <roleName>' -d \
  '{
    "title": "Sample 1",
    "id": 91
  }'
  "http://localhost:3000/api/samples:trigger/<:id>?triggerWorkflows=workflowKey"
```

ID와 데이터를 동시에 전달하면, 먼저 ID에 해당하는 데이터 행을 로드한 다음, 전달된 데이터 객체의 속성을 사용하여 원본 데이터 행을 덮어써서 최종적인 트리거 데이터 컨텍스트를 얻습니다.

:::warning{title="주의"}
관계 데이터가 전달되면 해당 데이터도 덮어쓰기 되므로, 특히 관계 데이터 항목을 미리 로드하도록 설정한 경우 관계 데이터가 예상치 않게 덮어쓰이지 않도록 전달 데이터를 신중하게 처리해야 합니다.
:::

또한 URL 매개변수 `triggerWorkflows`는 워크플로우의 key이며, 여러 워크플로우는 쉼표로 구분합니다. 해당 key는 워크플로우 캔버스 상단의 워크플로우 이름에 마우스를 올리면 확인할 수 있습니다:

![워크플로우_key_확인 방법](https://static-docs.nocobase.com/20240426135108.png)

위 호출이 성공하면 해당 `samples` 컬렉션의 사용자 정의 작업 이벤트가 트리거됩니다.

:::info{title="팁"}
HTTP API 호출을 통해 작업 트리거 시에도 워크플로우의 활성화 상태 및 컬렉션 설정 일치 여부에 주의해야 합니다. 그렇지 않으면 호출이 성공하지 않거나 오류가 발생할 수 있습니다.
:::

### 다중 데이터

단일 데이터 호출 방식과 유사하지만, 전달하는 데이터에는 여러 개의 기본 키 매개변수(`filterByTk[]`)만 필요하며 data 부분은 전달할 필요가 없습니다:

```bash
curl -X POST -H 'Authorization: Bearer <your token>' -H 'X-Role: <roleName>' \
  "http://localhost:3000/api/samples:trigger?filterByTk[]=1&filterByTk[]=2&triggerWorkflows=workflowKey"
```

해당 호출은 다중 데이터 모드의 사용자 정의 작업 이벤트를 트리거하며, id가 1과 2인 데이터를 트리거 컨텍스트 데이터로 사용합니다.
