# ctx.logger

Обертка для логирования на базе [pino](https://github.com/pinojs/pino), обеспечивающая высокопроизводительные структурированные логи в формате JSON. Рекомендуется использовать `ctx.logger` вместо `console` для упрощения сбора и анализа логов.

## Область применения

`ctx.logger` можно использовать во всех сценариях RunJS для отладки, отслеживания ошибок, анализа производительности и т. д.

## Определение типа

```ts
logger: pino.Logger;
```

`ctx.logger` является экземпляром `engine.logger.child({ module: 'flow-engine' })`, то есть дочерним логгером pino с контекстом `module`.

## Уровни логирования

pino поддерживает следующие уровни (от высшего к низшему):

| Уровень | Метод | Описание |
|------|------|------|
| `fatal` | `ctx.logger.fatal()` | Критическая ошибка, обычно приводящая к завершению процесса |
| `error` | `ctx.logger.error()` | Ошибка, указывающая на неудачный запрос или операцию |
| `warn` | `ctx.logger.warn()` | Предупреждение, указывающее на потенциальные риски или аномальные ситуации |
| `info` | `ctx.logger.info()` | Общая информация о работе системы |
| `debug` | `ctx.logger.debug()` | Отладочная информация, используемая при разработке |
| `trace` | `ctx.logger.trace()` | Подробная трассировка для глубокой диагностики |

## Рекомендуемый способ записи

Рекомендуется использовать формат `level(msg, meta)`: сначала сообщение, затем необязательный объект с метаданными.

```ts
ctx.logger.info('Загрузка блока завершена');
ctx.logger.info('Операция выполнена успешно', { recordId: 456 });
ctx.logger.warn('Предупреждение о производительности', { duration: 5000 });
ctx.logger.error('Ошибка операции', { userId: 123, action: 'create' });
ctx.logger.error('Ошибка запроса', { err });
```

pino также поддерживает форматы `level(meta, msg)` (объект первым аргументом) или `level({ msg, ...meta })` (единый объект), которые можно использовать по мере необходимости.

## Примеры

### Базовое использование

```ts
ctx.logger.info('Загрузка блока завершена');
ctx.logger.warn('Ошибка запроса, используется кэш', { err });
ctx.logger.debug('Сохранение...', { recordId: ctx.record?.id });
```

### Создание дочернего логгера с помощью child()

```ts
// Создание дочернего логгера с контекстом для текущей логики
const log = ctx.logger.child({ scope: 'myBlock' });
log.info('Выполнение шага 1');
log.debug('Выполнение шага 2', { step: 2 });
```

### Связь с console

Рекомендуется использовать `ctx.logger` напрямую для получения структурированных JSON-логов. Если вы привыкли использовать `console`, соответствия следующие: `console.log` → `ctx.logger.info`, `console.error` → `ctx.logger.error`, `console.warn` → `ctx.logger.warn`.

## Формат логов

pino выводит структурированный JSON, где каждая запись содержит:

- `level`: Уровень логирования (числовой)
- `time`: Метка времени (в миллисекундах)
- `msg`: Сообщение лога
- `module`: Фиксированное значение `flow-engine`
- Другие пользовательские поля (передаваемые через объекты)

## Примечания

- Логи представляют собой структурированный JSON, что облегчает их сбор, поиск и анализ.
- Для дочерних логгеров, созданных через `child()`, также рекомендуется использовать формат `level(msg, meta)`.
- Некоторые среды выполнения (например, рабочие процессы) могут использовать другие методы вывода логов.
