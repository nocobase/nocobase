# Сборка и упаковка

После завершения разработки плагина необходимо выполнить два шага — сборку (компиляция исходного кода) и упаковку (создание `.tgz`) — прежде чем распространять его в другие приложения NocoBase.

## Сборка плагина

Выполните команду сборки в каталоге исходного кода (`<app-path>/source/`). Сборка компилирует исходный код TypeScript из `src/` в JavaScript — клиентский код упаковывается с помощью Rsbuild, серверный код — с помощью tsup:

```bash
cd <app-path>/source
nb source build @my-project/plugin-hello
```

Артефакты сборки выводятся в каталог `dist/` в корне плагина.

:::tip Совет

Первая сборка может запустить полную проверку типов по всему репозиторию, что займёт больше времени. Рекомендуется убедиться, что зависимости установлены, а репозиторий находится в собираемом состоянии.

:::

## Упаковка плагина

Выполняется также в каталоге исходного кода (`<app-path>/source/`). С параметром `--tar` можно объединить сборку и упаковку в один шаг, сгенерировав архив `.tgz`:

```bash
cd <app-path>/source
nb source build @my-project/plugin-hello --tar
```

По умолчанию файл пакета выводится в каталог `source/storage/tar/`. После завершения команда выведет полный путь к tarball-файлу.

## Загрузка в другие приложения NocoBase

Загрузите и распакуйте файл `.tar.gz` в каталог `./storage/plugins` целевого приложения. Подробные шаги описаны в разделе [Установка и обновление плагинов](../get-started/install-upgrade-plugins.mdx).

### Включение плагина по умолчанию

После загрузки плагин не активируется автоматически — он появится в «Менеджере плагинов» и его потребуется включить вручную. Если вы поддерживаете собственное приложение NocoBase и хотите, чтобы плагин включался по умолчанию вместе с приложением, можно использовать переменную среды `APPEND_PRESET_BUILT_IN_PLUGINS` (добавить встроенные плагины). Подробнее см. в разделе [Предустановка и включение плагина по умолчанию](./write-your-first-plugin.md#предустановка-и-включение-плагина-по-умолчанию-необязательно).

## Пользовательская конфигурация сборки

Обычно конфигурации сборки по умолчанию достаточно. Если вам нужна настройка — например, изменение точки входа, добавление псевдонимов, настройка параметров сжатия и т. д. — создайте файл `build.config.ts` в корневом каталоге плагина:

```ts
import { defineConfig } from '@nocobase/build';

export default defineConfig({
  modifyRsbuildConfig: (config) => {
    // Изменение конфигурации Rsbuild для клиентской части (src/client-v2)
    // Справка: https://rsbuild.rs/guide/configuration/rsbuild
    return config;
  },
  modifyTsupConfig: (config) => {
    // Изменение конфигурации tsup для серверной части (src/server)
    // Справка: https://tsup.egoist.dev/#using-custom-configuration
    return config;
  },
  beforeBuild: (log) => {
    // Обратный вызов перед началом сборки, например очистка временных файлов, генерация кода и т. д.
  },
  afterBuild: (log) => {
    // Обратный вызов после завершения сборки, например копирование дополнительных ресурсов, вывод статистики и т. д.
  },
});
```

Ключевые моменты:

- `modifyRsbuildConfig` — используется для настройки клиентской сборки: добавление плагинов Rsbuild, изменение псевдонимов resolve, настройка стратегии разделения кода и т. д. Параметры конфигурации см. в [документации Rsbuild](https://rsbuild.rs/guide/configuration/rsbuild)
- `modifyTsupConfig` — используется для настройки серверной сборки: изменение target, externals, entry и т. д. Параметры конфигурации см. в [документации tsup](https://tsup.egoist.dev/#using-custom-configuration)
- `beforeBuild` / `afterBuild` — хуки до и после сборки, принимающие функцию `log` для вывода журнала. Например, в `beforeBuild` можно сгенерировать файлы кода, а в `afterBuild` — скопировать статические ресурсы в каталог артефактов
