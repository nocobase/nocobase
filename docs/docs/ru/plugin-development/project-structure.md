# Структура каталогов проекта

Приложение, инициализированное через NocoBase CLI (`nb init`), генерирует стандартную структуру каталогов. CLI поддерживает два источника — npm (`create-nocobase-app`) и Git, при этом структура верхнего уровня приложения одинакова.

## Обзор каталогов верхнего уровня

```bash
<app-path>/
├── .nb/                   # Метаданные CLI для текущего окружения
├── source/                # Исходный код приложения (ядро NocoBase + встроенные плагины)
├── storage/               # Каталог данных времени выполнения
│   ├── plugins/           # Скомпилированные плагины (загруженные или импортированные)
│   └── tar/               # Файлы пакетов плагинов (.tgz)
├── plugins/               # Исходный код ваших плагинов (nb scaffold plugin создаёт их здесь)
├── .env                   # Файл переменных окружения приложения
```

## plugins/ — каталог разработки плагинов

`plugins/` — это основное место для разработки пользовательских плагинов. Плагины, созданные с помощью `nb scaffold plugin`, размещаются здесь.

`nb` автоматически синхронизирует плагины из `plugins/` в `source/packages/plugins/` через символические ссылки для использования в процессе разработки и сборки. Вам не нужно вручную работать с содержимым каталога `source/`.

## source/ — каталог исходного кода

Каталог `source/` содержит полный исходный код NocoBase, его содержимое зависит от источника проекта:

- **Источник npm** (`create-nocobase-app`): по умолчанию содержит только базовые каталоги, такие как `packages/plugins/`.
- **Источник Git** (рекомендуется): содержит полный исходный код ядра фреймворка (`packages/core/`), встроенные плагины и т.д., что позволяет напрямую обращаться к ним при разработке с помощью AI.

## storage/ — каталог времени выполнения

В `storage/` хранятся данные, генерируемые во время выполнения, и результаты сборки:

- `plugins/`: упакованные плагины, загруженные через интерфейс или импортированные через CLI.
- `tar/`: сжатые пакеты плагинов, сгенерированные после выполнения `nb source build <plugin> --tar`.

## Пути загрузки плагинов и их приоритет

Плагины могут находиться в нескольких местах. При запуске NocoBase загружает их в следующем порядке приоритета:

1. Версия исходного кода в `source/packages/plugins` (для локальной разработки и отладки, автоматически синхронизируется `nb` из `plugins/`).
2. Упакованная версия в `storage/plugins` (загруженная через интерфейс или импортированная через CLI).
3. Пакеты зависимостей в `node_modules` (установленные через npm/yarn или встроенные во фреймворк).

Если плагин с одним и тем же именем существует как в каталоге исходного кода, так и в каталоге с упакованными версиями, NocoBase будет отдавать приоритет загрузке версии из исходного кода, что упрощает локальное переопределение и отладку.

## Шаблон каталога плагина

Создание плагина с помощью CLI:

```bash
nb scaffold plugin @my-project/plugin-hello
```

Сгенерированная структура каталогов выглядит следующим образом:

```bash
plugins/@my-project/plugin-hello/
├── dist/                    # Результаты сборки (генерируются по мере необходимости)
├── src/                     # Каталог исходного кода
│   ├── client-v2/           # Код фронтенда (блоки, страницы, модели и т.д.)
│   │   ├── plugin.ts        # Основной класс клиентского плагина
│   │   └── index.ts         # Точка входа клиентской части
│   ├── locale/              # Многоязычные ресурсы (общие для фронтенда и бэкенда)
│   ├── swagger/             # Документация OpenAPI/Swagger
│   └── server/              # Код бэкенда
│       ├── collections/     # Определения коллекций
│       ├── commands/        # Пользовательские команды
│       ├── migrations/      # Скрипты миграции базы данных
│       ├── plugin.ts        # Основной класс серверного плагина
│       └── index.ts         # Точка входа серверной части
├── index.ts                 # Экспорт для связи фронтенда и бэкенда
├── client-v2.d.ts           # Объявления типов фронтенда
├── client-v2.js             # Артефакт сборки фронтенда
├── server.d.ts              # Объявления типов бэкенда
├── server.js                # Артефакт сборки бэкенда
├── .npmignore               # Конфигурация игнорирования при публикации
└── package.json
```

:::tip Подсказка

После завершения сборки каталог `dist/` и файлы `client-v2.js`, `server.js` будут загружены при включении плагина. На этапе разработки вам нужно изменять только каталог `src/`. Перед публикацией выполните `nb source build <plugin>` или `nb source build <plugin> --tar`.

:::
