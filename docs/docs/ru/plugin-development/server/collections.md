# Коллекции

В разработке **плагинов** NocoBase **коллекция (таблица данных)** — одно из ключевых понятий. Вы можете добавлять или изменять структуру таблиц данных в плагинах, определяя или расширяя **коллекции**. В отличие от таблиц данных, созданных через интерфейс управления **источниками данных**, **коллекции**, определенные в коде, обычно являются системными метатаблицами и не отображаются в списке управления **источниками данных**.

## Определение таблиц данных

Согласно согласованной структуре каталогов, файлы **коллекций** размещаются в директории `./src/server/collections`. Используйте `defineCollection()` для создания новых таблиц и `extendCollection()` для расширения существующих.

```ts
import { defineCollection } from '@nocobase/database';

export default defineCollection({
  name: 'articles',
  title: '示例文章',
  fields: [
    { type: 'string', name: 'title', interface: 'input', uiSchema: { title: '标题', required: true } },
    { type: 'text', name: 'content', interface: 'textarea', uiSchema: { title: '正文' } },
    {
      type: 'belongsTo',
      name: 'author',
      target: 'users',
      foreignKey: 'authorId',
      interface: 'recordPicker',
      uiSchema: { title: '作者' },
    },
  ],
});
```

В примере выше:

- `name`: Имя таблицы (в базе данных автоматически создается таблица с таким же именем).
- `title`: Отображаемое имя таблицы в интерфейсе.
- `fields`: Коллекция полей, каждое поле содержит атрибуты, такие как `type`, `name` и другие.

Когда вам нужно добавить поля или изменить конфигурации для **коллекций** других **плагинов**, вы можете использовать `extendCollection()`:

```ts
import { extendCollection } from '@nocobase/database';

export default extendCollection({
  name: 'articles',
  fields: [
    {
      type: 'boolean',
      name: 'isPublished',
      defaultValue: false,
    },
  ],
});
```

После активации **плагина** система автоматически добавит поле `isPublished` в существующую таблицу `articles`.

:::tip
Согласованная структура каталогов загружается до выполнения методов `load()` всех **плагинов**, что позволяет избежать проблем с зависимостями, вызванных неполной загрузкой таблиц данных.
:::

## Справочник типов полей

В `defineCollection` свойство `type` в массиве `fields` определяет тип столбца в базе данных. Ниже перечислены все встроенные типы полей:

### Текст

| type | Тип в БД | Описание | Специфические параметры |
|------|----------|----------|-------------------------|
| `string` | VARCHAR(255) | Короткий текст | `length?: number` (пользовательская длина), `trim?: boolean` |
| `text` | TEXT | Длинный текст | `length?: 'tiny' \| 'medium' \| 'long'` (только MySQL) |

### Числа

| type | Тип в БД | Описание | Специфические параметры |
|------|----------|----------|-------------------------|
| `integer` | INTEGER | Целое число | -- |
| `bigInt` | BIGINT | Большое целое | -- |
| `float` | FLOAT | Число с плавающей точкой | -- |
| `double` | DOUBLE | Двойная точность | -- |
| `decimal` | DECIMAL(p,s) | Число с фиксированной точкой | `precision: number`, `scale: number` |

### Логический

| type | Тип в БД | Описание |
|------|----------|----------|
| `boolean` | BOOLEAN | Логическое значение |

### Дата и время

| type | Тип в БД | Описание | Специфические параметры |
|------|----------|----------|-------------------------|
| `date` | DATE(3) | Дата и время (с миллисекундами) | `defaultToCurrentTime?`, `onUpdateToCurrentTime?` |
| `dateOnly` | DATEONLY | Только дата, без времени | -- |
| `time` | TIME | Только время | -- |
| `unixTimestamp` | BIGINT | Unix-метка времени | `accuracy?: 'second' \| 'millisecond'` |

:::tip
`date` -- наиболее часто используемый тип даты. Если требуется различать обработку часовых поясов, также доступны `datetimeTz` (с часовым поясом) и `datetimeNoTz` (без часового пояса).
:::

### Структурированные данные

| type | Тип в БД | Описание | Специфические параметры |
|------|----------|----------|-------------------------|
| `json` | JSON / JSONB | Данные JSON | `jsonb?: boolean` (использовать JSONB в PostgreSQL) |
| `jsonb` | JSONB / JSON | Приоритет JSONB | -- |
| `array` | ARRAY / JSON | Массив | В PostgreSQL доступен нативный тип ARRAY |

### Генерация ID

| type | Тип в БД | Описание | Специфические параметры |
|------|----------|----------|-------------------------|
| `uid` | VARCHAR(255) | Автоматически генерируемый короткий ID | `prefix?: string` |
| `uuid` | UUID | UUID v4 | `autoFill?: boolean` (по умолчанию true) |
| `nanoid` | VARCHAR(255) | NanoID | `size?: number` (по умолчанию 12), `customAlphabet?: string` |
| `snowflakeId` | BIGINT | Snowflake ID | `autoFill?: boolean` (по умолчанию true) |

### Специальные типы

| type | Тип в БД | Описание |
|------|----------|----------|
| `password` | VARCHAR(255) | Автоматическое хеширование с солью |
| `virtual` | Нет реального столбца | Виртуальное поле, столбец в БД не создается |
| `context` | Настраиваемый | Автозаполнение из контекста запроса (например, `currentUser.id`) |

### Типы связей

Поля связей не создают столбцов в базе данных, а устанавливают отношения между таблицами на уровне ORM:

| type | Описание | Ключевые параметры |
|------|----------|--------------------|
| `belongsTo` | Многие к одному | `target` (целевая таблица), `foreignKey` (внешний ключ) |
| `hasOne` | Один к одному | `target`, `foreignKey` |
| `hasMany` | Один ко многим | `target`, `foreignKey` |
| `belongsToMany` | Многие ко многим | `target`, `through` (промежуточная таблица), `foreignKey`, `otherKey` |

Пример использования полей связей:

```ts
export default defineCollection({
  name: 'articles',
  fields: [
    { type: 'string', name: 'title' },
    // Многие к одному: статья принадлежит одному автору
    {
      type: 'belongsTo',
      name: 'author',
      target: 'users',
      foreignKey: 'authorId',
    },
    // Один ко многим: у статьи много комментариев
    {
      type: 'hasMany',
      name: 'comments',
      target: 'comments',
      foreignKey: 'articleId',
    },
    // Многие ко многим: у статьи много тегов
    {
      type: 'belongsToMany',
      name: 'tags',
      target: 'tags',
      through: 'articlesTags',  // имя промежуточной таблицы
    },
  ],
});
```

### Общие параметры

Все столбцовые поля поддерживают следующие параметры:

| Параметр | Тип | Описание |
|----------|-----|----------|
| `name` | `string` | Имя поля (обязательно) |
| `defaultValue` | `any` | Значение по умолчанию |
| `allowNull` | `boolean` | Допускается ли null |
| `unique` | `boolean` | Уникальность |
| `primaryKey` | `boolean` | Первичный ключ |
| `autoIncrement` | `boolean` | Автоинкремент |
| `index` | `boolean` | Создавать ли индекс |
| `comment` | `string` | Комментарий к полю |

## Синхронизация структуры базы данных

При первой активации **плагина** система автоматически синхронизирует конфигурации **коллекций** со структурой базы данных. Если **плагин** уже установлен и работает, после добавления или изменения **коллекций** необходимо выполнить команду обновления для синхронизации структуры базы данных:

```bash
nb app upgrade
```

Если при обновлении плагина требуется выполнить миграцию существующих данных -- например, переименовать поля, разделить таблицы, заполнить значения по умолчанию и т.д. -- это следует делать через [скрипты миграции (Migration)](./migration.md), а не вручную изменять базу данных.

## Как сделать Collection видимой в списке таблиц данных UI

Таблицы, определенные через `defineCollection`, являются внутренними серверными таблицами и по умолчанию **не отображаются** в списке "Управления источниками данных", а также не появляются в списке выбора таблиц при добавлении блока.

**Рекомендуемый подход**: добавьте соответствующую таблицу данных в интерфейсе NocoBase через "[Управление источниками данных](../../data-sources/data-source-main/index.md)", настройте поля и типы интерфейсов, после чего таблица автоматически появится в списке выбора таблиц при добавлении блока.

![Таблица доступна при добавлении блока](https://static-docs.nocobase.com/20260409143839.png)

Если действительно необходимо зарегистрировать таблицу в коде плагина (например, в демонстрационных сценариях), можно вручную зарегистрировать её в клиентском плагине через `addCollection`. Обратите внимание: регистрация должна выполняться через паттерн `eventBus`, а не напрямую в `load()` -- `ensureLoaded()` после `load()` очищает и заново устанавливает все коллекции. Полный пример см. в разделе [Создание полнофункционального плагина для управления данными](../client/examples/fullstack-plugin.md).

## Автоматическая генерация ресурсов (Resource)

После определения **коллекции** NocoBase автоматически генерирует для неё соответствующий ресурс REST API. Готовые к использованию CRUD-интерфейсы (`list`, `get`, `create`, `update`, `destroy`) не требуют дополнительного кода. Если встроенных CRUD-операций недостаточно -- например, вам нужен интерфейс "массового импорта" или "статистической сводки" -- можно зарегистрировать пользовательский action через `resourceManager`. Подробнее см. [ResourceManager. Управление ресурсами](./resource-manager.md).
