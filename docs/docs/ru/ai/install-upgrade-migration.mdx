# Руководство по установке, обновлению и миграции: использование NocoBase CLI

:::danger
Возможности CLI по установке и обслуживанию приложений все еще находятся в разработке. Сейчас он больше подходит для локальной разработки, тестовых сред и сценариев интеграции с AI Agent. Не рекомендуется напрямую использовать его для установки, обновления и эксплуатации в производственной среде.
:::

Новый NocoBase CLI (`nb`) объединяет установку, подключение, запуск, остановку, просмотр логов, обновление и очистку в едином наборе команд. По сравнению со старой документацией, где процессы отдельно поддерживались для Docker, create-nocobase-app и исходного кода Git, CLI лучше подходит для локальной разработки, обслуживания тестовых сред, а также для интеграции и совместной работы с AI Agent.

Эта статья подходит для следующих сценариев:

- Вы устанавливаете или обновляете NocoBase по старой документации и хотите перейти на новый способ через CLI.
- У вас уже есть NocoBase, развернутый старым способом, и вы хотите, чтобы CLI управлял им или подключался к нему.
- Вам нужно подготовить для AI Agent среду NocoBase, к которой можно подключаться и которую можно обслуживать.

## Отличия старого и нового способа

В старом подходе документация разделялась по источнику установки, и для каждого источника нужно было отдельно выполнять команды загрузки, настройки, установки, запуска и обновления.

**Старая документация по установке**

- [Установка через create-nocobase-app](/get-started/installation/create-nocobase-app)
- [Установка из исходного кода Git](/get-started/installation/git)

**Старая документация по обновлению**

- [Обновление установки через Docker](/get-started/upgrading/docker)

- [Обновление установки из исходного кода Git](/get-started/upgrading/git)

В новом подходе в качестве точки входа используется `nb init`. Один CLI env хранит информацию об источнике приложения, рабочем каталоге, каталоге хранения, базе данных и API-подключении. Далее, независимо от того, используется Docker, npm или Git, для обслуживания по возможности применяется один и тот же набор команд `nb app`, `nb source`, `nb db`.

| Сценарий                             | Старый способ                                                    | Новый способ CLI                                  |
| ------------------------------------ | ---------------------------------------------------------------- | ------------------------------------------------- |
| Точка входа для установки            | `docker compose`、`create-nocobase-app`、`git clone`             | `nb init` или `nb init --ui`                      |
| Управление запуском приложения       | Переход в каталог, изменение compose, выполнение команд yarn/pm2 | `nb app start/stop/restart/logs`、`nb env remove` |
| Обновление                           | Отдельный набор команд для каждого способа установки             | `nb app upgrade -e <env>`                         |
| Режим разработки                     | `yarn dev`                                                       | `nb source dev -e <env>`                          |
| Состояние базы данных                | Ручная проверка контейнера или службы БД                         | `nb db ps -e <env>`                               |
| Просмотр состояния и деталей среды   | Ручная фиксация путей, портов и адресов доступа                  | `nb env list`、`nb env info <env>`                |
| Подключение существующего приложения | Старая документация по установке не охватывала это единообразно  | `nb init --ui` или `nb init --api-base-url`       |

## Установка CLI

```bash
npm install -g @nocobase/cli@beta
nb --version
```

После установки можно посмотреть справку:

```bash
nb --help
nb init --help
```

## Установка NocoBase новым способом

### Рекомендуется: использовать визуальный мастер

```bash
nb init --ui
```

Мастер предложит выбрать:

- создание нового приложения или подключение существующего приложения
- источник установки: Docker, npm, Git
- версию NocoBase
- использование встроенной базы данных или внешней базы данных
- учетную запись и пароль администратора

### Использование интерактивного мастера в терминале

```bash
nb init
```

### Использование неинтерактивной команды

По умолчанию используется установка через Docker и встроенная база данных PostgreSQL:

```bash
nb init --yes --env app1
```

Установка через Docker с указанием версии:

```bash
nb init --yes --env app1 --source docker --version beta
```

Установка через npm:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Установка из исходного кода Git:

```bash
nb init --yes --env app1 --source git --version beta
```

Использование исходного кода Git с указанием типа встроенной базы данных:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Подключение существующего приложения NocoBase:

```bash
# По умолчанию используется аутентификация OAuth
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Использование аутентификации по token
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` — это имя среды в CLI. В последующих командах запуска, обновления, просмотра логов и других можно указывать целевое приложение через `-e app1`.
:::

## Обновление новым способом

В старом подходе команды обновления для Docker, create-nocobase-app и исходного кода Git отличались; в новом CLI это унифицировано:

```bash
nb app upgrade -e app1
```

Если вы хотите выполнить процесс обновления только с уже загруженным исходным кодом или образом, не подтягивая новый код или образ:

```bash
nb app upgrade -e app1 --skip-code-update
```

Или использовать короткую форму:

```bash
nb app upgrade -e app1 -s
```

## Часто используемые команды обслуживания

| Операция                                             | Команда                              |
| ---------------------------------------------------- | ------------------------------------ |
| Запустить приложение                                 | `nb app start -e app1`               |
| Остановить приложение                                | `nb app stop -e app1`                |
| Перезапустить приложение                             | `nb app restart -e app1`             |
| Просмотреть логи                                     | `nb app logs -e app1`                |
| Просмотреть env и статус аутентификации API          | `nb env list`                        |
| Просмотреть детали env                               | `nb env info app1`                   |
| Просмотреть состояние встроенной базы данных         | `nb db ps -e app1`                   |
| Обновить приложение                                  | `nb app upgrade -e app1`             |
| Остановить приложение и встроенную базу данных       | `nb app stop -e app1 --with-db`      |
| Полностью удалить env и локально размещенные ресурсы | `nb env remove app1 --purge --force` |

Если вы просто хотите остановить приложение вместе со встроенной базой данных, управляемой CLI, обычно достаточно сразу использовать `nb app stop -e app1 --with-db`.

Если вы уверены, что этот env больше не нужен, используйте `nb env remove app1 --purge --force`. Это удалит управляемые ресурсы запуска, данные storage, а также, при необходимости, локальные файлы app, которые были загружены и управляются CLI.

## Повседневные команды разработки

`nb source` используется для управления локальным проектом с исходным кодом и в основном подходит для env с источником npm и Git.

| Операция                                 | Команда                                             |
| ---------------------------------------- | --------------------------------------------------- |
| Запустить приложение в режиме разработки | `nb source dev -e app1`                             |
| Собрать исходный код                     | `nb source build --cwd /your/workspace/app1/source` |
| Запустить тесты                          | `nb source test --cwd /your/workspace/app1/source`  |
| Скачать или обновить исходный код/образ  | `nb source download --source git --version beta`    |

:::tip
Docker env обычно управляется через `nb app start`, `nb app logs` и `nb app upgrade`, а `nb source dev` не используется.
:::

## NB_CLI_ROOT и структура каталогов

CLI сохраняет глобальную конфигурацию и локальные файлы приложения в каталоге, соответствующем `NB_CLI_ROOT`.

По умолчанию `NB_CLI_ROOT` — это домашний каталог текущего пользователя, то есть путь, возвращаемый Node.js `os.homedir()`. Например, в macOS это обычно `/Users/<user>`. Поэтому, если не выполнять дополнительную настройку, файлы, созданные с помощью `nb init --yes --env app1`, обычно будут находиться здесь:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Вы также можете явно указать `NB_CLI_ROOT`:

```bash
export NB_CLI_ROOT=/your/workspace
```

После этого файл конфигурации и каталог приложения по умолчанию будут размещаться в этом каталоге:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Где:

- `.nocobase/config.json` хранит env, текущую среду, API-адрес, путь к исходному коду, путь storage, конфигурацию базы данных и другую информацию.
- `<envName>/source` — каталог исходного кода или приложения, управляемый CLI.
- `<envName>/storage` — каталог хранения приложения, управляемый CLI.

:::warning
После инициализации одного и того же приложения используйте один и тот же `NB_CLI_ROOT`. Если позже изменить `NB_CLI_ROOT`, CLI будет искать `.nocobase/config.json` в новом каталоге, а относительные пути `appRootPath` и `storagePath` тоже будут интерпретироваться относительно нового каталога, что может привести к невозможности найти приложение или к чтению/записи в неверный каталог.
:::

В настоящее время CLI использует только глобальный scope конфигурации. Если нужно изменить место хранения конфигурации и локальных файлов приложения, настройте `NB_CLI_ROOT`.

Для просмотра и поиска приложения в первую очередь используйте `nb env list` и `nb env info`:

```bash
# Просмотреть настроенные env и статус аутентификации API
nb env list

# Просмотреть подробную информацию об определенном env
nb env info app1
```

`nb env list` предназначен для быстрого обзора, где `App Status` — это статус аутентификации, полученный после доступа к API приложения с использованием сохраненных учетных данных Token/OAuth; статус базы данных больше не отображается. Чтобы просмотреть состояние работы встроенной базы данных, используйте:

```bash
nb db ps -e app1
```

Только если нужно переключить env по умолчанию, используйте:

```bash
nb env use app1
```

## Миграция со старого способа на CLI

Существует две стратегии миграции: **подключить существующее приложение** или **включить локальный каталог приложения в управление CLI**. Если ваше старое приложение уже стабильно работает, рекомендуется сначала использовать «подключить существующее приложение» — это наименее рискованный вариант.

### Способ 1: подключить существующее приложение

Подходит для NocoBase, уже работающего через Docker, create-nocobase-app, исходный код Git или серверное развертывание.

```bash
# По умолчанию используется аутентификация OAuth
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Использование аутентификации по token
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Этот способ сохраняет только API-подключение; CLI не берет на себя запуск, остановку, обновление, каталог исходного кода или каталог storage старого приложения. Он подходит для того, чтобы AI Agent или команды CLI API могли подключаться к существующему приложению.

Если нужно обновить аутентификацию:

```bash
nb env auth app1
```

### Способ 2: создать новый CLI env и затем перенести файлы

Подходит для сценариев, где в дальнейшем вы хотите управлять локальным приложением через `nb app` и `nb source`.

Перед миграцией сначала выполните:

1. Остановите старое приложение.
2. Сделайте резервную копию старой базы данных.
3. Сделайте резервную копию старого каталога `storage`.
4. Зафиксируйте ключевые переменные окружения старого приложения, например `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Миграция старой установки Docker

При старом способе с Docker данные приложения обычно хранятся в `storage` внутри каталога проекта:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Рекомендуется сначала определить новый `NB_CLI_ROOT`, затем создать новый Docker env через CLI. В примере ниже предполагается, что `NB_CLI_ROOT` — `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

Новый env по умолчанию будет использовать следующие каталоги:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Затем скопируйте старый `storage` в `/your/workspace/app2/storage` и убедитесь, что подключение к базе данных и параметры вроде `APP_KEY` совпадают со старой средой.

После завершения копирования и подтверждения конфигурации базы данных снова запустите приложение:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Миграция старой установки create-nocobase-app или исходного кода Git

При старом способе npm/Git исходный код, `.env` и `storage` обычно находятся в одном каталоге проекта:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Рекомендуется сначала определить новый `NB_CLI_ROOT`, затем создать новый env npm или Git. В примере ниже предполагается, что `NB_CLI_ROOT` — `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Затем при необходимости перенесите:

- исходный код пользовательских плагинов из старого проекта в соответствующий каталог `/your/workspace/app3/source`;
- старый `storage` в `/your/workspace/app3/storage`;
- ключевые настройки из старого `.env` в конфигурацию CLI env или параметры инициализации;
- при использовании исходной базы данных отключите встроенную базу данных при инициализации и укажите информацию о подключении к старой базе данных.

Пример инициализации с внешней базой данных:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

После завершения миграции запустите приложение и просмотрите логи:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
Не перезаписывайте напрямую `source`, `storage` и не выполняйте обновление с подключением к производственной базе данных без резервной копии. Перед официальной миграцией рекомендуется полностью отработать процесс один раз в тестовой среде.
:::

## Краткая памятка по миграции команд

| Старая команда                                        | Новая команда CLI                                                                       |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                            | `nb app start -e app1`                                                                  |
| `docker compose stop app`                             | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                          | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`        | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`    | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`      | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                            | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                               | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`            | `nb app upgrade -e app1 -s`                                                             |
| Ручная запись адресов нескольких приложений           | `nb env list`、`nb env info app1`                                                       |
| Ручное подключение существующего приложения           | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
