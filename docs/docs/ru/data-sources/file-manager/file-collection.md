# Таблица файлов

<PluginInfo name="file-manager"></PluginInfo>

## Введение

Таблица файлов предназначена для хранения метаданных файлов, таких как имя файла, расширение, размер, MIME-тип, путь, URL, адрес предварительного просмотра, место хранения и пользовательские meta. Сам файл сохраняется движком файлового хранилища, а таблица файлов хранит метаданные файла.

Таблицу файлов можно создать только через страницу основной базы данных. Внешние базы данных, источники данных REST API и внешние источники данных NocoBase не поддерживают создание таблиц файлов.

## Сценарии использования

Таблица файлов подходит для следующих бизнес-сценариев:

- Вложения к договорам, счета-фактуры, документы для подтверждения расходов
- Изображения продуктов, удостоверения сотрудников, проектная документация
- Файлы, загружаемые в бизнес-записи, файлы для предварительного просмотра и скачивания
- Библиотека вложений, требующая отдельного управления метаданными файлов

## Процесс использования

Таблица файлов обычно не используется непосредственно как основная бизнес-таблица. Типичный процесс выглядит так:

1. Создайте таблицу файлов для хранения заголовка файла, имени файла, размера, типа, URL, места хранения и других метаданных.
2. Создайте в бизнес-таблице поле связи с таблицей файлов. Например, свяжите таблицу «Договоры» с таблицей файлов «Вложения к договорам».
3. Добавьте поле связи в блок формы бизнес-таблицы, чтобы пользователи могли загружать файлы при создании или редактировании бизнес-записей.
4. После завершения загрузки NocoBase записывает метаданные файла в таблицу файлов и связывает запись о файле с текущей бизнес-записью через поле связи.
5. Отображайте поле вложений в блоке подробностей, табличном блоке или блоке списка бизнес-таблицы, чтобы пользователи могли просматривать, открывать предварительный просмотр или скачивать файлы.

## Настройка создания

В основной базе данных нажмите «Create collection» и выберите «File collection», чтобы создать таблицу файлов.

![20240324090414](https://static-docs.nocobase.com/20240324090414.png)

Настройки создания таблицы файлов в основном совпадают с настройками обычной таблицы. В таблице файлов заранее создаётся набор полей метаданных, предназначенных для хранения заголовка, пути, URL, места хранения и расширенной информации о загруженном файле.

| Настройка | Описание |
| --- | --- |
| Collection display name | Название таблицы, отображаемое в интерфейсе, например «Вложения к договорам», «Счета-фактуры», «Изображения продуктов». |
| Collection name | Идентификационное имя таблицы, используемое для внутренних ссылок в API, полях связи, разрешениях, рабочих процессах и т. д. |
| Categories | Категория таблицы. Категория влияет только на организацию интерфейса управления таблицами и не изменяет структуру таблицы. |
| Description | Описание таблицы. Здесь можно указать, какие файлы хранит эта таблица, кто их загружает и с какими бизнес-таблицами они связаны. |
| Preset fields | Предустановленные поля. При создании таблицы файлов рекомендуется сохранить системные поля и встроенные поля таблицы файлов. |

### Встроенные поля

После создания таблица файлов обычно содержит следующие встроенные поля. Сам файл хранится в файловом хранилище, а таблица файлов содержит эти метаданные.

| Поле | Имя поля | Описание |
| --- | --- | --- |
| ID | `id` | Поле первичного ключа по умолчанию, используемое для уникальной идентификации записи о файле. |
| Title | `title` | Заголовок файла, обычно отображаемый в интерфейсе. |
| File name | `filename` | Имя файла. |
| Extension name | `extname` | Расширение файла. |
| Size | `size` | Размер файла. |
| MIME type | `mimetype` | MIME-тип файла. |
| Path | `path` | Путь к файлу в хранилище. |
| URL | `url` | Адрес доступа к файлу. |
| Preview | `preview` | Адрес предварительного просмотра файла. |
| Storage | `storage` / `storageId` | Хранилище, которому принадлежит файл. `storage` — поле связи, а `storageId` — соответствующий внешний ключ. |
| Meta | `meta` | Расширенные метаданные файла. |
| Время создания | `createdAt` | Автоматически записывает время создания записи о файле. |
| Создатель | `createdBy` | Пользователь, загрузивший файл или создавший запись о файле; записывается автоматически. |
| Время обновления | `updatedAt` | Автоматически записывает время последнего обновления записи о файле. |
| Обновил | `updatedBy` | Пользователь, последним обновивший запись о файле; записывается автоматически. |
| Пространство | `space` | Доступно после включения [плагина нескольких пространств](../../multi-app/multi-space/index.md) и используется для изоляции данных по пространствам. При отключённой функции нескольких пространств это поле не отображается. |

![20240324090527](https://static-docs.nocobase.com/20240324090527.png)

### Поле первичного ключа

Как и обычная таблица, таблица файлов должна иметь поле первичного ключа. Поля вложений и поля связи используют первичный ключ для связывания с метаданными файлов.

Если в таблице файлов нет первичного ключа, при редактировании таблицы данных необходимо задать «Record unique key», иначе записи вложений могут быть неправильно связаны, отображены в предварительном просмотре или отредактированы.

## Редактирование настроек

В списке таблиц данных нажмите «Edit» справа от таблицы файлов, чтобы изменить отображаемое имя таблицы, категорию, описание, простой режим разбиения на страницы, «Record unique key» и другие настройки.

Поля метаданных файлов обычно заполняются автоматически в процессе загрузки. Не рекомендуется изменять назначение таких полей, как `url`, `path`, `storageId`. Если необходимо расширить бизнес-информацию о файлах, можно добавить новые поля, например «Тип файла», «Этап», «В архиве».

## Удаление таблицы данных

В списке таблиц данных нажмите «Delete» справа от таблицы файлов, чтобы удалить таблицу файлов.

Удаление таблицы файлов удаляет записи метаданных файлов и связанные метаданные Collection. Перед удалением убедитесь, что поля вложений, поля связи, блоки страниц, разрешения, рабочие процессы и API в бизнес-таблицах больше не зависят от неё.

:::danger Предупреждение

Таблица файлов хранит метаданные файлов. Удаление записей таблицы файлов может привести к тому, что ссылки на вложения в бизнес-записях перестанут работать; синхронное удаление самих файлов зависит от настроек файлового хранилища и бизнес-конфигурации. Перед выполнением операции убедитесь, что файлы больше не используются в бизнес-процессах.

:::
