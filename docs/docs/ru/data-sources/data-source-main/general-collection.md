# Обычная таблица

## Введение

Обычные таблицы — наиболее распространённый тип таблиц данных. Они подходят для хранения стандартных бизнес-данных: клиентов, заказов, договоров, заявок, отчётов о расходах, проектов, задач и т. д. Если к большинству бизнес-объектов не предъявляются особые требования к структуре, обычно достаточно использовать обычную таблицу.

Обычные таблицы могут поступать из следующих источников:

- Новые таблицы, созданные в основной базе данных
- Существующие реальные таблицы, синхронизированные из основной базы данных
- Существующие реальные таблицы, подключённые из внешней базы данных
- Ресурсы, сопоставленные через REST API
- Таблицы данных во внешнем приложении NocoBase

В NocoBase все эти данные используются как обычные таблицы. Разница заключается в следующем: обычные таблицы в основной базе данных могут создаваться и обслуживаться NocoBase вместе со структурой реальных таблиц; обычные таблицы во внешних источниках данных обычно только считывают существующую структуру, а структура реальных таблиц по-прежнему поддерживается внешней системой.

## Подходящие сценарии

Обычные таблицы подходят для следующих бизнес-сценариев:

- Данные CRM: клиенты, контакты, потенциальные сделки, договоры и т. д.
- Данные о транзакциях: заказы, накладные на отгрузку, возвраты, счета и т. д.
- Данные для совместной работы: заявки, задачи, проекты, требования и т. д.
- Данные процессов: отчёты о расходах, заявки на закупку, заявки на оплату и т. д.
- Основные данные: оборудование, активы, товары, магазины и т. д.



## Создание и настройка

В основной базе данных нажмите «Create collection» и выберите «General collection», чтобы создать обычную таблицу.

![20240324085739](https://static-docs.nocobase.com/20240324085739.png)

| Настройка | Описание |
| --- | --- |
| Collection display name | Название таблицы данных, отображаемое в интерфейсе, например «Клиенты», «Заказы», «Вложения к договорам». Рекомендуется использовать название, понятное сотрудникам бизнеса. |
| Collection name | Идентификатор таблицы данных, используемый для внутренних ссылок в API, полях связей, правах доступа, рабочих процессах и т. д. Он создаётся автоматически, но его можно изменить вручную; допускаются только буквы, цифры и символы подчёркивания, при этом название должно начинаться с буквы. |
| Categories | Категория таблицы данных. Категория влияет только на организацию интерфейса управления таблицами данных и не изменяет структуру таблицы. При большом количестве таблиц рекомендуется распределять их по бизнес-модулям, например «Управление клиентами», «Управление проектами», «Финансы». |
| Description | Описание таблицы данных. Здесь можно указать, какие данные хранятся в таблице, кто отвечает за её обслуживание и с какими бизнес-процессами она связана, чтобы упростить дальнейшее сопровождение. |
| Use simple pagination mode | Режим простой пагинации. После включения при разбиении данных в блоке таблицы пропускается подсчёт общего количества записей. Подходит для таблиц с большим объёмом данных и позволяет снизить нагрузку на запросы. |
| Preset fields | Предустановленные поля. При создании таблицы можно выбрать, добавлять ли автоматически распространённые поля, такие как ID, время создания, создатель, время обновления и обновивший пользователь. Для обычных бизнес-таблиц рекомендуется сохранить эти поля. |

### Встроенные поля

При создании обычной таблицы можно автоматически добавить распространённые системные поля с помощью `Preset fields`.

| Поле | Имя поля | Описание |
| --- | --- | --- |
| ID | `id` | Поле первичного ключа по умолчанию, используемое для уникальной идентификации записи. Тип первичного ключа по умолчанию — `Snowflake ID (53-bit)`. |
| Время создания | `createdAt` | Автоматически записывает время создания записи. Обычно используется для сортировки, фильтрации, аудита и условий рабочих процессов. |
| Создатель | `createdBy` | Автоматически записывает пользователя, создавшего запись. Обычно используется для просмотра только созданных мной данных, управления правами доступа и отслеживания ответственности. |
| Время обновления | `updatedAt` | Автоматически записывает время последнего обновления записи. Обычно используется для определения того, изменялись ли данные. |
| Обновивший пользователь | `updatedBy` | Автоматически записывает пользователя, последним обновившего запись. Обычно используется в сценариях аудита и совместной работы. |
| [Пространство](../../multi-app/multi-space/index.md) | `space` | Доступно после включения [плагина нескольких пространств](../../multi-app/multi-space/index.md) и используется для изоляции данных по пространствам. Если плагин нескольких пространств не включён, это поле не отображается среди предустановленных полей обычной таблицы. |

### Поле первичного ключа

**Primary key** обозначает поле первичного ключа. Оно используется для уникальной идентификации записи на уровне базы данных. При создании таблицы рекомендуется сохранить предустановленное поле ID; тип первичного ключа по умолчанию — `Snowflake ID (53-bit)`.

![20251209210153](https://static-docs.nocobase.com/20251209210153.png)

Наведите курсор на Interface поля ID, чтобы выбрать другой тип первичного ключа.

![20251209210517](https://static-docs.nocobase.com/20251209210517.png)

Доступны следующие типы первичного ключа:

- [Текст](../data-modeling/collection-fields/basic/input.md)
- [Целое число](../data-modeling/collection-fields/basic/integer.md)
- [Snowflake ID (53-bit)](../data-modeling/collection-fields/advanced/snowflake-id.md)
- [UUID](../data-modeling/collection-fields/advanced/uuid.md)
- [Nano ID](../data-modeling/collection-fields/advanced/nano-id.md)

:::warning Внимание

Для таблицы данных без первичного ключа необходимо при редактировании таблицы данных настроить «Record unique key», иначе невозможно будет создать блок на странице, а также корректно просматривать или редактировать записи.

:::

## Редактирование настроек

В списке таблиц данных нажмите «Edit» справа от обычной таблицы, чтобы изменить её основные настройки. Редактирование таблицы данных предназначено главным образом для изменения метаданных таблицы и некоторых параметров работы, но не для массового изменения структуры полей.

Чтобы добавить поле, изменить его тип, настроить тип интерфейса поля или удалить поле, перейдите в «Configure fields».

![edit_collection](https://static-docs.nocobase.com/edit_collection.png)

![edit_collection_configure](https://static-docs.nocobase.com/edit_collection_configure.png)

| Настройка | Можно редактировать | Описание |
| --- | --- | --- |
| Collection display name | Да | Название таблицы данных, отображаемое в интерфейсе, например «Клиенты», «Заказы», «Вложения к договорам». После изменения затрагивается только отображение в интерфейсе, идентификатор таблицы данных не изменяется. |
| Collection name | Нет | Идентификатор таблицы данных, используемый для внутренних ссылок в API, полях связей, правах доступа, рабочих процессах и т. д. После создания его нельзя изменить в форме редактирования таблицы. |
| Inherits | При наличии условий | Выбор родительской таблицы, от которой выполняется наследование. Доступно только для основной базы данных PostgreSQL, если эта настройка отображается в интерфейсе. Перед изменением наследования существующей таблицы необходимо убедиться, что структура полей, блоки страниц, права доступа и рабочие процессы не зависят от исходной структуры. |
| Categories | Да | Категория таблицы данных. Категория влияет только на организацию интерфейса управления таблицами данных и не изменяет структуру таблицы. |
| Description | Да | Описание таблицы данных. Подходит для указания назначения таблицы, ответственного за обслуживание, источника данных и связанных бизнес-процессов. |
| Use simple pagination mode | Да | Режим простой пагинации. После включения при разбиении данных в блоке таблицы пропускается подсчёт общего количества записей. Подходит для таблиц с большим объёмом данных. |
| Record unique key | Да | Уникальный идентификатор записи. Используется для определения записи в блоке; обычно выбирается первичный ключ или уникальное поле. Для таблиц данных без первичного ключа настройка обязательна, иначе невозможно корректно создать блок, просматривать или редактировать записи. |

:::warning Внимание

Редактирование таблицы данных не изменяет автоматически существующие поля. `Preset fields` действует только при создании таблицы; если после создания необходимо добавить поля времени создания, создателя, времени обновления, обновившего пользователя и т. д., их нужно добавить отдельно в «Configure fields».

:::

## Удаление таблицы данных

В списке таблиц данных нажмите «Delete» справа от обычной таблицы, чтобы удалить таблицу данных. Для обычных таблиц в основной базе данных также поддерживается массовое удаление после выбора нескольких таблиц.

![delete_collection](https://static-docs.nocobase.com/delete_collection.png)

При удалении появится запрос на повторное подтверждение. После подтверждения NocoBase удалит метаданные Collection этой обычной таблицы, а также реальную таблицу данных и содержащиеся в ней данные из основной базы данных.

![delete_collection_second_confirmation](https://static-docs.nocobase.com/delete_collection_second_confirmation.png)

В окне подтверждения удаления есть необязательный параметр: автоматически удалить объекты, зависящие от этой таблицы. После включения NocoBase попытается также удалить объекты базы данных, зависящие от этой таблицы, например представления базы данных, созданные на её основе, а также другие объекты, которые продолжают зависеть от этих объектов.

:::danger Предупреждение

Удаление обычной таблицы — операция высокого риска. После удаления могут перестать работать структура таблицы, данные таблицы, метаданные полей, а также блоки страниц, поля связей, права доступа, рабочие процессы и вызовы API, зависящие от этой таблицы. Перед включением автоматического удаления зависимых объектов убедитесь, что их также можно удалить.

:::
