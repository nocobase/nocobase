# Таблица данных

## Введение

В NocoBase **Collection (таблица данных)** — это модель данных, предназначенная для описания определённого типа бизнес-данных. Это не просто имя таблицы базы данных, а единое описание определённого типа данных в NocoBase.

Обычно Collection определяет три вещи:

| Определение | Описание |
| --- | --- |
| Где сохраняются данные | Данные могут поступать из таблицы основной базы данных, внешней таблицы базы данных, результатов SQL-запроса, представления базы данных, ресурса REST API или внешнего приложения NocoBase. |
| Какие есть поля | Поля описывают, какие сведения содержит каждая запись, например имя клиента, номер мобильного телефона, сумму заказа, время создания и ответственного. |
| Как используется NocoBase | Блоки страниц, права доступа, рабочие процессы, API и поля связей работают на основе Collection. |

Collection можно понимать как «структуру данных бизнес-объекта». Например, «клиент», «заказ», «договор» и «задача» могут быть отдельными Collection.

После создания или подключения таблицы данных обычно необходимо выполнить ещё три действия:

- настроить поля, чтобы таблица данных могла хранить необходимую бизнес-информацию
- на странице [добавить блок](../../interface-builder/blocks/index.md#添加区块), чтобы пользователи могли просматривать, вводить и обрабатывать данные
- настроить права доступа, рабочие процессы и API, чтобы данные можно было использовать и передавать в соответствии с бизнес-правилами

## Типы структуры таблиц

- **Обычная таблица** — подходит для хранения обычных бизнес-данных, таких как клиенты, заказы, договоры, заявки, авансовые отчёты, проекты и задачи
- **Древовидная таблица** — подходит для хранения иерархических данных, таких как организационная структура, категории товаров, уровни регионов, каталоги подразделений и каталоги базы знаний
- **Календарная таблица** — подходит для хранения данных с временными интервалами, таких как бронирование переговорных комнат, расписание проектов, расписание занятий, графики дежурств и мероприятия
- **Таблица комментариев** — подходит для хранения обсуждений, возникающих вокруг бизнес-записей, таких как комментарии к задачам и статьям, замечания по согласованию и отзывы клиентов; создайте в бизнес-таблице (обычной, древовидной или календарной таблице) [поле связи](./collection-fields/associations/index.md) с таблицей комментариев, а затем используйте всплывающую страницу бизнес-таблицы для создания [блока комментариев](../../plugins/@nocobase/plugin-comments/index.md) и комментирования бизнес-данных
- **Таблица файлов** — подходит для хранения метаданных файлов, таких как вложения к договорам, файлы счетов, изображения товаров и документы сотрудников; сами файлы сохраняются в подсистеме хранения файлов; создайте в бизнес-таблице (обычной, древовидной или календарной таблице) [поле связи](./collection-fields/associations/index.md) с таблицей файлов, а затем настройте загрузку файлов через поле связи в блоке, созданном на основе бизнес-таблицы, после чего метаданные файлов будут автоматически сохранены в таблице файлов
- **Представление базы данных** — уже существующее в базе данных представление, например представление финансового отчёта, отфильтрованное представление клиентов или агрегированное представление после синхронизации между системами
- **SQL-таблица** — подходит для использования результатов SQL-запросов в качестве таблицы данных, например сводки продаж, предупреждений о запасах, статистических отчётов по нескольким таблицам и операционных панелей
- **Наследуемая таблица** — подходит для случаев, когда несколько типов бизнес-объектов используют общий набор полей, но каждый тип также имеет собственные специализированные поля, например дочерние таблицы компьютерного оборудования, транспортных средств и офисной мебели, созданные на основе родительской таблицы активов
