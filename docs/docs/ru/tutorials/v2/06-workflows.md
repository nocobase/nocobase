# Глава 6. Рабочий процесс — пусть система работает за нас

В прошлой главе мы добавили права, чтобы разные роли видели разное. Но пока всё выполняется вручную: новая заявка пришла — надо самому посмотреть, статус сменился — никто не знает.

В этой главе с помощью [рабочего процесса](/workflow) NocoBase **система начнёт работать сама** — настроим [условные ветвления](/workflow/nodes/condition) и узлы [автоматического обновления](/workflow/nodes/update) для автоматического перевода статуса заявки и записи времени её создания.

## 6.1 Что такое [рабочий процесс](/workflow)

Рабочий процесс — это набор автоматизированных правил «если…, то…».

Аналогия: будильник на телефоне срабатывает каждый день в 8 утра. Будильник — простейший рабочий процесс: **выполнено условие (наступило 8 утра) — выполняется действие (звонит)**.

Рабочий процесс в NocoBase устроен так же:

![06-workflows-2026-03-20-13-25-38](https://static-docs.nocobase.com/06-workflows-2026-03-20-13-25-38.jpg)

- **[Триггер](/workflow/triggers/collection)**: точка входа. Например, «кто-то создал новую заявку» или «обновлена запись».
- **Условное ветвление**: опциональный шаг фильтрации. Например, «продолжаем только если обработчик не пустой».
- **Действия**: собственно работа. Например, «отправить уведомление» или «обновить поле».

Действия складываются в цепочку из узлов; типичные категории узлов:

- **Управление потоком**: условные ветвления, параллельные ветви, циклы, задержки
- **Работа с данными**: создание, обновление, чтение, удаление
- **Уведомления и внешние сервисы**: уведомления, HTTP-запросы, вычисления

В этом учебнике используются только самые распространённые из них — этого хватает на большинство сценариев.

### Типы триггеров

NocoBase предлагает несколько типов триггеров (выбираются при создании рабочего процесса):

| Триггер | Описание | Типичный сценарий |
|-------|------|---------|
| [**Событие коллекции**](/workflow/triggers/collection) | При создании, обновлении или удалении данных | Уведомление о новой заявке, журналирование смены статуса |
| [**Запланированная задача**](/workflow/triggers/schedule) | По cron или фиксированному времени | Ежедневный отчёт, регулярная очистка |
| [**Событие после действия**](../../workflow/triggers/post-action.md) | После действия пользователя в интерфейсе | Уведомление после отправки формы, журнал |
| **Согласование** | Запуск процесса согласования с многоступенчатой структурой | Согласование отпуска, закупки |
| **Событие пользовательского действия** | Привязка к пользовательской кнопке, срабатывает по клику | Архивирование, массовое действие |
| **Событие до действия** | Перехват действия, синхронное выполнение, потом проход | Валидация перед отправкой, автозаполнение |
| **ИИ-сотрудник** | Рабочий процесс доступен ИИ-сотруднику как инструмент | Автоматические бизнес-операции ИИ |

В учебнике мы используем **Событие коллекции** и **Событие пользовательского действия** — остальные настраиваются аналогично.

Рабочий процесс — встроенный плагин NocoBase, ничего дополнительно ставить не нужно.

## 6.2 Сценарий 1: новая заявка автоматически уведомляет обработчика

**Требование**: когда создаётся новая заявка с обозначенным обработчиком, система автоматически отправляет ему внутрисистемное сообщение «прилетела работа».

### Шаг 1. Создаём рабочий процесс

В правом верхнем углу откройте меню настройки плагинов и зайдите в **Управление рабочими процессами**.

![06-workflows-2026-03-14-23-50-45](https://static-docs.nocobase.com/06-workflows-2026-03-14-23-50-45.png)


Нажмите **«Создать»**, в диалоге:

- **Название**: «Уведомление обработчика о новой заявке»
- **Тип триггера**: **Событие коллекции**

![06-workflows-2026-03-14-23-53-37](https://static-docs.nocobase.com/06-workflows-2026-03-14-23-53-37.png)


Подтвердите, в списке нажмите ссылку **«Настроить»** и зайдёте в редактор процесса.

### Шаг 2. Настраиваем триггер

Кликните на карточку триггера, откроется панель настройки:

- **[Коллекция](/data-sources/data-modeling/collection)**: Основной источник данных / «Заявки»
- **Когда срабатывает**: «После создания или обновления данных»
- **Изменённые [поля](/data-sources/data-modeling/collection-fields)**: отметьте «Обработчик» — триггер будет срабатывать только при изменении этого поля, лишних уведомлений не будет (при создании все поля считаются изменёнными, поэтому новое создание тоже сработает)
- **Условие срабатывания**: режим «удовлетворяет **любое** из условий», добавьте два:
  - `assignee_id` не пуст
  - `Обработчик / ID` не пуст

  > Зачем два условия? Потому что в моменте триггера в форме может быть только внешний ключ (assignee_id) без подтянутого связанного объекта, или наоборот — связанный объект есть, а внешний ключ пуст. Связь по «ИЛИ» гарантирует срабатывание, если назначен обработчик.

- **Подгружать связанные данные**: отметьте «Обработчик» — узел уведомления будет использовать данные обработчика, их нужно подтянуть заранее в триггере.

![06-workflows-2026-03-14-23-58-31](https://static-docs.nocobase.com/06-workflows-2026-03-14-23-58-31.png)

Сохраните. Триггер сам уже выполнил фильтрацию по условию — отдельный узел условного ветвления не нужен.

### Шаг 3. Добавляем узел уведомления

Нажмите **+** под триггером, выберите узел **Уведомление**.

![06-workflows-2026-03-15-00-00-55](https://static-docs.nocobase.com/06-workflows-2026-03-15-00-00-55.png)

Откройте настройки узла. Первое поле — выбор **канала уведомления**, но ни одного канала ещё нет — список пустой. Создадим.

![06-workflows-2026-03-15-00-10-12](https://static-docs.nocobase.com/06-workflows-2026-03-15-00-10-12.png)


### Шаг 4. Создаём канал уведомлений

NocoBase поддерживает разные каналы:

| Тип канала | Описание |
|---------|------|
| **Встроенные сообщения** | Уведомление в браузере, в реальном времени попадает в центр уведомлений пользователя |
| **Электронная почта** | Через SMTP, нужен настроенный почтовый сервер |

В учебнике используем самый простой — **встроенные сообщения**:

1. Откройте настройки плагинов в правом верхнем углу, перейдите в **Управление уведомлениями**.
2. Нажмите **«Новый канал»**.

![06-workflows-2026-03-15-00-13-07](https://static-docs.nocobase.com/06-workflows-2026-03-15-00-13-07.png)

3. Тип канала — **встроенные сообщения**, имя — «Системные сообщения».
4. Сохраните.

![06-workflows-2026-03-15-00-17-55](https://static-docs.nocobase.com/06-workflows-2026-03-15-00-17-55.png)

### Шаг 5. Настраиваем узел уведомления

Возвращаемся в редактор рабочего процесса и открываем настройки узла **Уведомление**.

Параметры:

- **Канал уведомления**: выберите «Системные сообщения»
- **Получатель**: «Запрос пользователей → id = Переменная триггера / Триггерные данные / Обработчик / ID»
- **Заголовок**: например, «У Вас новая заявка». Поддерживаются переменные, можно добавить заголовок заявки: `Новая заявка: {{Триггерные данные / Заголовок}}`
- **Содержимое**: тело уведомления, тоже с переменными — приоритет, описание и т. д.

![06-workflows-2026-03-15-20-10-11](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-10-11.png)

(Сейчас пойдём за адресом заявки — перед выходом не забудьте сохранить!)

- **Десктопная страница деталей**: путь URL детальной страницы заявки. Как получить: откройте в интерфейсе любую заявку в виде окна деталей, скопируйте адрес из строки браузера, формат вроде `/admin/camcwbox2uc/view/d8f8e122d37/filterbytk/353072988225540`. Вставьте путь в поле; число после `filterbytk/` — это ID заявки. Замените эту часть переменной ID триггерных данных (выбор переменной → Триггерные данные → ID). После настройки клик по уведомлению в списке отправит пользователя сразу на страницу деталей и автоматически отметит уведомление как прочитанное.

![06-workflows-2026-03-15-00-28-32](https://static-docs.nocobase.com/06-workflows-2026-03-15-00-28-32.png)

![06-workflows-2026-03-15-20-15-19](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-15-19.png)

- **Продолжать при ошибке отправки**: опционально; если включено, рабочий процесс не прервётся при ошибке отправки уведомления.

> После отправки обработчик увидит сообщение в **центре уведомлений** в правом верхнем углу, у непрочитанных будет красная точка. Клик по уведомлению — переход на страницу деталей.

### Шаг 6. Тест и активация

> Полная цепочка сценария 1 включает только два узла: триггер (с фильтрами) → уведомление. Просто и эффективно.

Не торопитесь активировать — рабочий процесс поддерживает **ручной запуск** для проверки на конкретных данных:

1. Нажмите **«Выполнить»** в правом верхнем углу (а не переключатель активации).
2. Выберите существующую заявку как триггерные данные.
  > Если в выборе заявки отображается ID, в **Источники данных → Коллекции → Заявки** задайте «Заголовок» как **поле заголовка**.
![06-workflows-2026-03-15-19-47-57](https://static-docs.nocobase.com/06-workflows-2026-03-15-19-47-57.png)

3. Нажмите **Выполнить** — рабочий процесс выполнится и автоматически переключится на скопированную новую версию.
![06-workflows-2026-03-15-19-57-19](https://static-docs.nocobase.com/06-workflows-2026-03-15-19-57-19.png)

4. Нажмите три точки в правом верхнем углу и выберите **История запусков** — увидите запись только что выполненного запуска. Кликнув её, можно посмотреть детали: триггер, выполнение каждого узла, параметры.
![06-workflows-2026-03-15-19-58-34](https://static-docs.nocobase.com/06-workflows-2026-03-15-19-58-34.png)

![06-workflows-2026-03-15-20-01-02](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-01-02.png)


5. Заявка была назначена Alice, переключимся под её аккаунт — уведомление пришло!

![06-workflows-2026-03-15-20-16-22](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-16-22.png)

Клик переходит на страницу заявки и помечает уведомление прочитанным.

![06-workflows-2026-03-15-20-16-54](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-16-54.png)


Убедившись, что всё работает, переключите тумблер активации в правом верхнем углу.

![06-workflows-2026-03-15-20-18-16](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-18-16.png)

> **Внимание**: после первого выполнения рабочего процесса (включая ручное) он становится **только для чтения** (серый), редактировать его нельзя. Чтобы изменить, нажмите **«Скопировать в новую версию»** и работайте на новой версии. Старая версия автоматически отключится.

![06-workflows-2026-03-15-20-19-11](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-19-11.png)

Возвращаемся на страницу заявок, создаём новую с обработчиком, потом логинимся под обработчиком — в центре уведомлений должно появиться новое сообщение.

![06-workflows-2026-03-15-20-22-00](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-22-00.gif)

Поздравляем, первый автоматический сценарий запущен!

## 6.3 Сценарий 2: смена статуса автоматически фиксирует время завершения

**Требование**: когда статус заявки становится «Завершено», система автоматически проставляет в поле «Время завершения» текущее время. Не надо записывать вручную и ничего не забудется.

> Если в коллекции заявок ещё нет поля «Время завершения», добавьте поле типа **Дата** в **Управление коллекциями → Заявки** (см. Главу 2).
> ![06-workflows-2026-03-15-20-25-38](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-25-38.png)

### Шаг 1. Новый рабочий процесс

Возвращаемся в **Управление рабочими процессами**, нажимаем «Создать»:

- **Название**: «Автозапись времени завершения заявки»
- **Тип триггера**: **Событие пользовательского действия** (срабатывает по клику кнопки, привязанной к рабочему процессу)
- **Режим выполнения**: синхронный
> О синхронном и асинхронном:
> - Асинхронный: после действия можно делать другие дела, рабочий процесс выполняется в фоне и уведомляет о результате
> - Синхронный: интерфейс ждёт окончания рабочего процесса, и только потом можно работать дальше

![06-workflows-2026-03-19-22-56-34](https://static-docs.nocobase.com/06-workflows-2026-03-19-22-56-34.png)

### Шаг 2. Настраиваем триггер

Откройте настройки триггера:

- **Коллекция**: «Заявки»
- **Режим выполнения**: **однострочный** (за раз обрабатывается только текущая выбранная заявка)

![06-workflows-2026-03-19-22-58-21](https://static-docs.nocobase.com/06-workflows-2026-03-19-22-58-21.png)

<!-- TODO: дополнить скриншотом настройки триггера -->


### Шаг 3. Добавляем условное ветвление

В отличие от триггера события коллекции, у события пользовательского действия нет встроенных условий — добавляем узел условного ветвления сами:

![06-workflows-2026-03-15-20-39-14](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-39-14.png)

Рекомендуем выбрать «Продолжать раздельно по «Да» и «Нет»» — удобно расширять.

- Условие: **Триггерные данные → Статус** не равно **Завершено** (только незавершённые заявки проходят дальше, повторно не обрабатываются)

![06-workflows-2026-03-19-22-37-59](https://static-docs.nocobase.com/06-workflows-2026-03-19-22-37-59.png)

### Шаг 4. Добавляем узел обновления данных

В ветке «Да» условного узла нажмите **+**, выберите **Обновить запись**:

![06-workflows-2026-03-15-20-46-22](https://static-docs.nocobase.com/06-workflows-2026-03-15-20-46-22.png)

- **Коллекция**: «Заявки»
- **Условие фильтрации**: ID = Триггерные данные → ID (обновим только текущую заявку)
- **Значения полей**:
  - Статус = **Завершено**
  - Время завершения = **Системная переменная / Системное время**

![06-workflows-2026-03-19-22-39-27](https://static-docs.nocobase.com/06-workflows-2026-03-19-22-39-27.png)

> Один узел делает сразу два дела — меняет статус и записывает время — без необходимости настраивать значения полей в кнопке.

### Шаг 5. Создаём кнопку «Завершить»

Рабочий процесс готов, но событие пользовательского действия нужно привязать к кнопке. Создадим кнопку «Завершить» в колонке действий списка заявок:

1. В режиме UI Редактора в колонке действий таблицы заявок нажмите **«+»** и выберите действие **«Запустить рабочий процесс»**.

![06-workflows-2026-03-19-22-41-31](https://static-docs.nocobase.com/06-workflows-2026-03-19-22-41-31.png)

2. В настройках кнопки измените заголовок на **«Завершить»** и выберите подходящую иконку (например, галочку).

![06-workflows-2026-03-19-22-43-39](https://static-docs.nocobase.com/06-workflows-2026-03-19-22-43-39.png)

3. Настройте **правила связывания** для кнопки: при статусе «Завершено» кнопка скрывается (для уже завершённых заявок «Завершить» не нужен).
   - Условие: Текущая запись → Статус равен Завершено
   - Действие: скрыть

![06-workflows-2026-03-15-21-15-29](https://static-docs.nocobase.com/06-workflows-2026-03-15-21-15-29.png)

4. В настройках кнопки откройте **«Привязать рабочий процесс»** и выберите наш рабочий процесс «Автозапись времени завершения заявки».

![06-workflows-2026-03-19-23-00-53](https://static-docs.nocobase.com/06-workflows-2026-03-19-23-00-53.png)

### Шаг 6. Настраиваем поток событий для обновления

Кнопка готова, но после клика таблица не обновится сама — пользователь не увидит изменений. Настроим **поток событий** кнопки, чтобы после выполнения рабочего процесса таблица перезагружалась.

1. В настройках кнопки нажмите вторую иконку молнии (⚡) и откройте **Поток событий**.
2. Настройте срабатывание:
   - **Триггер события**: выберите **Клик**
   - **Время выполнения**: выберите **После встроенных потоков**
3. Нажмите **«Добавить шаг»** и выберите **«Обновить целевые блоки»**.

![06-workflows-2026-03-20-16-46-59](https://static-docs.nocobase.com/06-workflows-2026-03-20-16-46-59.png)

4. Найдите таблицу заявок текущей страницы, в её настройках выберите внизу **«Копировать UID»** и вставьте UID в шаг обновления.

![06-workflows-2026-03-20-16-48-39](https://static-docs.nocobase.com/06-workflows-2026-03-20-16-48-39.png)

После клика по «Завершить» рабочий процесс выполняется и таблица обновляется — пользователь сразу видит изменения статуса и времени завершения.

### Шаг 7. Активация и тестирование

Возвращаемся в **Управление рабочими процессами** и активируем рабочий процесс «Автозапись времени завершения заявки».

Откройте заявку со статусом «В обработке» и нажмите **«Завершить»** в колонке действий. Можно увидеть:

- Поле «Время завершения» автоматически заполнится текущим временем.
- Таблица обновится, кнопка «Завершить» исчезнет с этой заявки (правило связывания сработало).

![06-workflows-2026-03-15-21-25-11](https://static-docs.nocobase.com/06-workflows-2026-03-15-21-25-11.gif)

Удобно? Это второй типичный сценарий рабочего процесса — **автоматическое обновление данных**. И благодаря связке «событие пользовательского действия + кнопка» мы получили точный механизм запуска: рабочий процесс выполняется только при клике по конкретной кнопке.

## 6.4 Просмотр истории выполнения

Сколько раз отработал рабочий процесс? Не было ли ошибок? NocoBase всё фиксирует.

В списке **Управление рабочими процессами** у каждого рабочего процесса есть ссылка **Количество запусков** с числом запусков. По клику открывается подробная история:

- **Статус**: успех (зелёный) или провал (красный) — на виду
- **Время триггера**: когда сработал
- **Узлы**: внутри можно посмотреть результат каждого узла

![06-workflows-2026-03-15-21-25-38](https://static-docs.nocobase.com/06-workflows-2026-03-15-21-25-38.png)

Если выполнение провалилось, в деталях видно, какой узел сломался и какая ошибка. Это главный инструмент отладки.

![06-workflows-2026-03-15-21-36-16](https://static-docs.nocobase.com/06-workflows-2026-03-15-21-36-16.png)

## Итоги главы

В этой главе мы создали два простых, но полезных рабочих процесса:

- **Уведомление о новой заявке** (триггер — событие коллекции): автоматически уведомляет при создании или смене обработчика — кричать никому не нужно.
- **Автозапись времени завершения** (триггер — событие пользовательского действия): после клика по «Завершить» автоматически проставляется время — никаких пропусков.

Два рабочих процесса показывают два разных механизма запуска. Настройка занимает менее 10 минут — и система уже работает сама.

## Анонс следующей главы

Система автоматизирована, но не хватает **целостной картины**: всего сколько заявок? Какая категория преобладает? Сколько новых каждый день? В следующей главе с помощью [блоков](/interface-builder/blocks) графиков соберём **дашборд** — увидим всё одним взглядом.
