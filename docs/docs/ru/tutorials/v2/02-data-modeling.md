# Глава 2. Моделирование данных — две коллекции для системы заявок

В прошлой главе мы установили NocoBase и познакомились с интерфейсом. Теперь соберём скелет системы заявок — определим **модель данных**.

В этой главе мы создадим две [коллекции](/data-sources/data-modeling/collection) — заявки и категории, настроим [типы полей](/data-sources/data-modeling/collection-fields) (однострочный текст, выпадающий список, [многие-к-одному](/data-sources/data-modeling/collection-fields/associations/m2o) и т. д.) и установим связи между коллекциями. Модель данных — фундамент системы: сначала продумываем, что хранить и как данные связаны, и тогда сборка интерфейса и настройка прав пройдут естественно.


## 2.1 Что такое коллекция и поле

Если Вы работали с Excel, понять коллекцию легко:

| Понятие Excel | Понятие NocoBase | Описание |
|------------|--------------|------|
| Лист | Коллекция | Контейнер однотипных данных |
| Заголовок столбца | Поле | Атрибут данных |
| Каждая строка | Запись | Конкретная единица данных |

![02-data-modeling-2026-03-11-08-32-41](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-32-41.png)

Например, наша коллекция заявок — это как лист Excel: каждый столбец — поле (заголовок, статус, приоритет…), каждая строка — запись.

При этом NocoBase гораздо мощнее Excel. Поддерживаются разные **типы коллекций**, каждый со своими возможностями:

| Тип коллекции | Сценарий | Пример |
|--------|---------|------|
| **Обычная коллекция** | Большая часть бизнес-данных | Заявки, заказы, клиенты |
| **Древовидная коллекция** | Иерархические данные | Каталоги категорий, орг. структура |
| **Коллекция календаря** | Дата/события | Встречи, графики |
| **Коллекция файлов** | Управление вложениями | Документы, изображения |

Сегодня мы используем **обычную** и **древовидную** коллекции. Остальные типы рассмотрим, когда понадобятся.

**Открываем менеджер источников данных**: нажмите иконку **«Менеджер источников данных»** в левом нижнем углу (иконка БД рядом с шестерёнкой). Вы увидите [Основной источник данных](/data-sources) — туда мы и поместим все наши коллекции.

![02-data-modeling-2026-03-11-08-35-08](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-35-08.png)


## 2.2 Создаём основную коллекцию: заявки

Сразу к сути — создаём ядро системы, коллекцию заявок.

### Создание коллекции

1. На странице менеджера источников данных нажмите **Основной источник данных** и зайдите внутрь

![02-data-modeling-2026-03-11-08-36-06](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-36-06.png)

2. Нажмите **«Создать коллекцию»** и выберите **«Обычная коллекция»**

![02-data-modeling-2026-03-11-08-38-52](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-38-52.png)

3. Имя коллекции: `tickets`, заголовок: `Заявки`

![02-data-modeling-2026-03-11-08-40-34](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-40-34.png)

При создании коллекции система по умолчанию отмечает набор **системных полей**, которые автоматически фиксируют метаинформацию о записи:

| Поле | Описание |
|------|------|
| ID | Первичный ключ, распределённый уникальный идентификатор |
| Время создания | Когда запись создана |
| Создано пользователем | Кто создал запись |
| Дата последнего обновления | Когда последний раз обновлялась |
| Последнее обновление пользователем | Кто последним обновил |

Эти системные поля можно оставить по умолчанию — управлять ими вручную не требуется. Если в каком-то сценарии они не нужны, снимите галочки.

### Добавляем базовые поля

Коллекция создана, теперь добавим поля. Нажмите у коллекции заявок **«Настроить поля»** — Вы увидите, что системные поля уже в списке.

![02-data-modeling-2026-03-11-08-58-48](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-58-48.png)

![02-data-modeling-2026-03-11-08-59-47](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-08-59-47.png)

Нажмите кнопку **«Добавить поле»** в правом верхнем углу — раскроется список типов полей; выберите нужный.

![02-data-modeling-2026-03-11-09-00-22](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-00-22.png)

Сначала добавим собственные поля заявки, а поля связей — позже.

**1. Заголовок (однострочный текст)**

Каждой заявке нужен короткий заголовок-резюме. Нажмите **«Добавить поле»** → **[«Однострочный текст»](/data-sources/data-modeling/collection-fields/basic/input)**:

![02-data-modeling-2026-03-11-09-01-00](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-01-00.png)

- Имя поля: `title`, заголовок: `Заголовок`
- Нажмите **«Установка правил валидации»**, добавьте правило **«Обязательное»**

![02-data-modeling-2026-03-11-09-02-40](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-02-40.png)

**2. Описание**

Нужно подробно описать проблему с поддержкой форматирования, удобно вставлять картинки и код. В категории «Добавить поле» → «Медиа» три варианта:

| Тип поля | Особенности |
|---------|------|
| Markdown | Базовый Markdown, простые стили |
| Форматированный текст | Богатый текст, простые стили + загрузка вложений |
| **Markdown (Vditor)** | Самый функциональный: WYSIWYG, мгновенное отображение, режим исходного кода |

Выберем **Markdown (Vditor)**.

![02-data-modeling-2026-03-11-09-09-58](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-09-58.png)

- Имя поля: `description`, заголовок: `Описание`

![02-data-modeling-2026-03-11-09-10-50](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-10-50.png)

**3. Статус (одиночный выбор)**

![02-data-modeling-2026-03-11-09-12-00](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-12-00.png)
Заявка проходит путь от подачи до завершения, нужен статус для отслеживания.

- Имя поля: `status`, заголовок: `Статус`
- Добавьте варианты («значение опции» и «метка опции», цвет — опционально):

| Значение | Метка | Цвет |
|--------|---------|------|
| pending | На рассмотрении | Оранжевый |
| in_progress | В обработке | Синий |
| completed | Завершено | Зелёный |

![02-data-modeling-2026-03-11-09-17-44](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-17-44.png)

Сначала заполните опции и сохраните. Затем снова нажмите **«Редактировать»** на этом поле — теперь в «Значение по умолчанию» можно выбрать **«На рассмотрении»**.

![02-data-modeling-2026-03-11-09-20-28](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-20-28.png)

![02-data-modeling-2026-03-11-09-22-34](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-22-34.png)

> При первом создании опций ещё нет, поэтому значение по умолчанию выбрать нельзя — сохраните, затем вернитесь и настройте.

> Почему именно одиночный выбор? Статус — фиксированный набор значений. [Одиночный выбор](/data-sources/data-modeling/collection-fields/choices/select) предотвращает произвольный ввод и обеспечивает корректность данных.

**4. Приоритет (одиночный выбор)**

Отделяет срочные заявки и помогает сортировать обработку.

- Имя поля: `priority`, заголовок: `Приоритет`
- Добавьте варианты:

| Значение | Метка | Цвет |
|--------|---------|------|
| low | Низкий | |
| medium | Средний | |
| high | Высокий | Оранжевый |
| urgent | Срочный | Красный |

В коллекции заявок теперь 4 базовых поля. Но у заявки должна быть «категория»: сетевые проблемы, сбои ПО и т. д.

Можно сделать категорию выпадающим списком, но Вы быстро упрётесь: у категорий могут быть подкатегории («аппаратные проблемы» → «монитор», «клавиатура», «принтер»), и выпадающего списка уже не хватит.

Нам нужна **отдельная коллекция** для категорий. И лучше всего сделать её **древовидной** в NocoBase.


## 2.3 Создаём древовидную коллекцию категорий: добавляем иерархию

### Что такое древовидная коллекция

Это особый тип коллекции со встроенными **отношениями родитель-ребёнок** — у каждой записи может быть «родительский узел». Идеально подходит для иерархических данных:

```
Аппаратные проблемы          ← категория 1-го уровня
├── Монитор                   ← категория 2-го уровня
├── Клавиатура и мышь
└── Принтер
Сбои ПО
├── Офисное ПО
└── Системные проблемы
Сетевые проблемы
Учётные записи и права
```

В обычной коллекции пришлось бы вручную добавлять поле «родительская категория», а **древовидная коллекция сделает это автоматически**, плюс поддержит древовидное отображение, добавление дочерних записей и многое другое.

### Создание коллекции

1. Возвращаемся в менеджер источников данных, нажимаем **«Создать коллекцию»**.
2. На этот раз выбираем **«Древовидная коллекция»** (не обычную!).
![02-data-modeling-2026-03-11-09-26-07](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-26-07.png)

3. Имя коллекции: `categories`, заголовок: `Категории заявок`

![02-data-modeling-2026-03-11-09-26-55](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-26-55.png)

> После создания, помимо системных полей, в коллекции автоматически появятся два поля связей — **«Родитель»** и **«Дочерние»**. Это особенность древовидной коллекции. Через «Родитель» — доступ к родителю, через «Дочерние» — ко всем потомкам, добавлять вручную не нужно.

![02-data-modeling-2026-03-11-09-27-40](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-27-40.png)

### Добавляем поля

Нажмите **«Настроить поля»** — увидите системные поля и автоматически созданные «Родитель», «Дочерние».
Нажмите **«Добавить поле»** в правом верхнем углу:

**Поле 1: название категории**

1. Выберите **«Однострочный текст»**.
2. Имя поля: `name`, заголовок: `Название категории`.
3. Нажмите **«Установка правил валидации»**, добавьте **«Обязательное»**.

**Поле 2: цвет**

1. Выберите **«Цвет»**.
2. Имя поля: `color`, заголовок: `Цвет`.

![02-data-modeling-2026-03-11-09-28-59](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-28-59.png)

Поле цвета даёт каждой категории свой цвет — на интерфейсе это смотрится нагляднее.

![02-data-modeling-2026-03-11-09-29-23](https://static-docs.nocobase.com/02-data-modeling-2026-03-11-09-29-23.png)

Базовые поля обеих коллекций настроены. Теперь свяжем их.


## 2.4 Возвращаемся к коллекции заявок: добавляем поля связей

> **Поля связей могут показаться абстрактными.** Если непонятно, перейдите сразу к [Главе 3. Сборка страниц](./03-building-pages) — на практике увидите, как данные отображаются, и потом вернётесь к связям.

Заявка должна быть связана с категорией, заявителем и обработчиком. Эти поля называются **полями связей** — они хранят не текст, а ID записи из другой коллекции и через него получают связанные данные.

Посмотрим на конкретную заявку — слева её атрибуты; в полях «категория» и «заявитель» хранится не текст, а ID. Через ID система точно находит запись в правой коллекции:


![02-data-modeling-2026-03-12-00-50-10](https://static-docs.nocobase.com/02-data-modeling-2026-03-12-00-50-10.png)

В интерфейсе Вы видите имена («Сетевые проблемы», «Иван Иванов»), но за ними стоит связь по идентификатору. **Несколько заявок могут указывать на одну категорию или одного пользователя** — это связь [**многие-к-одному**](/data-sources/data-modeling/collection-fields/associations/m2o).

### Добавляем поля связей

В коллекции заявок нажмите **«Настроить поля»** → **«Добавить поле»** → **«Многие-к-одному»**.
![02-data-modeling-2026-03-12-00-52-39](https://static-docs.nocobase.com/02-data-modeling-2026-03-12-00-52-39.png)

В диалоге появятся такие настройки:

| Параметр | Описание | Что заполнять |
|--------|------|--------|
| Исходная коллекция | Текущая (заполняется автоматически) | Не меняем |
| **Целевая коллекция** | С какой коллекцией связь | Выбрать соответствующую |
| **Внешний ключ** | Имя столбца с ID в текущей коллекции | Дайте осмысленное имя |
| Поле идентификации в целевой коллекции | По умолчанию `id` | Оставить по умолчанию |
| **При удалении** | Что делать при удалении целевой записи | Оставить по умолчанию |

![02-data-modeling-2026-03-12-00-58-38](https://static-docs.nocobase.com/02-data-modeling-2026-03-12-00-58-38.png)

> Внешний ключ по умолчанию генерируется случайно (например, `f_xxxxx`) — рекомендуем заменить на осмысленное имя, чтобы было удобно сопровождать. Используйте латиницу в нижнем регистре с подчёркиваниями (например, `category_id`), без верблюжьего регистра.

Добавьте по очереди три поля:

**5. Категория → коллекция категорий заявок**

- Заголовок поля: `Категория`
- Целевая коллекция: **«Категории заявок»** (если в списке нет, введите имя — коллекция создастся)
- Внешний ключ: `category_id`

**6. Заявитель → коллекция пользователей**

Кто отправил заявку. В NocoBase есть встроенная коллекция пользователей, связываемся напрямую.

- Заголовок поля: `Заявитель`
- Целевая коллекция: **«Пользователи»**
- Внешний ключ: `submitter_id`
![02-data-modeling-2026-03-12-01-00-09](https://static-docs.nocobase.com/02-data-modeling-2026-03-12-01-00-09.png)

**7. Обработчик → коллекция пользователей**

Кто отвечает за заявку.

- Заголовок поля: `Обработчик`
- Целевая коллекция: **«Пользователи»**
- Внешний ключ: `assignee_id`

![02-data-modeling-2026-03-12-01-00-22](https://static-docs.nocobase.com/02-data-modeling-2026-03-12-01-00-22.png)

## Итоги главы

В этой главе мы построили скелет системы заявок:

1. **Коллекция заявок**: 4 базовых поля + 3 поля связей, тип — **обычная коллекция**.
2. **Коллекция категорий заявок**: 2 пользовательских поля + автоматические поля «Родитель» и «Дочерние», тип — **древовидная коллекция**, что естественно поддерживает иерархию.

Мы освоили несколько важных понятий:

- **Коллекция** = контейнер однотипных данных
- **Тип коллекции** = разные сценарии — разные типы (обычная, древовидная и т. д.)
- **Поле** = атрибут данных, создаётся через «Настроить поля» → «Добавить поле»
- **Системные поля** = ID, время создания, «Создано пользователем» и т. д. (отмечены при создании автоматически)
- **Поле связи (многие-к-одному)** = ссылка на запись в другой коллекции, основа для связей

> Возможно, Вы заметили, что на скриншотах далее уже есть данные — это тестовые записи для демонстрации, не переживайте. В NocoBase создание, чтение, изменение и удаление выполняются через интерфейс. В Главе 3 мы соберём блоки таблицы для отображения данных, в Главе 4 — формы для ввода. Раскроем тему пошагово.
