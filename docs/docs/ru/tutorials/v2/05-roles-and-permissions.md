# Глава 5. Пользователи и права — кто что видит

В прошлой главе мы доделали форму и страницу деталей — заявки можно вводить и просматривать. Но есть проблема: после входа все видят одно и то же. Обычный сотрудник видит управленческую страницу, технический специалист может удалять категории — это никуда не годится.

В этой главе ставим «турникеты»: создадим [роли](/users-permissions/acl/role), настроим [права на меню](/users-permissions/acl/permissions) и [область данных](/users-permissions/acl/permissions), чтобы **разные пользователи видели разное меню и работали с разными данными**.

## 5.1 Понимание [ролей](/users-permissions/acl/role)

В NocoBase **роль — это набор [прав](/users-permissions/acl/role)**. Не нужно каждому пользователю отдельно настраивать права: определяем несколько ролей и относим к ним пользователей.

После установки в NocoBase есть три встроенные роли:

- **Root**: суперадминистратор, все права, нельзя удалить
- **Admin**: администратор, по умолчанию имеет права настройки интерфейса
- **Member**: обычный пользователь, прав по умолчанию мало

Этих трёх ролей нам не хватит. Для системы заявок нужно более тонкое разделение, поэтому создадим 3 пользовательских роли.

## 5.2 Создаём три роли

Откройте меню настроек в правом верхнем углу, перейдите в **Пользователи и права → Управление ролями**.

Нажмите **«Добавить роль»** и создайте по очереди:

| Имя роли | Идентификатор | Описание |
|---------|---------|------|
| Администратор | admin-helpdesk | Видит все заявки, управляет категориями, назначает обработчиков |
| Технический специалист | technician | Видит только свои назначенные заявки, обрабатывает и закрывает |
| Обычный пользователь | user | Подаёт заявки и видит только свои |

![05-roles-and-permissions-2026-03-13-19-03-14](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-19-03-14.png)

> **Идентификатор роли** — внутренний уникальный ID, после создания не меняется. Используйте латиницу в нижнем регистре. Имя роли можно поменять в любое время.

![05-roles-and-permissions-2026-03-13-18-57-47](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-18-57-47.png)

После создания список ролей покажет три новые записи.


## 5.3 Настраиваем права на меню

Роли созданы — теперь укажем, какие пункты меню видит каждая роль.

Откройте конкретную роль, перейдите на вкладку **Права меню**. Здесь все пункты меню системы.

**Администратор**: всё включено
- Управление заявками, управление категориями, дашборд

**Технический специалист**: частично
- Управление заявками
- Дашборд
- ❌ Управление категориями

**Обычный пользователь**: минимум
- Управление заявками (только свои заявки)
- ❌ Управление категориями
- ❌ Дашборд

![05-roles-and-permissions-2026-03-13-19-09-11](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-19-09-11.png)

> **Подсказка**: в NocoBase есть удобная опция «Новые пункты меню доступны по умолчанию». Если не хотите при добавлении новых страниц каждый раз вручную включать доступ — включите её для роли администратора. Для обычного пользователя её лучше отключить.

## 5.4 Настраиваем права на данные

Права на меню решают, может ли пользователь зайти на страницу. Права на данные решают, какие записи он там увидит.

Ключевая концепция: **[область данных](/users-permissions/acl/permissions)**.

В правах роли переключитесь на вкладку **Права действий**. Найдите коллекцию заявок и настройте отдельно.

![05-roles-and-permissions-2026-03-13-19-51-06](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-19-51-06.png)

### Обычный пользователь: только свои заявки

1. Найдите право **Просмотр** для коллекции «Заявки».
2. Область данных → **Собственные записи**.
3. Так обычный пользователь увидит только заявки, которые он сам создал. (Внимание: по умолчанию учитывается системное поле «Создано пользователем», а не «Заявитель», но это можно изменить.)

То же самое — для прав **Редактировать** и **Удалить** (или вовсе не давайте **Удалить**).

![05-roles-and-permissions-2026-03-13-19-53-02](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-19-53-02.png)


О глобальной настройке: если настроить только коллекцию заявок, могут стать невидимыми остальные данные/конфиги (категории, обработчик). Так как наша система простая, в глобальных настройках включим **Просмотр** с областью **Все записи**, а у коллекций с чувствительной областью данных настроим права отдельно.

![05-roles-and-permissions-2026-03-13-19-57-24](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-19-57-24.png)


### Технический специалист: только назначенные ему заявки

1. Найдите право **Просмотр** для коллекции «Заявки».
2. Область данных → **Собственные записи**.
3. Тонкость: «Собственные записи» в NocoBase по умолчанию фильтруют по полю «Создано пользователем». Если хочется фильтровать по «Обработчику», корректируйте в глобальных [правах действий](/users-permissions/acl/permissions) или используйте на странице **условие фильтрации блока [данных](/interface-builder/blocks)**.

![05-roles-and-permissions-2026-03-13-20-01-54](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-20-01-54.png)

> **Полезный приём**: в блоке таблицы можно задать условия фильтра по умолчанию, например «Обработчик = Текущий пользователь». Но это глобальная настройка — администратора она тоже ограничит. Компромисс: «Обработчик = Текущий пользователь **ИЛИ** Заявитель = Текущий пользователь» — для обычных пользователей и технических специалистов одновременно. Администратору же сделайте отдельную страницу без этих фильтров.

![05-roles-and-permissions-2026-03-13-22-21-34](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-22-21-34.png)

### Администратор: видит все данные

Для администратора область данных = **Все записи**, все действия включены. Просто и надёжно.

![05-roles-and-permissions-2026-03-13-21-45-14](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-21-45-14.png)

## 5.5 Назначение заявки

Перед окончательной настройкой прав добавим в список заявок удобную функцию: **назначение обработчика**. Администратор сможет назначить заявку технику прямо из списка, без перехода на страницу редактирования.

Реализация простая — пользовательская кнопка-окно в колонке действий:

1. В режиме UI Редактора в колонке действий таблицы заявок нажмите **«+»** и добавьте действие **«Всплывающее окно»**.

![05-roles-and-permissions-2026-03-14-13-57-31](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-14-13-57-31.png)

2. Измените заголовок кнопки на **«Назначить»** (через настройки кнопки).

![05-roles-and-permissions-2026-03-14-13-59-22](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-14-13-59-22.png)


Поскольку у нас одно простое действие назначения, удобнее простое модальное окно, а не выезжающая панель. В настройках кнопки выберите **Настройки всплывающего окна → Диалог → Маленький → Подтвердить**.
![05-roles-and-permissions-2026-03-14-14-08-16](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-14-14-08-16.png)


3. Нажмите кнопку «Назначить», в окне выберите **«Добавить блок → Блок данных → Форма (Редактировать)»** и выберите текущую коллекцию.
4. В форме отметьте только поле **«Обработчик»** и в его настройках сделайте обязательным.
5. Добавьте кнопку **«Отправить»**.

![05-roles-and-permissions-2026-03-14-14-10-50](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-14-14-10-50.png)

Так администратор в списке заявок нажимает «Назначить», открывается лаконичная форма, выбирает обработчика и отправляет. Быстро, точно, без случайных правок.

### Скрытие кнопки через правило связывания

Кнопка «Назначить» нужна только администратору, а у обычных пользователей и технических специалистов она вызовет путаницу. Скроем её через **правило связывания** в зависимости от роли:

1. В режиме UI Редактора нажмите настройки кнопки «Назначить» и найдите **«Правила связывания»**.
2. Добавьте правило: **Текущий пользователь / Роль / Имя роли** не равно **Администратор** (название роли admin-helpdesk).
3. Действие при выполнении условия: **скрыть** кнопку.

Так кнопку видит только администратор; у других ролей при входе она автоматически скрывается.

![05-roles-and-permissions-2026-03-14-14-17-37](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-14-14-17-37.png)

## 5.6 Создаём тестовых пользователей и проверяем

Права настроены — проверим на практике.

Откройте **Управление пользователями** (центр настроек или собранная Вами ранее страница) и создайте 3 тестовых пользователей:

| Имя | Роль |
|-------|------|
| Alice | Администратор |
| Bob | Технический специалист |
| Charlie | Обычный пользователь |

![05-roles-and-permissions-2026-03-13-22-23-47](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-22-23-47.png)

Затем войдите под каждой учёткой и проверьте две вещи:

**1. Меню отображается, как задумано?**
- Alice → видит все пункты меню

![05-roles-and-permissions-2026-03-14-14-19-29](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-14-14-19-29.png)

- Bob → видит только Управление заявками и дашборд

![05-roles-and-permissions-2026-03-13-22-26-50](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-22-26-50.png)

- Charlie → видит только «Мои заявки»

![05-roles-and-permissions-2026-03-13-22-30-57](https://static-docs.nocobase.com/05-roles-and-permissions-2026-03-13-22-30-57.png)

**2. Данные фильтруются, как задумано?**
- Сначала под Alice создайте несколько заявок и распределите их разным обработчикам.
- Войдите под Bob — он увидит только назначенные ему заявки.
- Войдите под Charlie — он увидит только свои заявки.

Круто, правда? Одна и та же система, а пользователи видят совершенно разное! В этом сила прав.

## Итоги главы

В этой главе мы построили систему прав:

- **3 роли**: администратор, технический специалист, обычный пользователь
- **Права на меню**: какая роль попадает на какие страницы
- **Права на данные**: какая роль видит какие записи (через область данных)
- **Проверка**: вход разными учётками подтверждает корректность

Система заявок выглядит как настоящая: вводить, просматривать, контролировать доступ — всё умеет. Но пока всё выполняется вручную.

## Анонс следующей главы

В следующей главе разберём **рабочий процесс** — пусть система работает за нас сама. Например, при создании заявки автоматически уведомлять обработчика, при изменении статуса — записывать журнал.
