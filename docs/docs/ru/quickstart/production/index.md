# Развертывание в продакшене

Если ваше приложение NocoBase уже нормально работает на сервере, то для выхода в продакшен обычно нужно доделать только две вещи:

1. Сделать так, чтобы приложение автоматически восстанавливалось после перезагрузки машины
2. Добавить перед приложением reverse proxy, чтобы обеспечить стабильный внешний доступ

В NocoBase CLI для этого в основном нужны две группы команд:

- `nb app autostart`
- `nb env proxy`

Эта страница сначала объясняет общий путь. Если вам нужны детали по Nginx или Caddy, дальше переходите на соответствующие дочерние страницы.

## Шаг 1: Настройте автозапуск приложения

В продакшене первым делом важно не "сразу настроить домен", а убедиться, что сам сервис может стабильно восстанавливаться. Иначе после перезагрузки машины, пересоздания контейнера или обычных операций по обслуживанию приложение может не подняться автоматически.

В CLI `nb app autostart` — это группа подкоманд, а не одна команда, которая сразу делает все. Чаще всего используются:

- `nb app autostart enable`
- `nb app autostart list`
- `nb app autostart run`

Сначала включите автозапуск для текущего env:

```bash
nb app autostart enable
```

Если вы хотите явно указать другой env:

```bash
nb app autostart enable --env app1 --yes
```

После этого можно проверить, для каких env уже включен автозапуск:

```bash
nb app autostart list
```

После загрузки системы выполните следующую команду, чтобы запустить все env, у которых включен автозапуск:

```bash
nb app autostart run
```

Если при диагностике вы хотите видеть подробный вывод запуска, можно добавить:

```bash
nb app autostart run --verbose
```

:::tip Что именно делает этот шаг

`nb app autostart enable` помечает env, которым управляет CLI, как "разрешенный к автоматическому запуску"; `nb app autostart run` фактически запускает все env, для которых автозапуск уже включен.

То есть в реальной продакшен-среде обычно еще нужно встроить `nb app autostart run` в ваш собственный процесс системного старта — например, через `systemd`, стартовый скрипт контейнерной платформы или другой механизм автозапуска хоста, который вы уже используете.

:::

### Область применения

`nb app autostart` подходит только для env, у которых есть runtime, управляемый CLI, то есть:

- `local`
- `docker`

Если этот env представляет собой только удаленное API-подключение, или приложение на этой машине не запускается под локальным управлением CLI, то эта группа команд не подходит для настройки автозапуска.

## Шаг 2: Настройте обратный прокси

Когда приложение уже может автоматически восстанавливаться после перезапуска, вторым шагом настраивают внешний входной слой. В продакшене reverse proxy обычно отвечает за такие задачи:

- привязка домена или внешнего порта
- проксирование HTTP- и WebSocket-запросов в NocoBase
- управление HTTPS, сертификатами, кэшированием или контролем доступа

В NocoBase CLI рекомендуемые точки входа такие:

- `nb env proxy nginx`
- `nb env proxy caddy`

### Вариант по умолчанию

Если ваше приложение уже сохранено как CLI env и относится к типу `local` или `docker`, обычно достаточно просто дать CLI сгенерировать конфигурацию:

```bash
nb env proxy nginx --env app1 --host app.example.com
nb env proxy caddy --env app1 --host app.example.com
```

Если текущий env уже и есть целевой env, `--env` можно опустить:

```bash
nb env proxy nginx --host app.example.com
```

CLI заодно поможет закрыть несколько деталей, которые легко упустить при ручной настройке, например:

- проксирование WebSocket
- входные и статические пути в сценариях с подпутем
- SPA fallback page
- общие конфигурационные файлы provider

### Когда выбирать Nginx, а когда Caddy

Обычно можно ориентироваться так:

| Сценарий | Рекомендуется |
| --- | --- |
| Вы уже используете Nginx для сайтов, кэша, сертификатов или контроля доступа | [Nginx](./reverse-proxy/nginx.md) |
| У вас уже есть домен, и вы хотите быстрее поднять HTTPS с меньшим объемом ручного обслуживания TLS | [Caddy](./reverse-proxy/caddy.md) |
| Сначала хотите понять общую логику этой группы команд | [Reverse Proxy в продакшене](./reverse-proxy/index.md) |

Если вы меняли такие env-настройки, как `app-port` или `app-public-path`, которые влияют на результат проксирования, не забудьте заново выполнить соответствующую proxy-подкоманду.

## Базовый путь выхода в продакшен

Если вы хотите пройти самый простой путь до продакшена, обычно достаточно идти в таком порядке:

1. Сначала убедитесь, что приложение уже нормально запускается на самом сервере
2. Выполните `nb app autostart enable`
3. Встройте `nb app autostart run` в системный процесс запуска
4. Выберите Nginx или Caddy и выполните соответствующую подкоманду `nb env proxy`
5. Проверьте внешний доступ через домен или итоговый входной адрес

## Связанные команды

```bash
# Включить автозапуск для одного env
nb app autostart enable --env app1 --yes

# Посмотреть статус автозапуска
nb app autostart list

# Запустить все env, для которых включен автозапуск
nb app autostart run

# Сгенерировать конфигурацию reverse proxy для Nginx
nb env proxy nginx --env app1 --host app.example.com

# Сгенерировать конфигурацию reverse proxy для Caddy
nb env proxy caddy --env app1 --host app.example.com
```
