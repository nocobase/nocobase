# Контроль версий

:::tip Предварительные требования

- Перед чтением этой страницы установите NocoBase CLI и завершите инициализацию, как описано в [Быстрый старт Мастера ИИ-разработки](./index.md)
- Включите плагины «Управление резервным копированием» и «Управление версиями»
- Редакции Community и Standard не включают плагин управления версиями. Если нужна только точка отката перед ключевыми изменениями, используйте [Управление резервным копированием](../ops-management/backup-manager/index.mdx)

:::

## Введение

Навык контроля версий (`nocobase-revision`) создаёт восстанавливаемую версию приложения после завершения значимой вехи Мастером ИИ-разработки. Например, после построения страницы, создания группы коллекций или настройки рабочего процесса ИИ может выполнить `nb revision create`, чтобы сохранить текущее состояние.

Он не создаёт версию при каждом изменении поля. По умолчанию сохранение выполняется только после завершения и проверки понятной вехи — так список версий остаётся читаемым, и проще выбрать точку отката.

О списках версий, ручном создании версий, операциях восстановления и настройках хранения см. [руководство по плагину управления версиями](../ops-management/version-control/index.md).

## Возможности

Можно:

- Создавать версию после завершения и проверки вехи сборки
- Записывать краткое описание того, что было сохранено
- Создавать версии с использованием текущего окружения CLI

Нельзя:

- Заменять базовые возможности сохранения и восстановления плагина «Управление резервным копированием»
- Создавать версии, если плагин управления версиями не включён
- Автоматически восстанавливать версию. Для восстановления версий используйте [плагин управления версиями](../ops-management/version-control/index.md)

## Примеры запросов

### Сценарий A: сохранение завершённой конфигурации страницы

```text
Сохрани текущую сборку как версию: завершена страница управления клиентами, область фильтров и конфигурация формы редактирования
```

Навык превратит описание в краткую заметку версии и создаст версию.

Режим CLI:

```bash
nb revision create "Завершена страница управления клиентами, область фильтров и конфигурация формы редактирования"
```

### Сценарий B: сохранение вех модели данных и рабочего процесса

```text
Коллекции поставщиков и рабочий процесс согласования закупок проверены. Создай для меня версию.
```

Подходит для работы, охватывающей несколько возможностей. Например, создайте коллекции с помощью [Моделирования данных](./data-modeling), настройте процесс согласования с помощью [Управления рабочими процессами](./workflow), проверьте результат, затем сохраните версию.

### Сценарий C: создание версии в указанном окружении

```text
В окружении dev сохрани версию: завершена страница управления заявками и конфигурация поля SLA
```

Если указанное окружение не является текущим окружением CLI, навык сначала подтвердит целевое окружение, чтобы не сохранить версию в неправильное приложение.

Режим CLI:

```bash
nb revision create --env dev --yes "Завершена страница управления заявками и конфигурация поля SLA"
```

## Как писать описания версий

Описание версии должно указывать, что было завершено, а не просто использовать расплывчатую метку.

Рекомендуется:

- `Завершён клиентский реестр, страница деталей и поток отправки на согласование`
- `Завершены коллекции поставщиков, форма заявки на закупку и рабочий процесс согласования`
- `完成客户台账、详情页与审批提交流程配置`

Не рекомендуется:

- `снимок`
- `резервная копия`
- `тест`
- `версия 2`
- Только дата или метка времени

Также не включайте токены, URL, пароли или другую конфиденциальную информацию. Описания отображаются в списке версий и должны оставаться понятными, читаемыми и пригодными для аудита.

## Часто задаваемые вопросы

**Когда создавать версию?**

Создавайте после вехи, которую можно проверить отдельно. Например, страница открывается и редактируется корректно, связи коллекций проверены или рабочий процесс сохранён и его цепочка узлов проверена.

**Почему не создавать версию после каждой корректировки ИИ?**

Слишком много мелких версий быстро затрудняют чтение списка. Версия обычно должна представлять точку, к которой можно откатиться и продолжить работу, а не переименование одного поля или изменение положения кнопки.

**Нужно ли проверять результат перед созданием версии?**

Да. Навык контроля версий предназначен для сохранения завершённой и проверенной работы. Если страница всё ещё выдаёт ошибки или рабочий процесс не подтверждён, попросите ИИ исправить и проверить перед созданием версии.

**Где восстанавливать версию после её создания?**

Восстанавливайте в списке версий плагина управления версиями. Восстановление перезаписывает текущую конфигурацию приложения и данные, включённые в эту версию. Перед выполнением прочитайте [руководство по плагину управления версиями](../ops-management/version-control/index.md).
