# Как подключить старый метод установки к AI и перейти на CLI

Если Вы по-прежнему устанавливаете и обслуживаете NocoBase через Docker, `create-nocobase-app` или исходный код Git по старой документации, можно продолжать так же. Для доступа к ИИ не обязательно сразу переустанавливать приложение.

Эта страница помогает выбрать маршрут:

- продолжить использовать прежние способы установки и обновления;
- сначала дать существующим приложениям доступ к ИИ-агенту;
- перейти на новый подход на базе CLI.

По умолчанию сначала определите свою категорию и откройте соответствующий раздел — так стабильнее и меньше риск ошибок в производственном окружении.

## Какой способ выбрать?

| Если сейчас нужно... | Что делать по умолчанию |
| --- | --- |
| Продолжать устанавливать, обновлять и обслуживать приложения по-старому | Оставить прежний способ; см. ссылки на документацию ниже |
| Подключить стабильно работающее старое приложение к ИИ-агенту | Сначала удалённое подключение — наименьший риск |
| В будущем управлять приложениями через `nb app`, `nb env`, `nb source` | Создать новое CLI-приложение и перенести туда старые данные |

## Способ 1. Сначала дать существующим приложениям доступ к ИИ-агенту

Если старое приложение уже стабильно работает, по умолчанию используйте этот способ.

Суть: подключить существующие приложения к CLI и ИИ-агенту через удалённое подключение. Риск минимален — текущие процессы установки, запуска, остановки и обновления не затрагиваются.

Но важно понимать границы:

- нет возможностей, связанных с `nb app`;
- CLI не берёт на себя управление старым приложением в среде выполнения;
- возможности, связанные с разработкой через ИИ, доступны в обычном режиме.

Если сейчас важнее «сначала подключить ИИ», а не «сразу перевести всю эксплуатацию на CLI», по умолчанию идите этим путём.

При подключении к существующему приложению можно инициализировать CLI-окружение так:

```bash
# по умолчанию — OAuth-аутентификация
nb init --yes --env app1 \
  --api-base-url=http://your-app-host/api

# аутентификация по токену
nb init --yes --env app1 \
  --api-base-url=http://your-app-host/api \
  --auth-type=token \
  --access-token=<token>
```

Если позже понадобится повторная аутентификация:

```bash
nb env auth app1
```

Чтобы начать создавать возможности через ИИ, см. [Краткое руководство по созданию с AI](/ai-builder/).

## Способ 2. Переход на CLI

Если в будущем нужны `nb app`, `nb env` и `nb source` для локальных приложений, безопаснее не подключать существующее приложение напрямую, а создать новое и перенести туда данные.

Причина проста: возможность «подхватить существующее приложение» ещё в разработке.

Рекомендуемый маршрут миграции сейчас:

1. Создать новое CLI-приложение.
2. Перенести БД, `storage` и переменные окружения старого приложения.
3. Убедиться, что работа, обновление и возможности ИИ в новом приложении в порядке, и решить, переключать ли производственное окружение.

Сначала создайте новое CLI-окружение:

```bash
nb init --yes --env app1
```

Перед миграцией подготовьте:

1. Резервную копию БД.
2. Резервную копию каталога `storage`.
3. Запись ключевых переменных окружения старого приложения: `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

По умолчанию сначала переносите тестовое окружение. Производственное — только после проверки резервной копии, переменных окружения и конфигурации БД.
