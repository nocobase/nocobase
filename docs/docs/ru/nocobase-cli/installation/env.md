# Конфигурация приложения и `.env`

Эта страница относится только к приложениям, созданным или размещённым через NocoBase CLI.

Если Вы только что прочитали [Установить с помощью CLI (рекомендуется)](./cli.md) и раздел «Каталог установки», чаще всего возникают такие вопросы:

- Где лежит файл `.env`?
- Какие ещё настройки уместно писать в `.env`?
- Какие настройки лучше передавать через `nb env update`?

Краткий вывод:

- Для приложений, установленных через CLI, `.env` по умолчанию находится в `<app-path>/.env`.
- Файл необязателен — не каждое окружение нужно создавать вручную.
- Базовые параметры (`APP_KEY`, `TZ`, `APP_PORT`, `APP_PUBLIC_PATH`, `DB_*`) по умолчанию управляются через `nb env update`.
- `.env` в основном дополняет переменные среды выполнения, которые CLI не берёт на себя напрямую: хранилище, кэш, логи, телеметрия и переменные расширений плагинов.

## Сначала найдите `app-path`

В [Установить с помощью CLI (рекомендуется)](./cli.md#каталог-установки) структура каталогов CLI-окружения по умолчанию такая:

```text
<app-path>/
├── source/
├── storage/
└── .env
```

Если не уверены, где текущий `app-path`, проверьте:

```bash
nb env info app1 --field app.appPath
```

Замените `app1` на имя своего окружения.

Для приложения, созданного или размещённого через CLI, `.env` логичнее всего размещать здесь:

```text
<app-path>/.env
```

Обычно не нужно класть его в `source/.env` и не нужно искать `.env` в корне проекта Docker Compose по старой схеме установки.

## Когда нужно создавать `.env` вручную

`.env` необязателен.

Если нужно просто запустить приложение или изменить базовые параметры — порты, часовой пояс, подключение к БД, публичный путь — во многих случаях `.env` создавать не нужно.

Добавляйте переменные в `<app-path>/.env` только если нужны параметры среды выполнения, которые CLI не обрабатывает напрямую.

## По умолчанию — `nb env update`

В новом способе установки через CLI базовую конфигурацию приложения рекомендуется задавать через [`nb env update`](../../api/cli/env/update.md).

Это даёт два преимущества:

- конфигурация и само окружение хранятся в одной модели CLI — проще проверять и менять;
- Вы, скрипты и ИИ-агенты продолжаете использовать один набор команд, и не возникает ситуации «в файле одни значения, в CLI — другие».

### Эти параметры лучше передавать через `nb env update`

Раньше их часто писали прямо в `.env`. В режиме установки через CLI по умолчанию используйте `nb env update`:

| Мне нужно изменить... | Как задать значение |
| --- | --- |
| `APP_KEY` | `nb env update <name> --app-key <value>` |
| `TZ` | `nb env update <name> --timezone <value>` |
| `APP_PORT` | `nb env update <name> --app-port <value>` |
| `APP_PUBLIC_PATH` | `nb env update <name> --app-public-path <value>` |
| `CDN_BASE_URL` | `nb env update <name> --cdn-base-url <value>` |
| Тип БД и параметры подключения: `DB_DIALECT`, `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USER`, `DB_PASSWORD` | `nb env update <name> --db-dialect ... --db-host ... --db-port ... --db-database ... --db-user ... --db-password ...` |
| Схема PostgreSQL, префикс таблиц, подчёркивание в именах: `DB_SCHEMA`, `DB_TABLE_PREFIX`, `DB_UNDERSCORED` | `nb env update <name> --db-schema ... --db-table-prefix ... --db-underscored` |

Пример смены порта и часового пояса:

```bash
nb env update app1 --app-port 13080 --timezone Asia/Shanghai
```

Пример смены параметров БД:

```bash
nb env update app1 \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

После изменений CLI обычно предложит выполнить `nb app restart`. Подробнее о параметрах — в [`nb env update`](../../api/cli/env/update.md).

## Что лучше писать в `.env`

Если у переменной ещё нет параметра CLI или это расширенная настройка, передаваемая напрямую в среду выполнения приложения, пишите её в `<app-path>/.env`.

Обычно это:

- хранилище файлов и объектов: `LOCAL_STORAGE_*`, `AWS_S3_*`, `ALI_OSS_*`, `TX_COS_*`;
- кэш и Redis: `CACHE_*`, `REDIS_URL`;
- логи и телеметрия: `LOGGER_*`, `TELEMETRY_*`;
- переменные конкретных плагинов и расширений: экспорт, асинхронные задачи, рабочие процессы, расширения ИИ.

Например:

```bash
LOCAL_STORAGE_DEST=storage/uploads
AWS_S3_BUCKET=your-bucket
AWS_S3_REGION=ap-southeast-1
LOGGER_LEVEL=info
REDIS_URL=redis://127.0.0.1:6379
```

Такие переменные — конфигурация среды выполнения приложения; CLI пока не обрабатывает их по одной. Естественнее всего держать их в `.env`.

## Как разделить `.env` и `nb env update`

Если неясно, куда отнести настройку, по умолчанию:

- если у `nb env update` есть соответствующий параметр — используйте его в первую очередь;
- если параметра нет или это расширение среды выполнения (плагины, хранилище, кэш, логи) — пишите в `<app-path>/.env`.

В большинстве сценариев этого достаточно.

### Частое заблуждение

Не дублируйте одну и ту же настройку в двух местах.

Если `APP_PORT`, `TZ`, `APP_PUBLIC_PATH` и `DB_HOST` уже сохранены через `nb env update`, обычно не нужно повторять их в `.env`. Иначе при отладке трудно понять, какое значение реально применяется.

## Минимальный пример `.env`

Если базовая конфигурация уже в CLI, в `.env` часто остаются только расширения:

```bash
LOGGER_LEVEL=info
REDIS_URL=redis://127.0.0.1:6379
AWS_S3_BUCKET=your-bucket
AWS_S3_REGION=ap-southeast-1
```

Именно такой подход эта страница и предлагает:

`.env` по-прежнему полезен, но при установке через CLI он дополняет расширенную конфигурацию среды выполнения, а не заменяет все базовые параметры установки.
