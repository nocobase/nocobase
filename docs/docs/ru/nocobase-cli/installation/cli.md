# Установить с помощью CLI (рекомендуется)

Начиная с NocoBase 2.1.0 доступен официальный способ установки и управления через CLI: установка, подключение, обновление и ежедневное обслуживание, а также подготовка подключаемого и работоспособного окружения для ИИ-агента.

## Установите NocoBase CLI

Выполняется только при первой установке CLI.

Сначала установите CLI глобально:

```bash
npm install -g @nocobase/cli
nb --version
```

:::tip Рекомендуется сначала включить режим сессии

Если Вы открываете несколько терминалов или оболочек одновременно или хотите, чтобы ИИ-агент работал параллельно с Вами, по умолчанию сначала выполните [`nb session setup`](../../api/cli/session/setup.md). Тогда каждая сессия сохраняет своё текущее окружение и они не мешают друг другу.

```bash
nb session setup
```

:::

CLI по умолчанию проверяет наличие обновлений. Стратегию можно настроить:

- `prompt`: запрос при обнаружении новой версии.
- `auto`: автоматическое обновление.
– `off`: отключить автоматические обновления.

```bash
nb config set update.policy prompt
nb config set update.policy auto
nb config set update.policy off
```

Самообновление поддерживается только при стандартной глобальной установке через npm, pnpm или yarn. При запуске из исходников или дерева зависимостей локального проекта используйте [`nb self check`](../../api/cli/self/check.md), чтобы узнать способ установки, и обновляйте родительский проект.

Если разворачиваете NocoBase на сервере и хотите открыть мастер `nb init --ui` из удалённого браузера, рекомендуется задать хост CLI по умолчанию на IP текущего сервера:

```bash
nb config set default-ui-host <server-ip>
nb config set default-api-host <server-ip>
```

Замените `<server-ip>` на фактический доступный IP сервера.

`nb config` — глобальная конфигурация CLI. Обычно её задают один раз; при повторном `nb init --ui` эти значения подставляются автоматически.

В общем случае:

- `default-ui-host` — для URL мастера `nb init --ui`, открываемого в браузере; сам сервис мастера всегда слушает `0.0.0.0`;
- `default-api-host` — адрес API по умолчанию для новых установок.

На сервере оба значения обычно меняют на IP, доступный с этой машины, а не оставляют локальный адрес по умолчанию.

:::warning Это только мастер установки или временный способ доступа, а не рекомендуемая точка входа для производственного окружения.

Задание `default-ui-host` / `default-api-host` на IP сервера нужно в основном для открытия `nb init --ui` из удалённого браузера или временной проверки доступности сервиса после установки.

Это не означает, что в производственном окружении долгое время нужно отдавать сервис по `IP + port`. При формальном развёртывании рекомендуется доменное имя, единый доступ через обратный прокси (Nginx или Caddy) и HTTPS.

:::

## Установите NocoBase

### Способ 1. Установка через мастер UI

Рекомендуемая точка входа по умолчанию:

```bash
nb init --ui
```

Чтобы зафиксировать порт страницы мастера, добавьте `--ui-port`, например:

```bash
nb init --ui --ui-port 3000
```

![Мастер UI nb init](https://static-docs.nocobase.com/2026-06-03-20-54-01.png)

Мастер пошагово проведёт через настройку установки или подключения в зависимости от сценария.

### Способ 2. Работа через терминал

Если удобнее вводить шаги в терминале:

```bash
nb init
```

![2026-06-03-21-36-33](https://static-docs.nocobase.com/2026-06-03-21-36-33.png)

### Способ 3. Неинтерактивные команды

В скриптах, CI/CD и других неинтерактивных сценариях используйте `--yes`. В этом режиме `--env` нужно указывать явно; остальные параметры без явного значения берутся по умолчанию.

Минимальный пример:

```bash
nb init --yes --env app1
```

Типовые комбинации — разные источники установки, выбор версии, аутентификация `basic`, подключение CI/CD к существующему приложению, именование БД — см. [примеры в справочнике `nb init`](../../api/cli/init.md#example).

## Что проверить сразу после установки

`--env` — имя окружения в CLI. После установки дальнейшие действия обычно вращаются вокруг этого окружения.

Рекомендуется сначала подтвердить три вещи:

1. Окружение создано и успешно сохранено.
2. Приложение запускается, логи в норме.
3. Если планируется внешний доступ, продумана точка входа в производственное окружение, а не постоянное использование `IP + port`.

### Каталог установки

После `nb init --env app1` полный путь можно посмотреть так:

```bash
nb env info app1 --field app.appPath
```

По умолчанию CLI раскладывает файлы под `app-path` так:

```text
<app-path>/
├── source/   # каталог исходников приложения или загруженного содержимого
├── storage/  # каталог данных среды выполнения
└── .env      # необязательный файл переменных окружения приложения
```

В общем случае:

- `source/` — каталог локального npm/Git-приложения. Для Docker-окружения CLI тоже выводит эти пути, но вручную обычно не требуется.
- `storage/` — данные среды выполнения: встроенная БД, плагины, логи и т. д.
- `.env` — дополнительные переменные окружения приложения. Добавляйте в `<app-path>/.env` только при необходимости; если файл есть, его читают источники установки Docker, npm и Git.

Подробнее — в [справочнике `nb init`](../../api/cli/init.md).

### Напоминание о развёртывании в производственном окружении

Сразу после установки для проверки результата обычно достаточно открыть страницу по `IP + port`.

Если это окружение будет официально обслуживать внешних пользователей, обратите внимание:

- `nb init --ui` — временная страница мастера установки, не официальная внешняя точка входа приложения.
- После `nb init` текущий `IP + port` подходит для отладки, проверки или временного доступа в интрасети.
- В производственном окружении не рекомендуется долго открывать порт приложения NocoBase в публичную сеть.
- Для официального внешнего доступа — доменное имя и обратный прокси через Nginx или Caddy.
- В производственном окружении перед долгим использованием `http://IP:port` включите HTTPS.

`default-ui-host` и `default-api-host` упрощают мастер и генерацию адреса по умолчанию; это не точка входа в итоговое производственное окружение.

Когда окружение готово к запуску, следующий шаг после установки — «подключить обратный прокси и включить HTTPS», а не опциональная оптимизация.

Для формального развёртывания начните с [развёртывания производственного окружения](../production/index.md), затем при необходимости — [Nginx](../production/reverse-proxy/nginx.md) или [Caddy](../production/reverse-proxy/caddy.md).

### Повседневные операции

Проверьте, что окружение сохранено:

```bash
nb env current
nb env list
nb env status
nb env info app1
nb env info app1 --json
```

Дальнейшие шаги после установки:

| Мне нужно... | Где искать |
| --- | --- |
| Официально открыть это окружение наружу: обратный прокси производственного окружения, домен и HTTPS | [Nginx](../production/reverse-proxy/nginx.md) / [Caddy](../production/reverse-proxy/caddy.md) |
| Убедиться, что окружение сохранено, какое сейчас активно, переключение между окружениями | [`nb env`](../../api/cli/env/index.md), [Управление несколькими окружениями](../operations/multi-environment.md) |
| Запуск, остановка, перезапуск, логи, обновление приложения | [`nb app`](../../api/cli/app/index.md), [Управление приложением](../operations/manage-app.md) |
| Подключения к БД, состояние встроенной БД, проблемы с контейнером БД | [`nb db`](../../api/cli/db/index.md) |
| Установленные плагины, включение и отключение | [`nb plugin`](../../api/cli/plugin/index.md) |
| Коммерческая лицензия, статус авторизации, синхронизация коммерческих плагинов | [`nb license`](../../api/cli/license/index.md) |
| Локальные проекты исходного кода: загрузка, режим разработки, сборка, тесты (обычно с npm/Git-окружением) | [`nb source`](../../api/cli/source/index.md) |

После установки локального приложения часто достаточно:

```bash
nb env use app1
nb app start
nb app logs
nb plugin list
```

При нескольких окружениях см. [Управление несколькими окружениями](../operations/multi-environment.md).

Для обновления позже — [Управление приложением](../operations/manage-app.md) и [справочник `nb app upgrade`](../../api/cli/app/upgrade.md).
