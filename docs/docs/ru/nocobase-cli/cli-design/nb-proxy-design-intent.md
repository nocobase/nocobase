# Замысел проектирования `nb proxy`

`nb proxy` даёт набор более простых и стабильных команд вместо изначально сложного процесса на уровне входного слоя.

Если говорить только о базовом сценарии, достаточно запомнить три команды:

```bash
nb proxy nginx use docker
nb proxy nginx generate --env test2 --host c.local.nocobase.com
nb proxy nginx reload
```

В большинстве случаев работа с `nb proxy` сводится к трём шагам:

1. `use` — выбрать режим работы текущего провайдера.
2. `generate` — сгенерировать конфигурацию входа по окружению и доменному имени.
3. `reload` — применить конфигурацию.

Если используете Caddy, замените в команде `nginx` на `caddy`.

`use local` и `use docker` можно писать так:

- если Nginx или Caddy уже установлены локально — `use local`;
- если локальной установки нет и CLI будет управлять агентом через Docker — `use docker`.

Именно такой опыт даёт этот уровень `nb proxy`: не нужно сразу разбираться в деталях конфигурации Nginx или Caddy — достаточно пройти фиксированный процесс подключения входа.

## Почему сначала достаточно этих трёх шагов

Задача `nb proxy` узкая и понятная: **дать приложению стабильную точку внешнего доступа**.

Если Вы уже читали [Обзор развёртывания производственного окружения](../production/index.md), запомните различие с `nb app autostart` так:

- `nb app autostart` — «как снова запустить приложение после перезагрузки машины»;
- `nb proxy` — «как обеспечить стабильный внешний доступ к приложению через Nginx или Caddy».

В типичном сценарии развёртывания для `nb proxy` не нужно держать в голове много деталей. Чаще всего достаточно:

- выбрать режим работы;
- сгенерировать конфигурацию;
- перезагрузить, чтобы она вступила в силу.

## Что делают эти три шага

### `use`

`use` отвечает на вопрос «как сейчас управлять агентом».

Например:

- `nb proxy nginx use docker`
- `nb proxy nginx use local`

Проще всего ориентироваться так:

- Nginx или Caddy установлены локально — `use local`;
- локальной установки нет — `use docker`.

Можно также считать, что `use` выбирает режим работы провайдера по умолчанию; дальнейшие команды `start`, `reload` и `status` будут работать в этом режиме.

### `generate`

`generate` отвечает на вопрос «сформировать конфигурацию входа для текущего окружения».

Например:

```bash
nb proxy nginx generate --env test2 --host c.local.nocobase.com
```

На этом шаге данные окружения объединяются с параметрами входного слоя и получается готовая конфигурация прокси. Обычно важны:

- `--env` — какое CLI-окружение опубликовать;
- `--host` — к какому доменному имени привязать.

То есть `generate` управляет результатом конфигурации, а не состоянием процесса.

### `reload`

`reload` отвечает на вопрос «сделать только что сгенерированную конфигурацию действительно активной».

```bash
nb proxy nginx reload
```

Разделение полезно: генерация конфигурации и действия с процессом не смешиваются. При смене домена, порта или публичного пути сначала перегенерируйте конфигурацию, затем примените её — так проще понять, что происходит.

## Почему схема `use → generate → reload`

Эти три шага соответствуют трём устойчивым действиям входного слоя:

- сначала решить, как запускать агент;
- затем — какую запись входа сгенерировать для какого окружения;
- наконец — ввести конфигурацию в действие.

Если свернуть всё в одну «чёрный ящик»-команду, снаружи команд меньше, но при сбое трудно понять: неверно выбран режим, ошибка в конфигурации или не сделан `reload`.

После разбиения путь диагностики прямее:

- проблема в `use` — смените режим;
- проблема в `generate` — перегенерируйте конфигурацию;
- конфигурация верная, но не применилась — выполните `reload`.

## Преимущества такого уровня проектирования

Преимущество `nb proxy` не только в унификации команд Nginx и Caddy, но и в том, что частые действия входного слоя складываются в составной процесс.

Например, можно начать с входа агента:

```bash
nb proxy nginx generate --env test2 --reload
```

Или с конфигурации приложения:

```bash
nb env update --env test2 --app-port 13000 --proxy-generate --proxy-reload
```

Это два частых стартовых сценария:

- Вы меняете входной слой — достаточно `generate --reload`;
- сначала меняете окружение, затем запускаете `--proxy-generate --proxy-reload`.

В обоих случаях итоговый процесс один: обновить конфигурацию, сгенерировать вход, применить его на агенте.

## Зачем отдельный `nb proxy`

`nb proxy` унифицирует не конкретный файл конфигурации Nginx, а общие действия входного слоя.

Обычно не так важно:

- по какому пути лежит файл конфигурации;
- чем отличаются команды Nginx и Caddy;
- чем отличается работа в local и docker.

Важнее:

- как опубликовать это окружение;
- как сменить доменное имя;
- как ввести новую конфигурацию в действие.

`nb proxy` сводит эти действия к одному набору точек входа CLI. Запомнив базовый процесс, Вы покроете большинство сценариев; к страницам конкретного провайдера переходите только для отладки или особой настройки.

## Итого

- Базовая модель работы с `nb proxy` — `use → generate → reload`.
- Большинству пользователей достаточно запомнить эти три команды.
- Задача дизайна — не скрыть все детали, а зафиксировать самые частые процессы входного слоя.

Подробнее о командах — в [`nb proxy`](../../api/cli/proxy/index.md). Для подключения к официальному входу см. [Обратный прокси](../production/reverse-proxy/index.md).
