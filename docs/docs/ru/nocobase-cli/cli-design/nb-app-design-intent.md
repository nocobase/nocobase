# Замысел проектирования `nb app`

Команды `nb app` — это по сути адаптации поверх разных способов управления процессами, объединённые в набор стабильных точек входа для управления приложением. Цель — свести повседневную эксплуатацию и обслуживание к одному набору команд.

Сейчас CLI поддерживает в основном такие способы управления процессами приложения:

- Docker
- PM2

Если в будущем понадобится поддержка других способов, например Supervisor, адаптации будут добавляться на этом уровне. Для внешнего мира набор высокочастотных команд остаётся прежним:

```bash
nb app start
nb app restart
nb app logs
nb app upgrade
nb app stop
```

## Зачем объединять всё в `nb app`

Управление процессами можно реализовать по-разному, но большинству пользователей важны не детали нижнего уровня, а конкретные действия: «запустить приложение», «посмотреть лог», «обновить приложение».

Если различия нижнего уровня выставить наружу, пользователю сначала нужно понять, какой режим работы сейчас используется, и запомнить соответствующий набор операций. После объединения в `nb app` эти частые действия сводятся к стабильным точкам входа.

### Снизить порог обучения

Разные решения для управления процессами устроены по-разному:

- у Docker — своя система команд;
- у PM2 — своя система команд;
- у Supervisor — свой способ настройки.

Если показывать эти различия напрямую, придётся освоить несколько способов работы, и в частых сценариях — обновление, перезапуск, разбор логов — легко пропустить важные шаги.

После объединения под `nb app` для повседневного управления в большинстве случаев достаточно одного набора команд.

### Унифицировать бизнес-процессы

Управление жизненным циклом приложения — это не только управление процессом.

При запуске, обновлении и остановке CLI часто выполняет дополнительную логику, например:

- проверку окружения;
- обработку конфигурации;
- миграцию данных;
- обновление версии;
- управление логами.

Если использовать `nb app` как единую точку входа, поведение этих процессов остаётся согласованным. При дальнейшем расширении возможностей не нужно заново осваивать новый интерфейс эксплуатации.

## Зачем нужен `nb app autostart`

После единой точки входа для управления процессом нужен ещё один уровень — **управление автозапуском**, чтобы замкнуть весь цикл. Для этого и существует `nb app autostart`.

Типичное использование:

```bash
# включить автозапуск для текущего окружения
nb app autostart enable

# включить автозапуск для указанного окружения
nb app autostart enable --env app1

# посмотреть статус автозапуска
nb app autostart list

# запустить все окружения с включённым автозапуском
nb app autostart run

# при запуске показать вывод нижнего уровня
nb app autostart run --verbose
```

Этот набор команд сохраняет единообразие снаружи. На уровне `nb app` Вам не нужно знать, что внизу Docker, PM2 или что-то ещё — снаружи интерфейс остаётся таким же:

```bash
nb app autostart enable --env app1
nb app autostart disable --env app1
```

### К чему адаптируется `run`

`nb app autostart` тоже разделён на два уровня ответственности:

- `enable` / `disable` — управляют тем, разрешён ли автозапуск для конкретного окружения;
- `run` — при старте системы поднимает все окружения с включённым автозапуском.

То есть CLI даёт единую точку входа `run` для доступа к механизму автозапуска системы:

```bash
nb app autostart run
```

Здесь адаптируются механизмы запуска системы — `systemd`, `launchd`, скрипты запуска на хосте, — а не менеджеры процессов приложения вроде Supervisor.

## Итого

- Команды `nb app` — слой адаптации поверх разных способов управления процессами; снаружи они унифицированы и снижают путаницу.
- Реализацией могут быть Docker, PM2, Supervisor и др.; сейчас поддерживаются Docker и PM2.
- Поскольку настройки автозапуска у разных способов управления процессами различаются, для завершения цикла нужен единый набор возможностей `nb app autostart`.

Для повседневных операций см. [Управление приложением](../operations/manage-app.md). Если готовы развернуть приложение в производственном окружении, продолжайте с [Обзором развёртывания производственного окружения](../production/index.md).
