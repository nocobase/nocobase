# Управление несколькими окружениями

Если Вы ведёте несколько приложений NocoBase — `dev`, `test`, `staging`, `prod` и т. д., — сохраните каждое как отдельное CLI-окружение. Большинство команд `nb` по умолчанию действуют в текущем окружении, поэтому перед `nb app`, `nb api` и `nb db` важно проверить, какое окружение активно.

В этой версии CLI различает `current env` и `last env`. Обычно достаточно следить за `current env` — окружением, которое использует текущая оболочка или среда выполнения агента. К глобальному `last env` CLI возвращается только если режим сессии не включён.

## Быстрый указатель

| Мне нужно... | Какую команду использовать |
| --- | --- |
| Создать новое локальное окружение и завершить инициализацию | [`nb init`](../../api/cli/init.md) |
| Зарегистрировать существующее приложение как CLI-окружение | [`nb env add`](../../api/cli/env/add.md) |
| Посмотреть, какие окружения сохранены локально | [`nb env list`](../../api/cli/env/list.md) |
| Проверить подключение и аутентификацию всех окружений | [`nb env status --all`](../../api/cli/env/status.md) |
| Переключить окружение для последующих команд | [`nb env use`](../../api/cli/env/use.md) |
| Узнать, в какое окружение попадёт текущая команда | [`nb env current`](../../api/cli/env/current.md) и [`nb env status`](../../api/cli/env/status.md) |
| Просмотреть подробную конфигурацию окружения | [`nb env info`](../../api/cli/env/info.md) |
| Обновить сохранённую конфигурацию и при необходимости пересинхронизировать состояние | [`nb env update`](../../api/cli/env/update.md) |
| Повторно пройти аутентификацию после истечения сессии или сменить способ | [`nb env auth`](../../api/cli/env/auth.md) |
| Удалить неиспользуемое окружение и при необходимости очистить локальные ресурсы | [`nb env remove`](../../api/cli/env/remove.md) |

:::tip Рекомендуется сначала включить режим сессии

По умолчанию сначала выполните [`nb session setup`](../../api/cli/session/setup.md). Тогда разные терминалы, оболочки или среды выполнения агентов сохраняют свой `current env` и не мешают друг другу при параллельной работе.

Если режим сессии не включён, `nb env use` обновляет глобальный `last env`. Тогда переключение окружения в одном терминале может затронуть другой.

```bash
nb session setup
```

:::

## Создание нескольких окружений

Чтобы создать или восстановить локальное приложение, используйте `nb init` — инициализация сохранится в новом CLI-окружении.

```bash
nb init --env dev
nb init --env test
```

Если приложение уже есть и нужно только подключить его к CLI, проще `nb env add`:

```bash
nb env add staging --api-base-url http://staging.example.com/api --auth-type oauth
nb env add prod --api-base-url https://api.example.com/api --auth-type token --access-token <token>
```

Первый сценарий ближе к «инициализации окружения», второй — к «регистрации существующего окружения». Для подключения к уже работающему приложению по умолчанию используйте `nb env add`.

## Просмотр настроенных окружений

Сначала `nb env list` — какие окружения сохранены локально:

```bash
nb env list
```

Команда показывает только конфигурацию, без проверки состояния приложения. Для подключения и аутентификации — `nb env status --all`:

```bash
nb env status --all
```

Обычно встречаются статусы `ok`, `auth failed`, `unreachable`.

## Переключение текущего окружения

```bash
nb env use dev
```

После переключения команды без `--env` по умолчанию используют это окружение.

## Проверка текущего окружения

Если неясно, куда попадёт команда:

```bash
nb env current
nb env status
```

`nb env current` показывает имя, `nb env status` — доступность окружения и нормальность аутентификации.

## Подробности одного окружения

```bash
nb env info dev
nb env info dev --json
nb env info dev --field app.url
nb env info dev --show-secrets
```

`--field` удобен в скриптах для одного значения. `--show-secrets` выводит токены и пароли открытым текстом — только при явной необходимости для отладки.

## Обновление конфигурации окружения

`nb env update` меняет сохранённую конфигурацию: адрес API, способ аутентификации, источник кода, порт приложения, параметры БД. После обновления CLI выполняет последующие шаги по изменениям.

Чтобы CLI пересинхронизировался с последним состоянием текущего окружения:

```bash
nb env update
nb env update prod
```

Чтобы изменить параметры подключения или локальную конфигурацию:

```bash
nb env update prod --api-base-url https://api.example.com/api
nb env update prod --access-token <token>
nb env update dev --app-port 13080 --timezone Asia/Shanghai
```

Краткие правила:

- изменить подключение или локальную конфигурацию окружения — `nb env update`;
- изменился интерфейс приложения, плагины или доступные возможности CLI — снова `nb env update`;
- истекла сессия или нужна повторная аутентификация — `nb env auth`;
- посмотреть, что сохранено сейчас — `nb env info`.

При смене локальных параметров (`app-port`, `timezone`, `db-*`) `update` меняет только сохранённые значения и не перезапускает приложение. Обычно затем выполняют `nb app restart --env <name>`; если затронута встроенная БД, управляемая CLI, — `nb app restart --env <name> --with-db`.

## Повторная аутентификация

Если окружение сохранено, но сессия истекла или нужен другой способ входа:

```bash
nb env auth
nb env auth prod
nb env auth prod --auth-type oauth
nb env auth prod --auth-type basic --username admin --password secret
nb env auth prod --auth-type token --access-token <api-key>
```

Без имени окружения CLI использует текущее. После аутентификации выполняется синхронизация.

## Удаление окружения

Эти сценарии чаще всего путают. Запомните по умолчанию:

- только остановить приложение — `nb app stop`;
- остановить и встроенную БД на этой машине — `nb app stop --with-db`;
- окружение больше не нужно, но `storage` и локальные файлы приложения сохранить — `nb env remove`;
- удалить и локальные размещённые ресурсы — `nb env remove --purge`.

Удалить только сохранённую конфигурацию окружения:

```bash
nb env remove staging
```

Для локального или Docker-окружения, если нужно очистить и работающие ресурсы, и данные в `storage`:

```bash
nb env remove test --purge
```

В неинтерактивном режиме для `nb env remove` явно передайте `--force`:

```bash
nb env remove test --purge --force
```

`--purge` очищает только ресурсы, управляемые CLI на текущей машине. Для удалённого API-окружения сама удалённая служба не удаляется.

Остановить приложение и встроенную БД:

```bash
nb app stop --env app1 --with-db
```

Удалить окружение, сохранив `storage` и локальные файлы:

```bash
nb env remove app1 --force
```

Полностью очистить размещённое содержимое окружения:

```bash
nb env remove app1 --purge --force
```

Для локального npm/Git-окружения, размещённого через CLI, `--purge` также удаляет локальные файлы приложения. Для HTTP- или SSH-окружения удаляется только конфигурация в CLI, внешняя служба не затрагивается.
