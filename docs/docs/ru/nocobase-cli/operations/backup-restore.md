# Резервное копирование и восстановление

Если приложение NocoBase сохранено как CLI-окружение, ежедневное резервное копирование и восстановление выполняются группой команд `nb backup`. `nb backup create` создаёт резервную копию в целевом окружении и скачивает её локально. `nb backup restore` восстанавливает локальный файл резервной копии в целевое окружение.

В большинстве случаев достаточно запомнить совет по умолчанию: резервная копия перед обновлением, миграцией или пакетным изменением данных; восстановление — только когда Вы точно хотите перезаписать текущие данные.

## Быстрый указатель

| Мне нужно... | Какую команду использовать |
| --- | --- |
| Сначала сохранить резервную копию текущего окружения локально | [`nb backup create`](../../api/cli/backup/create.md) |
| Сохранить резервную копию в указанный каталог | [`nb backup create --output ./backups`](../../api/cli/backup/create.md) |
| Передать результат резервного копирования в скрипт | [`nb backup create --json-output`](../../api/cli/backup/create.md) |
| Восстановить локальную резервную копию в текущее окружение | [`nb backup restore --file ./backups/xxx.nbdata --force`](../../api/cli/backup/restore.md) |
| Восстановить локальную резервную копию в другое окружение | [`nb backup restore --env app1 --file ./backups/xxx.nbdata --yes --force`](../../api/cli/backup/restore.md) |

:::tip Сначала подтвердите текущее окружение

Команда `nb backup` по умолчанию действует в текущем окружении. Если одновременно ведётся несколько окружений, перед резервным копированием или восстановлением проверьте текущее окружение.

```bash
nb env current
nb env use app1
```

При явной передаче другого `--env` CLI обычно запрашивает подтверждение. В скриптах и неинтерактивных сценариях добавьте `--yes`, чтобы пропустить этот шаг.

:::

## Создать резервную копию

Самый простой вариант:

```bash
nb backup create
```

После успешного выполнения файл резервной копии скачан локально. Если `--output` не указан, CLI сохраняет файл в текущем рабочем каталоге с именем, возвращённым удалённой стороной — обычно `backup_*.nbdata`.

Чтобы складывать резервные копии в один каталог:

```bash
nb backup create --output ./backups
```

Если `./backups` уже существует и это каталог, CLI добавит к нему имя файла с удалённой стороны. Только если путь не существует, CLI воспримет его как путь к целевому файлу.

Для скриптов, CI или ссылок агента можно добавить `--json-output`:

```bash
nb backup create --env app1 --yes --json-output
```

В этом режиме CLI не выводит текст прогресса, а возвращает итоговый JSON — обычно с полями `env`, `name` и `output`.

## Восстановить резервную копию

Команда восстановления загружает локальный файл резервной копии в целевое окружение и перезаписывает текущие данные приложения:

```bash
nb backup restore --file ./backups/backup_20260520_190408_8397.nbdata --force
```

Чтобы восстановить не в текущее окружение, безопаснее указать его явно:

```bash
nb backup restore --env app1 --file ./backups/backup_20260520_190408_8397.nbdata --yes --force
```

:::warning Примечание

Восстановление — операция полной перезаписи. По умолчанию перед восстановлением рекомендуется сделать ещё одну резервную копию текущего целевого окружения.

```bash
nb backup create --env app1 --yes --output ./backups
nb backup restore --env app1 --file ./backups/backup_20260520_190408_8397.nbdata --yes --force
```

:::

`nb backup restore` сначала проверит, что путь из `--file` существует и это обычный файл. После успешной загрузки CLI дождётся повторной проверки работоспособности приложения — при успешном завершении команды приложение обычно снова доступно.

Без `--force` интерактивный терминал запросит подтверждение. В неинтерактивных терминалах, скриптах и сессиях ИИ-агента нужен `--force`.
