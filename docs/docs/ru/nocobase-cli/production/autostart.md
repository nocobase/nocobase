# Автозапуск приложения

В NocoBase CLI `nb app autostart` управляет тем, **каким окружениям разрешён автозапуск** и **как поднять их после старта системы**.

Если Вы официально запускаете на сервере приложение, размещённое через CLI, это обычно обязательный шаг в производственном окружении.

## Зачем нужен `nb app autostart`

Вопрос встречается часто.

При первом знакомстве кажется: раз внизу уже есть Docker, PM2 или `systemd`, зачем ещё один слой `nb app autostart`.

На самом деле эти уровни решают разные задачи:

- Docker, PM2 и Supervisor — **как обычно работает приложение и как управлять его процессами**;
- `systemd`, `launchd` и скрипты запуска хоста — **какую команду выполнить при старте системы**;
- `nb app autostart` — **на уровне NocoBase CLI: как единообразно разрешить автозапуск окружениям и поднять их после старта системы**.

CLI не отменяет Docker, PM2 или Supervisor. Он унифицирует разные способы управления процессами и сводит автозапуск к одному набору привычных команд — меньше путаницы.

При старте системы управление передаётся `systemd`, `launchd` или скрипту хоста. Они выполняют при загрузке машины:

```bash
nb app autostart run
```

Эта команда поднимает все приложения с включённым автозапуском.

Без этого слоя при смене нижнего способа работы пришлось бы помнить отдельные схемы автозапуска для Docker, PM2 и др. С `nb app autostart` достаточно одного набора привычек NocoBase CLI.

Подробнее о замысле — в [Замысле проектирования `nb app`](../cli-design/nb-app-design-intent.md#зачем-нужен-nb-app-autostart).

## Обязанности этой группы команд

Чаще всего используются:

- `nb app autostart enable`
- `nb app autostart disable`
- `nb app autostart list`
- `nb app autostart run`

Два уровня ответственности:

- `enable` / `disable` — разрешён ли автозапуск для конкретного окружения;
- `run` — поднять при старте системы все окружения с включённым автозапуском.

Включить автозапуск для текущего окружения:

```bash
nb app autostart enable
```

Для другого окружения:

```bash
nb app autostart enable --env app1 --yes
```

Проверить, какие окружения помечены для автозапуска:

```bash
nb app autostart list
```

После старта системы поднять все окружения с автозапуском:

```bash
nb app autostart run
```

При отладке — с выводом нижнего уровня:

```bash
nb app autostart run --verbose
```

Отключить автозапуск для окружения:

```bash
nb app autostart disable --env app1 --yes
```

## Связь с Docker, PM2 и systemd

Границу легко перепутать.

`nb app` отвечает на вопрос «как работает приложение». Нижний уровень может быть Docker, PM2 и т. д.

`nb app autostart` — «как поднять окружения с разрешённым автозапуском после загрузки машины». Это стабильная точка входа для механизма запуска хоста, а не замена конкретного менеджера процессов.

Итого:

- Docker, PM2, Supervisor — ближе к работе приложения;
- `systemd`, `launchd`, скрипты хоста — ближе к старту системы.

В производственном окружении обычно подключают `nb app autostart run` к своему процессу загрузки: `systemd`, `launchd`, скриптам платформы контейнеров или другому уже используемому автозапуску хоста.

## Область применения

`nb app autostart` только для окружений со средой выполнения, управляемой CLI:

- `local`
- `docker`

Если окружение — только удалённое API-подключение или приложение не работает под управлением CLI на этой машине, этот набор команд для автозапуска не подходит.

## Рекомендуемая последовательность

В большинстве случаев достаточно:

1. Убедиться, что приложение нормально запускается на этой машине.
2. Выполнить `nb app autostart enable --env <name> --yes`.
3. Подключить `nb app autostart run` к загрузке системы.
4. Перезагрузить машину или вручную выполнить `run` и проверить восстановление.

Для настройки входного уровня производственного окружения см. [Обратный прокси](./reverse-proxy/index.md).
