# Установка

Единая система управления бизнесом охватывает шесть модулей: **CRM, Управление продажами, Help Desk, Управление проектами, Управление активами, HR / Управление персоналом**. Доступны два способа восстановления — выбирайте подходящий, исходя из вашей версии NocoBase и технического опыта.

:::tip Предварительные требования

- Базовое окружение NocoBase уже развёрнуто. По установке основного приложения см. [официальную документацию](https://docs-cn.nocobase.com/welcome/getting-started/installation)
- Версия NocoBase — **v2.1.0-alpha.34 или выше**
- Скачан один из файлов единого решения:
  - **Резервная копия nbdata**: [nocobase_all_in_one_backup_260521.nbdata](https://static-docs.nocobase.com/nocobase_all_in_one_backup_260521.nbdata) — для метода 1
  - **SQL-архив**: [nocobase_all_in_one_sql_260521.zip](https://static-docs.nocobase.com/nocobase_all_in_one_sql_260521.zip) — для метода 2

:::

:::warning Внимание

- Решение подготовлено на базе **PostgreSQL 16**, окружение должно использовать именно эту версию
- **Переменная `DB_UNDERSCORED` не должна быть установлена в `true`** — проверьте `docker-compose.yml`: если значение `true`, восстановление завершится ошибкой

:::

Если у вас есть плагин «Менеджер резервных копий» — выбирайте метод 1, иначе метод 2. В текущей версии развёртывание идёт через **восстановление из резервной копии**, в последующих версиях планируется переход на инкрементальную миграцию, что упростит интеграцию решения в уже работающие установки NocoBase.

## Метод 1: Восстановление через Backup Manager (рекомендуется для Pro/Enterprise)

Этот вариант использует встроенный плагин «[Менеджер резервных копий](https://docs-cn.nocobase.com/handbook/backups)» и позволяет восстановить систему одним действием прямо из интерфейса. К окружению и набору плагинов предъявляются достаточно строгие требования.

### Особенности

**Преимущества:**

- **Простые действия** — всё выполняется в UI, восстанавливается вся конфигурация, включая плагины
- **Полное восстановление** — переносятся все системные файлы, включая шаблоны печати, файлы из полей вложений, аватары AI-сотрудников и т. п.

**Ограничения:**

- **Только Pro/Enterprise** — «Менеджер резервных копий» является корпоративным плагином и недоступен в Community
- **Жёсткие требования к окружению** — версия БД, регистрозависимость и прочие параметры должны быть в высокой степени совместимы с исходными
- **Сильная зависимость от плагинов** — все коммерческие плагины из резервной копии должны присутствовать локально, иначе восстановление прервётся

### Шаги

**Шаг 1: Запустить приложение из образа `full`**

Настоятельно рекомендуется использовать `full`-вариант Docker-образа — в нём уже есть клиент БД и прочие утилиты, дополнительная настройка не нужна:

```bash
docker pull nocobase/nocobase:alpha-full
```

После этого запускайте NocoBase из этого образа.

:::tip

При использовании другого образа потребуется вручную устанавливать клиент `pg_dump` внутри контейнера — процесс трудоёмкий и нестабильный.

:::

**Шаг 2: Включить плагин «Менеджер резервных копий»**

1. Войдите в NocoBase
2. Перейдите в **Управление плагинами**
3. Найдите и включите плагин **Менеджер резервных копий**

**Шаг 3: Восстановить из локального файла**

1. После включения плагина обновите страницу
2. В боковом меню перейдите в **Системное администрирование / Менеджер резервных копий**
3. Нажмите **Восстановить из локальной резервной копии** в правом верхнем углу
4. Перетащите скачанный файл `nocobase_all_in_one_backup_260521.nbdata` в область загрузки
5. Нажмите **Отправить** и дождитесь завершения восстановления — это может занять от нескольких десятков секунд до нескольких минут

### Замечания

- **Совместимость БД** — версия PostgreSQL, кодировка и настройки регистрозависимости должны совпадать с исходными; особенно важно совпадение имени `schema`
- **Соответствие коммерческих плагинов** — все используемые в резервной копии коммерческие плагины должны быть заранее установлены и включены локально, иначе восстановление прервётся. В единое решение входят: «Менеджер резервных копий», «Управление почтой», «Журнал аудита», «AI-сотрудники» и др.

## Метод 2: Прямой импорт SQL-файла (универсальный)

Этот вариант работает напрямую с базой данных, обходя «Менеджер резервных копий», и не имеет ограничений по версии и набору плагинов.

### Особенности

**Преимущества:**

- **Без ограничений по версии** — подходит всем пользователям NocoBase, включая Community
- **Высокая совместимость** — не зависит от внутренних утилит приложения, достаточно подключения к БД
- **Высокая устойчивость** — если каких-то коммерческих плагинов нет локально, они просто не активируются, остальные модули работают штатно

**Ограничения:**

- **Нужны базовые навыки работы с БД** — например, умение выполнить `.sql`-файл
- **Потеря системных файлов** — этот способ не переносит системные файлы, включая шаблоны печати, файлы из полей вложений, аватары AI-сотрудников и т. п.

### Шаги

**Шаг 1: Подготовить чистую базу данных**

Создайте новую пустую базу данных (PostgreSQL 16) для предстоящего импорта.

**Шаг 2: Импортировать `.sql`-файл в БД**

Распакуйте скачанный `nocobase_all_in_one_sql_260521.zip` и получите `.sql`-файл, затем импортируйте его в подготовленную базу. Доступны два варианта выполнения:

**Вариант A: Командная строка сервера (пример с Docker)**

Если NocoBase и БД развёрнуты в Docker, загрузите `.sql`-файл на сервер и выполните импорт через `docker exec`. Допустим, контейнер PostgreSQL называется `my-nocobase-db`:

```bash
# Скопировать SQL-файл внутрь контейнера
docker cp nocobase_all_in_one_sql_260521.sql my-nocobase-db:/tmp/
# Выполнить импорт внутри контейнера
docker exec -it my-nocobase-db psql -U nocobase -d nocobase -f /tmp/nocobase_all_in_one_sql_260521.sql
```

**Вариант B: Графический клиент БД (Navicat и т. п.)**

Если порт БД доступен извне, используйте любой графический клиент (Navicat, DBeaver, pgAdmin и т. п.):

1. Кликните правой кнопкой по нужной базе данных
2. Выберите «Выполнить SQL-файл» или «Выполнить SQL-скрипт»
3. Укажите распакованный `.sql`-файл и запустите выполнение

**Шаг 3: Подключить БД и запустить приложение**

Настройте параметры запуска NocoBase (`DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USER`, `DB_PASSWORD` и т. д.), чтобы приложение указывало на базу с импортированными данными, и запустите NocoBase обычным образом.

### Замечания

- **Права на БД** — для этого способа нужны логин и пароль с полным доступом к базе
- **Состояние плагинов** — после импорта данные коммерческих плагинов остаются в системе, но если соответствующие плагины не установлены локально, их функции будут недоступны. Приложение при этом не падает

## Сравнение двух методов

| Параметр | Метод 1: Менеджер резервных копий | Метод 2: Прямой импорт SQL |
| :--- | :--- | :--- |
| **Для каких пользователей** | Pro/Enterprise | Все пользователи (включая Community) |
| **Простота операций** | ⭐⭐⭐⭐⭐ (через UI) | ⭐⭐⭐ (нужны знания БД) |
| **Требования к окружению** | Жёсткие, нужна высокая совместимость БД и версии системы | Умеренные, достаточно совместимости БД |
| **Зависимость от плагинов** | Сильная, отсутствие плагина приводит к сбою восстановления | Данные импортируются независимо, без плагинов их функции просто недоступны |
| **Системные файлы** | Сохраняются полностью (шаблоны печати, загруженные файлы, аватары и т. п.) | Не переносятся (шаблоны печати, загруженные файлы, аватары и т. п.) |
| **Рекомендуется когда** | Корпоративные пользователи, управляемое окружение | Часть плагинов отсутствует, нужна высокая совместимость, Community |

## Обязательная настройка после установки

После восстановления система уже открывается, но **две настройки указывают на наше демо-окружение** и требуют замены на ваши собственные.

### 1. Движок хранилища файлов (OSS / локально)

В демо-резервной копии хранилище по умолчанию указывает на наш демонстрационный Alibaba Cloud OSS — Access Key наружу не публикуется, поэтому любая загрузка вложений, шаблонов печати и аватаров AI-сотрудников завершится ошибкой.

В большинстве случаев достаточно переключиться на локальное хранилище; собственный OSS нужен только при необходимости CDN-ускорения или работы с крупными файлами.

**Шаги переключения:**

1. Перейдите в **Управление плагинами / Менеджер файлов** (или откройте `/admin/settings/file-manager` напрямую)

2. **Вариант A — локальное хранилище** (самый простой, подходит для самостоятельного развёртывания):

   - Найдите автоматически созданную запись «Local Storage (локальное хранилище)»
   - Нажмите **Изменить**, в нижней части панели настроек отметьте «Использовать как хранилище по умолчанию» и нажмите **Отправить**

   ![Общие настройки движка хранения (внизу — «Использовать как хранилище по умолчанию»)](https://static-docs.nocobase.com/20240529115151.png)

   :::warning Внимание

   При развёртывании в Docker локальное хранилище находится внутри контейнера; при удалении контейнера файлы будут потеряны. Для production рекомендуется монтировать volume или использовать облачное хранилище.

   :::

3. **Вариант B — собственный OSS / S3 / COS**:

   - Нажмите **Добавить новое**, выберите нужный тип (Alibaba Cloud OSS / Amazon S3 / Tencent Cloud COS / S3 Pro)
   - Заполните Access Key, Bucket, Region, домен и прочие параметры, отметьте «Использовать как хранилище по умолчанию» и нажмите **Отправить**

   ![Пример настройки движка хранения Alibaba Cloud OSS](https://static-docs.nocobase.com/20240712220011.png)

4. Удалите или отключите преднастроенную в демо запись OSS, чтобы избежать ошибочного использования

Подробное описание параметров см. в [Обзоре движка хранилища](../../file-manager/storage/index.md).

### 2. Ключи LLM-сервисов для AI Employees

В демо-резервной копии предустановлены несколько записей LLM-сервисов (OpenAI, Claude, Gemini, DeepSeek, Qwen, Kimi и др.), в них прописаны наши API Key, **которые не работают снаружи**. До замены ключей функционал AI-сотрудников недоступен.

**Шаги настройки:**

1. Перейдите в **Системные настройки / AI Employees / LLM service** (или откройте `/admin/settings/ai/llm-services`)

   ![Переход на страницу настройки LLM service](https://static-docs.nocobase.com/00_QuickStart_cn-2025-09-29-15-40-47.png)

2. В списке предустановленных сервисов записи можно сортировать перетаскиванием, переключателем `Enabled` — включать и отключать

   ![Список LLM-сервисов (включение и сортировка)](https://static-docs.nocobase.com/ai-employees/2026-02-14/llm-service-list-enabled-and-sort.png)

3. Для каждой записи, которую планируете использовать:

   - Нажмите **Изменить**
   - Замените **API Key** на ваш собственный ключ (получите его в личном кабинете соответствующего провайдера: OpenAI, Anthropic, Google AI Studio, DeepSeek, Qwen, Kimi и др.)
   - Если используется прокси или внутренний шлюз, скорректируйте **Base URL**
   - В **Enabled Models** оставьте только те модели, которые планируете использовать, остальные можно убрать

   ![Изменение LLM-сервиса (API Key, Base URL, Enabled Models)](https://static-docs.nocobase.com/00_QuickStart_cn-2025-09-29-15-45-27.png)

4. Нажмите **Test flight** внизу для проверки подключения; после успешной проверки нажмите **Submit** для сохранения

   ![Test flight — проверка подключения](https://static-docs.nocobase.com/00_QuickStart_cn-2025-09-29-16-18-25.png)

5. Неиспользуемые сервисы достаточно перевести в **Disabled**, удалять необязательно

Подробную настройку см. в [Настройке LLM-сервиса](../../ai-employees/features/llm-service.md).

:::tip

Эти две настройки обязательны после восстановления демо. Прочие параметры (логотип сайта, SMTP, корпоративные плагины и т. п.) подстраиваются по необходимости.

:::

## Часто задаваемые вопросы

### Подойдёт ли Pro-версия? Не будет ли ошибок?

Подойдёт, ошибок не будет. В демо использованы некоторые Enterprise-плагины («Управление почтой», «Журнал аудита», «AI-сотрудники» и др.); в Pro эти плагины отсутствуют, и соответствующие пункты меню просто не появятся, остальные модули не пострадают. Например, исчезнет раздел «Журнал аудита», но CRM, продажи, тикеты, проекты, активы, HR и другие основные модули будут работать как обычно.

### Какую версию выбрать после восстановления?

Рекомендуется последний `alpha-full`-образ (`nocobase/nocobase:alpha-full`). Образ `full` включает клиент БД и другие зависимости, что позволяет избежать ошибок восстановления из-за отсутствующих утилит.

### После восстановления не отображается логотип?

Логотип на демо-сайте привязан к домену, поэтому на локальной установке он не загружается. Зайдите в **Системные настройки** и загрузите собственный логотип.

### Ошибка при загрузке файлов (неверный ключ OSS)?

После установки через SQL загрузка файлов может выдавать ошибку, связанную с OSS. Перейдите в **Управление плагинами / Менеджер файлов**, установите «Local Storage (локальное хранилище)» в качестве хранилища по умолчанию и сохраните — после этого загрузка пойдёт штатно.

Подробности см. выше в разделе [Движок хранилища файлов](#1-движок-хранилища-файлов-oss--локально).

### Как переключить язык?

В единое решение уже включена локализация на 20+ языков (пространство имён `nb_demo`). После восстановления по умолчанию установлен китайский; для переключения зайдите в **Системные настройки** и включите нужный язык.

### Как делать инкрементальные обновления?

В текущей версии обновление выполняется полной заменой, ваши доработки будут перезаписаны. Перед обновлением обязательно делайте резервную копию. Решение для инкрементальной миграции в планах и в первую очередь будет доступно в Pro/Enterprise. В Community поддержка осложнена отсутствием плагина управления миграциями.
