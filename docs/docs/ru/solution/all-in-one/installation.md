# Установка

> Текущая версия разворачивается через **восстановление из резервной копии**. В последующих версиях возможен переход на **инкрементальную миграцию**, что упростит интеграцию решения в уже работающие установки NocoBase.

Единая система управления бизнесом охватывает шесть модулей: **CRM (Управление клиентами), Управление продажами, Help Desk (Тикеты), Управление проектами, Управление активами и HR / Управление персоналом**. Чтобы вы могли быстро и без сбоев развернуть решение в своей среде NocoBase, доступны два способа восстановления. Выберите подходящий с учётом версии вашего тарифа и технического опыта.

Перед началом убедитесь, что:

- У вас уже развёрнуто базовое окружение NocoBase. По установке основного приложения см. [официальную документацию](https://docs-cn.nocobase.com/welcome/getting-started/installation).
- Версия NocoBase — **v2.1.0-alpha.34 или выше**.
- Скачаны соответствующие файлы единого решения:
  - **Файл резервной копии**: [nocobase_all_in_one_backup_260521.nbdata](https://static-docs.nocobase.com/nocobase_all_in_one_backup_260521.nbdata) — для способа 1.
  - **SQL-файл**: [nocobase_all_in_one_sql_260521.zip](https://static-docs.nocobase.com/nocobase_all_in_one_sql_260521.zip) — для способа 2.

**Важные замечания**:

- Решение подготовлено на базе **PostgreSQL 16** — убедитесь, что в вашей среде используется именно эта версия.
- **Переменная `DB_UNDERSCORED` не должна быть установлена в `true`**: проверьте `docker-compose.yml` и убедитесь, что эта переменная окружения не равна `true`, иначе восстановление завершится ошибкой из-за конфликта с резервной копией.

## Способ 1. Восстановление через менеджер резервных копий (рекомендуется для Pro/Enterprise)

Этот вариант использует встроенный плагин «[Менеджер резервных копий](https://docs-cn.nocobase.com/handbook/backups)» (Pro/Enterprise) и позволяет восстановить систему в один шаг прямо из интерфейса. Однако к окружению и тарифу предъявляются определённые требования.

### Особенности

* **Преимущества**:
  1. **Просто в работе**: всё делается через UI, восстанавливается вся конфигурация, включая плагины.
  2. **Полное восстановление**: **восстанавливаются все системные файлы**, включая шаблоны печати, файлы из полей «Файл» в таблицах, аватары AI-сотрудников и т. п.
* **Ограничения**:
  1. **Только для Pro/Enterprise**: «Менеджер резервных копий» — корпоративный плагин, доступный только в Pro/Enterprise.
  2. **Жёсткие требования к окружению**: версия БД, регистрозависимость и прочие настройки должны быть в высокой степени совместимы с теми, в которых формировалась резервная копия.
  3. **Зависимость от плагинов**: если в решении используются коммерческие плагины, которых нет в вашей установке, восстановление завершится ошибкой.

### Шаги

**Шаг 1. [Настоятельно рекомендуется] Запускать приложение из образа `full`**

Чтобы избежать ошибок из-за отсутствия клиента БД, настоятельно рекомендуется использовать `full`-вариант Docker-образа. В нём уже есть все необходимые утилиты, дополнительная настройка не требуется.

Пример команды для загрузки образа:

```bash
docker pull nocobase/nocobase:alpha-full
```

После этого запускайте NocoBase из этого образа.

> **Примечание**: при использовании другого образа потребуется вручную устанавливать клиент PostgreSQL (`pg_dump`) внутри контейнера — процесс трудоёмкий и нестабильный.

**Шаг 2. Включить плагин «Менеджер резервных копий»**

1. Войдите в NocoBase.
2. Перейдите в **`Управление плагинами`**.
3. Найдите и включите плагин **`Менеджер резервных копий`**.

**Шаг 3. Восстановить из локального файла**

1. После включения плагина обновите страницу.
2. В боковом меню перейдите: **`Системное администрирование`** → **`Менеджер резервных копий`**.
3. Нажмите **`Восстановить из локальной резервной копии`** в правом верхнем углу.
4. Перетащите скачанный файл `nocobase_all_in_one_backup_260521.nbdata` в область загрузки.
5. Нажмите **`Отправить`** и дождитесь завершения восстановления — это может занять от нескольких десятков секунд до нескольких минут.

### Важные нюансы

* **Совместимость БД** — критически важный момент. Версия PostgreSQL, кодировка, регистрозависимость и параметры базы должны соответствовать исходным; особенно важно, чтобы совпадало имя `schema`.
* **Совместимость коммерческих плагинов**: убедитесь, что у вас приобретены и включены все коммерческие плагины, требуемые решением, иначе восстановление прервётся. В единое решение входят: «Менеджер резервных копий», «Управление почтой», «Журнал аудита», «AI-сотрудники» и др.

## Способ 2. Прямой импорт SQL-файла (универсальный, подходит для Community)

Этот вариант работает напрямую с базой данных, минуя плагин «Менеджер резервных копий», и не требует Pro/Enterprise.

### Особенности

* **Преимущества**:
  1. **Без ограничений по тарифу**: подходит всем пользователям NocoBase, включая Community.
  2. **Высокая совместимость**: не зависит от внутренних утилит приложения — достаточно подключения к БД.
  3. **Высокая устойчивость**: если в решении есть коммерческие плагины, которых у вас нет, соответствующие функции просто не активируются, но остальной функционал работает штатно, приложение запускается без ошибок.
* **Ограничения**:
  1. **Нужны базовые навыки работы с БД**: вы должны уметь выполнить `.sql`-файл.
  2. **Потеря системных файлов**: **этот способ не переносит системные файлы**, включая шаблоны печати, файлы из полей «Файл» в таблицах, аватары AI-сотрудников и т. п.

### Шаги

**Шаг 1. Подготовить чистую базу данных**

Создайте новую пустую базу данных (PostgreSQL 16) для импорта.

**Шаг 2. Импортировать `.sql` в БД**

Скачайте `nocobase_all_in_one_sql_260521.zip`, распакуйте его, получите `.sql`-файл и импортируйте его в подготовленную базу. Способ зависит от вашего окружения:

* **Вариант A. Через командную строку сервера (пример с Docker)**

  Если NocoBase и PostgreSQL установлены в Docker, загрузите `.sql`-файл на сервер и выполните импорт через `docker exec`. Допустим, контейнер PostgreSQL называется `my-nocobase-db`:

  ```bash
  # Скопировать SQL-файл внутрь контейнера
  docker cp nocobase_all_in_one_sql_260521.sql my-nocobase-db:/tmp/
  # Выполнить импорт внутри контейнера
  docker exec -it my-nocobase-db psql -U nocobase -d nocobase -f /tmp/nocobase_all_in_one_sql_260521.sql
  ```

* **Вариант B. Через графический клиент БД (Navicat и т. п.)**

  Если порт БД доступен извне, используйте любой графический клиент (Navicat, DBeaver, pgAdmin и т. п.):

  1. Кликните правой кнопкой по нужной базе данных.
  2. Выберите «Выполнить SQL-файл» или «Выполнить SQL-скрипт».
  3. Укажите скачанный `.sql`-файл и запустите выполнение.

**Шаг 3. Подключить БД и запустить приложение**

Настройте параметры запуска NocoBase (переменные `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USER`, `DB_PASSWORD` и т. д.), чтобы приложение указывало на базу с только что импортированными данными, и запустите NocoBase обычным образом.

### Важные нюансы

* **Права на БД**: для этого способа нужны логин и пароль с полным доступом к базе.
* **Состояние плагинов**: после импорта данные коммерческих плагинов остаются в системе, но если соответствующие плагины не установлены и не включены локально, эти функции просто не будут видны и недоступны — приложение при этом не падает.

## Частые вопросы

### Подойдёт ли Pro-версия? Не будет ли ошибок?

Подойдёт, ошибок не будет. В демо использованы некоторые Enterprise-плагины (например, «Управление почтой», «Журнал аудита», «AI-сотрудники»). В Pro эти плагины отсутствуют — соответствующие пункты меню просто не появятся, но **остальные модули системы не пострадают**. Например, исчезнет раздел «Журнал аудита», но CRM, продажи, тикеты, проекты, активы, HR и другие основные модули будут работать как обычно.

### Какую версию выбрать после восстановления?

Рекомендуется последний `alpha-full`-образ (например, `nocobase/nocobase:alpha-full`). Образ `full` включает в себя клиент БД и другие зависимости, что снимает большинство ошибок восстановления, связанных с отсутствием утилит.

### После восстановления не отображается логотип?

Логотип на демо-сайте привязан к домену, поэтому на локальной установке он не загружается. Зайдите в **Системные настройки** и загрузите собственный логотип.

### Ошибка при загрузке файлов (ключ OSS)?

После установки через SQL загрузка файлов может выдавать ошибку, связанную с OSS. Решение: **Управление плагинами → Менеджер файлов**, установите **Local Storage** (локальное хранилище) в качестве хранилища по умолчанию и сохраните — после этого загрузка пойдёт штатно.

### Переключение языка?

В единое решение включена локализация на 20+ языков (пространство имён `nb_demo`). После восстановления по умолчанию установлен китайский; чтобы переключиться на другой язык — **Системные настройки → Включить нужный язык** (избегайте включения `ar-SA`: сейчас это вызывает аномалии при рендеринге NocoBase).

### Как делать инкрементальные обновления?

В текущей версии обновление выполняется полной заменой, ваши доработки будут перезаписаны. Перед обновлением обязательно делайте резервную копию. Решение для инкрементальной миграции в планах и в первую очередь будет доступно в Pro/Enterprise. В Community поддержка осложнена отсутствием плагина управления миграциями.

Надеемся, эта инструкция поможет успешно развернуть единую систему управления бизнесом. Если в процессе возникнут вопросы — свяжитесь с нами.
