# Guia de instalação e migração de upgrade: usando o NocoBase CLI

:::danger
As capacidades de instalação e manutenção de aplicações do CLI ainda estão em desenvolvimento. No momento, ele é mais adequado para desenvolvimento local, ambientes de teste e cenários de integração com AI Agent, e não é recomendado para instalação, upgrade e operação/manutenção diretamente em ambientes de produção.
:::

A nova versão do NocoBase CLI (`nb`) unifica operações como instalar, conectar, iniciar, parar, visualizar logs, fazer upgrade e limpar em um único conjunto de comandos. Em comparação com a documentação antiga, que mantinha fluxos separados para Docker, create-nocobase-app e código-fonte Git, o CLI é mais adequado para desenvolvimento local, manutenção de ambientes de teste, além de integração e colaboração com AI Agent.

Este artigo é aplicável aos seguintes cenários:

- Você está instalando ou atualizando o NocoBase com base na documentação antiga e quer migrar para a nova abordagem com CLI.
- Você já tem um NocoBase implantado de uma forma antiga e quer que o CLI o gerencie ou se conecte a ele.
- Você precisa preparar um ambiente NocoBase conectável e mantível para AI Agent.

## Diferenças entre a forma antiga e a nova

A forma antiga divide a documentação pela origem da instalação, e cada origem exige a execução separada de comandos de download, configuração, instalação, inicialização e upgrade.

**Documentação antiga de instalação**

- [Instalação com Docker](/get-started/installation/docker)
- [Instalação com create-nocobase-app](/get-started/installation/create-nocobase-app)
- [Instalação por código-fonte Git](/get-started/installation/git)

**Documentação antiga de upgrade**

- [Upgrade de instalação com Docker](/get-started/upgrading/docker)
- [Upgrade de instalação com create-nocobase-app](/get-started/upgrading/create-nocobase-app)
- [Upgrade de instalação por código-fonte Git](/get-started/upgrading/git)

A nova abordagem usa `nb init` como ponto de entrada e registra em um CLI env a origem da aplicação, diretório de execução, diretório de armazenamento, banco de dados e informações de conexão de API. Depois disso, independentemente de a origem ser Docker, npm ou Git, a manutenção deve usar, tanto quanto possível, o mesmo conjunto de comandos `nb app`, `nb source` e `nb db`.

| Cenário                                | Forma antiga                                                    | Nova forma com CLI                                |
| -------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------- |
| Entrada de instalação                  | `docker compose`, `create-nocobase-app`, `git clone`            | `nb init` ou `nb init --ui`                       |
| Gerenciamento de execução da aplicação | Trocar diretório, modificar compose, executar comandos yarn/pm2 | `nb app start/stop/restart/logs`, `nb env remove` |
| Upgrade                                | Um conjunto de comandos para cada forma de instalação           | `nb app upgrade -e <env>`                         |
| Modo de desenvolvimento                | `yarn dev`                                                      | `nb source dev -e <env>`                          |
| Estado do banco de dados               | Verificar manualmente contêineres ou serviços de banco          | `nb db ps -e <env>`                               |
| Ver status e detalhes do ambiente      | Registrar manualmente caminhos, portas e endereços de acesso    | `nb env list`, `nb env info <env>`                |
| Conectar aplicação existente           | A documentação antiga não cobria isso de forma unificada        | `nb init --ui` ou `nb init --api-base-url`        |

## Instalar o CLI

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Após a instalação, você pode ver a ajuda:

```bash
nb --help
nb init --help
```

## Instalar o NocoBase usando a nova abordagem

### Recomendado: usar o assistente visual

```bash
nb init --ui
```

O assistente vai guiá-lo para escolher:

- Criar uma nova aplicação ou conectar uma aplicação existente
- Origem da instalação: Docker, npm, Git
- Versão do NocoBase
- Usar banco de dados embutido ou banco de dados externo
- Conta e senha do administrador

### Usar o assistente interativo no terminal

```bash
nb init
```

### Usar comando não interativo

O método de instalação padrão é Docker, usando o banco de dados PostgreSQL embutido:

```bash
nb init --yes --env app1
```

Instalar uma versão específica com Docker:

```bash
nb init --yes --env app1 --source docker --version beta
```

Instalar com npm:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Instalar com código-fonte Git:

```bash
nb init --yes --env app1 --source git --version beta
```

Usar código-fonte Git e especificar o tipo de banco de dados embutido:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Conectar a uma aplicação NocoBase existente:

```bash
# Usa autenticação OAuth por padrão
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Usa autenticação por token
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` é o nome do ambiente no CLI. Depois, comandos como iniciar, fazer upgrade e visualizar logs podem usar `-e app1` para especificar a aplicação de destino.
:::

## Fazer upgrade usando a nova abordagem

Na forma antiga, os comandos de upgrade para Docker, create-nocobase-app e código-fonte Git eram diferentes; o novo CLI os unifica em:

```bash
nb app upgrade -e app1
```

Se você quiser apenas executar o processo de upgrade usando o código-fonte ou imagem já baixados, sem obter novo código ou imagem:

```bash
nb app upgrade -e app1 --skip-code-update
```

Ou usar a forma abreviada:

```bash
nb app upgrade -e app1 -s
```

## Comandos comuns de manutenção

| Operação                                                   | Comando                              |
| ---------------------------------------------------------- | ------------------------------------ |
| Iniciar aplicação                                          | `nb app start -e app1`               |
| Parar aplicação                                            | `nb app stop -e app1`                |
| Reiniciar aplicação                                        | `nb app restart -e app1`             |
| Ver logs                                                   | `nb app logs -e app1`                |
| Ver env e estado de autenticação da API                    | `nb env list`                        |
| Ver detalhes do env                                        | `nb env info app1`                   |
| Ver estado do banco de dados embutido                      | `nb db ps -e app1`                   |
| Fazer upgrade da aplicação                                 | `nb app upgrade -e app1`             |
| Parar aplicação e banco de dados embutido                  | `nb app stop -e app1 --with-db`      |
| Remover completamente env e recursos hospedados localmente | `nb env remove app1 --purge --force` |

Se você só quiser parar a aplicação junto com o banco de dados embutido hospedado pelo CLI, use diretamente `nb app stop -e app1 --with-db`.

Se tiver certeza de que esse env não é mais necessário, use então `nb env remove app1 --purge --force`. Isso removerá recursos de execução hospedados, dados de storage e, quando aplicável, arquivos locais da app baixados e gerenciados pelo CLI.

## Comandos de desenvolvimento diário

`nb source` é usado para gerenciar projetos locais de código-fonte, principalmente em envs npm e Git source.

| Operação                                        | Comando                                             |
| ----------------------------------------------- | --------------------------------------------------- |
| Executar a aplicação em modo de desenvolvimento | `nb source dev -e app1`                             |
| Compilar o código-fonte                         | `nb source build --cwd /your/workspace/app1/source` |
| Executar testes                                 | `nb source test --cwd /your/workspace/app1/source`  |
| Baixar ou atualizar código-fonte/imagem         | `nb source download --source git --version beta`    |

:::tip
Ambientes Docker normalmente são gerenciados com `nb app start`, `nb app logs` e `nb app upgrade`, e não usam `nb source dev`.
:::

## NB_CLI_ROOT e estrutura de diretórios

O CLI salva a configuração global e os arquivos da aplicação local no diretório correspondente a `NB_CLI_ROOT`.

Por padrão, `NB_CLI_ROOT` é o diretório home do usuário atual, isto é, o caminho retornado por `os.homedir()` do Node.js. Por exemplo, no macOS normalmente é `/Users/<user>`. Portanto, sem configuração adicional, os arquivos gerados por `nb init --yes --env app1` geralmente ficam em:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Você também pode especificar `NB_CLI_ROOT` explicitamente:

```bash
export NB_CLI_ROOT=/your/workspace
```

Depois de definido, o arquivo de configuração e o diretório padrão da aplicação serão colocados nesse diretório:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Onde:

- `.nocobase/config.json` salva informações como env, ambiente atual, endereço da API, caminho do código-fonte, caminho do storage e configuração do banco de dados.
- `<envName>/source` é o diretório de código-fonte ou aplicação gerenciado pelo CLI.
- `<envName>/storage` é o diretório de armazenamento da aplicação gerenciado pelo CLI.

:::warning
Depois de inicializar a mesma aplicação, mantenha o uso do mesmo `NB_CLI_ROOT`. Se você mudar `NB_CLI_ROOT` depois, o CLI procurará `.nocobase/config.json` no novo diretório, e caminhos relativos como `appRootPath` e `storagePath` também serão resolvidos com base nesse novo diretório, o que pode fazer com que a aplicação não seja encontrada ou que leitura/escrita ocorra no diretório errado.
:::

Atualmente o CLI usa apenas o scope de configuração global. Se precisar ajustar onde a configuração e os arquivos locais da aplicação são salvos, defina `NB_CLI_ROOT`.

Ao visualizar e localizar aplicações, priorize `nb env list` e `nb env info`:

```bash
# Ver envs configurados e estado de autenticação da API
nb env list

# Ver informações detalhadas de um env
nb env info app1
```

`nb env list` é voltado para uma visão geral rápida. Nele, `App Status` representa o estado de autenticação obtido após acessar a API da aplicação com as credenciais Token/OAuth salvas, e não exibe mais o estado do banco de dados. Para ver o estado de execução do banco de dados embutido, use:

```bash
nb db ps -e app1
```

Use o seguinte apenas quando precisar trocar o env padrão:

```bash
nb env use app1
```

## Migrar da forma antiga para o CLI

Há duas estratégias de migração: **conectar a uma aplicação existente** ou **trazer um diretório de aplicação local para a gestão do CLI**. Se sua aplicação antiga já estiver estável em execução, é recomendável usar primeiro “conectar a uma aplicação existente”, que tem o menor risco.

### Método 1: conectar a uma aplicação existente

Adequado para NocoBase já em execução via Docker, create-nocobase-app, código-fonte Git ou implantação em servidor.

```bash
# Usa autenticação OAuth por padrão
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Usa autenticação por token
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Essa abordagem salva apenas a conexão com a API. O CLI não assumirá o controle de iniciar, parar, fazer upgrade, nem do diretório de código-fonte ou do diretório de storage da aplicação antiga. É adequada para permitir que AI Agent ou comandos de API do CLI se conectem a uma aplicação existente.

Se precisar renovar a autenticação:

```bash
nb env auth app1
```

### Método 2: criar um novo CLI env e depois migrar os arquivos

Adequado para cenários em que você quer usar `nb app` e `nb source` para gerenciar a aplicação local posteriormente.

Antes da migração, conclua primeiro:

1. Pare a aplicação antiga.
2. Faça backup do banco de dados antigo.
3. Faça backup do diretório antigo `storage`.
4. Registre as variáveis de ambiente principais da aplicação antiga, como `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Migração de instalação Docker antiga

A forma antiga com Docker normalmente armazena os dados da aplicação em `storage` dentro do diretório do projeto:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Recomenda-se definir primeiro o novo `NB_CLI_ROOT` e então usar o CLI para criar um novo Docker env. O exemplo abaixo assume que `NB_CLI_ROOT` é `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

O novo env usará por padrão os seguintes diretórios:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Em seguida, copie o `storage` antigo para `/your/workspace/app2/storage` e certifique-se de que a conexão com o banco de dados e configurações como `APP_KEY` sejam consistentes com o ambiente antigo.

Depois que a cópia for concluída e a configuração do banco de dados for confirmada, inicie novamente a aplicação:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Migração de instalação antiga com create-nocobase-app ou código-fonte Git

Na forma antiga com npm/Git, o código-fonte, `.env` e `storage` normalmente ficam no mesmo diretório do projeto:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Recomenda-se definir primeiro o novo `NB_CLI_ROOT` e então criar um novo npm ou Git env. O exemplo abaixo assume que `NB_CLI_ROOT` é `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Depois migre conforme necessário:

- Migre o código-fonte dos plugins personalizados do projeto antigo para o diretório correspondente em `/your/workspace/app3/source`.
- Copie o `storage` do projeto antigo para `/your/workspace/app3/storage`.
- Sincronize as configurações principais do antigo `.env` com a configuração do CLI env ou com os parâmetros de inicialização.
- Ao usar o banco de dados original, desative o banco embutido durante a inicialização e preencha as informações de conexão do banco antigo.

Exemplo de inicialização usando banco de dados externo:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Após a migração, inicie e veja os logs:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
Não sobrescreva diretamente `source`, `storage` nem execute upgrades conectando-se ao banco de dados de produção sem backup. Antes da migração formal, recomenda-se fazer um ensaio completo primeiro em ambiente de teste.
:::

## Referência rápida de migração de comandos antigos

| Comando antigo                                        | Novo comando CLI                                                                        |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                            | `nb app start -e app1`                                                                  |
| `docker compose stop app`                             | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                          | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`        | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`    | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`      | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                            | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                               | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`            | `nb app upgrade -e app1 -s`                                                             |
| Registrar manualmente vários endereços de aplicação   | `nb env list`, `nb env info app1`                                                       |
| Conectar manualmente uma aplicação existente          | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
