# Panduan instalasi, upgrade, dan migrasi: menggunakan NocoBase CLI

:::danger
Kemampuan instalasi dan pemeliharaan aplikasi pada CLI masih dalam pengembangan. Saat ini, CLI lebih cocok untuk pengembangan lokal, lingkungan pengujian, dan skenario integrasi AI Agent, dan tidak disarankan untuk langsung digunakan untuk instalasi, upgrade, dan operasi pemeliharaan di lingkungan produksi.
:::

NocoBase CLI (`nb`) versi baru menyatukan operasi seperti instalasi, koneksi, start, stop, melihat log, upgrade, dan cleanup ke dalam satu set perintah. Dibandingkan dokumentasi lama yang memelihara alur secara terpisah untuk Docker, create-nocobase-app, dan source code Git, CLI lebih cocok untuk pengembangan lokal, pemeliharaan lingkungan pengujian, serta integrasi dan kolaborasi AI Agent.

Artikel ini berlaku untuk skenario berikut:

- Anda sedang menginstal atau meng-upgrade NocoBase berdasarkan dokumentasi lama dan ingin beralih ke cara baru dengan CLI.
- Anda sudah memiliki NocoBase yang di-deploy dengan cara lama dan ingin membiarkan CLI mengelola atau terhubung ke sana.
- Anda perlu menyiapkan lingkungan NocoBase yang dapat dihubungkan dan dipelihara untuk AI Agent.

## Perbedaan antara cara lama dan cara baru

Cara lama membagi dokumentasi berdasarkan sumber instalasi, dan setiap sumber memerlukan perintah download, konfigurasi, instalasi, start, dan upgrade yang dijalankan secara terpisah.

**Dokumentasi instalasi lama**

- [Instalasi Docker](/get-started/installation/docker)
- [Instalasi create-nocobase-app](/get-started/installation/create-nocobase-app)
- [Instalasi source code Git](/get-started/installation/git)

**Dokumentasi upgrade lama**

- [Upgrade untuk instalasi Docker](/get-started/upgrading/docker)
- [Upgrade untuk instalasi create-nocobase-app](/get-started/upgrading/create-nocobase-app)
- [Upgrade untuk instalasi source code Git](/get-started/upgrading/git)

Cara baru menggunakan `nb init` sebagai titik masuk, dengan satu CLI env untuk mencatat sumber aplikasi, direktori runtime, direktori storage, database, dan informasi koneksi API. Setelah itu, baik sumbernya Docker, npm, maupun Git, sebisa mungkin semuanya dikelola menggunakan set perintah yang sama: `nb app`, `nb source`, dan `nb db`.

| Skenario                              | Cara lama                                                  | Cara CLI baru                                     |
| ------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------- |
| Titik masuk instalasi                 | `docker compose`、`create-nocobase-app`、`git clone`       | `nb init` atau `nb init --ui`                     |
| Manajemen runtime aplikasi            | Pindah direktori, ubah compose, jalankan perintah yarn/pm2 | `nb app start/stop/restart/logs`、`nb env remove` |
| Upgrade                               | Satu set perintah untuk tiap metode instalasi              | `nb app upgrade -e <env>`                         |
| Mode development                      | `yarn dev`                                                 | `nb source dev -e <env>`                          |
| Status database                       | Cek container atau layanan database secara manual          | `nb db ps -e <env>`                               |
| Melihat status dan detail environment | Catat path, port, dan alamat akses secara manual           | `nb env list`、`nb env info <env>`                |
| Menghubungkan aplikasi yang sudah ada | Tidak dicakup secara terpadu dalam dokumentasi lama        | `nb init --ui` atau `nb init --api-base-url`      |

## Instalasi CLI

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Setelah terpasang, Anda dapat melihat bantuan:

```bash
nb --help
nb init --help
```

## Menginstal NocoBase dengan cara baru

### Rekomendasi: gunakan wizard visual

```bash
nb init --ui
```

Wizard akan memandu Anda memilih:

- membuat aplikasi baru atau menghubungkan aplikasi yang sudah ada
- sumber instalasi: Docker, npm, Git
- versi NocoBase
- menggunakan database bawaan atau database eksternal
- akun dan kata sandi administrator

### Menggunakan wizard interaktif terminal

```bash
nb init
```

### Menggunakan perintah non-interaktif

Metode instalasi default adalah Docker dengan database PostgreSQL bawaan:

```bash
nb init --yes --env app1
```

Menggunakan Docker untuk menginstal versi tertentu:

```bash
nb init --yes --env app1 --source docker --version beta
```

Menggunakan npm untuk instalasi:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Menggunakan source code Git untuk instalasi:

```bash
nb init --yes --env app1 --source git --version beta
```

Menggunakan source code Git dan menentukan jenis database bawaan:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Menghubungkan aplikasi NocoBase yang sudah ada:

```bash
# Default menggunakan autentikasi OAuth
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Menggunakan autentikasi token
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` adalah nama environment di dalam CLI. Setelahnya, perintah seperti start, upgrade, melihat log, dan lainnya dapat menentukan aplikasi target dengan `-e app1`.
:::

## Upgrade dengan cara baru

Pada cara lama, perintah upgrade untuk Docker, create-nocobase-app, dan source code Git berbeda-beda; pada CLI baru semuanya disatukan menjadi:

```bash
nb app upgrade -e app1
```

Jika Anda hanya ingin menjalankan proses upgrade menggunakan source code atau image yang sudah diunduh saat ini, tanpa menarik kode atau image baru:

```bash
nb app upgrade -e app1 --skip-code-update
```

Atau gunakan singkatan:

```bash
nb app upgrade -e app1 -s
```

## Perintah pemeliharaan yang umum digunakan

| Operasi                                                         | Perintah                             |
| --------------------------------------------------------------- | ------------------------------------ |
| Menjalankan aplikasi                                            | `nb app start -e app1`               |
| Menghentikan aplikasi                                           | `nb app stop -e app1`                |
| Memulai ulang aplikasi                                          | `nb app restart -e app1`             |
| Melihat log                                                     | `nb app logs -e app1`                |
| Melihat env dan status autentikasi API                          | `nb env list`                        |
| Melihat detail env                                              | `nb env info app1`                   |
| Melihat status database bawaan                                  | `nb db ps -e app1`                   |
| Upgrade aplikasi                                                | `nb app upgrade -e app1`             |
| Menghentikan aplikasi dan database bawaan                       | `nb app stop -e app1 --with-db`      |
| Menghapus env dan resource yang di-host secara lokal sepenuhnya | `nb env remove app1 --purge --force` |

Jika Anda hanya ingin menghentikan aplikasi bersama database bawaan yang dikelola CLI, cukup gunakan `nb app stop -e app1 --with-db`.

Jika Anda yakin env ini sudah tidak diperlukan lagi, gunakan `nb env remove app1 --purge --force`. Perintah ini akan menghapus resource runtime yang dikelola, data storage, serta file app lokal yang diunduh dan dikelola CLI jika berlaku.

## Perintah pengembangan harian

`nb source` digunakan untuk mengelola project source code lokal, terutama cocok untuk env sumber npm dan Git.

| Operasi                                      | Perintah                                            |
| -------------------------------------------- | --------------------------------------------------- |
| Menjalankan aplikasi dalam mode development  | `nb source dev -e app1`                             |
| Membangun source code                        | `nb source build --cwd /your/workspace/app1/source` |
| Menjalankan pengujian                        | `nb source test --cwd /your/workspace/app1/source`  |
| Mengunduh atau menyegarkan source code/image | `nb source download --source git --version beta`    |

:::tip
Docker env biasanya dikelola melalui `nb app start`, `nb app logs`, dan `nb app upgrade`, dan tidak menggunakan `nb source dev`.
:::

## NB_CLI_ROOT dan struktur direktori

CLI akan menyimpan konfigurasi global dan file aplikasi lokal di direktori yang sesuai dengan `NB_CLI_ROOT`.

Secara default, `NB_CLI_ROOT` adalah direktori home pengguna saat ini, yaitu path yang dikembalikan oleh Node.js `os.homedir()`. Misalnya, di macOS biasanya adalah `/Users/<user>`. Oleh karena itu, jika tidak ada konfigurasi tambahan, file yang dihasilkan oleh `nb init --yes --env app1` biasanya akan berada di:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Anda juga dapat menentukan `NB_CLI_ROOT` secara eksplisit:

```bash
export NB_CLI_ROOT=/your/workspace
```

Setelah diatur, file konfigurasi dan direktori aplikasi default akan ditempatkan di direktori tersebut:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Di antaranya:

- `.nocobase/config.json` menyimpan env, environment saat ini, alamat API, path source code, path storage, konfigurasi database, dan informasi lainnya.
- `<envName>/source` adalah source code atau direktori aplikasi yang dikelola CLI.
- `<envName>/storage` adalah direktori storage aplikasi yang dikelola CLI.

:::warning
Setelah aplikasi yang sama diinisialisasi, harap tetap menggunakan `NB_CLI_ROOT` yang sama. Jika nanti `NB_CLI_ROOT` diubah, CLI akan mencari `.nocobase/config.json` di direktori baru, dan `appRootPath` serta `storagePath` dalam bentuk path relatif juga akan di-resolve berdasarkan direktori baru, yang dapat menyebabkan aplikasi tidak ditemukan atau terbaca/tertulis ke direktori yang salah.
:::

Saat ini CLI hanya menggunakan scope konfigurasi global. Jika Anda perlu menyesuaikan lokasi penyimpanan konfigurasi dan file aplikasi lokal, atur `NB_CLI_ROOT`.

Saat melihat dan menemukan aplikasi, prioritaskan penggunaan `nb env list` dan `nb env info`:

```bash
# Melihat env yang telah dikonfigurasi dan status autentikasi API
nb env list

# Melihat informasi detail untuk env tertentu
nb env info app1
```

`nb env list` ditujukan untuk ringkasan cepat. Di dalamnya, `App Status` adalah status autentikasi yang diperoleh setelah mengakses API aplikasi menggunakan kredensial Token/OAuth yang tersimpan, dan status database tidak lagi ditampilkan. Jika perlu melihat status database bawaan, gunakan:

```bash
nb db ps -e app1
```

Hanya jika Anda perlu mengganti env default, barulah gunakan:

```bash
nb env use app1
```

## Migrasi dari cara lama ke CLI

Ada dua strategi migrasi: **menghubungkan aplikasi yang sudah ada**, atau **memasukkan direktori aplikasi lokal ke dalam pengelolaan CLI**. Jika aplikasi lama Anda sudah berjalan stabil, disarankan untuk terlebih dahulu menggunakan “menghubungkan aplikasi yang sudah ada”, karena risikonya paling rendah.

### Cara 1: menghubungkan aplikasi yang sudah ada

Cocok untuk NocoBase yang sudah berjalan melalui Docker, create-nocobase-app, source code Git, atau deployment server.

```bash
# Default menggunakan autentikasi OAuth
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Menggunakan autentikasi token
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Cara ini hanya menyimpan koneksi API; CLI tidak akan mengambil alih start, stop, upgrade, direktori source code, maupun direktori storage dari aplikasi lama. Cara ini cocok agar AI Agent atau perintah CLI API dapat terhubung ke aplikasi yang sudah ada.

Jika perlu menyegarkan autentikasi:

```bash
nb env auth app1
```

### Cara 2: buat CLI env baru lalu migrasikan file

Cocok untuk skenario ketika Anda ingin menggunakan `nb app` dan `nb source` untuk mengelola aplikasi lokal setelahnya.

Sebelum migrasi, selesaikan dulu:

1. Hentikan aplikasi lama.
2. Backup database lama.
3. Backup direktori `storage` lama.
4. Catat variabel environment penting aplikasi lama, seperti `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Migrasi instalasi Docker lama

Metode Docker lama biasanya menyimpan data aplikasi di `storage` dalam direktori project:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Disarankan untuk terlebih dahulu menentukan `NB_CLI_ROOT` yang baru, lalu membuat Docker env baru dengan CLI. Contoh berikut mengasumsikan `NB_CLI_ROOT` adalah `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

Env baru akan menggunakan direktori berikut secara default:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Lalu salin `storage` lama ke `/your/workspace/app2/storage`, dan pastikan koneksi database serta konfigurasi seperti `APP_KEY` konsisten dengan environment lama.

Setelah penyalinan selesai dan konfigurasi database dipastikan benar, jalankan kembali aplikasi:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Migrasi instalasi lama create-nocobase-app atau source code Git

Metode npm/Git lama biasanya menempatkan source code, `.env`, dan `storage` dalam satu direktori project:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Disarankan untuk terlebih dahulu menentukan `NB_CLI_ROOT` yang baru, lalu membuat env npm atau Git yang baru. Contoh berikut mengasumsikan `NB_CLI_ROOT` adalah `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Lalu migrasikan sesuai kebutuhan:

- Migrasikan source code plugin kustom dari project lama ke direktori yang sesuai di `/your/workspace/app3/source`.
- Salin `storage` lama ke `/your/workspace/app3/storage`.
- Sinkronkan konfigurasi penting dari `.env` lama ke konfigurasi CLI env atau parameter inisialisasi.
- Jika menggunakan database lama, nonaktifkan database bawaan saat inisialisasi dan isi informasi koneksi database lama.

Contoh inisialisasi dengan database eksternal:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Setelah migrasi selesai, jalankan aplikasi dan lihat log:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
Jangan langsung menimpa `source`, `storage`, atau melakukan upgrade dengan menghubungkan database produksi tanpa backup. Sebelum migrasi resmi, disarankan untuk melakukan simulasi lengkap sekali di lingkungan pengujian.
:::

## Ringkasan cepat migrasi perintah lama

| Perintah lama                                         | Perintah CLI baru                                                                       |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                            | `nb app start -e app1`                                                                  |
| `docker compose stop app`                             | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                          | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`        | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`    | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`      | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                            | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                               | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`            | `nb app upgrade -e app1 -s`                                                             |
| Mencatat alamat beberapa aplikasi secara manual       | `nb env list`、`nb env info app1`                                                       |
| Menghubungkan aplikasi yang sudah ada secara manual   | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
