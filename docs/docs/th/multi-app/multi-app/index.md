# การจัดการหลายแอปพลิเคชัน

## ภาพรวมฟังก์ชัน

การจัดการหลายแอปพลิเคชันเป็นโซลูชันการจัดการแอปพลิเคชันแบบรวมศูนย์ที่ NocoBase มอบให้ เพื่อใช้ในการสร้างและจัดการอินสแตนซ์แอปพลิเคชัน NocoBase หลายรายการที่แยกจากกันทางกายภาพในสภาพแวดล้อมการทำงานหนึ่งรายการหรือมากกว่านั้นครับ ผ่านตัวควบคุมแอปพลิเคชัน (AppSupervisor) ผู้ใช้สามารถสร้างและดูแลรักษาหลายแอปพลิเคชันได้จากจุดเข้าใช้งานเดียว เพื่อตอบสนองความต้องการของธุรกิจและขนาดที่แตกต่างกันในแต่ละช่วงครับ

## แอปพลิเคชันเดียว

ในช่วงเริ่มต้นของโครงการ ผู้ใช้ส่วนใหญ่มักจะเริ่มจากแอปพลิเคชันเดียวครับ

ในโหมดนี้ ระบบจำเป็นต้องปรับใช้ NocoBase เพียงอินสแตนซ์เดียวเท่านั้น โดยฟังก์ชันทางธุรกิจ ข้อมูล และผู้ใช้ทั้งหมดจะทำงานอยู่ในแอปพลิเคชันเดียวกันครับ การปรับใช้ทำได้ง่ายและมีต้นทุนการกำหนดค่าต่ำ ซึ่งเหมาะอย่างยิ่งสำหรับการตรวจสอบต้นแบบ โครงการขนาดเล็ก หรือเครื่องมือภายในครับ

อย่างไรก็ตาม เมื่อธุรกิจเริ่มมีความซับซ้อนมากขึ้น แอปพลิเคชันเดียวจะเผชิญกับข้อจำกัดตามธรรมชาติบางประการครับ:

- ฟังก์ชันต่างๆ เพิ่มพูนขึ้นเรื่อยๆ ทำให้ระบบเริ่มเทอะทะ
- การแยกส่วนระหว่างธุรกิจต่างๆ ทำได้ยาก
- ต้นทุนในการขยายและบำรุงรักษาแอปพลิเคชันเพิ่มสูงขึ้นอย่างต่อเนื่อง

ในจุดนี้ ผู้ใช้จะต้องการแยกธุรกิจต่างๆ ออกเป็นหลายแอปพลิเคชัน เพื่อเพิ่มความสามารถในการบำรุงรักษาและการขยายตัวของระบบครับ

## หลายแอปพลิเคชันแบบแชร์หน่วยความจำ

เมื่อผู้ใช้ต้องการแยกธุรกิจแต่ไม่ต้องการนำสถาปัตยกรรมการปรับใช้และการดำเนินงานที่ซับซ้อนเข้ามาเกี่ยวข้อง สามารถอัปเกรดเป็นโหมดหลายแอปพลิเคชันแบบแชร์หน่วยความจำได้ครับ

ในโหมดนี้ แอปพลิเคชันหลายรายการสามารถทำงานพร้อมกันได้ในอินสแตนซ์ NocoBase เดียวกันครับ แต่ละแอปพลิเคชันเป็นอิสระต่อกัน สามารถเชื่อมต่อกับฐานข้อมูลแยกกันได้ สามารถสร้าง เริ่มต้น และหยุดการทำงานแยกกันได้ แต่จะใช้กระบวนการ (Process) และพื้นที่หน่วยความจำร่วมกันครับ ผู้ใช้ยังคงต้องบำรุงรักษา NocoBase เพียงอินสแตนซ์เดียวเท่านั้นครับ

![](https://static-docs.nocobase.com/202512231055907.png)

วิธีการนี้ช่วยปรับปรุงให้ดีขึ้นอย่างเห็นได้ชัดครับ:

- ธุรกิจสามารถแยกตามมิติของแอปพลิเคชันได้
- ฟังก์ชันและการกำหนดค่าระหว่างแอปพลิเคชันมีความชัดเจนมากขึ้น
- เมื่อเทียบกับโซลูชันแบบหลายโปรเซสหรือหลายคอนเทนเนอร์ การใช้ทรัพยากรจะต่ำกว่าครับ

อย่างไรก็ตาม เนื่องจากแอปพลิเคชันทั้งหมดทำงานในโปรเซสเดียวกัน จึงมีการแชร์ทรัพยากร เช่น CPU และหน่วยความจำร่วมกันครับ ความผิดปกติหรือภาระงานสูงในแอปพลิเคชันเดียวอาจส่งผลกระทบต่อความเสถียรของแอปพลิเคชันอื่นๆ ได้ครับ

เมื่อจำนวนแอปพลิเคชันเพิ่มขึ้นอย่างต่อเนื่อง หรือเมื่อมีความต้องการด้านการแยกส่วนและความเสถียรที่สูงขึ้น จำเป็นต้องอัปเกรดสถาปัตยกรรมเพิ่มเติมครับ

## การปรับใช้แบบผสมผสานหลายสภาพแวดล้อม

เมื่อขนาดและความซับซ้อนของธุรกิจถึงระดับหนึ่ง และจำนวนแอปพลิเคชันจำเป็นต้องขยายตัวตามขนาด โหมดหลายแอปพลิเคชันแบบแชร์หน่วยความจำจะเผชิญกับความท้าทายต่างๆ เช่น การแย่งชิงทรัพยากร ความเสถียร และความปลอดภัยครับ ในช่วงของการขยายขนาด ผู้ใช้สามารถพิจารณาใช้วิธีการปรับใช้แบบผสมผสานหลายสภาพแวดล้อม เพื่อรองรับสถานการณ์ทางธุรกิจที่ซับซ้อนยิ่งขึ้นครับ

หัวใจสำคัญของสถาปัตยกรรมนี้คือการนำแอปพลิเคชันจุดเข้าใช้งาน (Entry App) เข้ามาใช้ กล่าวคือการปรับใช้ NocoBase หนึ่งรายการเพื่อเป็นศูนย์กลางการจัดการแบบรวมศูนย์ และในขณะเดียวกันก็ปรับใช้ NocoBase หลายรายการเพื่อเป็นสภาพแวดล้อมการทำงานของแอปพลิเคชัน สำหรับการรันแอปพลิเคชันทางธุรกิจจริงครับ

แอปพลิเคชันจุดเข้าใช้งานมีหน้าที่:

- การสร้าง การกำหนดค่า และการจัดการวงจรชีวิตของแอปพลิเคชัน
- การส่งคำสั่งการจัดการและการสรุปสถานะ

สภาพแวดล้อมแอปพลิเคชันอินสแตนซ์มีหน้าที่:

- รองรับและรันแอปพลิเคชันทางธุรกิจจริงผ่านโหมดหลายแอปพลิเคชันแบบแชร์หน่วยความจำครับ

สำหรับผู้ใช้ แอปพลิเคชันหลายรายการยังคงสามารถสร้างและจัดการผ่านจุดเข้าใช้งานเดียวได้ แต่ภายในระบบ:

- แอปพลิเคชันที่แตกต่างกันสามารถทำงานบนโหนดหรือคลัสเตอร์ที่แตกต่างกันได้
- แต่ละแอปพลิเคชันสามารถใช้ฐานข้อมูลและมิดเดิลแวร์ที่เป็นอิสระต่อกันได้
- สามารถขยายขนาดหรือแยกแอปพลิเคชันที่มีภาระงานสูงได้ตามต้องการครับ
