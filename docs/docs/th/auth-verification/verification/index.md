# การยืนยันตัวตน

:::info{title=ข้อแนะนำ}
ตั้งแต่เวอร์ชัน `1.6.0-alpha.30` เป็นต้นไป ฟังก์ชัน**รหัสยืนยัน**แบบเดิมได้รับการอัปเกรดเป็น**การจัดการการยืนยันตัวตน** ซึ่งรองรับการจัดการและเชื่อมต่อวิธีการยืนยันตัวตนของผู้ใช้ที่หลากหลาย หลังจากที่ผู้ใช้ผูกวิธีการยืนยันตัวตนที่เกี่ยวข้องแล้ว ก็สามารถทำการยืนยันตัวตนได้ในสถานการณ์ที่จำเป็น ฟังก์ชันนี้มีแผนที่จะรองรับอย่างเสถียรตั้งแต่เวอร์ชัน `1.7.0` เป็นต้นไป
:::



## บทนำ

**ศูนย์จัดการการยืนยันตัวตนรองรับการจัดการและเชื่อมต่อวิธีการยืนยันตัวตนของผู้ใช้ที่หลากหลาย** ตัวอย่างเช่น:

- รหัสยืนยันทาง SMS – มีให้ใช้งานโดยค่าเริ่มต้นจาก**ปลั๊กอิน**การยืนยันตัวตน ดูเพิ่มเติมที่: [การยืนยันตัวตน: SMS](./sms)
- TOTP Authenticator – ดูเพิ่มเติมที่: [การยืนยันตัวตน: TOTP Authenticator](../verification-totp/)

นักพัฒนาสามารถขยายประเภทการยืนยันตัวตนอื่นๆ ได้ในรูปแบบของ**ปลั๊กอิน** ดูเพิ่มเติมที่: [การขยายประเภทการยืนยันตัวตน](./dev/type)

**ผู้ใช้สามารถทำการยืนยันตัวตนได้ในสถานการณ์ที่จำเป็น หลังจากผูกวิธีการยืนยันตัวตนที่เกี่ยวข้องแล้ว** ตัวอย่างเช่น:

- การเข้าสู่ระบบด้วยรหัสยืนยันทาง SMS – ดูเพิ่มเติมที่: [การรับรองความถูกต้อง: SMS](./sms)
- การยืนยันตัวตนแบบสองปัจจัย (2FA) – ดูเพิ่มเติมที่: [การยืนยันตัวตนแบบสองปัจจัย (2FA)](../2fa/)
- การยืนยันตัวตนครั้งที่สองสำหรับการดำเนินการที่มีความเสี่ยง – รองรับในอนาคต

นักพัฒนาสามารถผสานรวมการยืนยันตัวตนเข้ากับสถานการณ์ที่จำเป็นอื่นๆ ได้ในรูปแบบของ**ปลั๊กอิน**เสริม ดูเพิ่มเติมที่: [การขยายสถานการณ์การยืนยันตัวตน](./dev/scene)

**ความแตกต่างและความเชื่อมโยงระหว่างโมดูลการยืนยันตัวตนและโมดูลการรับรองความถูกต้องของผู้ใช้:** โมดูลการรับรองความถูกต้องของผู้ใช้มีหน้าที่หลักในการรับรองความถูกต้องของตัวตนในสถานการณ์การเข้าสู่ระบบของผู้ใช้ โดยที่กระบวนการต่างๆ เช่น การเข้าสู่ระบบด้วย SMS และการยืนยันตัวตนแบบสองปัจจัย จะขึ้นอยู่กับตัวยืนยันที่โมดูลการยืนยันตัวตนจัดหาให้ ในขณะที่โมดูลการยืนยันตัวตนมีหน้าที่รับผิดชอบในการยืนยันตัวตนสำหรับการดำเนินการที่มีความเสี่ยงต่างๆ ซึ่งการเข้าสู่ระบบของผู้ใช้ก็เป็นหนึ่งในสถานการณ์ของการดำเนินการที่มีความเสี่ยงเหล่านั้น
