# การสนับสนุนการแปล

ภาษาเริ่มต้นของ NocoBase คือภาษาอังกฤษ ปัจจุบันแอปพลิเคชันหลักรองรับภาษาอังกฤษ, อิตาลี, ดัตช์, จีนตัวย่อ และญี่ปุ่น เราขอเชิญชวนคุณมาร่วมสนับสนุนการแปลภาษาอื่น ๆ เพื่อให้ผู้ใช้งานทั่วโลกได้รับประสบการณ์การใช้งาน NocoBase ที่สะดวกสบายยิ่งขึ้นครับ

---

## I. การแปลส่วนระบบ (System Localization)

### 1. การแปลส่วนติดต่อผู้ใช้งานและปลั๊กอิน

#### 1.1 ขอบเขตการแปล
ใช้สำหรับการแปลส่วนติดต่อผู้ใช้งาน (UI) ของระบบ NocoBase และปลั๊กอินเท่านั้น ไม่รวมถึงเนื้อหาที่กำหนดเองอื่น ๆ (เช่น ตารางข้อมูล หรือบล็อก Markdown) ครับ

![bbb6e0b44aeg](https://static-docs.nocobase.com/img_v3_02kh_8d429938-3aca-44b6-a437-bbb6e0b44aeg.jpg)

![20250319220127](https://static-docs.nocobase.com/20250319220127.png)


#### 1.2 ภาพรวมเนื้อหาการแปล
NocoBase ใช้ Git ในการจัดการเนื้อหาการแปล คลังข้อมูลหลัก (Repository) คือ:
https://github.com/nocobase/nocobase/tree/main/locales

แต่ละภาษาจะแสดงด้วยไฟล์ JSON ที่ตั้งชื่อตามรหัสภาษา (เช่น de-DE.json, fr-FR.json) โครงสร้างไฟล์จะจัดตามโมดูลของปลั๊กอิน โดยใช้คู่คีย์-ค่า (key-value pairs) ในการเก็บคำแปล ตัวอย่างเช่น:

```json
{
  // ปลั๊กอินฝั่งไคลเอนต์
  "@nocobase/client": {
    "(Fields only)": "(Fields only)",
    "12 hour": "12 hour",
    "24 hour": "24 hour"
    // ...คู่คีย์-ค่าอื่น ๆ
  },
  "@nocobase/plugin-acl": {
    // คู่คีย์-ค่าสำหรับปลั๊กอินนี้
  }
  // ...โมดูลปลั๊กอินอื่น ๆ
}
```

ในขณะที่ทำการแปล โปรดค่อย ๆ เปลี่ยนให้เป็นโครงสร้างที่คล้ายกับตัวอย่างด้านล่างนี้ครับ:

```json
{
  // ปลั๊กอินฝั่งไคลเอนต์
  "@nocobase/client": {
    "(Fields only)": "(เฉพาะฟิลด์ - แปลแล้ว)",
    "12 hour": "12 ชั่วโมง",
    "24 hour": "24 ชั่วโมง"
    // ...คู่คีย์-ค่าอื่น ๆ
  },
  "@nocobase/plugin-acl": {
    // คู่คีย์-ค่าสำหรับปลั๊กอินนี้
  }
  // ...โมดูลปลั๊กอินอื่น ๆ
}
```

#### 1.3 การทดสอบและการซิงค์การแปล
- หลังจากแปลเสร็จแล้ว โปรดทดสอบและตรวจสอบว่าข้อความทั้งหมดแสดงผลอย่างถูกต้อง
เราได้เปิดตัวปลั๊กอินสำหรับตรวจสอบการแปล - ค้นหา `Locale tester` ในตลาดปลั๊กอิน (Plugin Marketplace) ครับ
![20250422233152](https://static-docs.nocobase.com/20250422233152.png)
หลังจากติดตั้งแล้ว ให้คัดลอกเนื้อหา JSON จากไฟล์การแปลที่เกี่ยวข้องในคลังข้อมูล git มาวางไว้ด้านใน แล้วคลิกตกลงเพื่อตรวจสอบว่าเนื้อหาการแปลมีผลใช้งานหรือไม่
![20250422233950](https://static-docs.nocobase.com/20250422233950.png)

- เมื่อส่ง (Submit) แล้ว สคริปต์ของระบบจะซิงค์เนื้อหาการแปลไปยังคลังรหัสโดยอัตโนมัติครับ

#### 1.4 ปลั๊กอินการแปลของ NocoBase 2.0

> **หมายเหตุ:** ส่วนนี้กำลังอยู่ในระหว่างการพัฒนา ปลั๊กอินการแปลของ NocoBase 2.0 มีความแตกต่างจากเวอร์ชัน 1.x บางประการ รายละเอียดเพิ่มเติมจะแจ้งให้ทราบในการอัปเดตครั้งถัดไปครับ

## II. การแปลเอกสาร (NocoBase 2.0)

เอกสารสำหรับ NocoBase 2.0 ถูกจัดการด้วยโครงสร้างใหม่ ไฟล์ต้นฉบับของเอกสารอยู่ที่คลังข้อมูลหลักของ NocoBase:

https://github.com/nocobase/nocobase/tree/next/docs

### 2.1 โครงสร้างเอกสาร

เอกสารใช้ [Rspress](https://rspress.dev/) เป็นเครื่องมือสร้างไซต์แบบคงที่ (Static Site Generator) และรองรับ 22 ภาษา โครงสร้างมีการจัดระเบียบดังนี้:

```
docs/
├── docs/
│   ├── en/                    # ภาษาอังกฤษ (ภาษาต้นฉบับ)
│   ├── cn/                    # จีนตัวย่อ
│   ├── ja/                    # ญี่ปุ่น
│   ├── ko/                    # เกาหลี
│   ├── de/                    # เยอรมัน
│   ├── fr/                    # ฝรั่งเศส
│   ├── es/                    # สเปน
│   ├── pt/                    # โปรตุเกส
│   ├── ru/                    # รัสเซีย
│   ├── it/                    # อิตาลี
│   ├── tr/                    # ตุรกี
│   ├── uk/                    # ยูเครน
│   ├── vi/                    # เวียดนาม
│   ├── id/                    # อินโดนีเซีย
│   ├── th/                    # ไทย
│   ├── pl/                    # โปแลนด์
│   ├── nl/                    # ดัตช์
│   ├── cs/                    # เช็ก
│   ├── ar/                    # อาหรับ
│   ├── he/                    # ฮีบรู
│   ├── hi/                    # ฮินดี
│   ├── sv/                    # สวีเดน
│   └── public/                # ทรัพยากรส่วนกลาง (รูปภาพ ฯลฯ)
├── theme/                     # ธีมที่กำหนดเอง
├── rspress.config.ts          # การกำหนดค่า Rspress
└── package.json
```

### 2.2 เวิร์กโฟลว์การแปล

1. **ซิงค์กับต้นฉบับภาษาอังกฤษ**: การแปลทั้งหมดควรยึดตามเอกสารภาษาอังกฤษ (`docs/en/`) เมื่อเอกสารภาษาอังกฤษมีการอัปเดต คำแปลควรได้รับการอัปเดตตามไปด้วยครับ

2. **กลยุทธ์สาขา (Branch Strategy)**:
   - ใช้สาขา `develop` หรือ `next` เป็นข้อมูลอ้างอิงสำหรับเนื้อหาภาษาอังกฤษล่าสุด
   - สร้างสาขาการแปลของคุณจากสาขาเป้าหมาย

3. **โครงสร้างไฟล์**: ไดเรกทอรีของแต่ละภาษาควรสะท้อนโครงสร้างไดเรกทอรีของภาษาอังกฤษ ตัวอย่างเช่น:
   ```
   docs/en/get-started/index.md    →    docs/th/get-started/index.md
   docs/en/api/acl/acl.md          →    docs/th/api/acl/acl.md
   ```

### 2.3 การสนับสนุนการแปล

1. Fork คลังข้อมูล: https://github.com/nocobase/nocobase
2. Clone fork ของคุณและเช็คเอาต์สาขา `develop` หรือ `next`
3. ไปที่ไดเรกทอรี `docs/docs/`
4. ค้นหาไดเรกทอรีภาษาที่คุณต้องการสนับสนุน (เช่น ภาษาไทยคือ `th/`)
5. แปลไฟล์ markdown โดยรักษาโครงสร้างไฟล์ให้เหมือนกับเวอร์ชันภาษาอังกฤษ
6. ทดสอบการเปลี่ยนแปลงในเครื่องของคุณ:
   ```bash
   cd docs
   yarn install
   yarn dev
   ```
7. ส่ง Pull Request ไปยังคลังข้อมูลหลัก

### 2.4 แนวทางการแปล

- **รักษาความสม่ำเสมอของรูปแบบ**: รักษาโครงสร้าง markdown, หัวข้อ, บล็อกโค้ด และลิงก์ให้เหมือนกับไฟล์ต้นฉบับ
- **คงค่า Frontmatter ไว้**: คงค่า YAML frontmatter ที่ด้านบนสุดของไฟล์ไว้เหมือนเดิม เว้นแต่จะมีเนื้อหาที่สามารถแปลได้
- **การอ้างอิงรูปภาพ**: ใช้เส้นทางรูปภาพเดียวกันจาก `docs/public/` - รูปภาพจะถูกใช้ร่วมกันในทุกภาษา
- **ลิงก์ภายใน**: อัปเดตลิงก์ภายในให้ชี้ไปยังเส้นทางภาษาที่ถูกต้อง
- **ตัวอย่างโค้ด**: โดยปกติแล้ว ไม่ควรแปลตัวอย่างโค้ด แต่สามารถแปลคอมเมนต์ภายในโค้ดได้ครับ

### 2.5 การกำหนดค่าแถบนำทาง

โครงสร้างแถบนำทางของแต่ละภาษาจะถูกกำหนดในไฟล์ `_nav.json` และ `_meta.json` ภายในไดเรกทอรีของแต่ละภาษา เมื่อเพิ่มหน้าหรือส่วนใหม่ โปรดตรวจสอบให้แน่ใจว่าได้อัปเดตไฟล์กำหนดค่าเหล่านี้ด้วยครับ

### 3.2 โครงสร้างเนื้อหาและวิธีการแปล

เราใช้วิธีการจัดการเนื้อหาแบบผสมผสาน เนื้อหาและทรัพยากรภาษาอังกฤษ จีน และญี่ปุ่นจะถูกซิงค์และเขียนทับจากระบบ CMS เป็นระยะ ส่วนภาษาอื่น ๆ สามารถแก้ไขได้โดยตรงในไฟล์ภายในเครื่อง เนื้อหาภายในเครื่องจะถูกเก็บไว้ในไดเรกทอรี `content` โดยมีการจัดระเบียบดังนี้:

```
/content
  /articles        # บทความบล็อก
    /article-slug
      index.md     # เนื้อหาภาษาอังกฤษ (เริ่มต้น)
      index.cn.md  # เนื้อหาภาษาจีน
      index.ja.md  # เนื้อหาภาษาญี่ปุ่น
      metadata.json # เมทาดาต้าและคุณสมบัติการแปลอื่น ๆ
  /tutorials       # บทช่วยสอน
  /releases        # ข้อมูลการเปิดตัว
  /pages           # หน้าคงที่บางส่วน
  /categories      # ข้อมูลหมวดหมู่
    /article-categories.json  # รายการหมวดหมู่บทความ
    /category-slug            # รายละเอียดหมวดหมู่เดี่ยว
      /category.json
  /tags            # ข้อมูลแท็ก
    /article-tags.json        # รายการแท็กบทความ
    /release-tags.json        # รายการแท็กการเปิดตัว
    /tag-slug                 # รายละเอียดแท็กเดี่ยว
      /tag.json
  /help-center     # เนื้อหาศูนย์ช่วยเหลือ
    /help-center-tree.json    # โครงสร้างแถบนำทางศูนย์ช่วยเหลือ
  ....
```

### 3.3 แนวทางการแปลเนื้อหา

- เกี่ยวกับการแปลเนื้อหา Markdown

1. สร้างไฟล์ภาษาใหม่ตามไฟล์เริ่มต้น (เช่น `index.md` เป็น `index.th.md`)
2. เพิ่มคุณสมบัติการแปลในฟิลด์ที่เกี่ยวข้องในไฟล์ JSON
3. รักษาความสม่ำเสมอของโครงสร้างไฟล์ ลิงก์ และการอ้างอิงรูปภาพ

- เกี่ยวกับการแปลเนื้อหา JSON
เมทาดาต้าของเนื้อหาจำนวนมากถูกเก็บไว้ในไฟล์ JSON ซึ่งมักจะมีฟิลด์หลายภาษา:

```json
{
  "id": 123,
  "title": "English Title",       // ชื่อเรื่องภาษาอังกฤษ (เริ่มต้น)
  "title_cn": "中文标题",          // ชื่อเรื่องภาษาจีน
  "title_ja": "日本語タイトル",    // ชื่อเรื่องภาษาญี่ปุ่น
  "description": "English description",
  "description_cn": "中文描述",
  "description_ja": "日本語の説明",
  "slug": "article-slug",         // เส้นทาง URL (โดยปกติจะไม่แปล)
  "status": "published",
  "publishedAt": "2025-03-19T12:00:00Z"
}
```

**ข้อควรระวังในการแปล:**

1. **ข้อตกลงการตั้งชื่อฟิลด์**: ฟิลด์แปลมักจะใช้รูปแบบ `{ชื่อฟิลด์เดิม}_{รหัสภาษา}`
   - ตัวอย่างเช่น: title_th (ชื่อเรื่องภาษาไทย), description_th (คำอธิบายภาษาไทย)

2. **เมื่อเพิ่มภาษาใหม่**:
   - เพิ่มเวอร์ชันต่อท้ายภาษาสำหรับแต่ละฟิลด์ที่ต้องการแปล
   - อย่าแก้ไขค่าฟิลด์เดิม (เช่น title, description ฯลฯ) เนื่องจากฟิลด์เหล่านี้ทำหน้าที่เป็นเนื้อหาภาษาเริ่มต้น (ภาษาอังกฤษ)

3. **กลไกการซิงค์ของ CMS**:
   - ระบบ CMS จะอัปเดตเนื้อหาภาษาอังกฤษ จีน และญี่ปุ่นเป็นระยะ
   - ระบบจะอัปเดต/เขียนทับเฉพาะเนื้อหาของสามภาษานี้เท่านั้น (คุณสมบัติบางอย่างใน JSON) และ**จะไม่ลบ**ฟิลด์ภาษาที่ผู้สนับสนุนรายอื่นเพิ่มเข้ามา
   - ตัวอย่างเช่น: หากคุณเพิ่มคำแปลภาษาไทย (title_th) การซิงค์ของ CMS จะไม่ส่งผลกระทบต่อฟิลด์นี้ครับ


### 3.4 การกำหนดค่าการรองรับภาษาใหม่

หากต้องการเพิ่มการรองรับภาษาใหม่ จำเป็นต้องแก้ไขการกำหนดค่า `SUPPORTED_LANGUAGES` ในไฟล์ `src/utils/index.ts`:

```typescript
export const SUPPORTED_LANGUAGES = {
  en: {
    code: 'en',
    locale: 'en-US',
    name: 'English',
    default: true
  },
  cn: {
    code: 'cn',
    locale: 'zh-CN',
    name: 'Chinese'
  },
  ja: {
    code: 'ja',
    locale: 'ja-JP',
    name: 'Japanese'
  },
  // ตัวอย่างการเพิ่มภาษาใหม่:
  th: {
    code: 'th',
    locale: 'th-TH',
    name: 'Thai'
  }
};
```

### 3.5 ไฟล์เลย์เอาต์และสไตล์

แต่ละภาษาต้องการไฟล์เลย์เอาต์ที่เกี่ยวข้อง:

1. สร้างไฟล์เลย์เอาต์ใหม่ (เช่น สำหรับภาษาไทย ให้สร้าง `src/layouts/BaseTH.astro`)
2. คุณสามารถคัดลอกไฟล์เลย์เอาต์ที่มีอยู่ (เช่น `BaseEN.astro`) แล้วนำมาแปลได้ครับ
3. ไฟล์เลย์เอาต์ประกอบด้วยคำแปลสำหรับองค์ประกอบส่วนกลาง เช่น เมนูนำทาง, ส่วนท้าย (Footer) เป็นต้น
4. ตรวจสอบให้แน่ใจว่าได้อัปเดตการกำหนดค่าตัวสลับภาษา (Language Switcher) เพื่อให้สลับไปยังภาษาที่เพิ่มใหม่ได้อย่างถูกต้อง

### 3.6 การสร้างไดเรกทอรีหน้าภาษา

สร้างไดเรกทอรีหน้าแยกต่างหากสำหรับภาษาใหม่:

1. สร้างโฟลเดอร์ที่ตั้งชื่อตามรหัสภาษาในไดเรกทอรี `src` (เช่น `src/th/`)
2. คัดลอกโครงสร้างหน้าจากไดเรกทอรีภาษาอื่น (เช่น `src/en/`)
3. อัปเดตเนื้อหาหน้า โดยแปลชื่อเรื่อง, คำอธิบาย และข้อความให้เป็นภาษาเป้าหมาย
4. ตรวจสอบให้แน่ใจว่าหน้าต่าง ๆ ใช้ส่วนประกอบเลย์เอาต์ที่ถูกต้อง (เช่น `.layout: '@/layouts/BaseTH.astro'`)

### 3.7 การแปลส่วนประกอบ (Component)

ส่วนประกอบทั่วไปบางอย่างจำเป็นต้องได้รับการแปลเช่นกัน:

1. ตรวจสอบส่วนประกอบในไดเรกทอรี `src/components/`
2. ให้ความสำคัญเป็นพิเศษกับส่วนประกอบที่มีข้อความคงที่ (เช่น แถบนำทาง, ส่วนท้าย เป็นต้น)
3. ส่วนประกอบอาจใช้การแสดงผลแบบมีเงื่อนไข (Conditional Rendering) เพื่อแสดงเนื้อหาในภาษาต่าง ๆ:

```astro
{Astro.url.pathname.startsWith('/en') && <p>English content</p>}
{Astro.url.pathname.startsWith('/cn') && <p>中文内容</p>}
{Astro.url.pathname.startsWith('/th') && <p>เนื้อหาภาษาไทย</p>}
```

### 3.8 การทดสอบและการตรวจสอบ

หลังจากแปลเสร็จแล้ว ให้ทำการทดสอบอย่างละเอียด:

1. รันเว็บไซต์ในเครื่อง (โดยปกติจะใช้คำสั่ง `yarn dev`)
2. ตรวจสอบการแสดงผลของทุกหน้าในภาษาใหม่
3. ตรวจสอบว่าฟังก์ชันการสลับภาษาทำงานได้อย่างถูกต้อง
4. ตรวจสอบให้แน่ใจว่าลิงก์ทั้งหมดชี้ไปยังหน้าเวอร์ชันภาษาที่ถูกต้อง
5. ตรวจสอบเลย์เอาต์แบบตอบสนอง (Responsive Layout) เพื่อให้แน่ใจว่าข้อความที่แปลแล้วไม่ทำให้การออกแบบหน้าเว็บผิดเพี้ยน

## IV. วิธีเริ่มต้นการแปล

หากคุณต้องการสนับสนุนการแปลภาษาใหม่ให้กับ NocoBase โปรดทำตามขั้นตอนดังนี้:

| ส่วนประกอบ | คลังข้อมูล (Repository) | สาขา (Branch) | หมายเหตุ |
|------|------|------|------|
| ส่วนติดต่อระบบ | https://github.com/nocobase/nocobase/tree/main/locales | main | ไฟล์ JSON สำหรับการแปล |
| เอกสาร (2.0) | https://github.com/nocobase/nocobase | develop / next | ไดเรกทอรี `docs/docs/<lang>/` |
| เว็บไซต์ทางการ | https://github.com/nocobase/website | main | ดูรายละเอียดในส่วนที่ 3 |

หลังจากแปลเสร็จแล้ว โปรดส่ง Pull Request ไปยัง NocoBase ภาษาใหม่จะปรากฏในการกำหนดค่าระบบ ช่วยให้คุณสามารถเลือกภาษาที่ต้องการแสดงผลได้ครับ
