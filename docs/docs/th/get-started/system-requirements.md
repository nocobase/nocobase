# ข้อกำหนดของระบบ

ข้อกำหนดของระบบที่ระบุในเอกสารนี้คือข้อกำหนดสำหรับการรัน **บริการแอปพลิเคชัน NocoBase เอง** ซึ่งครอบคลุมเพียงทรัพยากรการคำนวณและหน่วยความจำที่จำเป็นสำหรับกระบวนการทำงานของแอปพลิเคชันเท่านั้น **ไม่รวมถึงบริการภายนอก (Third-party services) ที่เกี่ยวข้อง** ซึ่งรวมถึงแต่ไม่จำกัดเพียง:

- API Gateway / Reverse Proxy
- บริการฐานข้อมูล (เช่น MySQL, PostgreSQL)
- บริการแคช (เช่น Redis)
- มิดเดิลแวร์ เช่น Message Queue, Object Storage เป็นต้น

นอกเหนือจากการตรวจสอบฟังก์ชันการทำงานหรือการทดลองใช้งานแล้ว **ขอแนะนำอย่างยิ่งให้ปรับใช้ (Deploy) บริการภายนอกข้างต้นแยกต่างหาก** บนเซิร์ฟเวอร์หรือคอนเทนเนอร์ที่แยกจากกัน หรือเลือกใช้บริการคลาวด์ที่เกี่ยวข้องโดยตรงครับ

การกำหนดค่าระบบและการวางแผนความจุของบริการที่เกี่ยวข้องควรได้รับการประเมินและปรับแต่งแยกต่างหากตาม **ปริมาณข้อมูลจริง ภาระงาน และขนาดของการเข้าใช้งานพร้อมกัน (Concurrency)** ครับ

## โหมดการปรับใช้แบบโหนดเดียว (Single-Node Deployment)

โหมดการปรับใช้แบบโหนดเดียว หมายถึงการที่บริการแอปพลิเคชัน NocoBase รันอยู่บนเซิร์ฟเวอร์หรืออินสแตนซ์คอนเทนเนอร์เพียงแห่งเดียวครับ

### ข้อกำหนดฮาร์ดแวร์ขั้นต่ำ

| ทรัพยากร | ข้อกำหนด |
|---|---|
| CPU | 1 คอร์ |
| หน่วยความจำ | 2 GB |

**สถานการณ์ที่เหมาะสม**:

- ธุรกิจขนาดเล็กมาก
- การตรวจสอบฟังก์ชันการทำงาน (POC)
- สภาพแวดล้อมการพัฒนา / ทดสอบ
- สถานการณ์ที่แทบไม่มีการเข้าใช้งานพร้อมกัน

:::info{title=คำแนะนำ}

- การกำหนดค่านี้เพียงเพื่อให้ระบบสามารถทำงานได้เท่านั้น ไม่รับประกันประสิทธิภาพในการใช้งานครับ
- เมื่อปริมาณข้อมูลหรือการร้องขอพร้อมกันเพิ่มขึ้น ทรัพยากรระบบอาจกลายเป็นคอขวดได้อย่างรวดเร็วครับ
- สำหรับการ **พัฒนาซอร์สโค้ด, การพัฒนาปลั๊กอิน หรือการสร้าง (Build) และปรับใช้จากซอร์สโค้ด** แนะนำให้สำรอง **หน่วยความจำว่างมากกว่า 4 GB** เพื่อให้มั่นใจว่ากระบวนการติดตั้ง Dependency, การคอมไพล์ และการสร้างระบบจะเสร็จสมบูรณ์อย่างราบรื่นครับ

:::

### ข้อกำหนดฮาร์ดแวร์ที่แนะนำ

| ทรัพยากร | การกำหนดค่าที่แนะนำ |
|---|---|
| CPU | 2 คอร์ |
| หน่วยความจำ | ≥ 4 GB |

**สถานการณ์ที่เหมาะสม**:

เหมาะสำหรับธุรกิจขนาดเล็กถึงขนาดกลาง และสภาพแวดล้อมการใช้งานจริงที่มีการเข้าใช้งานพร้อมกันจำนวนไม่มากครับ

:::info{title=คำแนะนำ}

- ภายใต้การกำหนดค่านี้ ระบบสามารถรองรับการทำงานทั่วไปของระบบหลังบ้านและภาระงานทางธุรกิจระดับเบาได้ครับ
- เมื่อความซับซ้อนของธุรกิจ การเข้าใช้งานพร้อมกัน หรือภารกิจเบื้องหลังเพิ่มมากขึ้น ควรพิจารณาอัปเกรดสเปกฮาร์ดแวร์หรือเปลี่ยนไปใช้โหมดคลัสเตอร์ครับ

:::

## โหมดคลัสเตอร์ (Cluster Mode)

เหมาะสำหรับสถานการณ์ทางธุรกิจขนาดกลางถึงขนาดใหญ่และมีการเข้าใช้งานพร้อมกันจำนวนมาก โดยสามารถขยายระบบในแนวนอน (Horizontal Scaling) เพื่อเพิ่มความพร้อมใช้งาน (Availability) และความสามารถในการรองรับภาระงานของระบบ (รายละเอียดโปรดอ้างอิง: [โหมดคลัสเตอร์](/cluster-mode)) ครับ

### ข้อกำหนดฮาร์ดแวร์ของโหนด

ในโหมดคลัสเตอร์ ข้อแนะนำการกำหนดค่าฮาร์ดแวร์สำหรับแต่ละโหนดแอปพลิเคชัน (Pod / อินสแตนซ์) จะเหมือนกับโหมดการปรับใช้แบบโหนดเดียวครับ

**การกำหนดค่าขั้นต่ำต่อโหนด:**

- CPU: 1 คอร์
- หน่วยความจำ: 2 GB

**การกำหนดค่าที่แนะนำต่อโหนด:**

- CPU: 2 คอร์
- หน่วยความจำ: 4 GB

### การวางแผนจำนวนโหนด

- จำนวนโหนดในคลัสเตอร์สามารถขยายได้ตามต้องการ (2–N)
- จำนวนโหนดที่ต้องการจริงขึ้นอยู่กับ:
  - ปริมาณการเข้าใช้งานพร้อมกัน
  - ความซับซ้อนของตรรกะทางธุรกิจ
  - ภาระงานของภารกิจเบื้องหลังและการประมวลผลแบบอะซิงโครนัส
  - ความสามารถในการตอบสนองของบริการภายนอกที่เกี่ยวข้อง

คำแนะนำสำหรับสภาพแวดล้อมการใช้งานจริง:

- ปรับขนาดโหนดแบบไดนามิกโดยอ้างอิงจากตัวชี้วัดการตรวจสอบ (เช่น CPU, หน่วยความจำ, ความล่าช้าของการตอบสนอง เป็นต้น)
- สำรองทรัพยากรไว้จำนวนหนึ่งเพื่อรองรับการเพิ่มขึ้นของทราฟฟิกอย่างกะทันหันครับ
