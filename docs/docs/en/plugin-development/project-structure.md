# Project Structure

An application initialized via the NocoBase CLI (`nb init`) generates a standard application directory. The CLI supports two sources — npm (`create-nocobase-app`) and Git — and the top-level structure is the same for both.

## Top-Level Directory Overview

```bash
<app-path>/
├── .nb/                   # CLI metadata for the current env
├── source/                # Application source code project (NocoBase core + built-in plugins)
├── storage/               # Runtime data directory
│   ├── plugins/           # Compiled plugins (uploaded or imported)
│   └── tar/               # Plugin package files (.tgz)
├── plugins/               # Your plugin source code (generated by nb scaffold plugin)
├── .env                   # Application environment variables file
```

## plugins/ Plugin Development Directory

`plugins/` is the primary location for developing custom plugins. Plugins created via `nb scaffold plugin` are placed here.

`nb` automatically syncs plugins under `plugins/` as symlinks to `source/packages/plugins/` for the development and build workflow. You don't need to manually manage anything inside the `source/` directory.

## source/ Source Code Project Directory

The `source/` directory contains the full NocoBase source code project. Its contents depend on the project source:

- **npm source** (`create-nocobase-app`): Contains only base directories such as `packages/plugins/` by default.
- **Git source** (recommended): Contains the full framework core source code (`packages/core/`), built-in plugins, etc. When developing with AI, you can directly reference these.

## storage/ Runtime Directory

`storage/` stores runtime-generated data and build outputs:

- `plugins/`: Packaged plugins uploaded via UI or imported via CLI.
- `tar/`: Compressed plugin packages generated after running `nb source build <plugin> --tar`.

## Plugin Loading Paths and Priority

Plugins may exist in multiple locations. NocoBase loads them in the following priority order when starting:

1. Source code version in `source/packages/plugins` (for local development and debugging, automatically synced by `nb` from `plugins/`).
2. Packaged version in `storage/plugins` (uploaded via UI or imported via CLI).
3. Dependency packages in `node_modules` (installed via npm/yarn or framework built-in).

If a plugin with the same name exists in both the source directory and the packaged directory, NocoBase will prioritize loading the source version, facilitating local overrides and debugging.

## Plugin Directory Template

Create a plugin using the CLI:

```bash
nb scaffold plugin @my-project/plugin-hello
```

The generated directory structure is as follows:

```bash
plugins/@my-project/plugin-hello/
├── dist/                    # Build output (generated as needed)
├── src/                     # Source code directory
│   ├── client-v2/           # Frontend code (blocks, pages, models, etc.)
│   │   ├── plugin.ts        # Client-side plugin main class
│   │   └── index.ts         # Client-side entry
│   ├── locale/              # Multi-language resources (shared between frontend and backend)
│   ├── swagger/             # OpenAPI/Swagger documentation
│   └── server/              # Server-side code
│       ├── collections/     # Collection definitions
│       ├── commands/        # Custom commands
│       ├── migrations/      # Database migration scripts
│       ├── plugin.ts        # Server-side plugin main class
│       └── index.ts         # Server-side entry
├── index.ts                 # Frontend and backend bridge export
├── client-v2.d.ts           # Frontend type declarations
├── client-v2.js             # Frontend build artifact
├── server.d.ts              # Server-side type declarations
├── server.js                # Server-side build artifact
├── .npmignore               # Publish ignore configuration
└── package.json
```

:::tip

After the build completes, the `dist/` directory and `client-v2.js`, `server.js` files will be loaded when the plugin is enabled. During development, you only need to modify the `src/` directory. Before publishing, execute `nb source build <plugin>` or `nb source build <plugin> --tar`.

:::
