# وضع الذاكرة المشتركة

## مقدمة

عندما يرغب المستخدمون في تقسيم الأعمال على مستوى التطبيق دون إدخال بنية معقدة للنشر والصيانة، يمكن استخدام وضع التطبيقات المتعددة في الذاكرة المشتركة.

في هذا الوضع، يمكن تشغيل تطبيقات متعددة في مثيل NocoBase واحد في نفس الوقت. كل تطبيق مستقل، ويمكنه الاتصال بقاعدة بيانات مستقلة، ويمكن إنشاؤه وتشغيله وإيقافه بشكل منفصل، لكنها تشترك في نفس العملية ومساحة الذاكرة، ولا يزال المستخدم بحاجة فقط إلى صيانة مثيل NocoBase واحد.

## دليل الاستخدام

### إعداد متغيرات البيئة

قبل استخدام ميزة التطبيقات المتعددة، يرجى التأكد من ضبط متغيرات البيئة التالية عند بدء تشغيل NocoBase:

```bash
APP_DISCOVERY_ADAPTER=local
APP_PROCESS_ADAPTER=local
```

### إنشاء تطبيق

في قائمة إعدادات النظام، انقر فوق "مراقب التطبيقات" للدخول إلى صفحة إدارة التطبيقات.

![](https://static-docs.nocobase.com/202512291056215.png)

انقر فوق زر "إضافة جديد" لإنشاء تطبيق جديد.

![](https://static-docs.nocobase.com/202512291057696.png)

#### شرح خيارات التكوين

| خيار التكوين | الوصف |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **اسم التطبيق** | الاسم الذي يظهر للتطبيق في الواجهة |
| **معرف التطبيق** | معرف التطبيق، فريد عالميًا |
| **وضع البدء** | - البدء عند الزيارة الأولى: يبدأ التطبيق فقط عندما يصل المستخدم إليه عبر عنوان URL لأول مرة<br>- البدء مع التطبيق الرئيسي: يبدأ التطبيق الفرعي في نفس وقت بدء التطبيق الرئيسي (مما يزيد من وقت بدء التطبيق الرئيسي) |
| **البيئة** | في وضع الذاكرة المشتركة، تتوفر البيئة المحلية فقط، أي `local` |
| **اتصال قاعدة البيانات** | يستخدم لتكوين مصدر البيانات الرئيسي للتطبيق، ويدعم ثلاث طرق:<br>- قاعدة بيانات جديدة: إعادة استخدام خدمة قاعدة البيانات الحالية وإنشاء قاعدة بيانات مستقلة<br>- اتصال بيانات جديد: الاتصال بخدمة قاعدة بيانات أخرى<br>- وضع المخطط (Schema): عندما يكون مصدر البيانات الرئيسي الحالي هو PostgreSQL، يتم إنشاء مخطط مستقل للتطبيق |
| **الترقية** | إذا كانت قاعدة البيانات المتصلة تحتوي على بيانات تطبيق NocoBase من إصدار قديم، فهل يُسمح بالترقية التلقائية إلى إصدار التطبيق الحالي |
| **مفتاح JWT** | إنشاء مفتاح JWT مستقل تلقائيًا للتطبيق لضمان استقلالية جلسة التطبيق عن التطبيق الرئيسي والتطبيقات الأخرى |
| **نطاق مخصص** | تكوين نطاق وصول مستقل للتطبيق |

### بدء التطبيق

انقر فوق زر **بدء** لبدء التطبيق الفرعي.

> إذا تم تحديد *"البدء عند الزيارة الأولى"* أثناء الإنشاء، فسيتم البدء تلقائيًا عند الزيارة الأولى.

![](https://static-docs.nocobase.com/202512291121065.png)

### الوصول إلى التطبيق

انقر فوق زر **زيارة** لفتح التطبيق الفرعي في علامة تبويب جديدة.

يتم استخدام `/apps/:appName/admin/` افتراضيًا للوصول إلى التطبيق الفرعي، على سبيل المثال:

```bash
http://localhost:13000/apps/a_7zkxoarusnx/admin/
```

في الوقت نفسه، يمكن أيضًا تكوين نطاق مستقل للتطبيق الفرعي، ويجب توجيه النطاق إلى عنوان IP الحالي، وإذا تم استخدام nginx، فيجب أيضًا إضافة النطاق في تكوين nginx.

### إيقاف التطبيق

انقر فوق زر **إيقاف** لإيقاف التطبيق الفرعي.

![](https://static-docs.nocobase.com/202512291122113.png)

### حالة التطبيق

يمكنك عرض الحالة الحالية لكل تطبيق في القائمة.

![](https://static-docs.nocobase.com/202512291122339.png)

### حذف التطبيق

انقر فوق زر **حذف** لإزالة التطبيق.

![](https://static-docs.nocobase.com/202512291122178.png)

##常见问题 (الأسئلة الشائعة)

### 1. إدارة الإضافات

الإضافات التي يمكن للتطبيقات الأخرى استخدامها هي نفسها الموجودة في التطبيق الرئيسي (بما في ذلك الإصدارات)، ولكن يمكن تكوين واستخدام الإضافة بشكل مستقل.

### 2. عزل قاعدة البيانات

يمكن للتطبيقات الأخرى تكوين قواعد بيانات مستقلة. إذا كنت ترغب في مشاركة البيانات بين التطبيقات، يمكن تحقيق ذلك من خلال مصدر البيانات الخارجي.

### 3. النسخ الاحتياطي للبيانات ونقلها

حالياً، لا يدعم النسخ الاحتياطي للبيانات في التطبيق الرئيسي تضمين بيانات التطبيقات الأخرى (يحتوي فقط على معلومات التطبيق الأساسية)، ويجب إجراء النسخ الاحتياطي والنقل يدوياً داخل التطبيقات الأخرى.

### 4. النشر والتحديث

في وضع الذاكرة المشتركة، ستتبع إصدارات التطبيقات الأخرى تلقائياً ترقية التطبيق الرئيسي، مما يضمن اتساق إصدارات التطبيقات تلقائياً.

### 5. جلسات التطبيق

- إذا كان التطبيق يستخدم مفتاح JWT مستقلاً، فستكون جلسة التطبيق مستقلة عن التطبيق الرئيسي والتطبيقات الأخرى. إذا تم الوصول إلى تطبيقات مختلفة عبر مسارات فرعية لنفس النطاق، فبسبب تخزين رمز التطبيق (TOKEN) في LocalStorage، ستحتاج إلى تسجيل الدخول مرة أخرى عند التبديل بين التطبيقات المختلفة. يُنصح بتكوين نطاقات مستقلة للتطبيقات المختلفة لتحقيق عزل أفضل للجلسات.
- إذا لم يستخدم التطبيق مفتاح JWT مستقلاً، فسيشارك جلسة التطبيق الرئيسي، ولن تحتاج إلى تسجيل الدخول مرة أخرى عند العودة إلى التطبيق الرئيسي بعد زيارة تطبيقات أخرى في نفس المتصفح. ومع ذلك، هناك مخاطر أمنية؛ فإذا تكررت معرفات المستخدمين في تطبيقات مختلفة، فقد يؤدي ذلك إلى وصول غير مصرح به لبيانات التطبيقات الأخرى.
