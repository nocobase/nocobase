# المصادقة: OIDC

## مقدمة

تتبع إضافة المصادقة OIDC معيار بروتوكول OIDC (OpenID Connect)، باستخدام تدفق رمز التفويض، للسماح للمستخدمين بتسجيل الدخول إلى NocoBase باستخدام حسابات مقدمة من موفري خدمات مصادقة الهوية التابعين لجهات خارجية (IdP).

## التهيئة

### التهيئة الأساسية

![](https://static-docs.nocobase.com/202411130006341.png)

| التهيئة | الوصف | متطلبات الإصدار |
| ------- | ----- | --------------- |
| التسجيل التلقائي عند عدم وجود المستخدم | ما إذا كان سيتم إنشاء مستخدم جديد تلقائيًا إذا لم يتم العثور على مستخدم موجود مطابق. | - |
| Issuer | المُصدر المقدم من IdP، ينتهي عادةً بـ `/.well-known/openid-configuration`. | - |
| Client ID | معرف العميل | - |
| Client Secret | سر العميل | - |
| scope | اختياري، الافتراضي هو `openid email profile`. | - |
| خوارزمية توقيع id_token | خوارزمية التوقيع لـ `id_token`، الافتراضي هو `RS256`. | - |
| تفعيل تسجيل الخروج بمبادرة من RP | تفعيل تسجيل الخروج بمبادرة من RP. يُسجّل الخروج من جلسة IdP عند تسجيل خروج المستخدم. يجب أن يستخدم رد الاتصال لتسجيل الخروج من IdP رابط إعادة التوجيه بعد تسجيل الخروج المقدم في [الاستخدام](#usage). | `v1.3.44-beta` |

### تعيين الحقول

![](https://static-docs.nocobase.com/92d63c8f6f4082b50d9f475674cb5650.png)

| التهيئة | الوصف |
| ------- | ----- |
| خريطة الحقول | تعيين الحقول. يدعم NocoBase تعيين حقول مثل الاسم المستعار والبريد الإلكتروني ورقم الهاتف. الاسم المستعار الافتراضي يستخدم `openid`. |
| استخدام هذا الحقل لربط المستخدم | يُستخدم للمطابقة والربط مع المستخدمين الموجودين. يمكنك اختيار البريد الإلكتروني أو اسم المستخدم، مع البريد الإلكتروني كافتراضي. يجب أن يوفر IdP معلومات `email` أو `username`. |

### التهيئة المتقدمة

![](https://static-docs.nocobase.com/202411130013306.png)

| التهيئة | الوصف | متطلبات الإصدار |
| ------- | ----- | --------------- |
| HTTP | ما إذا كان رابط رد الاتصال لـ NocoBase يستخدم بروتوكول HTTP، الافتراضي هو `https`. | - |
| المنفذ | المنفذ لرابط رد الاتصال لـ NocoBase، الافتراضي هو `443/80`. | - |
| رمز الحالة | يُستخدم للتحقق من مصدر الطلب ومنع هجمات CSRF. يمكنك تقديم قيمة ثابتة، لكن **يُوصى بشدة بتركه فارغًا لتوليد قيم عشوائية افتراضيًا. إذا استخدمت قيمة ثابتة، قيّم بيئتك ومخاطر الأمان بعناية.** | - |
| تمرير المعاملات في تبادل منح رمز التفويض | قد تتطلب بعض موفري IdP تمرير Client ID أو Client Secret كمعاملات عند تبادل الرمز بمقابل رمز مميز. يمكنك تحديد هذا الخيار وتحديد أسماء المعاملات المقابلة. | - |
| طريقة استدعاء نقطة نهاية معلومات المستخدم | طريقة HTTP المستخدمة عند طلب API معلومات المستخدم. | - |
| مكان وضع رمز الوصول عند استدعاء نقطة نهاية معلومات المستخدم | كيفية تمرير رمز الوصول عند استدعاء API معلومات المستخدم:<br/>- الترويسة - في ترويسة الطلب (افتراضي).<br/>- الجسم - في جسم الطلب، يُستخدم مع طريقة `POST`.<br/>- معاملات الاستعلام - كمعاملات استعلام، يُستخدم مع طريقة `GET`. | - |
| تخطي التحقق من SSL | تخطي التحقق من SSL عند طلب API الخاص بـ IdP. **يعرض هذا الخيار نظامك لمخاطر هجمات الوسيط. فعّل هذا الخيار فقط إذا كنت تفهم غرضه وتداعياته. يُثبَّط بشدة في بيئات الإنتاج.** | `v1.3.40-beta` |

### الاستخدام

![](https://static-docs.nocobase.com/202411130019570.png)

| التهيئة | الوصف |
| ------- | ----- |
| رابط إعادة التوجيه | يُستخدم لتهيئة رابط رد الاتصال في IdP. |
| رابط إعادة التوجيه بعد تسجيل الخروج | يُستخدم لتهيئة رابط إعادة التوجيه بعد تسجيل الخروج في IdP عند تفعيل تسجيل الخروج بمبادرة من RP. |

:::info
عند الاختبار محليًا، استخدم `127.0.0.1` بدلاً من `localhost` للرابط، حيث يتطلب تسجيل الدخول عبر OIDC كتابة الحالة في ملف تعريف الارتباط للعميل للتحقق الأمني. إذا رأيت وميضًا لنافذة تسجيل الدخول لكنك فشلت في تسجيل الدخول بنجاح، تحقق من سجلات الخادم بحثًا عن مشاكل عدم تطابق الحالة وتأكد من تضمين معامل الحالة في ملف تعريف ارتباط الطلب. تحدث هذه المشكلة غالبًا عندما لا تتطابق الحالة في ملف تعريف ارتباط العميل مع الحالة في الطلب.
:::
