# Серверні журнали, аудиторські журнали та історія записів

## Серверні журнали

### Системні журнали

> Дивіться [Системні журнали](#)

- Записують інформацію про роботу системи, відстежують ланцюжки виконання коду та допомагають виявляти винятки чи помилки під час виконання.
- Журнали класифікуються за рівнями важливості та функціональними модулями.
- Виводяться через термінал або зберігаються у файлах.
- Переважно використовуються для діагностики та усунення несправностей системи під час її роботи.

### Журнали запитів

> Дивіться [Журнали запитів](#)

- Записують деталі HTTP API запитів та відповідей, зосереджуючись на ID запиту, шляху API, заголовках, коді стану відповіді та тривалості.
- Виводяться через термінал або зберігаються у файлах.
- Переважно використовуються для відстеження викликів API та продуктивності їх виконання.

## Аудиторські журнали

> Дивіться [Аудиторські журнали](../security/audit-logger/index.md)

- Записують дії користувачів (або API) над системними ресурсами, зосереджуючись на типі ресурсу, цільовому об'єкті, типі операції, інформації про користувача та статусі операції.
- Щоб краще відстежувати, що саме робили користувачі та які результати були отримані, параметри запитів та відповіді зберігаються як метадані. Ця інформація частково перетинається з журналами запитів, але не є ідентичною — наприклад, журнали запитів зазвичай не містять повних тіл запитів.
- Параметри запитів та відповіді **не є еквівалентними** знімкам даних. Вони можуть показати, які операції відбулися, але не точні дані до модифікації, тому їх не можна використовувати для контролю версій або відновлення даних після помилкових операцій.
- Зберігаються як у файлах, так і в таблицях бази даних.

## Історія записів

> Дивіться [Історія записів](/record-history/index.md)

- Записує **історію змін** вмісту даних.
- Відстежує тип ресурсу, об'єкт ресурсу, тип операції, змінені поля та значення до/після зміни.
- Корисно для **порівняння та аудиту даних**.
- Зберігається в таблицях бази даних.
