# ctx.logger

Обгортка для логування на базі [pino](https://github.com/pinojs/pino), що забезпечує високоефективні структуровані JSON-логи. Рекомендується використовувати `ctx.logger` замість `console` для полегшення збору та аналізу логів.

## Сценарії використання

`ctx.logger` можна використовувати у всіх сценаріях RunJS для налагодження, відстеження помилок, аналізу продуктивності тощо.

## Визначення типу

```ts
logger: pino.Logger;
```

`ctx.logger` є екземпляром `engine.logger.child({ module: 'flow-engine' })`, тобто дочірнім логером pino з контекстом `module`.

## Рівні логування

pino підтримує наступні рівні (від найвищого до найнижчого):

| Рівень | Метод | Опис |
|------|------|------|
| `fatal` | `ctx.logger.fatal()` | Критична помилка, що зазвичай призводить до завершення процесу |
| `error` | `ctx.logger.error()` | Помилка, що вказує на невдалий запит або операцію |
| `warn` | `ctx.logger.warn()` | Попередження, що вказує на потенційні ризики або аномальні ситуації |
| `info` | `ctx.logger.info()` | Загальна інформація про роботу системи |
| `debug` | `ctx.logger.debug()` | Інформація для налагодження, використовується під час розробки |
| `trace` | `ctx.logger.trace()` | Детальне трасування для глибокої діагностики |

## Рекомендований спосіб запису

Рекомендований формат — `level(msg, meta)`: спочатку повідомлення, потім необов'язковий об'єкт із метаданими.

```ts
ctx.logger.info('Блок завантажено');
ctx.logger.info('Операція успішна', { recordId: 456 });
ctx.logger.warn('Попередження продуктивності', { duration: 5000 });
ctx.logger.error('Операція не вдалася', { userId: 123, action: 'create' });
ctx.logger.error('Запит не вдався', { err });
```

pino також підтримує `level(meta, msg)` (об'єкт першим) або `level({ msg, ...meta })` (єдиний об'єкт), які можна використовувати за потреби.

## Приклади

### Базове використання

```ts
ctx.logger.info('Блок завантажено');
ctx.logger.warn('Запит не вдався, використовується кеш', { err });
ctx.logger.debug('Збереження...', { recordId: ctx.record?.id });
```

### Створення дочірнього логера за допомогою child()

```ts
// Створення дочірнього логера з контекстом для поточної логіки
const log = ctx.logger.child({ scope: 'myBlock' });
log.info('Виконання кроку 1');
log.debug('Виконання кроку 2', { step: 2 });
```

### Зв'язок із console

Рекомендується використовувати `ctx.logger` напряму для отримання структурованих JSON-логів. Якщо ви звикли використовувати `console`, відповідність наступна: `console.log` → `ctx.logger.info`, `console.error` → `ctx.logger.error`, `console.warn` → `ctx.logger.warn`.

## Формат логів

pino виводить структурований JSON, де кожен запис містить:

- `level`: рівень логування (число)
- `time`: мітка часу (мілісекунди)
- `msg`: повідомлення логу
- `module`: фіксоване значення `flow-engine`
- Інші користувацькі поля (передані через об'єкти)

## Примітки

- Логи є структурованим JSON, що полегшує їх збір, пошук та аналіз.
- Для дочірніх логерів, створених через `child()`, також рекомендується формат `level(msg, meta)`.
- Деякі середовища виконання (наприклад, робочі процеси) можуть використовувати інші методи виведення логів.
