# Системні вимоги

Системні вимоги, описані в цьому документі, стосуються **самого сервісу застосунку NocoBase** і охоплюють обчислювальні ресурси та пам'ять, необхідні для процесів застосунку. Вони **не включають залежні сторонні сервіси**, зокрема:

- API-шлюзи / зворотні проксі-сервери
- Сервіси баз даних (наприклад, MySQL, PostgreSQL)
- Сервіси кешування (наприклад, Redis)
- Проміжне програмне забезпечення, таке як черги повідомлень, об'єктні сховища тощо.

За винятком сценаріїв перевірки функціональності або експериментального використання, **наполегливо рекомендується розгортати вищезазначені сторонні сервіси окремо** на виділених серверах чи в контейнерах, або безпосередньо використовувати відповідні хмарні сервіси.

Конфігурацію системи та планування потужностей для цих сервісів слід оцінювати та налаштовувати окремо, виходячи з **фактичного обсягу даних, робочого навантаження та рівня конкурентності**.

## Режим розгортання на одному вузлі

Режим розгортання на одному вузлі означає, що сервіс застосунку NocoBase працює на одному сервері або в одному екземплярі контейнера.

### Мінімальні апаратні вимоги

| Ресурс | Вимога |
|---|---|
| CPU | 1 ядро |
| Пам'ять | 2 ГБ |

**Сценарії застосування**:

- Мікробізнес
- Перевірка концепції (POC)
- Середовища розробки / тестування
- Сценарії з майже повною відсутністю одночасних запитів

:::info{title=Підказка}

- Ця конфігурація лише гарантує працездатність системи, але не гарантує продуктивність.
- Зі зростанням обсягу даних або кількості одночасних запитів системні ресурси можуть швидко стати «вузьким місцем».
- Для сценаріїв **розробки вихідного коду, розробки плагінів або збірки та розгортання з вихідного коду** рекомендується зарезервувати **понад 4 ГБ вільної пам'яті**, щоб забезпечити успішне встановлення залежностей, компіляцію та процес збірки.

:::

### Рекомендовані апаратні вимоги

| Ресурс | Рекомендована конфігурація |
|---|---|
| CPU | 2 ядра |
| Пам'ять | ≥ 4 ГБ |

**Сценарії застосування**:

Підходить для малого та середнього бізнесу, а також для виробничих середовищ із невеликою кількістю одночасних запитів.

:::info{title=Підказка}

- За такої конфігурації система може забезпечити виконання стандартних операцій панелі керування та легких робочих навантажень.
- Коли складність бізнес-логіки, кількість одночасних запитів або фонових завдань зростає, слід розглянути можливість оновлення апаратних характеристик або переходу до кластерного режиму.

:::

## Кластерний режим

Підходить для середніх і великих бізнес-сценаріїв з високим рівнем конкурентності. Ви можете використовувати горизонтальне масштабування для підвищення доступності системи та пропускної здатності (детальніше див.: [Кластерний режим](/cluster-mode)).

### Вимоги до апаратного забезпечення вузла

У кластерному режимі рекомендована апаратна конфігурація для кожного вузла застосунку (Pod / екземпляр) відповідає вимогам режиму розгортання на одному вузлі.

**Мінімальна конфігурація вузла:**

- CPU: 1 ядро
- Пам'ять: 2 ГБ

**Рекомендована конфігурація вузла:**

- CPU: 2 ядра
- Пам'ять: 4 ГБ

### Планування кількості вузлів

- Кількість вузлів у кластері можна масштабувати за потреби (2–N).
- Фактична необхідна кількість вузлів залежить від:
  - Кількість одночасних запитів
  - Складність бізнес-логіки
  - Навантаження від фонових завдань та асинхронної обробки
  - Швидкість відповіді зовнішніх залежних сервісів

Рекомендації для виробничих середовищ:

- Динамічно коригуйте кількість вузлів на основі показників моніторингу (CPU, пам'ять, затримка запитів тощо).
- Резервуйте певну кількість ресурсів для обробки сплесків трафіку.
