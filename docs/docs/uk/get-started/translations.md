# Внесок у переклад

Мовою за замовчуванням у NocoBase є англійська. Наразі основний додаток підтримує англійську, італійську, нідерландську, спрощену китайську та японську мови. Ми щиро запрошуємо вас зробити свій внесок у переклад іншими мовами, щоб користувачі в усьому світі могли насолоджуватися ще зручнішим досвідом роботи з NocoBase.

---

## I. Локалізація системи

### 1. Переклад інтерфейсу системи та плагінів

#### 1.1 Область перекладу
Це стосується лише локалізації інтерфейсу системи NocoBase та її плагінів і не охоплює інший користувацький контент (наприклад, таблиці даних або блоки Markdown).

![20250319220127](https://static-docs.nocobase.com/20250319220127.png)


#### 1.2 Огляд вмісту локалізації
NocoBase використовує Git для керування вмістом локалізації. Основним репозиторієм є:
https://github.com/nocobase/nocobase/tree/main/locales

Кожна мова представлена файлом JSON, названим відповідно до її мовного коду (наприклад, de-DE.json, fr-FR.json). Структура файлу організована за модулями плагінів, де для зберігання перекладів використовуються пари "ключ-значення". Наприклад:

```json
{
  // Клієнтський плагін
  "@nocobase/client": {
    "(Fields only)": "(Fields only)",
    "12 hour": "12 hour",
    "24 hour": "24 hour"
    // ...інші пари ключ-значення
  },
  "@nocobase/plugin-acl": {
    // Пари ключ-значення для цього плагіна
  }
  // ...інші модулі плагінів
}
```

Під час перекладу, будь ласка, поступово перетворюйте його на структуру, подібну до наступної:

```json
{
  // Клієнтський плагін
  "@nocobase/client": {
    "(Fields only)": "(Тільки поля - перекладено)",
    "12 hour": "12-годинний",
    "24 hour": "24-годинний"
    // ...інші пари ключ-значення
  },
  "@nocobase/plugin-acl": {
    // Пари ключ-значення для цього плагіна
  }
  // ...інші модулі плагінів
}
```

#### 1.3 Тестування та синхронізація перекладу

Ми також випустили плагін для перевірки перекладу — знайдіть `Locale tester` у маркетплейсі плагінів.

## II. Локалізація документації (NocoBase 2.0)

Документація для NocoBase 2.0 керується за новою структурою. Вихідні файли документації знаходяться в основному репозиторії NocoBase:

https://github.com/nocobase/nocobase/tree/next/docs

### 2.1 Структура документації

Документація використовує [Rspress](https://rspress.dev/) як генератор статичних сайтів і підтримує 22 мови. Структура організована наступним чином:

```

│   ├── en/                    # Англійська (вихідна мова)
│   ├── cn/                    # Спрощена китайська
│   ├── ja/                    # Японська
│   ├── ko/                    # Корейська
│   ├── de/                    # Німецька
│   ├── fr/                    # Французька
│   ├── es/                    # Іспанська
│   ├── pt/                    # Португальська
│   ├── ru/                    # Російська
│   ├── it/                    # Італійська
│   ├── tr/                    # Турецька
│   ├── uk/                    # Українська
│   ├── vi/                    # В'єтнамська
│   ├── id/                    # Індонезійська
│   ├── th/                    # Тайська
│   ├── pl/                    # Польська
│   ├── nl/                    # Нідерландська
│   ├── cs/                    # Чеська
│   ├── ar/                    # Арабська
│   ├── he/                    # Іврит
│   ├── hi/                    # Гінді
│   ├── sv/                    # Шведська
│   └── public/                # Спільні ресурси (зображення тощо)
├── theme/                     # Користувацька тема
├── rspress.config.ts          # Конфігурація Rspress
└── package.json
```

### 2.2 Робочий процес перекладу

1. **Синхронізація з англійським джерелом**: Усі переклади мають базуватися на англійській документації (`docs/en/`). Коли англійська документація оновлюється, переклади мають бути оновлені відповідно.

2. **Стратегія розгалуження**:
   - Використовуйте гілку `develop` або `next` як еталон для останнього англійського вмісту
   - Створіть свою гілку перекладу з цільової гілки

3. **Структура файлів**: Кожна мовна директорія повинна дзеркально відображати структуру англійської директорії. Наприклад:
   ```
   docs/en/get-started/index.md    →    docs/uk/get-started/index.md
   docs/en/api/acl/acl.md          →    docs/uk/api/acl/acl.md
   ```

### 2.3 Внесок у переклад

1. Зробіть Fork репозиторію: https://github.com/nocobase/nocobase
2. Клонуйте свій fork та перейдіть у гілку `develop` або `next`
3. Перейдіть до директорії `docs/docs/`
4. Знайдіть мовну директорію, у яку ви хочете зробити внесок (наприклад, `uk/` для української)
5. Перекладіть markdown-файли, зберігаючи ту саму структуру файлів, що й в англійській версії
6. Протестуйте зміни локально:
   ```bash
   cd docs
   yarn install
   yarn dev
   ```
7. Надішліть Pull Request до основного репозиторію

### 2.4 Рекомендації щодо перекладу

- **Дотримуйтесь послідовності форматування**: Зберігайте ту саму структуру markdown, заголовки, блоки коду та посилання, що й у джерелі
- **Зберігайте frontmatter**: Залишайте будь-який YAML frontmatter у верхній частині файлів без змін, якщо він не містить вмісту для перекладу
- **Посилання на зображення**: Використовуйте ті самі шляхи до зображень із `docs/public/` — зображення є спільними для всіх мов
- **Внутрішні посилання**: Оновлюйте внутрішні посилання, щоб вони вказували на правильний мовний шлях
- **Приклади коду**: Як правило, приклади коду не слід перекладати, але коментарі в коді перекладати можна

### 2.5 Конфігурація навігації

Структура навігації для кожної мови визначена у файлах `_nav.json` та `_meta.json` у кожній мовній директорії. При додаванні нових сторінок або розділів обов'язково оновіть ці конфігураційні файли.

### 3.2 Структура вмісту та метод локалізації

Ми використовуємо змішаний підхід до керування вмістом. Англійський, китайський та японський вміст і ресурси регулярно синхронізуються з CMS-системи та перезаписуються, тоді як інші мови можна редагувати безпосередньо в локальних файлах. Локальний вміст зберігається в директорії `content`, організованій наступним чином:

```
/content
  /articles        # Статті блогу
    /article-slug
      index.md     # Англійський вміст (за замовчуванням)
      index.cn.md  # Китайський вміст
      index.ja.md  # Японський вміст
      metadata.json # Метадані та інші властивості локалізації
  /tutorials       # Навчальні посібники
  /releases        # Інформація про релізи
  /pages           # Деякі статичні сторінки
  /categories      # Інформація про категорії
    /article-categories.json  # Список категорій статей
    /category-slug            # Деталі окремої категорії
      /category.json
  /tags            # Інформація про теги
    /article-tags.json        # Список тегів статей
    /release-tags.json        # Список тегів релізів
    /tag-slug                 # Деталі окремого тегу
      /tag.json
  /help-center     # Вміст центру допомоги
    /help-center-tree.json    # Навігаційна структура центру допомоги
  ....
```

### 3.3 Рекомендації щодо перекладу вмісту

- Про переклад вмісту Markdown

1. Створіть новий мовний файл на основі файлу за замовчуванням (наприклад, з `index.md` у `index.uk.md`)
2. Додайте локалізовані властивості у відповідні поля у файлі JSON
3. Дотримуйтесь послідовності в структурі файлів, посиланнях та посиланнях на зображення

- Переклад вмісту JSON
Багато метаданих вмісту зберігаються у файлах JSON, які зазвичай містять багатомовні поля:

```json
{
  "id": 123,
  "title": "English Title",       // Англійський заголовок (за замовчуванням)
  "title_cn": "中文标题",          // Китайський заголовок
  "title_ja": "日本語タイトル",    // Японський заголовок
  "description": "English description",
  "description_cn": "中文描述",
  "description_ja": "日本語の説明",
  "slug": "article-slug",         // URL-шлях (зазвичай не перекладається)
  "status": "published",
  "publishedAt": "2025-03-19T12:00:00Z"
}
```

**Примітки до перекладу:**

1. **Угода про іменування полів**: Поля перекладу зазвичай використовують формат `{оригінальне_поле}_{код_мови}`
   - Наприклад: title_uk (український заголовок), description_uk (український опис)

2. **При додаванні нової мови**:
   - Додайте відповідну версію з мовним суфіксом для кожного поля, яке потребує перекладу
   - Не змінюйте оригінальні значення полів (таких як title, description тощо), оскільки вони служать вмістом мови за замовчуванням (англійської)

3. **Механізм синхронізації CMS**:
   - CMS-система періодично оновлює англійський, китайський та японський вміст
   - Система лише оновлюватиме/перезаписуватиме вміст для цих трьох мов (деякі властивості в JSON) і **не видалятиме** мовні поля, додані іншими учасниками
   - Наприклад: якщо ви додали український переклад (title_uk), синхронізація CMS не вплине на це поле


### 3.4 Налаштування підтримки нової мови

Щоб додати підтримку нової мови, вам потрібно змінити конфігурацію `SUPPORTED_LANGUAGES` у файлі `src/utils/index.ts`:

```typescript
export const SUPPORTED_LANGUAGES = {
  en: {
    code: 'en',
    locale: 'en-US',
    name: 'English',
    default: true
  },
  cn: {
    code: 'cn',
    locale: 'zh-CN',
    name: 'Chinese'
  },
  ja: {
    code: 'ja',
    locale: 'ja-JP',
    name: 'Japanese'
  },
  // Приклад додавання нової мови:
  uk: {
    code: 'uk',
    locale: 'uk-UA',
    name: 'Ukrainian'
  }
};
```

### 3.5 Файли макетів та стилі

Кожна мова потребує відповідних файлів макетів:

1. Створіть новий файл макета (наприклад, для української створіть `src/layouts/BaseUK.astro`)
2. Ви можете скопіювати існуючий файл макета (наприклад, `BaseEN.astro`) та перекласти його
3. Файл макета містить переклади для глобальних елементів, таких як меню навігації, футери тощо
4. Обов'язково оновіть конфігурацію перемикача мов, щоб правильно перемикатися на щойно додану мову

### 3.6 Створення мовних директорій сторінок

Створіть незалежні директорії сторінок для нової мови:

1. Створіть папку з назвою коду мови в директорії `src` (наприклад, `src/uk/`)
2. Скопіюйте структуру сторінок з інших мовних директорій (наприклад, `src/en/`)
3. Оновіть вміст сторінок, переклавши заголовки, описи та текст цільовою мовою
4. Переконайтеся, що сторінки використовують правильний компонент макета (наприклад, `.layout: '@/layouts/BaseUK.astro'`)

### 3.7 Локалізація компонентів

Деякі загальні компоненти також потребують перекладу:

1. Перевірте компоненти в директорії `src/components/`
2. Зверніть особливу увагу на компоненти з фіксованим текстом (наприклад, панелі навігації, футери тощо)
3. Компоненти можуть використовувати умовний рендеринг для відображення вмісту різними мовами:

```astro
{Astro.url.pathname.startsWith('/en') && <p>English content</p>}
{Astro.url.pathname.startsWith('/uk') && <p>Український вміст</p>}
```

### 3.8 Тестування та перевірка

Після завершення перекладу проведіть ретельне тестування:

1. Запустіть вебсайт локально (зазвичай за допомогою `yarn dev`)
2. Перевірте, як усі сторінки відображаються новою мовою
3. Переконайтеся, що функціональність перемикання мов працює належним чином
4. Переконайтеся, що всі посилання вказують на сторінки правильної мовної версії
5. Перевірте адаптивні макети, щоб переконатися, що перекладений текст не порушує дизайн сторінки

## IV. Як почати переклад

Якщо ви хочете зробити внесок у переклад NocoBase новою мовою, будь ласка, виконайте наступні кроки:

| Компонент | Репозиторій | Гілка | Примітки |
|-----------|------------|--------|-------|
