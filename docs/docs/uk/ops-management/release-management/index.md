# Управління релізами

## Вступ

Зазвичай, у реальних застосунках, для забезпечення безпеки даних та стабільної роботи програми, розгортають кілька середовищ: середовище розробки, середовище попереднього випуску та виробниче середовище. У цьому документі ми розглянемо два типові процеси безкодового розроблення та детально пояснимо, як реалізувати управління релізами в NocoBase.

## Встановлення

Для управління релізами необхідні три плагіни. Будь ласка, переконайтеся, що всі вони активовані.

### Змінні середовища

- Вбудований плагін, встановлений та активований за замовчуванням.
- Забезпечує централізовану конфігурацію та управління змінними середовища та ключами, що використовуються для зберігання конфіденційних даних, повторного використання даних конфігурації, ізоляції конфігурацій за середовищами тощо ([Переглянути документацію](#)).

### Менеджер резервного копіювання

- Доступний лише у версії Professional або вище ([Дізнатися більше](https://www.nocobase.com/en/commercial)).
- Підтримує функції резервного копіювання та відновлення, включно з плановим резервним копіюванням, забезпечуючи безпеку даних та швидке відновлення ([Переглянути документацію](../backup-manager/index.mdx)).

### Менеджер міграцій

- Доступний лише у версії Professional або вище ([Дізнатися більше](https://www.nocobase.com/en/commercial)).
- Використовується для міграції конфігурацій застосунку з одного середовища застосунку до іншого ([Переглянути документацію](../migration-manager/index.md)).

## Поширені процеси безкодового розроблення

### Одне середовище розробки, односторонній випуск

Цей підхід підходить для простих процесів розроблення. Існує одне середовище розробки, одне середовище попереднього випуску та одне виробниче середовище. Зміни послідовно переміщуються із середовища розробки до середовища попереднього випуску, а потім розгортаються у виробничому середовищі. У цьому процесі лише середовище розробки може змінювати конфігурації; ані середовище попереднього випуску, ані виробниче середовище не дозволяють модифікацій.

![20250106234710](https://static-docs.nocobase.com/20250106234710.png)

При налаштуванні правил міграції для вбудованих таблиць ядра та плагінів оберіть правило «Перезаписати пріоритетно». Для інших таблиць, якщо немає особливих вимог, можете залишити налаштування за замовчуванням.

![20250105194845](https://static-docs.nocobase.com/20250105194845.png)

### Кілька середовищ розробки, об'єднаний випуск

Цей підхід підходить для сценаріїв спільної роботи кількох осіб або складних проєктів. Кілька паралельних середовищ розробки можуть використовуватися незалежно, а всі зміни об'єднуються в єдине середовище попереднього випуску для тестування та перевірки, перш ніж бути розгорнутими у виробничому середовищі. У цьому процесі також лише середовище розробки може змінювати конфігурації; ані середовище попереднього випуску, ані виробниче середовище не дозволяють модифікацій.

![20250107103829](https://static-docs.nocobase.com/20250107103829.png)

При налаштуванні правил міграції для вбудованих таблиць ядра та плагінів оберіть правило «Вставити або оновити пріоритетно». Для інших таблиць, якщо немає особливих вимог, можете залишити налаштування за замовчуванням.

![20250105194942](https://static-docs.nocobase.com/20250105194942.png)

## Відкат

Перед виконанням міграції система автоматично створює резервну копію поточного застосунку. Якщо міграція не вдалася або результати не відповідають очікуванням, ви можете виконати відкат та відновити дані за допомогою [Менеджера резервного копіювання](../backup-manager/index.mdx).
