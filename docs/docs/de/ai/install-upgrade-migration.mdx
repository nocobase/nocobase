# Anleitung zu Installation, Upgrade und Migration mit der NocoBase CLI

:::danger
Die Installations- und Wartungsfunktionen der CLI befinden sich noch in Entwicklung. Sie eignen sich derzeit eher für lokale Entwicklungs- und Testumgebungen sowie für die Anbindung von AI Agents und werden nicht für die Installation, das Upgrade oder den laufenden Betrieb in Produktionsumgebungen empfohlen.
:::

Die neue NocoBase CLI (`nb`) bündelt Installation, Verbindung, Start, Stopp, Logs, Upgrade, Bereinigung und weitere Operationen in einem einheitlichen Befehlssatz. Im Vergleich zu den bisherigen Abläufen, die in der alten Dokumentation getrennt nach Docker, create-nocobase-app und Git-Quellcode gepflegt wurden, eignet sich die CLI besser für die Wartung lokaler Entwicklungs- und Testumgebungen sowie für die Anbindung und Zusammenarbeit mit AI Agents.

Dieser Artikel ist für die folgenden Szenarien geeignet:

- Sie installieren oder aktualisieren NocoBase derzeit gemäß der alten Dokumentation und möchten auf die neue CLI-Methode umsteigen.
- Sie betreiben bereits eine NocoBase-Instanz, die nach der alten Methode installiert wurde, und möchten sie über die CLI verwalten oder verbinden.
- Sie müssen für einen AI Agent eine verbindbare und wartbare NocoBase-Umgebung bereitstellen.

## Unterschiede zwischen alter und neuer Methode

Die alte Methode teilt die Dokumentation nach Installationsquelle auf; jede Quelle erfordert eigene Befehle für Download, Konfiguration, Installation, Start und Upgrade.

**Alte Installationsdokumente**

- [Docker-Installation](/get-started/installation/docker)
- [create-nocobase-app-Installation](/get-started/installation/create-nocobase-app)
- [Git-Quellcode-Installation](/get-started/installation/git)

**Alte Upgrade-Dokumente**

- [Upgrade einer Docker-Installation](/get-started/upgrading/docker)
- [Upgrade einer create-nocobase-app-Installation](/get-started/upgrading/create-nocobase-app)
- [Upgrade einer Git-Quellcode-Installation](/get-started/upgrading/git)

Die neue Methode nutzt `nb init` als Einstiegspunkt und speichert die Anwendungsquelle, das Lauf- und Speicherverzeichnis sowie die Datenbank- und API-Verbindungsinformationen in einer CLI-env. Unabhängig davon, ob die Quelle Docker, npm oder Git ist, werden anschließend nach Möglichkeit dieselben Befehle `nb app`, `nb source` und `nb db` zur Wartung verwendet.

| Szenario | Alte Methode | Neue CLI-Methode |
| --- | --- | --- |
| Installations-Einstiegspunkt | `docker compose`, `create-nocobase-app`, `git clone` | `nb init` oder `nb init --ui` |
| Anwendungsbetrieb | Verzeichnis wechseln, Compose-Datei bearbeiten, yarn/pm2-Befehle ausführen | `nb app start/stop/restart/logs/down` |
| Upgrade | Pro Installationsmethode eigene Befehle | `nb app upgrade -e <env>` |
| Entwicklungsmodus | `yarn dev` | `nb source dev -e <env>` |
| Datenbankstatus | Manuell Container oder Datenbankdienst prüfen | `nb db ps -e <env>` |
| Umgebungsstatus und Details anzeigen | Pfade, Ports und Adressen manuell pflegen | `nb env list`, `nb env info <env>` |
| Bestehende Anwendung anbinden | In der alten Dokumentation nicht einheitlich abgedeckt | `nb init --ui` oder `nb init --api-base-url` |

## CLI installieren

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Nach der Installation können Sie die Hilfe aufrufen:

```bash
nb --help
nb init --help
```

## NocoBase mit der neuen Methode installieren

### Empfohlen: visueller Assistent

```bash
nb init --ui
```

Der Assistent führt Sie durch folgende Auswahl:

- Neue Anwendung erstellen oder bestehende Anwendung verbinden
- Installationsquelle: Docker, npm, Git
- NocoBase-Version
- Eingebaute Datenbank verwenden oder externe Datenbank
- Administratorkonto und Passwort

### Interaktiver Assistent im Terminal

```bash
nb init
```

### Nicht-interaktiver Befehl

Standardinstallationsmethode ist Docker mit eingebauter PostgreSQL-Datenbank:

```bash
nb init --yes --env app1
```

Mit Docker eine bestimmte Version installieren:

```bash
nb init --yes --env app1 --source docker --version beta
```

Mit npm installieren:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Mit Git-Quellcode installieren:

```bash
nb init --yes --env app1 --source git --version beta
```

Mit Git-Quellcode installieren und den Typ der eingebauten Datenbank festlegen:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Bestehende NocoBase-Anwendung anbinden:

```bash
# Standardmäßig OAuth-Authentifizierung
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Token-Authentifizierung
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` ist der Umgebungsname in der CLI; nachfolgende Befehle wie Start, Upgrade und Logs lassen sich mit `-e app1` auf die gewünschte Anwendung beziehen.
:::

## Mit der neuen Methode aktualisieren

In der alten Methode unterscheiden sich die Upgrade-Befehle für Docker, create-nocobase-app und Git-Quellcode; die neue CLI vereinheitlicht sie zu:

```bash
nb app upgrade -e app1
```

Wenn Sie nur den bereits heruntergeladenen Quellcode oder das vorhandene Image für den Upgrade-Vorgang nutzen möchten, ohne neuen Code oder ein neues Image abzurufen:

```bash
nb app upgrade -e app1 --skip-code-update
```

Oder als Kurzform:

```bash
nb app upgrade -e app1 -s
```

## Häufige Wartungsbefehle

| Aktion | Befehl |
| --- | --- |
| Anwendung starten | `nb app start -e app1` |
| Anwendung stoppen | `nb app stop -e app1` |
| Anwendung neu starten | `nb app restart -e app1` |
| Logs ansehen | `nb app logs -e app1` |
| env- und API-Authentifizierungsstatus | `nb env list` |
| env-Details ansehen | `nb env info app1` |
| Status der eingebauten Datenbank ansehen | `nb db ps -e app1` |
| Anwendung aktualisieren | `nb app upgrade -e app1` |
| Stoppen und Laufzeitressourcen bereinigen | `nb app down -e app1` |

`nb app down` stoppt standardmäßig die Anwendung und entfernt die von der CLI verwalteten Laufcontainer sowie die lokalen Anwendungsdateien, behält aber die storage-Daten und die env-Konfiguration. Um alles vollständig zu löschen, ist eine explizite Bestätigung erforderlich:

```bash
nb app down -e app1 --all --yes
```

:::warning
`--all --yes` löscht die storage-Daten und die CLI-env-Konfiguration dieser env. Verwenden Sie es nur, wenn Sie sicher gesichert haben und die Umgebung nicht mehr benötigen.
:::

## Tägliche Entwicklungsbefehle

`nb source` dient zur Verwaltung des lokalen Quellcode-Projekts und ist hauptsächlich für npm- und Git-source-envs gedacht.

| Aktion | Befehl |
| --- | --- |
| Anwendung im Entwicklungsmodus ausführen | `nb source dev -e app1` |
| Quellcode bauen | `nb source build --cwd /your/workspace/app1/source` |
| Tests ausführen | `nb source test --cwd /your/workspace/app1/source` |
| Quellcode/Image herunterladen oder aktualisieren | `nb source download --source git --version beta` |

:::tip
Docker-envs werden in der Regel über `nb app start`, `nb app logs` und `nb app upgrade` verwaltet; `nb source dev` wird hier nicht verwendet.
:::

## NB_CLI_ROOT und Verzeichnisstruktur

Die CLI speichert ihre globale Konfiguration und die lokalen Anwendungsdateien im Verzeichnis, das `NB_CLI_ROOT` entspricht.

Standardmäßig ist `NB_CLI_ROOT` das Home-Verzeichnis des aktuellen Nutzers, also der von Node.js `os.homedir()` zurückgegebene Pfad. Auf macOS ist das beispielsweise üblicherweise `/Users/<user>`. Ohne zusätzliche Konfiguration werden die durch `nb init --yes --env app1` erzeugten Dateien daher typischerweise hier abgelegt:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Sie können `NB_CLI_ROOT` auch explizit festlegen:

```bash
export NB_CLI_ROOT=/your/workspace
```

Nach dem Setzen werden Konfigurationsdateien und das standardmäßige Anwendungsverzeichnis in diesem Verzeichnis abgelegt:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Dabei gilt:

- `.nocobase/config.json` speichert envs, die aktuelle Umgebung, API-Adresse, Quellcode-Pfad, storage-Pfad, Datenbankkonfiguration und weitere Informationen.
- `<envName>/source` ist das von der CLI verwaltete Quellcode- bzw. Anwendungsverzeichnis.
- `<envName>/storage` ist das von der CLI verwaltete Speicherverzeichnis der Anwendung.

:::warning
Verwenden Sie nach der Initialisierung einer Anwendung weiterhin dasselbe `NB_CLI_ROOT`. Wenn Sie später `NB_CLI_ROOT` ändern, sucht die CLI im neuen Verzeichnis nach `.nocobase/config.json`, und die als relative Pfade gespeicherten `appRootPath` und `storagePath` werden ebenfalls anhand des neuen Verzeichnisses aufgelöst, was dazu führen kann, dass die Anwendung nicht gefunden oder in falsche Verzeichnisse geschrieben wird.
:::

Die CLI nutzt derzeit ausschließlich den globalen Konfigurations-Scope. Wenn Sie den Speicherort der Konfiguration und der lokalen Anwendungsdateien ändern möchten, setzen Sie bitte `NB_CLI_ROOT`.

Verwenden Sie zum Anzeigen und Auffinden von Anwendungen vorzugsweise `nb env list` und `nb env info`:

```bash
# Konfigurierte envs und API-Authentifizierungsstatus anzeigen
nb env list

# Detailinformationen einer env anzeigen
nb env info app1
```

`nb env list` ist auf einen schnellen Überblick ausgelegt; `App Status` zeigt den Authentifizierungsstatus, der durch den Zugriff auf die API mit den gespeicherten Token/OAuth-Anmeldedaten ermittelt wird, und nicht mehr den Datenbankstatus. Den Laufstatus der eingebauten Datenbank prüfen Sie mit:

```bash
nb db ps -e app1
```

Nur wenn Sie die Standard-env wechseln möchten, verwenden Sie:

```bash
nb env use app1
```

## Migration von der alten Methode zur CLI

Es gibt zwei Migrationsstrategien: **Bestehende Anwendung anbinden** oder **das lokale Anwendungsverzeichnis unter CLI-Verwaltung stellen**. Wenn Ihre alte Anwendung bereits stabil läuft, empfehlen wir zunächst „Bestehende Anwendung anbinden", da das Risiko am geringsten ist.

### Variante 1: Bestehende Anwendung anbinden

Geeignet für NocoBase-Instanzen, die bereits per Docker, create-nocobase-app, Git-Quellcode oder Server-Deployment laufen.

```bash
# Standardmäßig OAuth-Authentifizierung
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Token-Authentifizierung
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Diese Variante speichert nur die API-Verbindung; die CLI übernimmt nicht das Starten, Stoppen, Aktualisieren, das Quellcode-Verzeichnis oder das storage-Verzeichnis der alten Anwendung. Sie eignet sich, um einem AI Agent oder den CLI-API-Befehlen die Verbindung zu einer bestehenden Anwendung zu ermöglichen.

Wenn die Authentifizierung aktualisiert werden muss:

```bash
nb env auth app1
```

### Variante 2: Neue CLI-env anlegen und Dateien migrieren

Geeignet, wenn Sie die lokale Anwendung künftig mit `nb app` und `nb source` verwalten möchten.

Bevor Sie mit der Migration beginnen:

1. Stoppen Sie die alte Anwendung.
2. Sichern Sie die alte Datenbank.
3. Sichern Sie das alte `storage`-Verzeichnis.
4. Notieren Sie wichtige Umgebungsvariablen der alten Anwendung, z. B. `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Migration einer alten Docker-Installation

Bei der alten Docker-Methode liegen die Anwendungsdaten üblicherweise im `storage`-Verzeichnis innerhalb des Projektordners:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Es wird empfohlen, zuerst das neue `NB_CLI_ROOT` festzulegen und anschließend mit der CLI eine neue Docker-env zu erstellen. Im folgenden Beispiel wird angenommen, dass `NB_CLI_ROOT` `/your/workspace` ist.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

Die neue env verwendet standardmäßig folgende Verzeichnisse:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Kopieren Sie anschließend das alte `storage` nach `/your/workspace/app2/storage` und stellen Sie sicher, dass Datenbankverbindung, `APP_KEY` und weitere Konfigurationen mit der alten Umgebung übereinstimmen.

Sobald die Kopie abgeschlossen und die Datenbankkonfiguration bestätigt ist, starten Sie die Anwendung:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Migration einer alten create-nocobase-app- oder Git-Quellcode-Installation

Bei der alten npm-/Git-Methode liegen Quellcode, `.env` und `storage` üblicherweise im selben Projektverzeichnis:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Es wird empfohlen, zuerst das neue `NB_CLI_ROOT` festzulegen und anschließend eine neue npm- oder Git-env zu erstellen. Im folgenden Beispiel wird angenommen, dass `NB_CLI_ROOT` `/your/workspace` ist.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Anschließend nach Bedarf migrieren:

- Quellcode eigener Plugins aus dem alten Projekt in das entsprechende Verzeichnis unter `/your/workspace/app3/source` übertragen.
- `storage` aus dem alten Projekt nach `/your/workspace/app3/storage` kopieren.
- Wichtige Konfigurationswerte aus dem alten `.env` in die CLI-env-Konfiguration oder in die Initialisierungsparameter übernehmen.
- Wenn Sie die ursprüngliche Datenbank weiter verwenden, deaktivieren Sie bei der Initialisierung die eingebaute Datenbank und tragen Sie die Verbindungsdaten der alten Datenbank ein.

Beispiel für die Initialisierung mit einer externen Datenbank:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Nach der Migration starten und Logs prüfen:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
Überschreiben Sie `source`, `storage` nicht ohne Backup und führen Sie kein Upgrade gegen eine produktive Datenbank durch. Vor einer offiziellen Migration sollten Sie den gesamten Ablauf zunächst in einer Testumgebung vollständig durchspielen.
:::

## Schnellreferenz: alte Befehle und neue CLI-Befehle

| Alter Befehl | Neuer CLI-Befehl |
| --- | --- |
| `docker compose up -d app` | `nb app start -e app1` |
| `docker compose stop app` | `nb app stop -e app1` |
| `docker compose logs -f app` | `nb app logs -e app1` |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1` |
| `yarn create nocobase-app my-nocobase-app ...` | `nb init --yes --env app1 --source npm --version beta` |
| `npx create-nocobase-app@beta my-nocobase-app ...` | `nb init --yes --env app1 --source npm --version beta` |
| `git clone ... -b next --depth=1 my-nocobase` | `nb init --yes --env app1 --source git --version beta` |
| `git clone ... -b develop --depth=1 my-nocobase` | `nb init --yes --env app1 --source git --version alpha` |
| `yarn dev` | `nb source dev -e app1` |
| `yarn nocobase upgrade` | `nb app upgrade -e app1` |
| `yarn nocobase upgrade --skip-code-update` | `nb app upgrade -e app1 -s` |
| Mehrere Anwendungsadressen manuell pflegen | `nb env list`, `nb env info app1` |
| Bestehende Anwendung manuell anbinden | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
