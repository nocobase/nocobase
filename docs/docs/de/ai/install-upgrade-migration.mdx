# Leitfaden für Installation und Upgrade-Migration: mit der NocoBase CLI

:::danger
Die Installations- und App-Wartungsfunktionen der CLI befinden sich noch in Entwicklung. Derzeit eignet sie sich eher für lokale Entwicklung, Testumgebungen und AI-Agent-Integrationsszenarien und wird nicht für Installation, Upgrade sowie Betrieb und Wartung direkt in Produktionsumgebungen empfohlen.
:::

Die neue NocoBase CLI (`nb`) vereinheitlicht Vorgänge wie Installieren, Verbinden, Starten, Stoppen, Anzeigen von Logs, Upgraden und Bereinigen in einem einzigen Befehlssatz. Im Vergleich zu den alten Dokumenten, in denen Workflows für Docker, create-nocobase-app und Git-Quellcode getrennt gepflegt wurden, eignet sich die CLI besser für lokale Entwicklung, Wartung von Testumgebungen sowie für die Integration und Zusammenarbeit mit AI Agent.

Dieser Artikel ist für die folgenden Szenarien gedacht:

- Du installierst oder aktualisierst NocoBase nach alter Dokumentation und möchtest auf die neue CLI-Methode umsteigen.
- Du hast bereits ein NocoBase, das auf eine alte Weise bereitgestellt wurde, und möchtest es von der CLI verwalten oder mit ihr verbinden lassen.
- Du musst für AI Agent eine verbindbare und wartbare NocoBase-Umgebung vorbereiten.

## Unterschiede zwischen der alten und der neuen Methode

Die alte Methode teilt die Dokumentation nach Installationsquelle auf, und für jede Quelle müssen Download-, Konfigurations-, Installations-, Start- und Upgrade-Befehle separat ausgeführt werden.

**Alte Installationsdokumentation**

- [Docker-Installation](/get-started/installation/docker)
- [create-nocobase-app-Installation](/get-started/installation/create-nocobase-app)
- [Installation über Git-Quellcode](/get-started/installation/git)

**Alte Upgrade-Dokumentation**

- [Upgrade einer Docker-Installation](/get-started/upgrading/docker)
- [Upgrade einer create-nocobase-app-Installation](/get-started/upgrading/create-nocobase-app)
- [Upgrade einer Installation über Git-Quellcode](/get-started/upgrading/git)

Die neue Methode nutzt `nb init` als Einstieg und speichert in einer CLI-env die Herkunft der App, Laufverzeichnis, Speicherverzeichnis, Datenbank und API-Verbindungsinformationen. Unabhängig davon, ob die Quelle Docker, npm oder Git ist, soll die weitere Wartung möglichst mit denselben `nb app`-, `nb source`- und `nb db`-Befehlen erfolgen.

| Szenario                             | Alte Methode                                                     | Neue CLI-Methode                                  |
| ------------------------------------ | ---------------------------------------------------------------- | ------------------------------------------------- |
| Installations-Einstieg               | `docker compose`, `create-nocobase-app`, `git clone`             | `nb init` oder `nb init --ui`                     |
| Laufzeitverwaltung der App           | Verzeichnis wechseln, Compose ändern, yarn/pm2-Befehle ausführen | `nb app start/stop/restart/logs`, `nb env remove` |
| Upgrade                              | Für jede Installationsart ein eigener Befehlssatz                | `nb app upgrade -e <env>`                         |
| Entwicklungsmodus                    | `yarn dev`                                                       | `nb source dev -e <env>`                          |
| Datenbankstatus                      | Container oder Datenbankdienst manuell prüfen                    | `nb db ps -e <env>`                               |
| Umgebungsstatus und Details anzeigen | Pfade, Ports und Zugriffsadressen manuell notieren               | `nb env list`, `nb env info <env>`                |
| Vorhandene App anbinden              | Alte Installationsdokumentation deckte das nicht einheitlich ab  | `nb init --ui` oder `nb init --api-base-url`      |

## CLI installieren

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Nach der Installation kannst du die Hilfe anzeigen:

```bash
nb --help
nb init --help
```

## NocoBase mit der neuen Methode installieren

### Empfohlen: visuellen Assistenten verwenden

```bash
nb init --ui
```

Der Assistent führt dich durch die Auswahl von:

- Neue App erstellen oder vorhandene App verbinden
- Installationsquelle: Docker, npm, Git
- NocoBase-Version
- Integrierte Datenbank oder externe Datenbank verwenden
- Administrator-Konto und Passwort

### Interaktiven Terminal-Assistenten verwenden

```bash
nb init
```

### Nicht-interaktiven Befehl verwenden

Die Standardinstallationsmethode ist Docker mit integrierter PostgreSQL-Datenbank:

```bash
nb init --yes --env app1
```

Docker-Installation mit bestimmter Version:

```bash
nb init --yes --env app1 --source docker --version beta
```

Installation mit npm:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Installation mit Git-Quellcode:

```bash
nb init --yes --env app1 --source git --version beta
```

Git-Quellcode verwenden und integrierten Datenbanktyp festlegen:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Mit einer vorhandenen NocoBase-App verbinden:

```bash
# Verwendet standardmäßig OAuth-Authentifizierung
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Verwendet Token-Authentifizierung
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` ist der Umgebungsname in der CLI. Spätere Befehle wie Starten, Upgrade und Log-Anzeige können mit `-e app1` die Ziel-App angeben.
:::

## Mit der neuen Methode upgraden

Bei der alten Methode unterschieden sich die Upgrade-Befehle für Docker, create-nocobase-app und Git-Quellcode; die neue CLI vereinheitlicht sie zu:

```bash
nb app upgrade -e app1
```

Wenn du den Upgrade-Prozess nur mit dem bereits heruntergeladenen Quellcode oder Image ausführen möchtest, ohne neuen Code oder ein neues Image zu holen:

```bash
nb app upgrade -e app1 --skip-code-update
```

Oder in Kurzform:

```bash
nb app upgrade -e app1 -s
```

## Häufige Wartungsbefehle

| Vorgang                                                   | Befehl                               |
| --------------------------------------------------------- | ------------------------------------ |
| App starten                                               | `nb app start -e app1`               |
| App stoppen                                               | `nb app stop -e app1`                |
| App neu starten                                           | `nb app restart -e app1`             |
| Logs anzeigen                                             | `nb app logs -e app1`                |
| env und API-Authentifizierungsstatus anzeigen             | `nb env list`                        |
| env-Details anzeigen                                      | `nb env info app1`                   |
| Status der integrierten Datenbank anzeigen                | `nb db ps -e app1`                   |
| App upgraden                                              | `nb app upgrade -e app1`             |
| App und integrierte Datenbank stoppen                     | `nb app stop -e app1 --with-db`      |
| env und lokal verwaltete Ressourcen vollständig entfernen | `nb env remove app1 --purge --force` |

Wenn du nur die App zusammen mit der von der CLI verwalteten integrierten Datenbank stoppen möchtest, verwende standardmäßig einfach `nb app stop -e app1 --with-db`.

Wenn du sicher bist, dass diese env nicht mehr benötigt wird, verwende anschließend `nb env remove app1 --purge --force`. Dadurch werden verwaltete Laufzeitressourcen, storage-Daten und – falls zutreffend – lokal von der CLI heruntergeladene und verwaltete App-Dateien entfernt.

## Tägliche Entwicklungsbefehle

`nb source` wird zur Verwaltung lokaler Quellcode-Projekte verwendet und ist hauptsächlich für npm- und Git-source-envs gedacht.

| Vorgang                                          | Befehl                                              |
| ------------------------------------------------ | --------------------------------------------------- |
| App im Entwicklungsmodus ausführen               | `nb source dev -e app1`                             |
| Quellcode bauen                                  | `nb source build --cwd /your/workspace/app1/source` |
| Tests ausführen                                  | `nb source test --cwd /your/workspace/app1/source`  |
| Quellcode/Image herunterladen oder aktualisieren | `nb source download --source git --version beta`    |

:::tip
Docker-envs werden normalerweise mit `nb app start`, `nb app logs` und `nb app upgrade` verwaltet und verwenden nicht `nb source dev`.
:::

## NB_CLI_ROOT und Verzeichnisstruktur

Die CLI speichert globale Konfigurationen und lokale App-Dateien im Verzeichnis, das `NB_CLI_ROOT` entspricht.

Standardmäßig ist `NB_CLI_ROOT` das Home-Verzeichnis des aktuellen Benutzers, also der von `os.homedir()` in Node.js zurückgegebene Pfad. Auf macOS ist das zum Beispiel normalerweise `/Users/<user>`. Wenn keine zusätzliche Konfiguration erfolgt, befinden sich die durch `nb init --yes --env app1` erzeugten Dateien normalerweise hier:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Du kannst `NB_CLI_ROOT` auch explizit festlegen:

```bash
export NB_CLI_ROOT=/your/workspace
```

Danach werden die Konfigurationsdatei und das Standard-App-Verzeichnis dort abgelegt:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Dabei gilt:

- `.nocobase/config.json` speichert Informationen wie env, aktuelle Umgebung, API-Adresse, Quellcodepfad, storage-Pfad und Datenbankkonfiguration.
- `<envName>/source` ist das von der CLI verwaltete Quellcode- oder App-Verzeichnis.
- `<envName>/storage` ist das von der CLI verwaltete Speicherverzeichnis der App.

:::warning
Nachdem dieselbe App initialisiert wurde, solltest du denselben `NB_CLI_ROOT` beibehalten. Wenn du `NB_CLI_ROOT` später änderst, sucht die CLI nach `.nocobase/config.json` im neuen Verzeichnis, und relative Pfade wie `appRootPath` und `storagePath` werden ebenfalls relativ zum neuen Verzeichnis aufgelöst. Das kann dazu führen, dass die App nicht gefunden wird oder in das falsche Verzeichnis gelesen/geschrieben wird.
:::

Die CLI verwendet derzeit nur den globalen Konfigurations-Scope. Wenn du anpassen musst, wo die Konfiguration und lokale App-Dateien gespeichert werden, setze `NB_CLI_ROOT`.

Beim Anzeigen und Auffinden von Apps solltest du zuerst `nb env list` und `nb env info` verwenden:

```bash
# Konfigurierte envs und API-Authentifizierungsstatus anzeigen
nb env list

# Detailinformationen zu einer env anzeigen
nb env info app1
```

`nb env list` ist für einen schnellen Überblick gedacht. Dabei ist `App Status` der Authentifizierungsstatus, der nach dem Zugriff auf die App-API mit den gespeicherten Token-/OAuth-Anmeldedaten ermittelt wird; der Datenbankstatus wird nicht mehr angezeigt. Um den Laufstatus der integrierten Datenbank zu sehen, verwende:

```bash
nb db ps -e app1
```

Nur wenn du die Standard-env wechseln musst, verwende:

```bash
nb env use app1
```

## Von der alten Methode zur CLI migrieren

Es gibt zwei Migrationsstrategien: **mit einer vorhandenen App verbinden** oder **ein lokales App-Verzeichnis in die CLI-Verwaltung übernehmen**. Wenn deine alte App bereits stabil läuft, wird empfohlen, zuerst „mit vorhandener App verbinden“ zu verwenden – das ist am risikoärmsten.

### Methode 1: Mit einer vorhandenen App verbinden

Geeignet für NocoBase, das bereits über Docker, create-nocobase-app, Git-Quellcode oder eine Serverbereitstellung läuft.

```bash
# Verwendet standardmäßig OAuth-Authentifizierung
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Verwendet Token-Authentifizierung
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Diese Methode speichert nur die API-Verbindung. Die CLI übernimmt nicht das Starten, Stoppen, Upgraden sowie das Verwalten von Quellcode- oder storage-Verzeichnis der alten App. Sie eignet sich, um AI Agent oder CLI-API-Befehle mit einer vorhandenen App zu verbinden.

Falls die Authentifizierung erneuert werden muss:

```bash
nb env auth app1
```

### Methode 2: Neue CLI-env erstellen und dann Dateien migrieren

Geeignet für Szenarien, in denen du die lokale App später mit `nb app` und `nb source` verwalten möchtest.

Vor der Migration solltest du zuerst Folgendes erledigen:

1. Die alte App stoppen.
2. Die alte Datenbank sichern.
3. Das alte `storage`-Verzeichnis sichern.
4. Die wichtigen Umgebungsvariablen der alten App notieren, z. B. `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Migration einer alten Docker-Installation

Bei der alten Docker-Methode liegen die App-Daten normalerweise im `storage`-Verzeichnis des Projektordners:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Es wird empfohlen, zuerst den neuen `NB_CLI_ROOT` festzulegen und dann mit der CLI eine neue Docker-env zu erstellen. Das folgende Beispiel geht davon aus, dass `NB_CLI_ROOT` `/your/workspace` ist.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

Die neue env verwendet standardmäßig folgende Verzeichnisse:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Kopiere dann das alte `storage` nach `/your/workspace/app2/storage` und stelle sicher, dass Datenbankverbindung und Konfigurationen wie `APP_KEY` mit der alten Umgebung übereinstimmen.

Nachdem das Kopieren abgeschlossen und die Datenbankkonfiguration bestätigt wurde, starte die App erneut:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Migration einer alten create-nocobase-app- oder Git-Quellcode-Installation

Bei der alten npm/Git-Methode befinden sich Quellcode, `.env` und `storage` normalerweise im selben Projektverzeichnis:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Es wird empfohlen, zuerst den neuen `NB_CLI_ROOT` festzulegen und dann eine neue npm- oder Git-env zu erstellen. Das folgende Beispiel geht davon aus, dass `NB_CLI_ROOT` `/your/workspace` ist.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Migriere dann nach Bedarf:

- Migriere den Quellcode benutzerdefinierter Plugins aus dem alten Projekt in das entsprechende Verzeichnis unter `/your/workspace/app3/source`.
- Kopiere das `storage` des alten Projekts nach `/your/workspace/app3/storage`.
- Übernimm die wichtigen Konfigurationen aus der alten `.env` in die CLI-env-Konfiguration oder die Initialisierungsparameter.
- Wenn du die ursprüngliche Datenbank verwendest, deaktiviere bei der Initialisierung die integrierte Datenbank und trage die Verbindungsinformationen der alten Datenbank ein.

Beispiel für Initialisierung mit externer Datenbank:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Nach der Migration starten und Logs anzeigen:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
Überschreibe `source` oder `storage` nicht direkt und führe keine Upgrades gegen eine Produktionsdatenbank ohne Backup aus. Vor der formellen Migration wird empfohlen, den gesamten Ablauf zuerst vollständig in einer Testumgebung durchzuspielen.
:::

## Schnelle Zuordnung alter Befehle zur Migration

| Alter Befehl                                          | Neuer CLI-Befehl                                                                        |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                            | `nb app start -e app1`                                                                  |
| `docker compose stop app`                             | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                          | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`        | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`    | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`      | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                            | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                               | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`            | `nb app upgrade -e app1 -s`                                                             |
| Mehrere App-Adressen manuell notieren                 | `nb env list`, `nb env info app1`                                                       |
| Vorhandene App manuell verbinden                      | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
