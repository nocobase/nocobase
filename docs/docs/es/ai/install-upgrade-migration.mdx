# Guía de instalación y migración de actualizaciones: uso de NocoBase CLI

:::danger
Las capacidades de instalación y mantenimiento de aplicaciones del CLI siguen en desarrollo. Actualmente es más adecuado para desarrollo local, entornos de prueba y escenarios de integración con AI Agent, y no se recomienda usarlo directamente para la instalación, actualización, operación y mantenimiento en entornos de producción.
:::

La nueva NocoBase CLI (`nb`) unifica la instalación, conexión, inicio, detención, visualización de logs, actualización y limpieza en un solo conjunto de comandos. En comparación con los flujos separados en la documentación antigua para Docker, create-nocobase-app y código fuente Git, el CLI es más adecuado para desarrollo local, mantenimiento de entornos de prueba e integración y colaboración con AI Agent.

Este artículo aplica a los siguientes escenarios:

- Estás instalando o actualizando NocoBase siguiendo la documentación antigua y quieres cambiar al nuevo método con CLI.
- Ya tienes una instancia de NocoBase desplegada con el método antiguo y quieres que el CLI la gestione o se conecte a ella.
- Necesitas preparar un entorno de NocoBase conectable y mantenible para un AI Agent.

## Diferencias entre el método antiguo y el nuevo

El método antiguo divide la documentación según el origen de instalación, y cada origen requiere ejecutar por separado comandos de descarga, configuración, instalación, inicio y actualización.

**Documentación antigua de instalación**

- [Instalación con Docker](/get-started/installation/docker)
- [Instalación con create-nocobase-app](/get-started/installation/create-nocobase-app)
- [Instalación desde código fuente Git](/get-started/installation/git)

**Documentación antigua de actualización**

- [Actualización de una instalación con Docker](/get-started/upgrading/docker)
- [Actualización de una instalación con create-nocobase-app](/get-started/upgrading/create-nocobase-app)
- [Actualización de una instalación desde código fuente Git](/get-started/upgrading/git)

El nuevo método usa `nb init` como punto de entrada y registra en un CLI env el origen de la aplicación, el directorio de ejecución, el directorio de almacenamiento, la base de datos y la información de conexión a la API. Después, tanto si el origen es Docker, npm o Git, se intenta mantener todo con el mismo conjunto de comandos `nb app`, `nb source` y `nb db`.

| Escenario                              | Método antiguo                                                       | Nuevo método con CLI                              |
| -------------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------- |
| Punto de entrada de instalación        | `docker compose`, `create-nocobase-app`, `git clone`                 | `nb init` o `nb init --ui`                        |
| Gestión de ejecución de la aplicación  | Cambiar de directorio, modificar compose, ejecutar comandos yarn/pm2 | `nb app start/stop/restart/logs`, `nb env remove` |
| Actualización                          | Un conjunto de comandos por cada método de instalación               | `nb app upgrade -e <env>`                         |
| Modo de desarrollo                     | `yarn dev`                                                           | `nb source dev -e <env>`                          |
| Estado de la base de datos             | Revisar manualmente contenedores o servicios de base de datos        | `nb db ps -e <env>`                               |
| Ver estado y detalles del entorno      | Registrar manualmente rutas, puertos y direcciones de acceso         | `nb env list`, `nb env info <env>`                |
| Integración de aplicaciones existentes | La documentación antigua no lo cubre de forma unificada              | `nb init --ui` o `nb init --api-base-url`         |

## Instalar el CLI

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Después de instalarlo, puedes ver la ayuda:

```bash
nb --help
nb init --help
```

## Instalar NocoBase con el nuevo método

### Recomendado: usar el asistente visual

```bash
nb init --ui
```

El asistente te guiará para elegir:

- Crear una nueva aplicación o conectarte a una existente
- Origen de instalación: Docker, npm, Git
- Versión de NocoBase
- Usar la base de datos integrada o una base de datos externa
- Cuenta y contraseña del administrador

### Usar el asistente interactivo en terminal

```bash
nb init
```

### Usar comandos no interactivos

El método de instalación predeterminado es Docker y usa la base de datos PostgreSQL integrada:

```bash
nb init --yes --env app1
```

Instalar una versión específica con Docker:

```bash
nb init --yes --env app1 --source docker --version beta
```

Instalar con npm:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Instalar desde código fuente Git:

```bash
nb init --yes --env app1 --source git --version beta
```

Usar código fuente Git y especificar el tipo de base de datos integrada:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Conectarse a una aplicación NocoBase existente:

```bash
# Usa autenticación OAuth por defecto
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Usar autenticación por token
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` es el nombre del entorno en el CLI. Los comandos posteriores, como iniciar, actualizar y ver logs, pueden especificar la aplicación de destino con `-e app1`.
:::

## Actualizar con el nuevo método

En el método antiguo, los comandos de actualización para Docker, create-nocobase-app y código fuente Git eran diferentes; el nuevo CLI los unifica como:

```bash
nb app upgrade -e app1
```

Si solo quieres ejecutar el proceso de actualización usando el código fuente o la imagen que ya se ha descargado, sin extraer nuevo código o imágenes:

```bash
nb app upgrade -e app1 --skip-code-update
```

O usar la forma abreviada:

```bash
nb app upgrade -e app1 -s
```

## Comandos comunes de mantenimiento

| Operación                                                       | Comando                              |
| --------------------------------------------------------------- | ------------------------------------ |
| Iniciar aplicación                                              | `nb app start -e app1`               |
| Detener aplicación                                              | `nb app stop -e app1`                |
| Reiniciar aplicación                                            | `nb app restart -e app1`             |
| Ver logs                                                        | `nb app logs -e app1`                |
| Ver estado del env y de la autenticación API                    | `nb env list`                        |
| Ver detalles del env                                            | `nb env info app1`                   |
| Ver estado de la base de datos integrada                        | `nb db ps -e app1`                   |
| Actualizar aplicación                                           | `nb app upgrade -e app1`             |
| Detener aplicación y base de datos integrada                    | `nb app stop -e app1 --with-db`      |
| Eliminar por completo el env y los recursos alojados localmente | `nb env remove app1 --purge --force` |

Si solo quieres detener la aplicación junto con la base de datos integrada gestionada por el CLI, normalmente basta con usar directamente `nb app stop -e app1 --with-db`.

Si estás seguro de que este env ya no es necesario, entonces usa `nb env remove app1 --purge --force`. Esto eliminará los recursos de ejecución gestionados, los datos de storage y, cuando corresponda, los archivos locales de la app descargados y administrados por el CLI.

## Comandos de desarrollo diario

`nb source` se usa para gestionar proyectos locales de código fuente y es principalmente adecuado para envs de origen npm y Git.

| Operación                                   | Comando                                             |
| ------------------------------------------- | --------------------------------------------------- |
| Ejecutar la aplicación en modo desarrollo   | `nb source dev -e app1`                             |
| Compilar el código fuente                   | `nb source build --cwd /your/workspace/app1/source` |
| Ejecutar pruebas                            | `nb source test --cwd /your/workspace/app1/source`  |
| Descargar o actualizar código fuente/imagen | `nb source download --source git --version beta`    |

:::tip
Los envs de Docker normalmente se gestionan mediante `nb app start`, `nb app logs` y `nb app upgrade`, y no usan `nb source dev`.
:::

## NB_CLI_ROOT y estructura de directorios

El CLI guarda la configuración global y los archivos locales de la aplicación en el directorio correspondiente a `NB_CLI_ROOT`.

De forma predeterminada, `NB_CLI_ROOT` es el directorio personal del usuario actual, es decir, la ruta devuelta por `os.homedir()` de Node.js. Por ejemplo, en macOS normalmente es `/Users/<user>`. Por lo tanto, si no haces configuración adicional, los archivos generados por `nb init --yes --env app1` suelen estar en:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

También puedes especificar explícitamente `NB_CLI_ROOT`:

```bash
export NB_CLI_ROOT=/your/workspace
```

Después de configurarlo, el archivo de configuración y el directorio predeterminado de la aplicación se colocarán en ese directorio:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Donde:

- `.nocobase/config.json` guarda información como envs, entorno actual, dirección API, ruta del código fuente, ruta de storage y configuración de la base de datos.
- `<envName>/source` es el directorio de código fuente o de la aplicación gestionado por el CLI.
- `<envName>/storage` es el directorio de almacenamiento de la aplicación gestionado por el CLI.

:::warning
Después de inicializar la misma aplicación, mantén el uso del mismo `NB_CLI_ROOT`. Si más adelante cambias `NB_CLI_ROOT`, el CLI buscará `.nocobase/config.json` en el nuevo directorio, y las rutas relativas `appRootPath` y `storagePath` también se resolverán según el nuevo directorio, lo que puede hacer que no se encuentre la aplicación o que se lea y escriba en un directorio incorrecto.
:::

Actualmente el CLI solo usa el alcance de configuración global. Si necesitas ajustar la ubicación donde se guardan la configuración y los archivos locales de la aplicación, configura `NB_CLI_ROOT`.

Al ver y ubicar aplicaciones, da prioridad a `nb env list` y `nb env info`:

```bash
# Ver los envs configurados y el estado de autenticación API
nb env list

# Ver la información detallada de un env
nb env info app1
```

`nb env list` está orientado a una vista general rápida. Allí, `App Status` es el estado de autenticación obtenido al acceder a la API de la aplicación con las credenciales Token/OAuth guardadas, y ya no muestra el estado de la base de datos. Si necesitas ver el estado de ejecución de la base de datos integrada, usa:

```bash
nb db ps -e app1
```

Solo cuando necesites cambiar el env predeterminado, usa:

```bash
nb env use app1
```

## Migrar del método antiguo al CLI

Hay dos estrategias de migración: **conectarse a una aplicación existente** o **poner el directorio de una aplicación local bajo gestión del CLI**. Si tu aplicación antigua ya funciona de forma estable, se recomienda usar primero "conectarse a una aplicación existente", que es la opción de menor riesgo.

### Método 1: conectarse a una aplicación existente

Adecuado para NocoBase que ya se ejecuta mediante Docker, create-nocobase-app, código fuente Git o despliegue en servidor.

```bash
# Usa autenticación OAuth por defecto
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Usar autenticación por token
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Este método solo guarda la conexión API. El CLI no asumirá la gestión del inicio, detención, actualización, directorio de código fuente ni directorio de storage de la aplicación antigua. Es adecuado para permitir que AI Agent o los comandos API del CLI se conecten a aplicaciones existentes.

Si necesitas renovar la autenticación:

```bash
nb env auth app1
```

### Método 2: crear un nuevo CLI env y luego migrar los archivos

Adecuado para escenarios en los que quieres usar `nb app` y `nb source` para gestionar aplicaciones locales a partir de entonces.

Antes de la migración, completa primero lo siguiente:

1. Detén la aplicación antigua.
2. Haz una copia de seguridad de la base de datos antigua.
3. Haz una copia de seguridad del directorio `storage` antiguo.
4. Registra las variables de entorno clave de la aplicación antigua, como `APP_KEY`, `TZ`, `DB_*` y `DB_UNDERSCORED`.

#### Migración desde una instalación antigua con Docker

El método antiguo con Docker normalmente coloca los datos de la aplicación en `storage` dentro del directorio del proyecto:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Se recomienda determinar primero el nuevo `NB_CLI_ROOT` y luego usar el CLI para crear un nuevo env de Docker. El siguiente ejemplo asume que `NB_CLI_ROOT` es `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

El nuevo env usará por defecto los siguientes directorios:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Luego copia el `storage` antiguo a `/your/workspace/app2/storage`, y asegúrate de que la conexión a la base de datos y configuraciones como `APP_KEY` coincidan con el entorno antiguo.

Después de copiar y confirmar la configuración de la base de datos, inicia la aplicación:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Migración desde una instalación antigua con create-nocobase-app o código fuente Git

El método antiguo con npm/Git normalmente coloca el código fuente, `.env` y `storage` en el mismo directorio del proyecto:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Se recomienda determinar primero el nuevo `NB_CLI_ROOT` y luego crear un nuevo env de npm o Git. El siguiente ejemplo asume que `NB_CLI_ROOT` es `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Luego migra según sea necesario:

- Migra el código fuente de los plugins personalizados del proyecto antiguo al directorio correspondiente en `/your/workspace/app3/source`.
- Copia el `storage` del proyecto antiguo a `/your/workspace/app3/storage`.
- Sincroniza las configuraciones clave del `.env` antiguo en la configuración del CLI env o en los parámetros de inicialización.
- Si usas la base de datos original, desactiva la base de datos integrada durante la inicialización e introduce la información de conexión de la base de datos antigua.

Ejemplo de inicialización con una base de datos externa:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Después de la migración, inicia la aplicación y revisa los logs:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
No sobrescribas directamente `source`, `storage` ni te conectes a la base de datos de producción para ejecutar actualizaciones sin tener una copia de seguridad. Antes de la migración formal, se recomienda hacer primero un ensayo completo en un entorno de prueba.
:::

## Referencia rápida de migración de comandos antiguos

| Comando antiguo                                          | Nuevo comando CLI                                                                       |
| -------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                               | `nb app start -e app1`                                                                  |
| `docker compose stop app`                                | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                             | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app`    | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`           | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`       | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`            | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                               | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                                  | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`               | `nb app upgrade -e app1 -s`                                                             |
| Registrar manualmente varias direcciones de aplicaciones | `nb env list`, `nb env info app1`                                                       |
| Conectarse manualmente a una aplicación existente        | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
