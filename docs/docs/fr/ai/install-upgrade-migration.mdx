# Guide d’installation et de migration de mise à niveau : utiliser NocoBase CLI

:::danger
Les capacités d’installation et de maintenance des applications du CLI sont encore en cours de développement. Pour le moment, il convient davantage au développement local, aux environnements de test et aux scénarios d’intégration avec AI Agent, et il n’est pas recommandé pour l’installation, la mise à niveau, l’exploitation et la maintenance directement en environnement de production.
:::

La nouvelle version de NocoBase CLI (`nb`) unifie dans un même ensemble de commandes des opérations telles que l’installation, la connexion, le démarrage, l’arrêt, la consultation des logs, la mise à niveau et le nettoyage. Par rapport à l’ancienne documentation, qui maintenait séparément les workflows pour Docker, create-nocobase-app et le code source Git, le CLI est mieux adapté au développement local, à la maintenance des environnements de test, ainsi qu’à l’intégration et à la collaboration avec AI Agent.

Cet article s’applique aux scénarios suivants :

- Vous installez ou mettez à niveau NocoBase en suivant l’ancienne documentation et souhaitez passer à la nouvelle méthode via CLI.
- Vous disposez déjà d’un NocoBase déployé avec une ancienne méthode et souhaitez que le CLI le gère ou s’y connecte.
- Vous devez préparer un environnement NocoBase connectable et maintenable pour AI Agent.

## Différences entre l’ancienne et la nouvelle méthode

L’ancienne méthode sépare la documentation selon la source d’installation, et chaque source nécessite d’exécuter séparément des commandes de téléchargement, configuration, installation, démarrage et mise à niveau.

**Ancienne documentation d’installation**

- [Installation Docker](/get-started/installation/docker)
- [Installation avec create-nocobase-app](/get-started/installation/create-nocobase-app)
- [Installation via le code source Git](/get-started/installation/git)

**Ancienne documentation de mise à niveau**

- [Mise à niveau d’une installation Docker](/get-started/upgrading/docker)
- [Mise à niveau d’une installation create-nocobase-app](/get-started/upgrading/create-nocobase-app)
- [Mise à niveau d’une installation via le code source Git](/get-started/upgrading/git)

La nouvelle méthode utilise `nb init` comme point d’entrée et enregistre dans un CLI env la source de l’application, le répertoire d’exécution, le répertoire de stockage, la base de données et les informations de connexion API. Ensuite, que la source soit Docker, npm ou Git, la maintenance doit autant que possible utiliser le même ensemble de commandes `nb app`, `nb source` et `nb db`.

| Scénario                                      | Ancienne méthode                                                         | Nouvelle méthode CLI                              |
| --------------------------------------------- | ------------------------------------------------------------------------ | ------------------------------------------------- |
| Point d’entrée d’installation                 | `docker compose`, `create-nocobase-app`, `git clone`                     | `nb init` ou `nb init --ui`                       |
| Gestion de l’exécution de l’application       | Changer de répertoire, modifier compose, exécuter des commandes yarn/pm2 | `nb app start/stop/restart/logs`, `nb env remove` |
| Mise à niveau                                 | Un ensemble de commandes pour chaque méthode d’installation              | `nb app upgrade -e <env>`                         |
| Mode développement                            | `yarn dev`                                                               | `nb source dev -e <env>`                          |
| État de la base de données                    | Vérifier manuellement les conteneurs ou services de base de données      | `nb db ps -e <env>`                               |
| Voir l’état et les détails de l’environnement | Noter manuellement les chemins, ports et adresses d’accès                | `nb env list`, `nb env info <env>`                |
| Intégration d’une application existante       | L’ancienne documentation ne couvrait pas cela de manière unifiée         | `nb init --ui` ou `nb init --api-base-url`        |

## Installer le CLI

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Après l’installation, vous pouvez afficher l’aide :

```bash
nb --help
nb init --help
```

## Installer NocoBase avec la nouvelle méthode

### Recommandé : utiliser l’assistant visuel

```bash
nb init --ui
```

L’assistant vous guidera pour choisir :

- Créer une nouvelle application ou connecter une application existante
- Source d’installation : Docker, npm, Git
- Version de NocoBase
- Utiliser une base de données intégrée ou une base de données externe
- Compte et mot de passe administrateur

### Utiliser l’assistant interactif dans le terminal

```bash
nb init
```

### Utiliser une commande non interactive

La méthode d’installation par défaut est Docker, avec la base de données PostgreSQL intégrée :

```bash
nb init --yes --env app1
```

Installer une version spécifique avec Docker :

```bash
nb init --yes --env app1 --source docker --version beta
```

Installer avec npm :

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Installer avec le code source Git :

```bash
nb init --yes --env app1 --source git --version beta
```

Utiliser le code source Git et spécifier le type de base de données intégrée :

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Se connecter à une application NocoBase existante :

```bash
# Utilise OAuth par défaut
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Utilise l’authentification par token
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` est le nom d’environnement dans le CLI. Par la suite, les commandes de démarrage, mise à niveau, consultation des logs, etc. peuvent cibler l’application avec `-e app1`.
:::

## Mettre à niveau avec la nouvelle méthode

Dans l’ancienne méthode, les commandes de mise à niveau différaient selon Docker, create-nocobase-app ou le code source Git ; le nouveau CLI les unifie en :

```bash
nb app upgrade -e app1
```

Si vous souhaitez uniquement exécuter le processus de mise à niveau avec le code source ou l’image déjà téléchargés, sans récupérer de nouveau code ou de nouvelle image :

```bash
nb app upgrade -e app1 --skip-code-update
```

Ou utiliser la forme abrégée :

```bash
nb app upgrade -e app1 -s
```

## Commandes de maintenance courantes

| Opération                                                           | Commande                             |
| ------------------------------------------------------------------- | ------------------------------------ |
| Démarrer l’application                                              | `nb app start -e app1`               |
| Arrêter l’application                                               | `nb app stop -e app1`                |
| Redémarrer l’application                                            | `nb app restart -e app1`             |
| Consulter les logs                                                  | `nb app logs -e app1`                |
| Voir l’env et l’état d’authentification API                         | `nb env list`                        |
| Voir les détails de l’env                                           | `nb env info app1`                   |
| Voir l’état de la base de données intégrée                          | `nb db ps -e app1`                   |
| Mettre à niveau l’application                                       | `nb app upgrade -e app1`             |
| Arrêter l’application et la base de données intégrée                | `nb app stop -e app1 --with-db`      |
| Supprimer complètement l’env et les ressources hébergées localement | `nb env remove app1 --purge --force` |

Si vous voulez simplement arrêter l’application avec la base de données intégrée hébergée par le CLI, utilisez directement `nb app stop -e app1 --with-db`.

Si vous êtes certain que cet env n’est plus nécessaire, utilisez ensuite `nb env remove app1 --purge --force`. Cela supprimera les ressources d’exécution gérées, les données de storage ainsi que, le cas échéant, les fichiers d’application locaux téléchargés et gérés par le CLI.

## Commandes de développement au quotidien

`nb source` sert à gérer les projets de code source locaux et s’applique principalement aux envs de type npm et Git source.

| Opération                                        | Commande                                            |
| ------------------------------------------------ | --------------------------------------------------- |
| Exécuter l’application en mode développement     | `nb source dev -e app1`                             |
| Compiler le code source                          | `nb source build --cwd /your/workspace/app1/source` |
| Exécuter les tests                               | `nb source test --cwd /your/workspace/app1/source`  |
| Télécharger ou actualiser le code source/l’image | `nb source download --source git --version beta`    |

:::tip
Les envs Docker sont généralement gérés via `nb app start`, `nb app logs` et `nb app upgrade`, et n’utilisent pas `nb source dev`.
:::

## NB_CLI_ROOT et structure des répertoires

Le CLI enregistre la configuration globale et les fichiers de l’application locale dans le répertoire correspondant à `NB_CLI_ROOT`.

Par défaut, `NB_CLI_ROOT` correspond au répertoire personnel de l’utilisateur courant, c’est-à-dire le chemin renvoyé par `os.homedir()` de Node.js. Par exemple, sur macOS, il s’agit généralement de `/Users/<user>`. Ainsi, sans configuration supplémentaire, les fichiers générés par `nb init --yes --env app1` se trouvent généralement dans :

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Vous pouvez aussi spécifier explicitement `NB_CLI_ROOT` :

```bash
export NB_CLI_ROOT=/your/workspace
```

Après configuration, le fichier de configuration et le répertoire d’application par défaut seront placés dans ce répertoire :

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Où :

- `.nocobase/config.json` enregistre des informations telles que l’env, l’environnement courant, l’adresse API, le chemin du code source, le chemin du storage et la configuration de la base de données.
- `<envName>/source` est le répertoire du code source ou de l’application géré par le CLI.
- `<envName>/storage` est le répertoire de stockage de l’application géré par le CLI.

:::warning
Après avoir initialisé une même application, veillez à conserver le même `NB_CLI_ROOT`. Si vous changez ensuite `NB_CLI_ROOT`, le CLI recherchera `.nocobase/config.json` dans le nouveau répertoire, et les chemins relatifs comme `appRootPath` et `storagePath` seront aussi résolus à partir de ce nouveau répertoire, ce qui peut conduire à ne plus trouver l’application ou à lire/écrire dans le mauvais répertoire.
:::

Le CLI utilise actuellement uniquement le scope de configuration global. Si vous devez ajuster l’emplacement d’enregistrement de la configuration et des fichiers locaux de l’application, définissez `NB_CLI_ROOT`.

Pour afficher et localiser les applications, utilisez en priorité `nb env list` et `nb env info` :

```bash
# Afficher les envs configurés et l’état d’authentification API
nb env list

# Afficher les informations détaillées d’un env
nb env info app1
```

`nb env list` est conçu pour une vue d’ensemble rapide. Dans cette commande, `App Status` correspond à l’état d’authentification obtenu après accès à l’API de l’application avec les identifiants Token/OAuth enregistrés, et n’affiche plus l’état de la base de données. Pour consulter l’état d’exécution de la base de données intégrée, utilisez :

```bash
nb db ps -e app1
```

Utilisez la commande suivante uniquement si vous devez changer l’env par défaut :

```bash
nb env use app1
```

## Migrer de l’ancienne méthode vers le CLI

Il existe deux stratégies de migration : **se connecter à une application existante** ou **intégrer un répertoire d’application local dans la gestion du CLI**. Si votre ancienne application fonctionne déjà de manière stable, il est recommandé de commencer par « se connecter à une application existante », qui présente le risque le plus faible.

### Méthode 1 : se connecter à une application existante

Convient à un NocoBase déjà en cours d’exécution via Docker, create-nocobase-app, code source Git ou déploiement sur serveur.

```bash
# Utilise OAuth par défaut
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Utilise l’authentification par token
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Cette méthode enregistre uniquement la connexion API. Le CLI ne prendra pas en charge le démarrage, l’arrêt, la mise à niveau, ni les répertoires de code source ou de storage de l’ancienne application. Elle convient pour permettre à AI Agent ou aux commandes API du CLI de se connecter à une application existante.

Si vous devez actualiser l’authentification :

```bash
nb env auth app1
```

### Méthode 2 : créer un nouvel env CLI puis migrer les fichiers

Convient aux scénarios où vous souhaitez ensuite gérer l’application locale avec `nb app` et `nb source`.

Avant la migration, veuillez d’abord effectuer les étapes suivantes :

1. Arrêter l’ancienne application.
2. Sauvegarder l’ancienne base de données.
3. Sauvegarder l’ancien répertoire `storage`.
4. Noter les variables d’environnement clés de l’ancienne application, par exemple `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Migration d’une ancienne installation Docker

Avec l’ancienne méthode Docker, les données de l’application sont généralement stockées dans `storage` au sein du répertoire du projet :

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Il est recommandé de définir d’abord le nouveau `NB_CLI_ROOT`, puis d’utiliser le CLI pour créer un nouvel env Docker. L’exemple ci-dessous suppose que `NB_CLI_ROOT` vaut `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

Le nouvel env utilisera par défaut les répertoires suivants :

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Copiez ensuite l’ancien `storage` vers `/your/workspace/app2/storage`, et assurez-vous que la connexion à la base de données ainsi que les paramètres tels que `APP_KEY` sont cohérents avec l’ancien environnement.

Une fois la copie terminée et la configuration de la base de données confirmée, redémarrez l’application :

```bash
nb app start -e app2
nb app logs -e app2
```

#### Migration d’une ancienne installation create-nocobase-app ou via code source Git

Avec l’ancienne méthode npm/Git, le code source, `.env` et `storage` se trouvent généralement dans le même répertoire de projet :

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Il est recommandé de définir d’abord le nouveau `NB_CLI_ROOT`, puis de créer un nouvel env npm ou Git. L’exemple ci-dessous suppose que `NB_CLI_ROOT` vaut `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Puis migrez selon vos besoins :

- Migrez le code source des plugins personnalisés de l’ancien projet vers le répertoire correspondant dans `/your/workspace/app3/source`.
- Copiez le `storage` de l’ancien projet vers `/your/workspace/app3/storage`.
- Synchronisez les configurations clés de l’ancien `.env` dans la configuration du CLI env ou dans les paramètres d’initialisation.
- Si vous utilisez la base de données d’origine, désactivez la base de données intégrée lors de l’initialisation et renseignez les informations de connexion à l’ancienne base de données.

Exemple d’initialisation avec une base de données externe :

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Une fois la migration terminée, démarrez l’application et consultez les logs :

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
N’écrasez pas directement `source`, `storage` et n’exécutez pas de mise à niveau sur une base de données de production sans sauvegarde. Avant une migration officielle, il est recommandé de faire d’abord une répétition complète dans un environnement de test.
:::

## Tableau rapide de migration des anciennes commandes

| Ancienne commande                                     | Nouvelle commande CLI                                                                   |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                            | `nb app start -e app1`                                                                  |
| `docker compose stop app`                             | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                          | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`        | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`    | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`      | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                            | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                               | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`            | `nb app upgrade -e app1 -s`                                                             |
| Noter manuellement plusieurs adresses d’application   | `nb env list`, `nb env info app1`                                                       |
| Connecter manuellement une application existante      | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
