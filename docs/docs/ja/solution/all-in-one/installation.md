# インストール方法

一体型業務管理システムは **CRM 顧客管理、販売管理、ヘルプデスク、プロジェクト管理、固定資産管理、HR 人事管理** の 6 モジュールを網羅します。ここでは 2 種類の復元方法を用意しており、NocoBase のバージョンや技術背景に合わせて 1 つを選べば構いません。

:::tip 前提条件

- 基本的な NocoBase 実行環境がすでにあること。本体のインストールは[公式インストールガイド](https://docs-cn.nocobase.com/welcome/getting-started/installation)を参照
- NocoBase のバージョンが **v2.1.0-alpha.34 以降**
- 一体型ソリューションのバックアップファイルのいずれかをダウンロード済み:
  - **nbdata バックアップ**:[nocobase_all_in_one_backup_260521.nbdata](https://static-docs.nocobase.com/nocobase_all_in_one_backup_260521.nbdata) — 方法 1 で使用
  - **SQL 圧縮ファイル**:[nocobase_all_in_one_sql_260521.zip](https://static-docs.nocobase.com/nocobase_all_in_one_sql_260521.zip) — 方法 2 で使用

:::

:::warning 注意

- 本ソリューションは **PostgreSQL 16** で作成されており、環境も PostgreSQL 16 である必要があります
- **`DB_UNDERSCORED` を `true` にしない** — `docker-compose.yml` を確認し、`true` に設定されていると復元に失敗します

:::

一般的には、バックアップマネージャープラグインがあれば方法 1、なければ方法 2 を選びます。現バージョンは**バックアップ復元**方式でデプロイしますが、今後のバージョンでは差分マイグレーション方式に切り替え、既存の NocoBase 環境への組み込みを容易にする予定です。

## 方法 1:バックアップマネージャーで復元(プロ/エンタープライズ版推奨)

NocoBase 内蔵の「[バックアップマネージャー](https://docs-cn.nocobase.com/handbook/backups)」プラグインでワンクリック復元する方法で、UI 操作が最も簡単です。ただし環境とプラグインへの依存条件が比較的厳しくなります。

### 特徴

**メリット:**

- **操作が手軽** — UI 上で完結し、プラグイン設定を含むすべての内容を復元できる
- **完全復元** — テンプレート印刷ファイル、添付ファイルフィールドのアップロードファイル、AI 社員のアバターなど、すべてのシステムファイルを復元できる

**制約:**

- **プロ/エンタープライズ版限定** — バックアップマネージャーはエンタープライズ向けプラグインで、コミュニティ版では利用できない
- **環境要件が厳しい** — データベースのバージョン、大文字小文字の感度設定などがバックアップ元と高い互換性を持つ必要がある
- **プラグイン依存が強い** — バックアップに含まれる商用プラグインがローカル環境にも存在しないと復元に失敗する

### 手順

**Step 1:`full` イメージでアプリを起動**

データベースクライアントなどの周辺コンポーネントがすべて含まれており追加設定が不要な、`full` 版の Docker イメージを強く推奨します。

```bash
docker pull nocobase/nocobase:alpha-full
```

このイメージで NocoBase サービスを起動します。

:::tip

`full` イメージを使わない場合、コンテナ内に `pg_dump` クライアントを手動でインストールする必要があり、手順が煩雑で不安定になります。

:::

**Step 2:「バックアップマネージャー」プラグインを有効化**

1. NocoBase システムにログイン
2. 「プラグイン管理」に入る
3. 「バックアップマネージャー」プラグインを見つけて有効化

**Step 3:ローカルバックアップファイルから復元**

1. プラグインを有効化したらページを更新
2. 左メニューの「システム管理 / バックアップマネージャー」に進む
3. 右上の「ローカルバックアップから復元」ボタンをクリック
4. ダウンロードした `nocobase_all_in_one_backup_260521.nbdata` ファイルをアップロード領域にドラッグ
5. 「送信」をクリックし、復元完了を待つ。通常は数十秒から数分

### 注意事項

- **データベース互換性** — PostgreSQL のバージョン、文字セット、大文字小文字の感度設定がバックアップ元と一致している必要があり、特に `schema` 名は完全に一致させる
- **商用プラグインの整合性** — バックアップで使用されている商用プラグインをローカルにすべて有効化しておく必要があり、不足があると復元が中断する。一体型ソリューションが利用する商用プラグインにはバックアップマネージャー、メール管理、監査ログ、AI 社員などがある

## 方法 2:SQL ファイル直接インポート(全ユーザー対応)

データベースを直接操作して復元する方法で、バックアップマネージャーを介さないためバージョンやプラグインの制限がありません。

### 特徴

**メリット:**

- **バージョン制限なし** — コミュニティ版を含むすべての NocoBase ユーザーで利用可能
- **高い互換性** — アプリ内の dump ツールに依存せず、データベースに接続できれば操作できる
- **高い耐障害性** — バックアップに含まれる商用プラグインがローカルになくても有効化されないだけで、他のモジュールの利用には影響しない

**制約:**

- **データベース操作のスキルが必要** — `.sql` ファイルの実行など、基本操作ができる必要がある
- **システムファイルが失われる** — テンプレート印刷ファイル、添付ファイルフィールドのアップロードファイル、AI 社員のアバターなど、すべてのシステムファイルが失われる

### 手順

**Step 1:クリーンなデータベースを準備**

これからインポートするデータ用に、新規の空データベース(PostgreSQL 16)を用意します。

**Step 2:`.sql` ファイルをデータベースにインポート**

ダウンロードした `nocobase_all_in_one_sql_260521.zip` を解凍して得られる `.sql` ファイルを、Step 1 で用意したデータベースにインポートします。実行方法は 2 通りあります。

**オプション A:サーバーコマンドライン(Docker の例)**

NocoBase とデータベースを Docker で稼働させている場合、`.sql` ファイルをサーバーにアップロードしてから `docker exec` でインポートします。PostgreSQL コンテナ名が `my-nocobase-db` だと仮定します:

```bash
# sql ファイルをコンテナにコピー
docker cp nocobase_all_in_one_sql_260521.sql my-nocobase-db:/tmp/
# コンテナ内でインポートを実行
docker exec -it my-nocobase-db psql -U nocobase -d nocobase -f /tmp/nocobase_all_in_one_sql_260521.sql
```

**オプション B:リモートデータベースクライアント(Navicat など)**

データベースがポートを公開している場合、Navicat、DBeaver、pgAdmin などの GUI クライアントで接続し、以下を実行します:

1. 対象データベースを右クリック
2. 「SQL ファイルを実行」または「SQL スクリプトを実行」を選択
3. 解凍した `.sql` ファイルを選んで実行

**Step 3:データベースに接続してアプリを起動**

NocoBase の起動パラメータ(`DB_HOST`、`DB_PORT`、`DB_DATABASE`、`DB_USER`、`DB_PASSWORD` など)を、データをインポートしたデータベースに向くよう設定し、NocoBase サービスを通常通り起動します。

### 注意事項

- **データベース権限** — データベースを直接操作できるアカウントとパスワードが必要
- **プラグインの状態** — インポート成功後、商用プラグインのデータは存在するものの、対応プラグインがローカルにインストールされていなければ該当機能は利用できない。ただしアプリがクラッシュすることはない

## 二つの方法の比較

| 項目 | 方法 1:バックアップマネージャー | 方法 2:SQL 直接インポート |
| :--- | :--- | :--- |
| **対象ユーザー** | プロ/エンタープライズ版 | 全ユーザー(コミュニティ版を含む) |
| **操作の手軽さ** | ⭐⭐⭐⭐⭐(UI 操作) | ⭐⭐⭐(データベース知識が必要) |
| **環境要件** | 厳しい、データベース・システムバージョンに高い互換性が必要 | 一般的、データベースが互換であれば OK |
| **プラグイン依存** | 強い依存、プラグイン不足で復元失敗 | データは単体でインポート可能、プラグイン不足で該当機能は利用不可 |
| **システムファイル** | 完全保持(印刷テンプレート、アップロードファイル、アバターなど) | 失われる(印刷テンプレート、アップロードファイル、アバターなど) |
| **おすすめシーン** | エンタープライズユーザー、環境を統制できる場合 | プラグインが一部不足、互換性重視、コミュニティ版 |

## インストール後の必須設定

復元が完了するとシステムは開けるようになりますが、**弊社のデモ環境を指している**設定が 2 か所あり、自分の環境に切り替える必要があります。

### 1. ファイルストレージエンジン(OSS / ローカル)

デモバックアップのデフォルトストレージエンジンは弊社デモ用の Alibaba Cloud OSS を指しており、Access Key は外部に公開していないため、添付ファイルフィールド、テンプレート印刷、AI 社員アバターのアップロードはすべて失敗します。

通常はローカルストレージに切り替えれば十分で、CDN 高速化や大容量ファイルが必要な場合のみ自前の OSS を使います。

**切り替え手順:**

1. 「プラグイン管理 / ファイルマネージャー」(または `/admin/settings/file-manager` に直接アクセス)に進む

2. **オプション A — ローカルストレージを使う**(最も簡単、セルフホスト向け):

   - 自動作成された「Local Storage(ローカルストレージ)」項目を探す
   - 「編集」をクリックし、設定パネル下部の「デフォルトストレージエンジンに設定」にチェックを入れて送信

   ![ストレージエンジン共通設定(下部「デフォルトストレージエンジンに設定」)](https://static-docs.nocobase.com/20240529115151.png)

   :::warning 注意

   Docker デプロイの場合、ローカルストレージはコンテナ内にあり、コンテナを削除するとファイルが失われます。本番環境では volume をマウントするかクラウドストレージへの切り替えを推奨します。

   :::

3. **オプション B — 自前の OSS / S3 / COS を使う**:

   - 「新規追加」をクリックし、対応するタイプ(Alibaba Cloud OSS / Amazon S3 / Tencent Cloud COS / S3 Pro)を選択
   - Access Key、Bucket、Region、ドメインなどのパラメータを入力し、「デフォルトストレージエンジンに設定」にチェックを入れて送信

   ![Alibaba Cloud OSS ストレージエンジン設定例](https://static-docs.nocobase.com/20240712220011.png)

4. デモにプリセットされた OSS 項目は削除または無効化し、誤用を防ぐ

詳細なパラメータ説明は[ストレージエンジン概要](../../file-manager/storage/index.md)を参照。

### 2. AI 社員の LLM サービスキー

デモバックアップには複数の LLM サービス項目(OpenAI、Claude、Gemini、DeepSeek、Qwen、Kimi など)がプリセットされていますが、入力されているのは弊社の API Key であり、**外部では機能しません**。AI 社員機能は切り替えるまで利用できません。

**設定手順:**

1. 「システム設定 / AI 社員 / LLM service」(または `/admin/settings/ai/llm-services` にアクセス)に進む

   ![LLM service 設定ページに入る](https://static-docs.nocobase.com/00_QuickStart_cn-2025-09-29-15-40-47.png)

2. プリセットのサービス一覧は、ドラッグで並び替え、`Enabled` スイッチで有効/無効を切り替えできる

   ![LLM サービス一覧(有効/無効 + 並び替え)](https://static-docs.nocobase.com/ai-employees/2026-02-14/llm-service-list-enabled-and-sort.png)

3. 利用予定の各サービスについて:

   - 「編集」をクリック
   - 「API Key」を自分のキーに置き換える(対応するサービス提供元のアカウントから取得:OpenAI、Anthropic、Google AI Studio、DeepSeek、Qwen、Kimi など)
   - プロキシや国内中継を使う場合は「Base URL」を調整
   - 「Enabled Models」では利用するモデルを残し、不要なものは削除

   ![LLM サービスの編集(API Key、Base URL、Enabled Models)](https://static-docs.nocobase.com/00_QuickStart_cn-2025-09-29-15-45-27.png)

4. 下部の「Test flight」で接続性をテストし、通過したら「Submit」で保存

   ![Test flight で接続をテスト](https://static-docs.nocobase.com/00_QuickStart_cn-2025-09-29-16-18-25.png)

5. 利用しないサービスは Disabled にすればよく、削除する必要はない

詳細な設定は[LLM サービスの設定](../../ai-employees/features/llm-service.md)を参照。

:::tip

この 2 点はデモ復元後に必ず変更すべき箇所です。その他の設定(サイトロゴ、SMTP、エンタープライズ版プラグインなど)は必要に応じて調整してください。

:::

## よくある質問

### プロフェッショナル版でも使えますか?エラーは出ますか?

そのまま使え、エラーは出ません。デモではエンタープライズ版プラグイン(メール管理、監査ログ、AI 社員など)を一部使用しています。プロフェッショナル版にこれらが無い場合、該当機能の入口は表示されませんが、他のモジュールには影響しません。例えば監査ログの入口は表示されなくなりますが、CRM、販売、チケット、プロジェクト、資産、HR などのコアモジュールは問題なく利用できます。

### 復元後はどのバージョンを選ぶべきですか?

最新の `alpha-full` イメージ(`nocobase/nocobase:alpha-full`)を推奨します。`full` イメージにはデータベースクライアントなどの依存が同梱されており、復元時のツール不足を防げます。

### 復元後にロゴが表示されません

公式デモのロゴはドメイン制限が設定されており、ローカルドメインでは読み込めません。「システム設定」から自分のロゴを再アップロードしてください。

### ファイルアップロードでエラー(OSS Key エラー)が出ます

SQL 方式でインストールするとファイルアップロードで OSS 関連のエラーが出ることがあります。「プラグイン管理 / ファイルマネージャー」に進み、「Local Storage(ローカルストレージ)」をデフォルトストレージに設定して保存すれば正常にアップロードできます。

詳細な対処は上記[ファイルストレージエンジン](#1-ファイルストレージエンジンoss--ローカル)セクションを参照。

### 多言語の切り替えは?

一体型ソリューションは 20 以上の言語にローカライズ済みです(`nb_demo` 名前空間)。復元後はデフォルトで中国語表示となるため、他言語に切り替えるには「システム設定 / 対象言語を有効化」してください。

### 差分アップグレードはどうすればよいですか?

現バージョンのアップグレードは全置換であり、独自カスタマイズは上書きされます。アップグレード前のバックアップを必ず取得してください。差分マイグレーション方式は計画中で、まずプロ/エンタープライズ版から対応する予定です。コミュニティ版はマイグレーション管理プラグインがないため、現時点では対応が困難です。
