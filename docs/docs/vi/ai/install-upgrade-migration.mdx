# Hướng dẫn cài đặt, nâng cấp và di chuyển: sử dụng NocoBase CLI

:::danger
Khả năng cài đặt và bảo trì ứng dụng của CLI vẫn đang được phát triển. Hiện tại, công cụ này phù hợp hơn cho phát triển cục bộ, môi trường kiểm thử và các kịch bản tích hợp AI Agent, chưa khuyến nghị dùng trực tiếp cho cài đặt, nâng cấp và vận hành bảo trì trong môi trường production.
:::

NocoBase CLI (`nb`) phiên bản mới hợp nhất các thao tác như cài đặt, kết nối, khởi động, dừng, xem log, nâng cấp và dọn dẹp vào cùng một bộ lệnh. So với tài liệu cũ duy trì riêng các quy trình cho Docker, create-nocobase-app và mã nguồn Git, CLI phù hợp hơn cho phát triển cục bộ, bảo trì môi trường kiểm thử, cũng như tích hợp và cộng tác với AI Agent.

Bài viết này áp dụng cho các tình huống sau:

- Bạn đang cài đặt hoặc nâng cấp NocoBase theo tài liệu cũ và muốn chuyển sang cách mới bằng CLI.
- Bạn đã có một NocoBase được triển khai theo cách cũ và muốn để CLI quản lý hoặc kết nối tới nó.
- Bạn cần chuẩn bị một môi trường NocoBase có thể kết nối và bảo trì cho AI Agent.

## Khác biệt giữa cách cũ và cách mới

Cách cũ chia tài liệu theo nguồn cài đặt, và mỗi nguồn đều cần thực hiện riêng các lệnh tải xuống, cấu hình, cài đặt, khởi động và nâng cấp.

**Tài liệu cài đặt cũ**

- [Cài đặt Docker](/get-started/installation/docker)
- [Cài đặt create-nocobase-app](/get-started/installation/create-nocobase-app)
- [Cài đặt từ mã nguồn Git](/get-started/installation/git)

**Tài liệu nâng cấp cũ**

- [Nâng cấp cho cài đặt Docker](/get-started/upgrading/docker)
- [Nâng cấp cho cài đặt create-nocobase-app](/get-started/upgrading/create-nocobase-app)
- [Nâng cấp cho cài đặt từ mã nguồn Git](/get-started/upgrading/git)

Cách mới lấy `nb init` làm điểm vào, sử dụng một CLI env để lưu nguồn ứng dụng, thư mục chạy, thư mục storage, cơ sở dữ liệu và thông tin kết nối API. Sau đó, dù nguồn là Docker, npm hay Git, việc bảo trì đều cố gắng dùng chung một bộ lệnh `nb app`, `nb source`, `nb db`.

| Tình huống                            | Cách cũ                                                | Cách CLI mới                                      |
| ------------------------------------- | ------------------------------------------------------ | ------------------------------------------------- |
| Điểm vào cài đặt                      | `docker compose`、`create-nocobase-app`、`git clone`   | `nb init` hoặc `nb init --ui`                     |
| Quản lý chạy ứng dụng                 | Chuyển thư mục, sửa compose, chạy lệnh yarn/pm2        | `nb app start/stop/restart/logs`、`nb env remove` |
| Nâng cấp                              | Mỗi cách cài đặt có một bộ lệnh riêng                  | `nb app upgrade -e <env>`                         |
| Chế độ phát triển                     | `yarn dev`                                             | `nb source dev -e <env>`                          |
| Trạng thái cơ sở dữ liệu              | Kiểm tra thủ công container hoặc dịch vụ cơ sở dữ liệu | `nb db ps -e <env>`                               |
| Xem trạng thái và chi tiết môi trường | Ghi thủ công đường dẫn, cổng và địa chỉ truy cập       | `nb env list`、`nb env info <env>`                |
| Kết nối ứng dụng hiện có              | Tài liệu cài đặt cũ không bao phủ thống nhất           | `nb init --ui` hoặc `nb init --api-base-url`      |

## Cài đặt CLI

```bash
npm install -g @nocobase/cli@beta
nb --version
```

Sau khi cài đặt, bạn có thể xem trợ giúp:

```bash
nb --help
nb init --help
```

## Cài đặt NocoBase theo cách mới

### Khuyến nghị: sử dụng trình hướng dẫn trực quan

```bash
nb init --ui
```

Trình hướng dẫn sẽ giúp bạn chọn:

- tạo ứng dụng mới hoặc kết nối ứng dụng hiện có
- nguồn cài đặt: Docker, npm, Git
- phiên bản NocoBase
- sử dụng cơ sở dữ liệu tích hợp hay cơ sở dữ liệu bên ngoài
- tài khoản và mật khẩu quản trị viên

### Sử dụng trình hướng dẫn tương tác trong terminal

```bash
nb init
```

### Sử dụng lệnh không tương tác

Cách cài đặt mặc định là Docker và sử dụng cơ sở dữ liệu PostgreSQL tích hợp:

```bash
nb init --yes --env app1
```

Dùng Docker để cài phiên bản chỉ định:

```bash
nb init --yes --env app1 --source docker --version beta
```

Dùng npm để cài đặt:

```bash
nb init --yes --env app1 --source npm --version beta --app-port 13080
```

Dùng mã nguồn Git để cài đặt:

```bash
nb init --yes --env app1 --source git --version beta
```

Dùng mã nguồn Git và chỉ định loại cơ sở dữ liệu tích hợp:

```bash
nb init --yes --env app1 --source git --version beta --db-dialect mysql
```

Kết nối ứng dụng NocoBase hiện có:

```bash
# Mặc định dùng xác thực OAuth
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Dùng xác thực token
nb init --yes --env app1 \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

:::tip
`--env` là tên môi trường trong CLI. Về sau, các lệnh như khởi động, nâng cấp, xem log... đều có thể chỉ định ứng dụng đích bằng `-e app1`.
:::

## Nâng cấp theo cách mới

Trong cách cũ, lệnh nâng cấp cho Docker, create-nocobase-app và mã nguồn Git là khác nhau; với CLI mới, tất cả được thống nhất thành:

```bash
nb app upgrade -e app1
```

Nếu bạn chỉ muốn dùng mã nguồn hoặc image hiện đã tải về để thực hiện quy trình nâng cấp, mà không kéo mã hoặc image mới:

```bash
nb app upgrade -e app1 --skip-code-update
```

Hoặc dùng dạng rút gọn:

```bash
nb app upgrade -e app1 -s
```

## Các lệnh bảo trì thường dùng

| Thao tác                                                  | Lệnh                                 |
| --------------------------------------------------------- | ------------------------------------ |
| Khởi động ứng dụng                                        | `nb app start -e app1`               |
| Dừng ứng dụng                                             | `nb app stop -e app1`                |
| Khởi động lại ứng dụng                                    | `nb app restart -e app1`             |
| Xem log                                                   | `nb app logs -e app1`                |
| Xem env và trạng thái xác thực API                        | `nb env list`                        |
| Xem chi tiết env                                          | `nb env info app1`                   |
| Xem trạng thái cơ sở dữ liệu tích hợp                     | `nb db ps -e app1`                   |
| Nâng cấp ứng dụng                                         | `nb app upgrade -e app1`             |
| Dừng ứng dụng và cơ sở dữ liệu tích hợp                   | `nb app stop -e app1 --with-db`      |
| Xóa hoàn toàn env và tài nguyên được host trên máy cục bộ | `nb env remove app1 --purge --force` |

Nếu bạn chỉ muốn dừng ứng dụng cùng với cơ sở dữ liệu tích hợp do CLI quản lý, mặc định chỉ cần dùng trực tiếp `nb app stop -e app1 --with-db`.

Nếu bạn chắc chắn env này không còn cần nữa, hãy dùng `nb env remove app1 --purge --force`. Lệnh này sẽ xóa tài nguyên chạy được quản lý, dữ liệu storage, cũng như các file app cục bộ do CLI tải về và quản lý khi áp dụng.

## Các lệnh phát triển hằng ngày

`nb source` dùng để quản lý dự án mã nguồn cục bộ, chủ yếu phù hợp với env nguồn npm và Git.

| Thao tác                              | Lệnh                                                |
| ------------------------------------- | --------------------------------------------------- |
| Chạy ứng dụng ở chế độ phát triển     | `nb source dev -e app1`                             |
| Build mã nguồn                        | `nb source build --cwd /your/workspace/app1/source` |
| Chạy kiểm thử                         | `nb source test --cwd /your/workspace/app1/source`  |
| Tải xuống hoặc làm mới mã nguồn/image | `nb source download --source git --version beta`    |

:::tip
Docker env thường được quản lý bằng `nb app start`, `nb app logs` và `nb app upgrade`, không dùng `nb source dev`.
:::

## NB_CLI_ROOT và cấu trúc thư mục

CLI sẽ lưu cấu hình toàn cục và file ứng dụng cục bộ trong thư mục tương ứng với `NB_CLI_ROOT`.

Theo mặc định, `NB_CLI_ROOT` là thư mục home của người dùng hiện tại, tức là đường dẫn do Node.js `os.homedir()` trả về. Ví dụ trên macOS thường là `/Users/<user>`. Vì vậy, nếu không cấu hình thêm, các file được tạo bởi `nb init --yes --env app1` thường sẽ nằm tại:

```text
~
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Bạn cũng có thể chỉ định rõ `NB_CLI_ROOT`:

```bash
export NB_CLI_ROOT=/your/workspace
```

Sau khi thiết lập, file cấu hình và thư mục ứng dụng mặc định sẽ được đặt trong thư mục này:

```text
/your/workspace
├── .nocobase
│   └── config.json
└── app1
    ├── source
    └── storage
```

Trong đó:

- `.nocobase/config.json` lưu env, môi trường hiện tại, địa chỉ API, đường dẫn mã nguồn, đường dẫn storage, cấu hình cơ sở dữ liệu và các thông tin khác.
- `<envName>/source` là thư mục mã nguồn hoặc ứng dụng do CLI quản lý.
- `<envName>/storage` là thư mục lưu trữ ứng dụng do CLI quản lý.

:::warning
Sau khi cùng một ứng dụng đã được khởi tạo, hãy luôn dùng cùng một `NB_CLI_ROOT`. Nếu sau này đổi `NB_CLI_ROOT`, CLI sẽ tìm `.nocobase/config.json` trong thư mục mới, và các `appRootPath`, `storagePath` ở dạng đường dẫn tương đối cũng sẽ được phân giải theo thư mục mới, có thể dẫn đến không tìm thấy ứng dụng hoặc đọc/ghi nhầm thư mục.
:::

Hiện tại CLI chỉ dùng scope cấu hình toàn cục. Nếu cần điều chỉnh vị trí lưu cấu hình và file ứng dụng cục bộ, hãy thiết lập `NB_CLI_ROOT`.

Khi xem và định vị ứng dụng, ưu tiên dùng `nb env list` và `nb env info`:

```bash
# Xem các env đã cấu hình và trạng thái xác thực API
nb env list

# Xem thông tin chi tiết của một env
nb env info app1
```

`nb env list` phục vụ cho việc xem nhanh tổng quan, trong đó `App Status` là trạng thái xác thực thu được sau khi truy cập API ứng dụng bằng thông tin xác thực Token/OAuth đã lưu, và không còn hiển thị trạng thái cơ sở dữ liệu. Khi cần xem trạng thái chạy của cơ sở dữ liệu tích hợp, hãy dùng:

```bash
nb db ps -e app1
```

Chỉ khi cần chuyển env mặc định, mới dùng:

```bash
nb env use app1
```

## Di chuyển từ cách cũ sang CLI

Có hai chiến lược di chuyển: **kết nối ứng dụng hiện có**, hoặc **đưa thư mục ứng dụng cục bộ vào quản lý của CLI**. Nếu ứng dụng cũ của bạn đã chạy ổn định, nên ưu tiên dùng “kết nối ứng dụng hiện có” trước để giảm rủi ro.

### Cách 1: kết nối ứng dụng hiện có

Phù hợp với NocoBase đã chạy qua Docker, create-nocobase-app, mã nguồn Git hoặc triển khai trên máy chủ.

```bash
# Mặc định dùng xác thực OAuth
nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api

# Dùng xác thực token
nb init --yes --env "app$RANDOM" \
  --api-base-url=http://next.v2.test.nocobase.com/nocobase/api \
  --auth-type=token \
  --access-token=<token>
```

Cách này chỉ lưu kết nối API; CLI sẽ không tiếp quản việc khởi động, dừng, nâng cấp, thư mục mã nguồn hay thư mục storage của ứng dụng cũ. Nó phù hợp để AI Agent hoặc các lệnh CLI API kết nối tới ứng dụng hiện có.

Nếu cần làm mới xác thực:

```bash
nb env auth app1
```

### Cách 2: tạo CLI env mới rồi di chuyển file

Phù hợp với trường hợp bạn muốn về sau dùng `nb app` và `nb source` để quản lý ứng dụng cục bộ.

Trước khi di chuyển, hãy hoàn thành trước các việc sau:

1. Dừng ứng dụng cũ.
2. Sao lưu cơ sở dữ liệu cũ.
3. Sao lưu thư mục `storage` cũ.
4. Ghi lại các biến môi trường quan trọng của ứng dụng cũ, chẳng hạn `APP_KEY`, `TZ`, `DB_*`, `DB_UNDERSCORED`.

#### Di chuyển cài đặt Docker cũ

Cách Docker cũ thường lưu dữ liệu ứng dụng trong `storage` của thư mục dự án:

```text
/old-docker-project
├── docker-compose.yml
└── storage
```

Nên xác định `NB_CLI_ROOT` mới trước, sau đó dùng CLI tạo một Docker env mới. Ví dụ dưới đây giả sử `NB_CLI_ROOT` là `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app2 --source docker --version beta
nb app stop -e app2
```

Env mới mặc định sẽ dùng các thư mục sau:

```text
/your/workspace
└── app2
    ├── source
    └── storage
```

Sau đó sao chép `storage` cũ vào `/your/workspace/app2/storage`, và đảm bảo kết nối cơ sở dữ liệu cùng các cấu hình như `APP_KEY` khớp với môi trường cũ.

Sau khi sao chép xong và xác nhận cấu hình cơ sở dữ liệu, hãy khởi động lại ứng dụng:

```bash
nb app start -e app2
nb app logs -e app2
```

#### Di chuyển cài đặt cũ bằng create-nocobase-app hoặc mã nguồn Git

Cách npm/Git cũ thường đặt mã nguồn, `.env` và `storage` trong cùng một thư mục dự án:

```text
/old-nocobase
├── .env
├── package.json
├── packages
└── storage
```

Nên xác định `NB_CLI_ROOT` mới trước, sau đó tạo env npm hoặc Git mới. Ví dụ dưới đây giả sử `NB_CLI_ROOT` là `/your/workspace`.

```bash
export NB_CLI_ROOT=/your/workspace
nb init --yes --env app3 --source git --version beta
nb app stop -e app3
```

Sau đó di chuyển theo nhu cầu:

- Di chuyển mã nguồn plugin tùy chỉnh trong dự án cũ sang thư mục tương ứng trong `/your/workspace/app3/source`.
- Sao chép `storage` cũ vào `/your/workspace/app3/storage`.
- Đồng bộ các cấu hình quan trọng trong `.env` cũ vào cấu hình CLI env hoặc tham số khởi tạo.
- Khi dùng cơ sở dữ liệu cũ, hãy tắt cơ sở dữ liệu tích hợp lúc khởi tạo và điền thông tin kết nối của cơ sở dữ liệu cũ.

Ví dụ khởi tạo với cơ sở dữ liệu bên ngoài:

```bash
nb init --yes --env app3 --source git --version beta \
  --no-builtin-db \
  --db-dialect postgres \
  --db-host 127.0.0.1 \
  --db-port 5432 \
  --db-database nocobase \
  --db-user nocobase \
  --db-password nocobase
```

Sau khi di chuyển xong, khởi động ứng dụng và xem log:

```bash
nb app start -e app3
nb app logs -e app3
```

:::warning
Không ghi đè trực tiếp `source`, `storage` hoặc kết nối cơ sở dữ liệu production để nâng cấp khi chưa có bản sao lưu. Trước khi di chuyển chính thức, nên diễn tập đầy đủ một lần trong môi trường kiểm thử.
:::

## Tra cứu nhanh việc chuyển đổi lệnh cũ

| Lệnh cũ                                               | Lệnh CLI mới                                                                            |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `docker compose up -d app`                            | `nb app start -e app1`                                                                  |
| `docker compose stop app`                             | `nb app stop -e app1`                                                                   |
| `docker compose logs -f app`                          | `nb app logs -e app1`                                                                   |
| `docker compose pull app && docker compose up -d app` | `nb app upgrade -e app1`                                                                |
| `yarn create nocobase-app my-nocobase-app ...`        | `nb init --yes --env app1 --source npm --version beta`                                  |
| `npx create-nocobase-app@beta my-nocobase-app ...`    | `nb init --yes --env app1 --source npm --version beta`                                  |
| `git clone ... -b next --depth=1 my-nocobase`         | `nb init --yes --env app1 --source git --version beta`                                  |
| `git clone ... -b develop --depth=1 my-nocobase`      | `nb init --yes --env app1 --source git --version alpha`                                 |
| `yarn dev`                                            | `nb source dev -e app1`                                                                 |
| `yarn nocobase upgrade`                               | `nb app upgrade -e app1`                                                                |
| `yarn nocobase upgrade --skip-code-update`            | `nb app upgrade -e app1 -s`                                                             |
| Ghi thủ công địa chỉ của nhiều ứng dụng               | `nb env list`、`nb env info app1`                                                       |
| Kết nối thủ công ứng dụng hiện có                     | `nb init --yes --env app1 --api-base-url=http://next.v2.test.nocobase.com/nocobase/api` |
