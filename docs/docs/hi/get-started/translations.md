# अनुवाद में योगदान

NocoBase की डिफ़ॉल्ट भाषा अंग्रेज़ी है। वर्तमान में, मुख्य एप्लिकेशन अंग्रेज़ी, इतालवी, डच, सरलीकृत चीनी और जापानी का समर्थन करता है। हम आपको अन्य भाषाओं के अनुवाद में योगदान देने के लिए सादर आमंत्रित करते हैं, ताकि दुनिया भर के उपयोगकर्ता और भी सुविधाजनक NocoBase अनुभव का आनंद ले सकें।

---

## I. सिस्टम स्थानीयकरण (System Localization)

### 1. सिस्टम इंटरफ़ेस और प्लगइन अनुवाद

#### 1.1 अनुवाद का दायरा
यह केवल NocoBase सिस्टम इंटरफ़ेस और प्लगइन के स्थानीयकरण पर लागू होता है, और इसमें अन्य कस्टम सामग्री (जैसे डेटा टेबल या Markdown ब्लॉक) शामिल नहीं हैं।

![20250319220127](https://static-docs.nocobase.com/20250319220127.png)


#### 1.2 स्थानीयकरण सामग्री का अवलोकन
NocoBase अपनी स्थानीयकरण सामग्री को प्रबंधित करने के लिए Git का उपयोग करता है। मुख्य रिपॉजिटरी है:
https://github.com/nocobase/nocobase/tree/main/locales

प्रत्येक भाषा को उसके भाषा कोड (जैसे, de-DE.json, fr-FR.json) के नाम वाली JSON फ़ाइल द्वारा दर्शाया जाता है। फ़ाइल संरचना प्लगइन मॉड्यूल द्वारा व्यवस्थित की गई है, जिसमें अनुवादों को स्टोर करने के लिए की-वैल्यू (key-value) पेयर का उपयोग किया जाता है। उदाहरण के लिए:

```json
{
  // क्लाइंट प्लगइन
  "@nocobase/client": {
    "(Fields only)": "(Fields only)",
    "12 hour": "12 hour",
    "24 hour": "24 hour"
    // ...अन्य की-वैल्यू पेयर
  },
  "@nocobase/plugin-acl": {
    // इस प्लगइन के लिए की-वैल्यू पेयर
  }
  // ...अन्य प्लगइन मॉड्यूल
}
```

अनुवाद करते समय, कृपया इसे धीरे-धीरे निम्नलिखित संरचना में बदलें:

```json
{
  // क्लाइंट प्लगइन
  "@nocobase/client": {
    "(Fields only)": "(केवल फ़ील्ड - अनुवादित)",
    "12 hour": "12 घंटे",
    "24 hour": "24 घंटे"
    // ...अन्य की-वैल्यू पेयर
  },
  "@nocobase/plugin-acl": {
    // इस प्लगइन के लिए की-वैल्यू पेयर
  }
  // ...अन्य प्लगइन मॉड्यूल
}
```

#### 1.3 अनुवाद परीक्षण और सिंक्रनाइज़ेशन
- अपना अनुवाद पूरा करने के बाद, कृपया परीक्षण करें और सत्यापित करें कि सभी टेक्स्ट सही ढंग से प्रदर्शित हो रहे हैं।
हमने एक अनुवाद सत्यापन प्लगइन भी जारी किया है - प्लगइन मार्केटप्लेस में `Locale tester` खोजें।
![20250422233152](https://static-docs.nocobase.com/20250422233152.png)
इंस्टालेशन के बाद, git रिपॉजिटरी की संबंधित स्थानीयकरण फ़ाइल से JSON सामग्री कॉपी करें, उसे अंदर पेस्ट करें, और अनुवाद प्रभावी है या नहीं यह सत्यापित करने के लिए OK पर क्लिक करें।

## II. दस्तावेज़ स्थानीयकरण (NocoBase 2.0)

NocoBase 2.0 के दस्तावेज़ों को एक नई संरचना में प्रबंधित किया जाता है। दस्तावेज़ स्रोत फ़ाइलें मुख्य NocoBase रिपॉजिटरी में स्थित हैं:

https://github.com/nocobase/nocobase/tree/next/docs

### 2.1 दस्तावेज़ संरचना

दस्तावेज़ स्टैटिक साइट जनरेटर के रूप में [Rspress](https://rspress.dev/) का उपयोग करते हैं और 22 भाषाओं का समर्थन करते हैं। संरचना इस प्रकार व्यवस्थित है:

```
docs/
├── docs/
│   ├── en/                    # अंग्रेज़ी (स्रोत भाषा)
│   ├── cn/                    # सरलीकृत चीनी
│   ├── ja/                    # जापानी
│   ├── ko/                    # कोरियाई
│   ├── de/                    # जर्मन
│   ├── fr/                    # फ्रांसीसी
│   ├── es/                    # स्पेनिश
│   ├── pt/                    # पुर्तगाली
│   ├── ru/                    # रूसी
│   ├── it/                    # इतालवी
│   ├── tr/                    # तुर्की
│   ├── uk/                    # यूक्रेनी
│   ├── vi/                    # वियतनामी
│   ├── id/                    # इंडोनेशियाई
│   ├── th/                    # थाई
│   ├── pl/                    # पोलिश
│   ├── nl/                    # डच
│   ├── cs/                    # चेक
│   ├── ar/                    # अरबी
│   ├── he/                    # हिब्रू
│   ├── hi/                    # हिन्दी
│   ├── sv/                    # स्वीडिश
│   └── public/                # साझा संसाधन (इमेज आदि)
├── theme/                     # कस्टम थीम
├── rspress.config.ts          # Rspress कॉन्फ़िगरेशन
└── package.json
```

### 2.2 अनुवाद वर्कफ़्लो

1. **अंग्रेज़ी स्रोत के साथ सिंक करें**: सभी अनुवाद अंग्रेज़ी दस्तावेज़ों (`docs/en/`) पर आधारित होने चाहिए। जब अंग्रेज़ी दस्तावेज़ अपडेट हों, तो अनुवादों को तदनुसार अपडेट किया जाना चाहिए।

2. **ब्रांच रणनीति**:
   - नवीनतम अंग्रेज़ी सामग्री के संदर्भ के रूप में `develop` या `next` ब्रांच का उपयोग करें
   - टारगेट ब्रांच से अपनी अनुवाद ब्रांच बनाएं

3. **फ़ाइल संरचना**: प्रत्येक भाषा निर्देशिका को अंग्रेज़ी निर्देशिका संरचना के समान होना चाहिए। उदाहरण के लिए:
   ```
   docs/en/get-started/index.md    →    docs/hi/get-started/index.md
   docs/en/api/acl/acl.md          →    docs/hi/api/acl/acl.md
   ```

### 2.3 अनुवाद में योगदान देना

1. रिपॉजिटरी को फोर्क (Fork) करें: https://github.com/nocobase/nocobase
2. अपने फोर्क को क्लोन करें और `develop` या `next` ब्रांच को चेकआउट करें
3. `docs/docs/` निर्देशिका पर जाएँ
4. वह भाषा निर्देशिका ढूँढें जिसमें आप योगदान देना चाहते हैं (उदाहरण के लिए, हिन्दी के लिए `hi/`)
5. Markdown फ़ाइलों का अनुवाद करें, फ़ाइल संरचना को अंग्रेज़ी संस्करण के समान ही रखें
6. अपने परिवर्तनों का स्थानीय स्तर पर परीक्षण करें:
   ```bash
   cd docs
   yarn install
   yarn dev
   ```
7. मुख्य रिपॉजिटरी में पुल रिक्वेस्ट (Pull Request) सबमिट करें

### 2.4 अनुवाद दिशानिर्देश

- **फॉर्मेटिंग एक समान रखें**: स्रोत के समान Markdown संरचना, हेडिंग, कोड ब्लॉक और लिंक बनाए रखें
- **Frontmatter सुरक्षित रखें**: फ़ाइलों के शीर्ष पर किसी भी YAML frontmatter को तब तक अपरिवर्तित रखें जब तक कि उसमें अनुवाद योग्य सामग्री न हो
- **इमेज संदर्भ**: `docs/public/` से समान इमेज पाथ का उपयोग करें - इमेज सभी भाषाओं में साझा की जाती हैं
- **आंतरिक लिंक**: सही भाषा पाथ पर इंगित करने के लिए आंतरिक लिंक अपडेट करें
- **कोड उदाहरण**: आम तौर पर, कोड उदाहरणों का अनुवाद नहीं किया जाना चाहिए, लेकिन कोड के भीतर की टिप्पणियों (comments) का अनुवाद किया जा सकता है

### 2.5 नेविगेशन कॉन्फ़िगरेशन

प्रत्येक भाषा के लिए नेविगेशन संरचना प्रत्येक भाषा निर्देशिका के भीतर `_nav.json` और `_meta.json` फ़ाइलों में परिभाषित की गई है। नए पेज या सेक्शन जोड़ते समय, इन कॉन्फ़िगरेशन फ़ाइलों को अपडेट करना सुनिश्चित करें।

### 3.2 सामग्री संरचना और स्थानीयकरण विधि

हम एक मिश्रित सामग्री प्रबंधन दृष्टिकोण का उपयोग करते हैं। अंग्रेज़ी, चीनी और जापानी सामग्री और संसाधनों को नियमित रूप से CMS सिस्टम से सिंक्रनाइज़ और ओवरराइट किया जाता है, जबकि अन्य भाषाओं को सीधे स्थानीय फ़ाइलों में संपादित किया जा सकता है। स्थानीय सामग्री `content` निर्देशिका में स्टोर की जाती है, जो इस प्रकार व्यवस्थित है:

```
/content
  /articles        # ब्लॉग लेख
    /article-slug
      index.md     # अंग्रेज़ी सामग्री (डिफ़ॉल्ट)
      index.cn.md  # चीनी सामग्री
      index.ja.md  # जापानी सामग्री
      metadata.json # मेटाडेटा और अन्य स्थानीयकरण गुण
  /tutorials       # ट्यूटोरियल
  /releases        # रिलीज़ जानकारी
  /pages           # कुछ स्टैटिक पेज
  /categories      # श्रेणी जानकारी
    /article-categories.json  # लेख श्रेणी सूची
    /category-slug            # व्यक्तिगत श्रेणी विवरण
      /category.json
  /tags            # टैग जानकारी
    /article-tags.json        # लेख टैग सूची
    /release-tags.json        # रिलीज़ टैग सूची
    /tag-slug                 # व्यक्तिगत टैग विवरण
      /tag.json
  /help-center     # सहायता केंद्र सामग्री
    /help-center-tree.json    # सहायता केंद्र नेविगेशन संरचना
  ....
```

### 3.3 सामग्री अनुवाद दिशानिर्देश

- **Markdown सामग्री अनुवाद के बारे में**

1. डिफ़ॉल्ट फ़ाइल के आधार पर एक नई भाषा फ़ाइल बनाएँ (उदाहरण के लिए, `index.md` से `index.hi.md`)
2. JSON फ़ाइल के संबंधित फ़ील्ड में स्थानीयकृत गुण (properties) जोड़ें
3. फ़ाइल संरचना, लिंक और इमेज संदर्भों में निरंतरता बनाए रखें

- **JSON सामग्री अनुवाद**
कई सामग्री मेटाडेटा JSON फ़ाइलों में स्टोर किए जाते हैं, जिनमें आम तौर पर बहुभाषी फ़ील्ड होते हैं:

```json
{
  "id": 123,
  "title": "English Title",       // अंग्रेज़ी शीर्षक (डिफ़ॉल्ट)
  "title_cn": "中文标题",          // चीनी शीर्षक
  "title_ja": "日本語タイトル",    // जापानी शीर्षक
  "description": "English description",
  "description_cn": "中文描述",
  "description_ja": "日本語の説明",
  "slug": "article-slug",         // URL पाथ (आम तौर पर अनुवादित नहीं)
  "status": "published",
  "publishedAt": "2025-03-19T12:00:00Z"
}
```

**अनुवाद नोट:**

1. **फ़ील्ड नामकरण परंपरा**: अनुवाद फ़ील्ड आम तौर पर `{original_field}_{language_code}` फॉर्मेट का उपयोग करते हैं
   - उदाहरण के लिए: title_hi (हिन्दी शीर्षक), description_hi (हिन्दी विवरण)

2. **नई भाषा जोड़ते समय**:
   - प्रत्येक फ़ील्ड के लिए एक संबंधित भाषा सफ़िक्स (suffix) संस्करण जोड़ें जिसे अनुवाद की आवश्यकता है
   - मूल फ़ील्ड मानों (जैसे title, description आदि) को न बदलें, क्योंकि वे डिफ़ॉल्ट भाषा (अंग्रेज़ी) सामग्री के रूप में काम करते हैं

3. **CMS सिंक्रनाइज़ेशन तंत्र**:
   - CMS सिस्टम समय-समय पर अंग्रेज़ी, चीनी और जापानी सामग्री को अपडेट करता है
   - सिस्टम केवल इन तीन भाषाओं की सामग्री (JSON में कुछ गुण) को अपडेट/ओवरराइट करेगा, और अन्य योगदानकर्ताओं द्वारा जोड़े गए भाषा फ़ील्ड को **हटाएगा नहीं**
   - उदाहरण के लिए: यदि आपने हिन्दी अनुवाद (title_hi) जोड़ा है, तो CMS सिंक्रनाइज़ेशन इस फ़ील्ड को प्रभावित नहीं करेगा


### 3.4 नई भाषा समर्थन कॉन्फ़िगर करना

नई भाषा के लिए समर्थन जोड़ने के लिए, आपको `src/utils/index.ts` फ़ाइल में `SUPPORTED_LANGUAGES` कॉन्फ़िगरेशन को संशोधित करना होगा:

```typescript
export const SUPPORTED_LANGUAGES = {
  en: {
    code: 'en',
    locale: 'en-US',
    name: 'English',
    default: true
  },
  cn: {
    code: 'cn',
    locale: 'zh-CN',
    name: 'Chinese'
  },
  ja: {
    code: 'ja',
    locale: 'ja-JP',
    name: 'Japanese'
  },
  // नई भाषा जोड़ने का उदाहरण:
  hi: {
    code: 'hi',
    locale: 'hi-IN',
    name: 'Hindi'
  }
};
```

### 3.5 लेआउट फ़ाइलें और स्टाइल

प्रत्येक भाषा के लिए संबंधित लेआउट फ़ाइलों की आवश्यकता होती है:

1. एक नई लेआउट फ़ाइल बनाएँ (उदाहरण के लिए, हिन्दी के लिए, `src/layouts/BaseHI.astro` बनाएँ)
2. आप एक मौजूदा लेआउट फ़ाइल (जैसे `BaseEN.astro`) को कॉपी कर सकते हैं और उसका अनुवाद कर सकते हैं
3. लेआउट फ़ाइल में नेविगेशन मेनू, फुटर आदि जैसे वैश्विक तत्वों के अनुवाद होते हैं
4. नई जोड़ी गई भाषा पर ठीक से स्विच करने के लिए भाषा स्विचर कॉन्फ़िगरेशन को अपडेट करना सुनिश्चित करें।

### 3.6 भाषा पेज निर्देशिकाएँ बनाना

नई भाषा के लिए स्वतंत्र पेज निर्देशिकाएँ बनाएँ:

1. `src` निर्देशिका में भाषा कोड के नाम से एक फ़ोल्डर बनाएँ (उदाहरण के लिए `src/hi/`)
2. अन्य भाषा निर्देशिकाओं (जैसे `src/en/`) से पेज संरचना को कॉपी करें
3. पेज सामग्री को अपडेट करें, शीर्षकों, विवरणों और टेक्स्ट का टारगेट भाषा में अनुवाद करें
4. सुनिश्चित करें कि पेज सही लेआउट घटक का उपयोग करते हैं (उदाहरण के लिए, `.layout: '@/layouts/BaseHI.astro'`)

### 3.7 घटक (Component) स्थानीयकरण

कुछ सामान्य घटकों को भी अनुवाद की आवश्यकता होती है:

1. `src/components/` निर्देशिका में घटकों की जाँच करें
2. निश्चित टेक्स्ट वाले घटकों (जैसे नेविगेशन बार, फुटर आदि) पर विशेष ध्यान दें
3. घटक विभिन्न भाषाओं में सामग्री प्रदर्शित करने के लिए कंडीशनल रेंडरिंग का उपयोग कर सकते हैं:

```astro
{Astro.url.pathname.startsWith('/en') && <p>English content</p>}
{Astro.url.pathname.startsWith('/hi') && <p>हिन्दी सामग्री</p>}
```

### 3.8 परीक्षण और सत्यापन

अनुवाद पूरा करने के बाद, गहन परीक्षण करें:

1. वेबसाइट को स्थानीय स्तर पर चलाएँ (आम तौर पर `yarn dev` का उपयोग करके)
2. जाँचें कि नई भाषा में सभी पेज कैसे प्रदर्शित होते हैं
3. सत्यापित करें कि भाषा स्विचिंग कार्यक्षमता ठीक से काम कर रही है
4. सुनिश्चित करें कि सभी लिंक सही भाषा संस्करण पेजों की ओर इंगित करते हैं
5. रिस्पॉन्सिव लेआउट की जाँच करें ताकि यह सुनिश्चित हो सके कि अनुवादित टेक्स्ट पेज डिज़ाइन को खराब नहीं करता है।

यदि आप NocoBase में नई भाषा के अनुवाद में योगदान देना चाहते हैं, तो कृपया इन चरणों का पालन करें:

| घटक | रिपॉजिटरी | ब्रांच | नोट |
|------|------|------|------|
| सिस्टम इंटरफ़ेस | https://github.com/nocobase/nocobase/tree/main/locales | main | JSON स्थानीयकरण फ़ाइलें |
| दस्तावेज़ (2.0) | https://github.com/nocobase/nocobase | develop / next | `docs/docs/<lang>/` निर्देशिका |
| वेबसाइट | https://github.com/nocobase/website | main | अनुभाग III देखें |

अनुवाद पूरा करने के बाद, कृपया NocoBase को पुल रिक्वेस्ट (Pull Request) सबमिट करें। नई भाषाएँ सिस्टम कॉन्फ़िगरेशन में दिखाई देंगी, जिससे आप चुन सकेंगे कि कौन सी भाषा प्रदर्शित करनी है।
