# 数据表

## 介绍

在 NocoBase 中，**Collection（数据表）** 是用来描述一类业务数据的数据模型，**它不是单纯的数据库表名，而是对一类数据的统一描述。**它主要定义三件事：

| 定义 | 说明 |
| --- | --- |
| 数据保存在哪里 | 主数据库表、外部数据库表、SQL 查询结果、数据库视图、REST API 资源等。 |
| 有哪些字段 | 比如客户名称、手机号、订单金额、创建时间、负责人。 |
| 如何被 NocoBase 使用 | 页面区块、权限、工作流、API、关系字段都会基于 Collection 工作。 |

它可以对应：

- 主数据库里真实创建的数据表
- 外部数据库里已有的数据表
- 数据库视图
- SQL 查询结果
- REST API 映射出来的资源
- 外部 NocoBase 应用中的数据表

把它理解为“业务对象的数据结构”。比如：

| 业务对象 | 对应 Collection |
| --- | --- |
| 客户 | 客户表 |
| 订单 | 订单表 |
| 合同 | 合同表 |
| 任务 | 任务表 |
| 文件 | 文件表 |

数据表创建后，通常还需要继续做三件事：

- 配置字段，让数据表能保存业务需要的信息
- 在页面中[添加区块](../../interface-builder/blocks/index.md#添加区块)，让用户可以查看、录入和处理数据
- 配置权限、工作流和 API，让数据可以按业务规则被访问和流转

## 表结构类型

- **普通表** — 适合保存客户、订单、合同、工单、报销单、项目、任务等常规业务数据
- **树表** — 适合保存组织架构、商品分类、地区层级、部门目录、知识库目录等上下级数据
- **日历表** — 适合保存会议室预约、项目排期、课程安排、值班计划、活动日程等带时间范围的数据
- **评论表** — 适合保存任务评论、文章评论、审批意见、客户反馈等围绕业务记录产生的讨论内容；在业务表（普通表、树表、日历表）中创建[关系字段](./field.md)关联评论表，使用业务表弹窗页面创建[评论区块](../../plugins/@nocobase/plugin-comments/index.md)，对业务数据进行评论
- **文件表** — 适合保存合同附件、发票文件、产品图片、员工证件等文件元信息，文件实际由文件存储引擎保存；在业务表（普通表、树表、日历表）中创建[关系字段](./field.md)关联文件表，使用业务表创建区块配置关系字段上传文件，文件元信息会自动保存到文件表中
- **SQL 表** — 适合把销售汇总、库存预警、跨表统计报表、运营看板等 SQL 查询结果作为数据表使用
- **数据库视图** — 数据库里已有的 view，比如财务报表视图、过滤后的客户视图、跨系统同步后的聚合视图
- **继承表** — 多类业务对象共享一组公共字段，同时每一类又有自己的专属字段，比如资产父表派生电脑资产、车辆资产、办公家具

## 数据表增删改

- **主数据库** — 可以在 NocoBase 中创建普通表、树表、日历表、评论表、文件表、连接数据库视图、创建 SQL 数据表，也可以同步主数据库里已有的真实表。主数据库使用 PostgreSQL 时，还可以创建继承表。操作步骤见[主数据库管理](../main/database.md#主数据库管理)、[创建 SQL 数据表](../main/collection-sql.md#创建-sql-数据表)、[连接数据库视图](../main/collection-view.md#连接数据库视图)
- **外部数据库** — 接入外部数据库里已有的真实表和 view。真实表结构通常由数据库侧维护。操作步骤见[选择数据表](../external/database.md#选择数据表)、[外部数据库管理](../external/database.md#外部数据库管理)
- **REST API 数据源** — 把第三方接口资源映射成 Collection，并配置请求动作、响应转换和字段映射。操作步骤见[REST API 数据源](../external/rest-api.md)
- **外部 NocoBase** — 使用另一个 NocoBase 应用中已经存在的数据表。更多说明见[外部 NocoBase](../external/nocobase.md)

## 普通表

普通表是最常用的数据表类型，适合保存客户、订单、合同、工单、报销单、项目、任务等常规业务数据。
通过主数据库页面创建，从主数据库中同步已有的真实表，从外部数据库同步表，REST API 映射的数据表在 NocoBase 中都按照普通表方式使用。

可以在界面中使用的区块：
| 区块 | 用途 |
| --- | --- |
| [表格区块](../../interface-builder/blocks/data-blocks/table.md) | 查看、筛选、排序、批量处理记录。 |
| [表单区块](../../interface-builder/blocks/data-blocks/form.md) | 新增或编辑单条记录。 |
| [详情区块](../../interface-builder/blocks/data-blocks/details.md) | 查看单条记录详情。 |
| [列表区块](../../interface-builder/blocks/data-blocks/list.md) | 以列表方式展示记录。 |
| [网格卡片区块](../../interface-builder/blocks/data-blocks/grid-card.md) | 以卡片网格展示图片、文件、商品、资产等记录。 |
| [看板区块](../../interface-builder/blocks/data-blocks/kanban.md) | 按状态、阶段、负责人等字段分组展示记录。 |
| [日历区块](../../interface-builder/blocks/data-blocks/calendar.md) | 按日期或时间范围展示记录。 |
| [图表区块](../../interface-builder/blocks/data-blocks/chart.md) | 基于记录生成统计图表。 |
| [地图区块](../../interface-builder/blocks/data-blocks/map.md) | 按地理位置展示记录。 |
| [甘特图区块](../../interface-builder/blocks/data-blocks/gantt.md) | 按开始、结束时间展示项目计划、任务排期。 |
| [表单筛选区块](../../interface-builder/blocks/filter-blocks/form.md) | 使用表单条件筛选页面中的数据区块。 |
| [树筛选区块](../../interface-builder/blocks/filter-blocks/tree.md) | 使用树结构筛选页面中的数据区块，常用于分类、组织、地区等层级筛选。 |
| [评论区块](../../plugins/@nocobase/plugin-comments/index.md) | 在业务记录弹窗或详情页中展示和回复评论，需要启用评论插件并配置评论表。 |

| 配置 | 说明 |
| --- | --- |
| Collection display name | 数据表在界面中显示的名称，比如「客户」「订单」「合同附件」。建议使用业务人员能直接理解的名称。 |
| Collection name | 数据表的标识名称，用于 API、关系字段、权限、工作流等内部引用。它会自动生成，也可以手动修改；只支持字母、数字和下划线，并且必须以字母开头。 |
| Categories | 数据表分类。分类只影响数据表管理界面的组织方式，不改变数据表结构。数据表较多时，建议按业务模块分类，比如「客户管理」「项目管理」「财务」。 |
| Description | 数据表说明。可以写数据表保存什么数据、由谁维护、和哪些业务流程有关，方便后续维护。 |
| Use simple pagination mode | 简单分页模式。启用后，表格区块分页时会跳过总记录数统计，适合数据量很大的表，可以减少查询压力。 |
| Preset fields | 预设字段。创建表时可以选择是否自动添加 ID、创建时间、创建人、更新时间、更新人等常用字段。普通业务表建议保留这些字段。 |


普通表创建时可以通过 `Preset fields` 自动添加常用系统字段。

| 字段 | 字段名 | 说明 |
| --- | --- | --- |
| ID | `id` | 默认主键字段，用于唯一标识一条记录。默认主键类型是 `Snowflake ID (53-bit)`。 |
| 创建时间 | `createdAt` | 自动记录这条记录的创建时间。常用于排序、筛选、审计和工作流条件。 |
| 创建人 | `createdBy` | 自动记录创建这条记录的用户。常用于“只看我创建的数据”、权限控制和责任追踪。 |
| 更新时间 | `updatedAt` | 自动记录这条记录最后一次更新的时间。常用于判断数据是否被修改过。 |
| 更新人 | `updatedBy` | 自动记录最后一次更新这条记录的用户。常用于审计和协作场景。 |
| [空间](../../multi-app/multi-space/index.md) | `space` | 启用[多空间插件](../../multi-app/multi-space/index.md)后可用，用于按空间隔离数据。没有启用多空间时不会出现在普通表预设字段中。 |

**Primary key** 标识主键字段。用于在数据库层唯一标识一条记录。创建表时建议保留 ID 预设字段，默认主键类型是 `Snowflake ID (53-bit)`；如果有特殊需要，也可以选择文本、整数、UUID 或 Nano ID 作为主键。

![create_collection_primary_key](https://static-docs.nocobase.com/create_collection_primary_key.png)

:::warning 注意

无主键数据表，需要[**编辑数据表**](../main/database.md#编辑数据表)，设置**记录唯一标识**，否则无法在页面中创建区块、无法正确查看或编辑记录。

:::

## 树表

树表适合保存有上下级关系的数据，比如组织架构、商品分类、地区层级、部门目录、知识库目录。树表只能通过主数据库页面创建，其他数据源不支持树表模板。

树表可以使用[普通表](#普通表)的所有区块，仅用作数据表数据增删改查，如果配合：
- [表格区块](../../interface-builder/blocks/data-blocks/table.md#启用树表)可以展示树形记录，用于查看和维护上下级结构
- [树筛选区块](../../interface-builder/blocks/filter-blocks/tree.md)可以用树结构筛选其他数据区块

树表的创建配置与普通表一致。`Preset fields` 会在普通表内置字段之外，增加用于保存层级关系的字段。

| 字段 | 字段名 | 说明 |
| --- | --- | --- |
| Parent ID | `parentId` | 保存父节点 ID。根节点通常为空。 |
| Parent | `parent` | 多对一关系字段，指向当前表中的父节点。 |
| Children | `children` | 一对多关系字段，表示当前节点的子节点。 |

:::warning 注意

树表数据要避免形成循环关系，比如 A 的父节点是 B，B 的父节点又是 A。

:::

## 日历表

日历表适合保存带时间范围的数据，比如会议室预约、项目排期、课程安排、值班计划、活动日程。日历表只能通过主数据库页面创建，其他数据源不支持日历表模板。

日历表可以使用[普通表](#普通表)的所有区块，仅用作数据表数据增删改查，如果配合:
- [日历区块](../../interface-builder/blocks/data-blocks/calendar.md)可以按日、周、月等视图展示事件记录，并在日历中创建和编辑事件

日历表的创建配置与普通表一致。`Preset fields` 会在普通表内置字段之外，增加用于保存重复事件的字段。日历区块使用时，还需要自定义配置标题字段、开始日期字段和结束日期字段。

| 字段 | 字段名 | 说明 |
| --- | --- | --- |
| Repeats | `cron` | 保存重复事件规则，比如每天、每周、每月或每年重复。 |
| 标题字段 | 自定义 | 确定事件标题。 |
| 开始日期字段 | 自定义 | 确定事件开始时间。 |
| 结束日期字段 | 自定义 | 确定事件结束时间。 |

## 评论表

评论表适合保存围绕业务记录产生的讨论内容，比如任务评论、文章评论、审批意见、客户反馈。评论表只能通过主数据库页面创建，其他数据源不支持评论表模板。

评论表可以使用[普通表](#普通表)的所有区块，仅用作数据表数据增删改查。如果配合[评论区块](../../plugins/@nocobase/plugin-comments/index.md)，可以在业务记录弹窗或详情页中展示评论、创建评论和回复评论。常见流程是：

1. 创建评论表，用来保存评论内容、创建人和创建时间等信息。
2. 在业务表中创建关系字段，关联到评论表。比如在「任务」表中关联「任务评论」表。
3. 在业务表的详情页或弹窗页面中添加评论区块，选择对应的评论表和关系字段。
4. 用户在评论区块中发表评论，评论记录会写入评论表，并通过关系字段关联回当前业务记录。

评论表的创建配置与普通表基本一致。`Preset fields` 会在普通表内置字段之外，增加评论内容字段。评论记录和某条业务记录的关联关系，通常通过关系字段和评论区块配置决定。

| 字段 | 字段名 | 说明 |
| --- | --- | --- |
| Comment Content | `content` | 评论内容字段，默认使用 Markdown / 富文本类编辑体验。 |

:::warning 注意

评论表不适合保存主业务数据。它适合作为业务记录的附属。

:::

## 文件表

文件表适合保存文件元信息，比如文件名、扩展名、大小、MIME 类型、路径、URL、预览地址、存储位置和自定义 meta。文件表只能通过主数据库页面创建，其他数据源不支持文件表模板。

文件表可以使用[普通表](#普通表)的所有区块，只用作数据展示，禁止使用区块编辑数据，文件表的数据是通过上传文件自动生成的。使用流程是：

1. 创建文件表，用来保存文件标题、文件名、大小、类型、URL、存储位置等元信息。
2. 在业务表中创建关系字段，关联到文件表。比如在「合同」表中关联「合同附件」文件表。
3. 在业务表的表单区块中添加关系字段，让用户在新增或编辑业务记录时上传文件。
4. 上传完成后，NocoBase 会把文件元信息写入文件表，并通过关系字段把文件记录关联到当前业务记录。
5. 在业务表的详情区块、表格区块或列表区块中展示附件字段，让用户查看、预览或下载文件。

文件表的创建配置与普通表基本一致。`Preset fields` 会在普通表内置字段之外，增加一组文件元信息字段。文件本体由文件存储引擎保存，文件表保存的是文件元数据。

| 字段 | 字段名 | 说明 |
| --- | --- | --- |
| Title | `title` | 文件标题，通常用于界面展示。 |
| File name | `filename` | 文件名。 |
| Extension name | `extname` | 文件扩展名。 |
| Size | `size` | 文件大小。 |
| MIME type | `mimetype` | 文件 MIME 类型。 |
| Path | `path` | 文件在存储中的路径。 |
| URL | `url` | 文件访问地址。 |
| Preview | `preview` | 文件预览地址。 |
| Storage | `storage` / `storageId` | 文件所属存储。 |
| Meta | `meta` | 文件扩展元信息。 |

:::warning 注意

文件元信息字段通常由上传过程自动写入，不能当作普通业务字段随意修改。删除文件表记录可能会导致业务记录里的附件引用失效。是否同步删除文件本体取决于文件存储和业务配置，删除前先确认文件是否仍被业务表引用。

:::

## SQL 表

SQL 表适合把一段 SQL 查询结果作为数据表使用，不会实际创建新的数据库表。常见场景包括销售汇总、库存预警、跨表统计报表、运营看板等。详细创建方式见[创建 SQL 数据表](../main/collection-sql.md)。

SQL 表只能通过主数据库页面创建。它只支持查询展示，不支持新增、编辑和删除记录。


## 数据库视图

数据库视图适合连接数据库里已经存在的 view，比如 DBA 维护的财务报表视图、过滤后的客户视图、跨系统同步后的聚合视图。它用于复用数据库侧已经定义好的查询逻辑。详细配置见[连接数据库视图](../main/collection-view.md)。

在外部数据库中，数据库视图来自外部数据库本身。NocoBase 读取外部数据库账号所有者范围内可识别的 view，并把它作为外部数据源下的数据表使用。详细配置见[外部数据库](../external/database.md)。

## 继承表

继承表是在普通表基础上的扩展，适合多个数据表共享一组公共字段，同时每个子表又有自己的专属字段。比如先创建「资产」父表，保存资产编号、资产名称、购入日期、负责人等公共字段，再派生「电脑资产」「车辆资产」「办公家具」等子表。

继承表只能在主数据库使用 PostgreSQL 时创建。其他主数据库、外部数据库不能创建继承表。

继承表可以使用[普通表](#普通表)的所有区块，与普通区块操作没有区别。

| 配置 | 说明 |
| --- | --- |
| Inherits | 选择要继承的父表。当前数据表会继承父表字段结构，同时可以继续定义自己的字段。 |

:::warning 注意

继承表适合结构高度相似的业务对象。如果不同对象的业务流程、权限和页面差异很大，拆成普通表再用关系字段连接通常更清晰。

:::
