# 字段

字段决定一条记录里可以保存哪些信息，也决定这些信息在页面中如何输入和展示。比如客户表中的「客户名称」「手机号」「负责人」都是字段。

NocoBase 的字段有两个层次：

1. **Field type** — 字段在数据层如何保存，比如字符串、整数、日期、JSON、关系
2. **Field interface** — 字段在界面层如何输入和展示，比如单行文本、手机号、日期选择器、附件、关系选择

这个区分很重要。同一个 Field type 可以对应多个 Field interface。比如数据库里的字符串字段，可以在 NocoBase 中显示为单行文本、邮箱、手机号、URL、颜色或图标。

## 字段的基本配置

新增或编辑字段时，通常会遇到这些配置：

| 配置 | 说明 |
| --- | --- |
| Field display name | 字段在界面中显示的名称，比如「客户名称」「联系电话」「预计完成时间」。 |
| Field name | 字段内部名称，用于 API、权限、工作流等内部引用。字段创建后通常不再修改。 |
| Field interface | 字段的界面类型。它会影响表单控件、展示组件和后续可用配置。 |
| Field type | 字段的数据类型。通常由 Field interface 自动带出，部分字段可以手动选择。 |
| Default value | 默认值。创建记录时，如果用户没有填写该字段，可以自动填入默认值。 |
| Validation rules | 校验规则。比如必填、唯一、格式限制等，实际可用规则取决于字段类型。 |
| Description | 字段说明。适合写字段含义、录入要求、数据来源或维护人。 |

## 标题字段

标题字段用于决定一条记录在关系选择、详情标题、关联数据展示等位置如何显示。

比如客户表通常可以把「客户名称」设为标题字段，订单表可以把「订单编号」设为标题字段。这样在其他表选择客户或订单时，看到的是业务人员能理解的文本，而不是内部 ID。

:::tip 提示

标题字段只影响记录在界面中的默认显示方式，不等同于主键，也不等同于记录唯一标识。

:::

#### 字段映射
在 NocoBase 中，字段的定义分为两个维度：
Interface（界面类型 / UI 类型）：决定字段在前端如何展示和交互。比如「单行文本」「数字」「下拉菜单」「日期时间」等，它是用户视角的字段分类。
Type（数据类型）：决定字段在数据库中实际存储的数据格式。比如 `string`、`double`、`float`、`decimal`、`boolean`、`datetime` 等。

字段映射用于告诉 NocoBase：一个数据库字段在页面里应该怎么展示、怎么输入、怎么校验。
[了解更多字段配置信息](../field)
| 配置 | 作用 | 例子 |
| --- | --- | --- |
| Field type | 决定字段在数据层怎么保存。 | `string`、`integer`、`boolean`、`date`、`json`。 |
| Field interface | 决定字段在界面层怎么展示和输入。 | 单行文本、邮箱、手机号、数字、日期、下拉菜单、附件、关系选择。 |

通常来说，新增字段时先选择 UI 类型（Field interface），NocoBase 会自动带出合适的 Field type。比如选择「手机号」时，数据层通常仍然是字符串；选择「货币」时，数据层通常是数值类型。

如果字段来自数据库里已有的表，NocoBase 会先根据数据库列类型推断 Field type，再给出可用的 Field interface。这个时候需要你检查推断结果是否符合业务含义。比如数据库字段是 `varchar`，它可能是普通文本，也可能是邮箱、手机号、URL、颜色或图标。

新增字段后，仍然可以在相近类型之间调整 UI 类型。能否切换取决于字段底层的 Field type 和当前字段配置，编辑字段时以表单里实际可选项为准。

常见可以相互切换的 UI 类型如下：

| 大类 | 可以相互切换的 UI 类型 | 说明 |
| --- | --- | --- |
| 文本类 | 单行文本、邮箱、手机号、URL、颜色、图标。 | 底层通常都是字符串。适合姓名、标题、编号、联系方式、链接等短文本。 |
| 长文本类 | 多行文本、Markdown、Markdown (Vditor)、富文本。 | 适合备注、说明、正文、富文本内容。切换前先确认已有内容格式是否匹配。 |
| 数字类 | 数字、整数、货币、百分比、排序。 | 适合金额、数量、比例、排序值等数值字段。 |
| 日期时间类 | 日期时间、日期、时间。 | 适合开始时间、结束时间、预约日期、提醒时间等时间类字段。 |
| 选项类 | 下拉菜单、单选框、多选框。 | 适合状态、类型、优先级、标签等取值范围明确的字段。 |
| 布尔类 | 勾选、开关。 | 适合启用 / 禁用、完成 / 未完成等二值状态。 |

有些字段不适合作为普通 UI 类型来回切换：

| 字段类型 | 说明 |
| --- | --- |
| 主键字段 | 用于唯一标识记录，不建议切换成普通输入控件。 |
| 系统字段 | 创建时间、创建人、更新时间、更新人等字段由 NocoBase 维护，通常不需要切换 UI 类型。 |
| 关系字段 | 一对一、一对多、多对一、多对多依赖关联关系配置，不能直接切换成普通文本或数字字段。 |
| 附件字段 | 附件字段依赖文件表和文件存储配置，不建议切换成普通字段。 |
| 计算、公式、序列等特殊字段 | 这类字段有自己的生成或计算规则，通常只能编辑专属配置。 |
| JSON 字段 | 适合保存结构化对象或数组，不建议切换成普通文本字段后继续承载结构化数据。 |

新增字段时，Field interface 是必选项。字段创建后，字段名称通常不再修改，界面类型也不建议随意切换。如果只是显示名称不准确，可以编辑 Field display name；如果输入方式、校验方式或业务含义不对，再考虑调整 Field interface。

:::warning 注意

切换 Field interface 不等于修改数据库字段类型。它主要影响页面控件、展示方式、校验规则和后续可用配置。切换前先确认已有数据是否能被新的界面类型正确识别。

:::


## 主键和记录唯一标识

**Primary key** 是数据库层面的主键，用来唯一标识一条记录。普通业务表通常使用默认的 ID 字段即可。

**Record unique key** 是 NocoBase 定位记录时使用的唯一字段。SQL 表、数据库视图、外部数据库表或没有标准主键的数据表，需要特别关注这个配置。没有可用的记录唯一标识时，页面区块可能无法正确查看、编辑或删除记录。

## 默认值

默认值适合用于减少重复录入。比如状态字段可以默认是「待处理」，日期字段可以默认是当前日期，负责人字段可以默认是当前用户。

使用默认值时，需要注意两点：

- 默认值只在创建记录时生效，通常不会自动修改已有记录
- 如果字段有唯一约束，不建议再配置固定默认值，否则容易产生重复值

## 字段描述

字段描述用于补充字段的业务含义。它适合写给后续维护者看，比如：

- 这个字段的数据从哪里来
- 由谁负责维护
- 填写时有什么格式要求
- 是否会被权限、工作流或外部系统使用

字段多、业务流程复杂时，建议给关键字段补充描述。这样后续调整页面、权限和工作流时，会更容易判断字段能不能改。

## 字段类型分类

NocoBase 的字段类型可以按使用场景理解：

| 分类 | 常见字段 | 适合场景 |
| --- | --- | --- |
| 基本类型 | 单行文本、多行文本、整数、数字、百分比、密码、颜色、图标 | 保存普通文本、数字和基础格式数据。 |
| 选择类型 | 勾选、下拉菜单、单选框、复选框、中国行政区 | 从固定选项中选择值。 |
| 多媒体 | Markdown、Markdown(Vditor)、富文本、附件、附件 URL、手写签名 | 保存富文本、说明文档、图片、文件和签名。 |
| 日期与时间 | 日期时间、日期、时间、Unix 时间戳 | 保存时间点、日期范围、定时触发相关数据。 |
| 几何图形 | 点、线、圆、多边形 | 保存空间位置或几何数据。 |
| 高级类型 | UUID、Nano ID、排序、计算公式、自动编码、JSON、数据表选择器、加密 | 保存标识符、结构化数据、自动生成值或加密内容。 |
| 系统信息 | 创建日期、最后修改日期、创建人、最后修改人、Table OID | 保存系统自动维护的信息。 |
| 关系类型 | 一对一、一对多、多对一、多对多 | 建立数据表之间的关系。 |

## 按场景选择字段

很多时候你不是先知道要选哪个字段，而是先知道业务上要保存什么。可以先从场景判断，再回到字段类型。

### 文本和联系方式

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 简短文本 | 单行文本 | 客户名称、项目名称、合同编号、城市、职位。 |
| 较长说明 | 多行文本 | 备注、需求描述、处理意见、问题说明。 |
| 邮箱地址 | 电子邮箱 | 客户邮箱、员工邮箱、供应商联系邮箱。 |
| 手机号码 | 手机号码 | 客户手机号、联系人电话、员工联系方式。 |
| 链接地址 | URL | 官网、合同链接、外部系统地址、图片或文件链接。 |
| 密码或敏感输入 | 密码、加密 | 密码类输入、密钥、证件号等不适合明文展示的数据。 |

如果只是保存普通文字，默认用单行文本或多行文本就够了。只有当字段有明确格式，比如邮箱、手机号、URL 时，才需要选择对应的界面类型，这样 NocoBase 可以提供更合适的输入和校验。

### 数字、金额和比例

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 整数 | 整数 | 数量、人数、库存、次数、排序值。 |
| 小数 | 数字 | 重量、面积、评分、用量、折扣前的计算值。 |
| 百分比 | 百分比 | 完成率、折扣率、毛利率、进度。 |
| 金额 | 数字或货币类界面 | 合同金额、订单金额、预算、费用。 |
| 排序值 | 排序 | 手动调整列表顺序、流程节点顺序、菜单顺序。 |

数字字段要提前想清楚精度。比如金额、比例、面积这类字段，通常不要用整数；库存数量、人员数量这类字段，通常用整数更直观。

### 日期和时间

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 具体时间点 | 日期时间 | 预约开始时间、任务截止时间、合同签署时间。 |
| 只有日期 | 日期 | 生日、入职日期、发票日期、项目开始日期。 |
| 只有时间 | 时间 | 每日打卡时间、营业开始时间、提醒时间。 |
| 时间戳 | Unix 时间戳 | 接入外部系统、日志数据、已有数据库中使用时间戳的字段。 |
| 创建或更新时间 | 创建日期、最后修改日期 | 系统自动记录数据创建和更新的时间。 |

如果业务需要跨时区协作，要特别注意日期时间字段是否包含时区。比如全球团队的会议时间、跨地区工单 SLA，通常需要更严格地确认时间字段含义。

### 状态和选项

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 是 / 否 | 勾选 | 是否启用、是否归档、是否已付款、是否需要回访。 |
| 单个固定选项 | 下拉菜单（单选）、单选框 | 订单状态、客户等级、工单优先级、合同类型。 |
| 多个固定选项 | 下拉菜单（多选）、复选框 | 客户标签、适用渠道、技能标签、产品特性。 |
| 地区选择 | 中国行政区 | 省市区选择、收货地区、服务区域。 |

选项字段适合值集合稳定的场景。比如「订单状态」通常是固定流程，用单选更合适；「客户标签」可能同时有多个值，用多选更合适。

### 富文本、文件和外部附件

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 格式化说明 | Markdown、Markdown(Vditor)、富文本 | 知识库正文、公告内容、产品说明、处理方案。 |
| 上传文件 | 附件 | 合同附件、发票图片、员工证件、项目资料。 |
| 外部文件地址 | 附件 URL、URL | 文件已经在对象存储、网盘或第三方系统中，只需要保存访问地址。 |
| 签名图片 | 手写签名 | 合同签收、现场确认、审批签名。 |

附件字段依赖文件表和文件存储引擎。如果文件本体不由 NocoBase 管理，只保存外部地址时，优先考虑附件 URL 或 URL 字段。

### 关系数据

| 你要表达 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 一条记录只关联另一条记录 | 一对一 | 员工和员工档案、用户和扩展资料。 |
| 当前表的一条记录拥有多条子记录 | 一对多 | 客户拥有多张订单、项目拥有多个任务。 |
| 当前表的多条记录属于另一张表的一条记录 | 多对一 | 订单属于客户、任务属于项目、联系人属于公司。 |
| 两边都可以关联多条记录 | 多对多 | 用户和角色、学生和课程、产品和标签。 |

关系字段适合表达业务对象之间的连接。比如订单里不要重复保存客户名称、客户电话、客户地址的所有信息；通常做法是用关系字段关联客户表，再按需要展示客户字段。

### 自动标识和编码

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 全局唯一 ID | UUID、Nano ID、Snowflake ID | 对外同步、跨系统引用、接口集成。 |
| 业务编号 | 自动编码 | 订单编号、合同编号、工单编号、资产编号。 |
| 数据表引用 | 数据表选择器 | 需要让用户选择某张数据表作为配置项。 |

如果字段只是数据库内部主键，默认 ID 通常够用。如果这个编号会展示给业务人员或外部系统使用，建议用自动编码这类更有业务含义的字段。

### 计算和结构化数据

| 你要保存 | 建议字段 | 适合场景 |
| --- | --- | --- |
| 根据其他字段计算出的结果 | 计算公式 | 小计金额、利润、年龄、剩余天数、风险等级。 |
| 不固定结构的数据 | JSON | 第三方 API 原始响应、扩展配置、动态表单结果。 |
| 空间数据 | 点、线、圆、多边形 | 门店位置、配送范围、巡检路线、地理围栏。 |

计算公式适合展示派生值，不适合保存需要人工反复修改的业务字段。JSON 适合扩展性很强的数据，不过如果某个属性经常用于筛选、排序或权限判断，建议拆成独立字段。

## Field type 和 Field interface 的映射

Field type 决定数据怎么存，Field interface 决定界面怎么用。外部数据库、SQL 表和数据库视图会先根据数据库字段类型推断 Field type，再给出可选的 Field interface。

常见映射可以这样理解：

| 数据层类型 | 常见 Field interface | 说明 |
| --- | --- | --- |
| `string` | 单行文本、邮箱、手机号、URL、颜色、图标、下拉菜单、单选框 | 同样是字符串，界面上可以按业务含义选择不同控件。 |
| `text` | 多行文本、Markdown、Markdown(Vditor)、富文本、URL、JSON | 适合较长文本或结构化文本。 |
| `integer` / `bigInt` | 整数、排序、勾选、下拉菜单、单选框、Unix 时间戳 | 整数字段既可以表示数量，也可以表示状态、排序或时间戳。 |
| `float` / `decimal` | 数字、百分比、货币 | 适合小数、金额、比例等数值。 |
| `boolean` | 勾选、开关 | 适合是 / 否、启用 / 禁用等二值状态。 |
| `date` / `time` | 日期时间、日期、时间、创建日期、最后修改日期 | 适合时间类字段。 |
| `json` | JSON | 适合保存结构化对象或数组。 |
| 关系类型 | 一对一、一对多、多对一、多对多 | 用于连接不同数据表中的记录。 |

如果字段来自已有数据库结构，可以先按数据库字段类型理解常见映射：

| 数据库字段类型 | 常见 Field type | 常见 Field interface |
| --- | --- | --- |
| `varchar` / `char` | `string` | 单行文本、邮箱、手机号、URL、颜色、图标。 |
| `text` | `text` | 多行文本、Markdown、富文本、URL。 |
| `int` / `bigint` | `integer` / `bigInt` | 整数、排序、Unix 时间戳、下拉菜单、单选框。 |
| `decimal` / `numeric` / `float` / `double` | `decimal` / `float` / `double` | 数字、货币、百分比。 |
| `boolean` / `tinyint(1)` | `boolean` | 勾选、开关。 |
| `date` / `datetime` / `timestamp` / `time` | `date` / `dateOnly` / `time` | 日期时间、日期、时间、创建时间、更新时间。 |
| `json` / `jsonb` | `json` | JSON。 |
| `enum` / `set` | `enum` / `set` | 下拉菜单、单选框、多选框。 |

如果 NocoBase 推断出的 Field interface 不符合业务含义，可以在「Configure fields」里调整。比如数据库字段是 `varchar`，但它保存的是邮箱，就可以把界面类型改成邮箱。

:::warning 注意

切换 Field interface 不等于修改数据库字段类型。它主要影响界面输入、展示和校验方式。修改前请确认已有数据能被新的界面类型正确识别。

:::

## 唯一约束

唯一约束用于保证字段中非空值不重复。它适合邮箱、手机号、工号、合同编号、订单编号等业务标识。

启用唯一约束前，需要确认已有数据已经满足唯一性。如果已有重复值，新增约束可能失败，后续创建或更新记录也会遇到校验错误。

唯一约束会增加数据库检查成本。数据量很大时，只给真正需要唯一性的字段开启。

## 常见字段操作

| 操作 | 说明 |
| --- | --- |
| 新增字段 | 在数据表中添加新的业务属性。主数据库支持最完整的字段新增能力。 |
| 编辑字段 | 修改显示名称、界面类型、校验规则、描述或字段专属配置。 |
| 删除字段 | 删除不再使用的字段。删除前需要确认页面、权限、工作流和 API 是否仍在引用。 |
| 设置标题字段 | 设置记录的默认显示标题。 |
| 从数据库同步 | 外部数据库、SQL 表和数据库视图结构变化后，同步最新字段结构。 |

:::danger 警告

删除字段可能导致已有数据、页面区块、权限、工作流和外部 API 不可用。删除前请先确认依赖关系，并做好备份或迁移。

:::
