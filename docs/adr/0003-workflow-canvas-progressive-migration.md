# Workflow canvas migrates as parallel worlds, with pure logic relocated to client-v2 and the legacy canvas importing it back

The v1 workflow canvas (node-graph editor) and the new client-v2 canvas coexist as **parallel worlds** over the same `workflows` + `flow_nodes` data: distinguished only by URL/entry list (`/admin/settings/workflow/workflows/:id` reached from the legacy settings list vs `/admin/workflow/workflows/:id` reached from the `WorkflowPane` list), with **no per-workflow flag and no runtime flip**. The framework-agnostic pure logic the canvas depends on (`linkNodes`, the `getCollectionFieldOptions` field-tree builder, `nodeVariableUtils`, the `NodeDragContext`/`NodeClipboardContext` pointer logic, the `@emotion/css` stylesheet) is **relocated into `src/client-v2/` and imported back by the legacy canvas** via the allowed `v1 â†’ v2` direction â€” one shared copy, not two. The legacy canvas retires by deleting its settings list + route once the modern canvas reaches parity.

## Considered Options

- **(A, chosen) Parallel worlds + pure logic relocated to client-v2.** Two canvases over one dataset, switched by URL only. Pure (no-Formily) logic lives once in client-v2; v1 imports it back. Formily is load-bearing in exactly three v1 spots (config drawer, add-node menu/preset, remove-branch modal) â€” these are rebuilt natively for v2 (antd `Form` + the Instruction's `FieldsetLoader`/`PresetFieldsetLoader`), never ported.
- **(B) One canvas URL, runtime flip by flag.** A single route renders v1 or v2 based on a per-workflow column / feature toggle. Rejected: forces the two canvases to share a mount point (dragging the settings/ProLayout shell back in) and creates a "half-migrated workflow" state to reason about, for no benefit over (A).
- **(C) Rewrite the canvas pure logic independently in v2.** Rejected: the field-tree builder (~250 lines: relation lazy-load, type filtering, foreign keys) and the ~700-line drag logic are the most bug-prone parts of the system; maintaining two copies during the dual-canvas period is exactly the risk (A) avoids. v2 reuses the same code, only adapting `VariableOption â†’ MetaTreeNode` at the very end.

## Consequences

- **Import direction inverts for the relocated modules.** After relocation, `src/client/` imports `getCollectionFieldOptions`, `linkNodes`, the drag/clipboard contexts, and the stylesheet from `@nocobase/client-v2` (the workflow plugin's client-v2 entry). A future reader seeing the legacy canvas import canvas logic from v2 should expect this â€” it is the deliberate mechanism that lets one implementation serve both canvases. The injected dependencies (`compile`, `collectionManager`) keep the logic runtime-agnostic: v1 passes `useCompile()`, v2 passes `useT()`.
- **`compile` equivalence is a load-bearing, tested constraint.** The relocated field-tree builder uses `compile` only to expand **field titles** (plain strings or `{{t("â€¦")}}` i18n templates). Within that scope, v1's `useCompile` (Formily `Schema.compile`) and v2's `useT()` (`flowEngine.translate`, which natively expands `{{t(â€¦)}}`) behave identically. They are *not* equivalent for arbitrary scope expressions (`{{fn(arg)}}`, `{{$deps[0]}}`) â€” but field titles never carry those. A characterization test pins this with an assertion that both expand the same `{{t(â€¦)}}` title to the same translation; if a non-i18n expression ever reaches a field title, that test fails and surfaces the drift.
- **Two render-extension points become per-node loaders, distinguished by field name (not case).** A node's in-canvas render and its config UI are independent migration points. The modern canvas reads uppercase loader fields on the Instruction â€” `ComponentLoader` (in-canvas node render, mirroring v1's lowercase `Component`; absent â†’ default card), `FieldsetLoader` (config drawer; absent â†’ "not yet migrated" placeholder, never a Formily fallback), `PresetFieldsetLoader` (add-time preset). These are `() => Promise<{ default: ComponentType }>` loaders matching the existing `createConfigFormLoader` convention, rendered with Suspense + a skeleton card. The legacy lowercase data fields (`fieldset`, `view`, `scope`, `components`) remain pass-through, consumed only by the legacy canvas. This supersedes ADR-0002's "case-sensitive `fieldset`/`Fieldset`" switch: the switch is now **field-name-based** (`fieldset` = legacy Formily data, `FieldsetLoader` = modern loader).
- **`Branch`/`CanvasContent`/`BranchContext` are relocated as a *second copy*, not shared.** Unlike the pure logic, these couple to `<Node>` (whose card differs between v1 and v2), so v2 gets its own `Branch` and v1 keeps its own until retirement. The shared-one-copy rule applies only to Formily-free, Node-independent logic.
- **`useVariables` stays untouched; the adapter never reaches v1.** Each Instruction's `useVariables` keeps returning the legacy `VariableOption`. Only the v2 aggregator wraps it in the `VariableOption â†’ MetaTreeNode` adapter; the v1 aggregator (`client/variable.tsx`) consumes `VariableOption` directly and has no code path to the adapter. The adapter is a v2-only consumer of a shared data source, provably isolated.
- **Test strategy: characterization baseline before the move.** Before relocating, golden characterization tests are written on the v1 side (injected mock `compile`/`collectionManager`) covering the full pure-logic surface; after relocation the same v1 tests re-run green (proving v1's behavior is unchanged through the back-import), and equivalent v2 tests run the same mock inputs. DOM/pointer side effects of drag/clipboard stay covered by the existing e2e; only their pure functions (`getDropImpact` math, `collectUpstreams`/`collectDownstreams`) get unit baselines. The v1 characterization tests are deleted with v1 at retirement; the v2 tests persist.
